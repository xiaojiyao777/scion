## 2. 按依赖排序的实现顺序

```
Sprint 1: 分支迭代修复（不需要新模块，改现有代码）
  ├── campaign.py: CONTINUE_EXPLORE 保留 workspace
  ├── branch.py: get_code_base 返回 workspace path
  ├── context_manager.py: 增加 branch_code
  └── scheduler.py: max_active_branches 检查
  验证: 跑 5 轮 campaign，确认 Round N+1 基于 Round N 代码

Sprint 2: Verification Gate 补齐
  ├── scion/verification/ 新建 7 个文件
  ├── campaign.py: 替换 VerificationGate stub
  └── 确认 surrogate oracle.py 接口
  验证: 构造 3 个有 bug 的算子，确认被正确拦截

Sprint 3: SQLite Lineage 补齐
  ├── scion/lineage/ 新建 3 个文件（按 engineering arch §4.4 DDL）
  ├── scion/memory/hypothesis_store.py（含 blacklist）
  ├── campaign.py: 每步写 lineage
  └── cli inspect: 查询验证
  验证: campaign 后 scion inspect 可查完整历史

Sprint 4: 实例 + 集成验证
  ├── 分析退化 instances，生成新的有区分度 instances
  ├── 重划 split_manifest
  └── 完整 campaign 端到端审计（逐步审查每个阶段输入输出）
```

## 4. 有意偏离 v0.1 Spec（合理）

| 偏离 | 理由 |
|---|---|
| f-string 代替 jinja2 模板 | 更简单，无外部依赖，效果相同 |
| tool_use 代替 json.loads 解析 | 优于 spec（spec 当时没有考虑 tool_use） |
| DecisionFeatures.expand_count | 合理扩展，解决 expand 死循环 |
| Prompt caching (1h TTL) | 优于 spec（降成本） |
| Case-level screening feedback | 优于 spec（更丰富的 LLM 上下文） |
