# Scion v0.4 Algorithm Design Space Upgrade

*Date: 2026-05-05*
*Status: integrated design source for the v0.4 algorithm-space upgrade*

## Purpose

v0.4 exposed a structural limit in the current Scion design. The framework has
largely preserved the v3 governance boundaries, and CVRP is now integrated
through a problem adapter. However, CVRP campaigns still collapse into a narrow
optimization pattern:

```text
choose an operator surface
-> generate one post-baseline operator
-> evaluate it after a strong ALNS+VNS baseline
-> observe tie/no-op dominated evidence
```

This is not primarily a Decision or Protocol failure. It is a research-space and
proposal-process limitation.

The v0.4 upgrade has two main modules:

```text
1. research_surfaces v2
   Model the algorithm design space explicitly.

2. AgenticProposalSession
   Let the LLM use that design space through a bounded multi-step proposal
   agent instead of single-shot calls.
```

Together they move Scion from:

```text
operator design space optimization
```

to:

```text
heuristic algorithm research-surface optimization
```

while preserving the v3 rule:

```text
LLM proposes; deterministic gates decide.
```

## Design Thesis

`research_surfaces v2` and `AgenticProposalSession` solve different problems.

| Upgrade | Solves | Does Not Solve |
|---|---|---|
| `research_surfaces v2` | What algorithm components a problem package exposes for research. | How the LLM reasons across multiple steps. |
| `AgenticProposalSession` | How the LLM inspects surfaces, diagnoses evidence, drafts alternatives, and emits a bounded proposal. | Which domain-specific algorithm components exist. |

They are coupled but not interchangeable.

```text
research_surfaces v2 = algorithm-space contract
AgenticProposalSession = proposal-time control loop
ProblemAdapter = semantic oracle and context renderer
Scion core = governance and evidence discipline
```

## Non-Negotiable Boundaries

Scion remains the framework, not the research object.

Framework core may own:

- branch lifecycle;
- scheduler and budget rules;
- Contract / Verification / Protocol / Decision orchestration;
- generic research-surface schema handling;
- generic proposal-tool framework;
- context exposure policy;
- evidence refs and lineage;
- deterministic promotion and abandonment rules.

Problem packages own:

- domain semantics;
- solution and instance models;
- objective metrics and comparison specs;
- surface definitions and interface contracts;
- domain-specific prompt/context rendering;
- taxonomy aliases and novelty signatures;
- solver wrapper integration;
- runtime audit fields and final evidence interpretation.

Framework core must not hardcode CVRP, warehouse, or future problem semantics.
Problem-specific words may appear in problem packages, adapter-rendered context,
checked-in problem specs, and problem-owned tests. They should not appear as
decision logic in core governance modules.

## Module 1: research_surfaces v2

### Why v1 Is Not Enough

The current research-surface model is a useful bridge from operator categories:

```text
name
kind
description
target_files
prompt_hint
required_functions
create_new_allowed
modify_allowed
remove_allowed
```

This mostly says:

```text
which files may the LLM edit?
```

Algorithm design space needs a stronger contract:

```text
which algorithm component does this surface control,
when is it invoked,
what are its safe degrees of freedom,
how is it verified,
and what runtime evidence proves it mattered?
```

### Surface v2 Concept

A research surface is a problem-owned, bounded, auditable algorithm component.

It should answer:

- what algorithm role it controls;
- where it is invoked in the solver pipeline;
- which files or artifacts may change;
- which functions/classes/config fields are required;
- which values/components are allowed;
- what static and runtime safety bounds apply;
- what evidence proves loaded/executed/changed behavior;
- how novelty and memory should group proposals.

### Proposed Schema Shape

The exact Python/YAML names can evolve, but the conceptual schema should include
these sections:

```yaml
research_surfaces:
  - name: neighborhood_portfolio
    kind: portfolio

    algorithm:
      role: main_search_portfolio
      invocation_point: during_main_search
      description: Controls which predeclared search components run and how
        candidate limits are bounded.

    targets:
      files:
        - policies/neighborhood_portfolio.py
      create_new_allowed: false
      modify_allowed: true
      remove_allowed: false
      singleton: true

    interface:
      required_functions:
        - enabled_components
        - component_weights
        - candidate_limits
      return_contract: problem_defined

    bounds:
      allowed_components:
        - relocate
        - swap
        - two_opt_star
        - or_opt
        - ruin_repair
      numeric_ranges:
        top_k_neighbors: [3, 30]
        max_route_pairs: [10, 500]
      complexity_scale_terms:
        - routes
        - customers
        - route_pairs

    evidence:
      required_runtime_fields:
        - surface_loaded
        - surface_errors
        - enabled_components
        - component_attempts
        - component_accepted
        - component_runtime_ms
        - surface_stop_reason

    novelty:
      strategy: semantic_signature
      signature_fields:
        - enabled_components
        - component_weight_pattern
        - candidate_limit_pattern

    prompt:
      hypothesis_guidance: problem-owned text
      implementation_guidance: problem-owned text
      anti_patterns: problem-owned text
```

The framework should not interpret `routes`, `customers`, or component names as
domain concepts. It should treat them as problem-provided metadata.

### Surface Kinds

Initial generic kinds:

| Kind | Purpose |
|---|---|
| `operator` | Candidate modifies a local transformation component. |
| `policy` | Candidate modifies bounded solver behavior functions. |
| `config` | Candidate modifies structured parameters or schedules. |
| `portfolio` | Candidate chooses/weights predeclared components. |
| `construction` | Candidate controls initial solution construction policy. |
| `acceptance_restart` | Candidate controls bounded acceptance, restart, or perturbation policy. |

Only the framework-level behavior of a kind should be generic. Domain behavior
must come from problem package metadata.

### Algorithm Role And Invocation Point

Every surface should declare both.

Example roles:

```text
construction
main_search_portfolio
destroy_repair_policy
local_improvement
acceptance_policy
restart_policy
budget_schedule
post_baseline_polish
```

Example invocation points:

```text
before_search
during_main_search
after_each_iteration
on_stagnation
after_baseline
post_processing
```

This lets Scion and the proposal agent distinguish a late polish operator from a
main-search design lever without hardcoding CVRP.

### Evidence Contract

Every surface must define runtime evidence that proves three facts:

```text
loaded
executed
affected behavior or produced a meaningful no-op signal
```

Minimum generic shape:

```json
{
  "surface_name": "neighborhood_portfolio",
  "surface_loaded": true,
  "surface_errors": 0,
  "surface_executed": true,
  "surface_behavior_changed": true,
  "surface_stop_reason": "budget_exhausted"
}
```

Problem packages can add domain fields, but the field contract must be declared
on the surface so context, audit, and proposal tools can reason over it.

### Novelty Contract

Novelty cannot remain file-only. Singleton policy files can have many
meaningfully different proposals.

Surface v2 should allow problem-owned novelty strategies:

| Strategy | Example |
|---|---|
| `target_file` | Ordinary operator modify/remove. |
| `symbol_signature` | New operator class or function family. |
| `semantic_signature` | Singleton policy changes with different behavior. |
| `component_mix` | Portfolio surfaces. |
| `mode_and_bias` | Construction surfaces. |

Core executes the novelty strategy generically. Problem packages supply the
fields and signature renderer.

### Prompt Contract

Prompt text should be problem-owned and surface-owned.

Core prompt shell says:

```text
Choose one declared research surface.
Respect the surface contract.
Return the required structured proposal.
Do not request forbidden evidence.
```

Problem package prompt blocks say:

```text
This surface controls construction before the main search.
Allowed modes are ...
Required functions are ...
Avoid these complexity patterns ...
Runtime audit fields are ...
```

This keeps Scion core generic while still giving the LLM rich domain guidance.

## Module 2: AgenticProposalSession

`AgenticProposalSession` is the proposal-time engine that uses
`research_surfaces v2`.

It is defined in detail in
[`v0.4-agentic-proposal-session-design.md`](v0.4-agentic-proposal-session-design.md).

In this integrated design, its role is:

```text
read declared surfaces
-> diagnose recent allowed evidence
-> choose one surface
-> inspect that surface's contract
-> draft hypothesis
-> draft patch
-> run static preview
-> emit structured proposal
```

The session is still tainted. It cannot:

- run verification or protocol;
- read validation/frozen raw metrics;
- write campaign workspace;
- mutate champion state;
- write decision features.

### Required Tools

The agent should use generic tools:

```text
context.read_problem
context.list_surfaces
context.read_surface
context.read_objective_policy
context.read_champion_summary
memory.query
feedback.query_screening
feedback.query_runtime
feedback.query_holdout_summary
proposal.draft_hypothesis
proposal.draft_patch
proposal.contract_preview
proposal.finalize
```

Surface v2 is what makes these tools useful. Without surface metadata, the agent
can only read a file and guess.

### Surface-Aware Workflow

The agent should make the following decision explicit:

```text
Is the current surface still worth exploring,
or should the branch shift to another declared surface?
```

For CVRP, if repeated post-baseline operator attempts have near-zero accepted
move rate, the agent should be able to shift toward a main-search surface such
as portfolio, construction, or policy. That shift should come from surface
metadata and proposal memory, not from CVRP logic in Scion core.

## Problem Package Responsibilities

For each problem, the package must provide:

```text
problem-v1.yaml
research_surfaces v2 declarations
adapter rendering for problem summary
adapter rendering for each surface interface
objective metric specs
family taxonomy and aliases
novelty signature metadata
solver wrapper hooks for surfaces
runtime audit fields
verification and objective recomputation
final evidence interpretation if needed
focused tests
```

If a problem does not provide surface v2 metadata, Scion may use the legacy
operator-surface bridge, but it should not pretend that it has full algorithm
design-space access.

## Framework Responsibilities

Scion core should implement generic support:

- parse and validate surface v2 metadata;
- bridge legacy operator categories into surface v1/v2 compatibility shape;
- render generic surface lists into proposal context;
- enforce target/action permissions;
- expose surface metadata through proposal tools;
- run generic novelty strategies;
- enforce context exposure policy;
- persist proposal-session and surface evidence refs;
- keep `DecisionFeatures` free of proposal text.

Core should not implement CVRP-specific or warehouse-specific surface logic.

## CVRP v2 Surface Plan

CVRP should move beyond post-baseline operator polish.

Recommended first surfaces:

### `construction_policy`

Purpose:

```text
Control initial solution construction before main search.
```

Example allowed modes:

```text
baseline_default
nearest_capacity
savings_seed
sweep_seed
cluster_then_route
```

Required evidence:

```text
construction_surface_loaded
construction_mode
construction_elapsed_ms
construction_routes
construction_distance
construction_feasible
```

### `neighborhood_portfolio`

Purpose:

```text
Control which predeclared neighborhoods/components run during main search.
```

Example functions:

```text
enabled_components(instance, time_limit_sec)
component_weights(instance, time_limit_sec)
candidate_limits(instance, time_limit_sec)
```

Required evidence:

```text
portfolio_surface_loaded
enabled_components
component_attempts
component_accepted
component_runtime_ms
candidate_limits
surface_stop_reason
```

### Later Surfaces

```text
destroy_repair_policy
acceptance_restart_policy
budget_schedule
perturbation_policy
```

These should be introduced only when the problem package can provide bounded
interfaces, runtime audit, and verification coverage.

## Warehouse Compatibility

Warehouse can continue using operator surfaces because, for that research
object, the operator pool is close to the algorithm body.

The compatibility mapping should be explicit:

```text
legacy operator category
-> surface kind: operator
-> role: local_improvement or assignment_repair
-> invocation_point: during_search
```

Warehouse-specific prompt hints, taxonomy, objective names, and complexity terms
should still live in the warehouse problem package, not in Scion core.

## Data Flow

Integrated data flow:

```text
ProblemSpecV1.research_surfaces v2
      |
      v
ProblemAdapter renders problem/surface context
      |
      v
AgenticProposalSession reads surface metadata through tools
      |
      v
Agent emits HypothesisProposal / PatchProposal
      |
      v
ContractGate validates schema, target, action, interface, complexity
      |
      v
WorkspaceLifecycle materializes candidate
      |
      v
VerificationGate checks adapter semantics and runtime audit
      |
      v
ExperimentProtocol runs screening/validation/frozen
      |
      v
SafeFeatureExtractor builds DecisionFeatures
      |
      v
DecisionEngine decides deterministically
```

Proposal memory and transcripts are side artifacts:

```text
AgenticProposalSession
-> tainted proposal memory
-> future proposal context only
```

They do not enter the Decision path.

## Implementation Plan

### RS2-0: Lock Design

- Land this integrated design.
- Keep
  [`v0.4-agentic-proposal-session-design.md`](v0.4-agentic-proposal-session-design.md)
  as the detailed agent design.
- Treat
  [`v0.4-algorithm-design-space-optimization.md`](v0.4-algorithm-design-space-optimization.md)
  as the motivation and evidence document.

### RS2-1: Surface v2 Schema

- Extend `ProblemSpecV1.research_surfaces` with algorithm, target, interface,
  bounds, evidence, novelty, and prompt metadata.
- Preserve legacy surface compatibility.
- Add tests for loading and bridging.

### RS2-2: Surface Rendering

- Status: implemented as the first minimal context slice.
- `ContextManager` now renders declared v2 metadata into the Research Surfaces
  block as problem-provided context: algorithm role/invocation point, targets,
  interface, bounds, evidence, novelty, and prompt guidance.
- Adapter-owned interface rendering remains preferred for active code context;
  generic metadata rendering is used as the fallback for non-operator surfaces.
- Core prompt shells remain generic and do not interpret problem-specific
  bounds, component names, scale terms, or runtime fields.

### RS2-3: Contract And Novelty

- Status: implemented as the first generic ContractGate slice.
- `ContractGate` now reads v2 `targets` and `interface` metadata as the
  authority for target files, action permissions, singleton metadata, and
  required function presence, while preserving legacy v1 surface fields as the
  compatibility input.
- Problem spec loading fails closed on explicit legacy/v2 conflicts for
  `target_files` vs `targets.files`, action permission flags, and
  `required_functions` vs `interface.required_functions`.
- Novelty supports `target_file` and `semantic_signature` strategies
  generically. Ordinary operator modify/remove remains file-keyed; singleton
  policy/config/portfolio-style surfaces can opt into deterministic signatures
  only over declared bounded structured fields currently persisted on the
  proposal/record (`target_objectives`, `protected_objectives`,
  `predicted_direction`, `suggested_weight`). Free-text rationale fields such
  as `hypothesis_text`, `expected_effect`, and `no_op_condition` are not
  identity inputs; unavailable or unsupported signature fields fall back to
  strict target-file identity.
- Research surface `kind` now fails closed unless it is one of the generic
  supported kinds: `operator`, `policy`, `config`, `portfolio`,
  `construction`, or `acceptance_restart`.
- C9c complexity checks now prefer v2 surface
  `bounds.complexity_scale_terms`; the old route/customer/order/vehicle names
  are retained only as an explicit legacy fallback for surfaces without v2
  bounds metadata.
- Hypothesis context renders screening aggregate evidence only.
  Validation/frozen `ProtocolResult` aggregate stats and raw holdout details
  are omitted from proposal context.
- Return-contract semantics and problem-specific signature renderers remain
  problem/APS work; this slice only checks declared interface presence.
- Runtime audit enforcement for `evidence.required_runtime_fields` is wired into
  adapter-backed verification through RS2-4.

### RS2-4: Runtime Audit Contract

- Let problem packages declare required runtime fields per surface.
- `VerificationGate.run()` accepts selected-surface metadata directly or derives
  it from `HypothesisProposal.change_locus`; campaign explore and stale
  reconcile paths pass the hypothesis metadata through the compatibility helper.
- Runtime audit checks the selected declared surface's
  `evidence.required_runtime_fields` against solver `runtime` output. Missing,
  `None`, or empty required values fail closed as
  `surface_runtime_contract_error`.
- Generic evidence semantics are intentionally small: `*_errors` fields must be
  numeric zero, and `*_loaded` / `*_executed` fields must be truthy when
  declared. Other fields only require presence with non-empty value.
- Unknown selected surfaces fail closed. Calls without a selected surface remain
  the explicit legacy path and keep pre-RS2-4 behavior.
- CVRP `search_policy` declares runtime fields that match the solver output:
  `policy_loaded`, `policy_errors`, `baseline_time_fraction`,
  `operator_round_limit`, and `post_baseline_operators_enabled`.
- Remaining integration gap: `ExperimentProtocol.run_experiment()` still only
  receives action metadata, so protocol pair raw metrics record legacy runtime
  audit failures but do not yet enforce selected-surface required fields.

### RS2-5: Agentic Proposal Integration

- Add `AgenticProposalSession` behind `ProposalPipeline`.
- Expose surface v2 through proposal tools.
- Persist tainted transcript and memory refs.
- Keep single-shot proposal as fallback.

### RS2-6: CVRP First Algorithm Surfaces

- First slice complete: add `construction_policy`.
  - Checked-in default: `policies/construction_policy.py`.
  - Surface kind: `construction`.
  - Required functions: `construction_mode(instance, time_limit_sec)` and
    `construction_bias(instance, time_limit_sec)`.
  - Solver audit fields: `construction_surface_loaded`,
    `construction_errors`, `construction_mode`, `construction_elapsed_ms`,
    `construction_routes`, `construction_distance`, and
    `construction_feasible`.
  - Invalid policy outputs fail closed through runtime audit.
- Second slice complete: add `neighborhood_portfolio`.
  - Checked-in default: `policies/neighborhood_portfolio.py`.
  - Surface kind: `portfolio`.
  - Required functions: `enabled_components(instance, time_limit_sec)`,
    `component_weights(instance, time_limit_sec)`, and
    `candidate_limits(instance, time_limit_sec)`.
  - Solver audit fields: `portfolio_surface_loaded`, `portfolio_errors`,
    `enabled_components`, `component_weights`, `candidate_limits`,
    `component_attempts`, `component_accepted`, `component_runtime_ms`, and
    `portfolio_stop_reason`.
  - Invalid unknown components, weights, or limits fail closed through runtime
    audit.
- Controlled CVRP fixtures now prove non-operator surfaces change runtime audit
  fields and search behavior without editing `solver.py`.

### RS2-7: Dual Problem Regression

- Run warehouse and CVRP smoke campaigns.
- Warehouse should still operate through operator surfaces.
- CVRP should show nontrivial non-operator surface execution.

## Acceptance Criteria

Design acceptance:

- v0.4 has two explicit upgrade modules: `research_surfaces v2` and
  `AgenticProposalSession`.
- Docs distinguish algorithm-space modeling from proposal-agent control flow.
- Docs preserve the v3 tainted Creative Layer boundary.

Framework acceptance:

- Scion core can load and expose surface v2 metadata without problem-specific
  conditionals.
- Agent tools can inspect surfaces without reading forbidden holdout data.
- Contract and novelty are surface-aware and problem-metadata-driven.
- `DecisionFeatures` remains free of proposal text.

CVRP acceptance:

- CVRP exposes at least one main-algorithm surface beyond post-baseline
  operators.
- Runtime audit proves selected surface loaded and executed.
- Controlled CVRP campaign demonstrates a non-operator surface can affect search
  behavior.

Regression acceptance:

- Warehouse operator path remains functional.
- Legacy problem packages still work through compatibility mapping.
- No CVRP or warehouse logic is added to Decision, Protocol, CampaignLoop, or
  BranchStepRunner.
