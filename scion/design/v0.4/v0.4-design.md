# Scion v0.4 Design

*Date: 2026-05-01*

## Positioning

v0.4 is the first post-v0.3 hardening release. Its purpose is not to run more
campaigns on a new benchmark set. Its purpose is to prove that Scion's
branch-governed, evidence-driven autoresearch loop can generalize to a second
real combinatorial optimization problem while preserving the audit boundaries
defined in the v3 architecture.

v0.4 has three linked goals:

1. Make performance and evidence completeness first-class promotion
   constraints.
2. Onboard CVRP as the second real problem class through the `ProblemAdapter`
   boundary.
3. Continue splitting `campaign.py` so the framework remains understandable and
   auditable as new problem classes are added.

The target claim is:

```text
Scion can optimize a modular CVRP heuristic baseline through the same
hypothesis -> branch -> code -> verification -> protocol -> frozen promotion
loop used in v0.3, with complete evidence and runtime-aware gates.
```

## Non-Negotiable Invariants

The v3 architecture remains the controlling design source.

- LLM output is tainted. It can influence proposal generation and context, but
  cannot directly influence promotion decisions.
- The Decision Layer reads only structured `DecisionFeatures`.
- Contract Gate, Verification Gate, Experiment Protocol, and Decision Engine
  remain separate control surfaces.
- Frozen holdout evidence is required for promotion.
- A champion is a pool-level immutable snapshot, not a mutable workspace.
- Promotion count is search-process evidence, not final solver-quality
  evidence.
- Persisted facts should be enough to rebuild views; mutable summaries must not
  become a second source of truth.

## Scope

### In Scope

- Performance-aware promotion constraints.
- CVRP adapter, objective policy, solver wrapper, fixture tests, and split
  manifests.
- Final quality/runtime evidence harness shared by warehouse delivery and CVRP.
- Surgical `campaign.py` decomposition.
- Runtime feedback to LLM context, isolated from Decision.
- Legacy fallback reduction where it blocks adapter-native CVRP execution.

### Out of Scope

- Beating state-of-the-art CVRP solvers.
- Full CVRPLIB-scale formal claims before the adapter and harness are stable.
- VRPTW, pickup-and-delivery, heterogeneous fleets, or other VRP variants.
- FCMCNF/Benders integration.
- Treating MILP or BKS references as promotion oracles without adapter-defined
  provenance.

## Design Decisions

### D1: Runtime Is A Promotion Constraint

A generated operator is valid only if it satisfies all three conditions:

```text
objective improvement
+ complete evidence
+ acceptable runtime/complexity
```

Consequences:

- Timeout/crash pairs are counted, never skipped.
- Validation and frozen gates fail closed on incomplete evidence.
- Static contract rejects unbounded high-order enumeration.
- `V9_perf_guard` must report candidate time, champion time, ratio, case id,
  and timeout status.
- Runtime failures become structured search feedback for future proposals, but
  do not become free-text Decision inputs.

### D2: CVRP Promotion Objective

CVRP promotion uses a lexicographic objective:

```text
minimize:
1. fleet_violation = max(0, routes - allowed_routes)
2. total_distance
```

`allowed_routes` should come from an explicit fleet limit when available. When
the benchmark instance does not expose a reliable fleet limit, Scion may use the
reference `.sol` route count as the benchmark-comparability limit.

This rule prevents a candidate from winning by using more routes to reduce
distance. Such a candidate can be CVRP-feasible under the current baseline model,
but it is not benchmark-comparable.

### D3: BKS Gap Is Final Quality Evidence, Not Promotion Evidence

CVRPLIB reference solutions provide a best-known cost and, through their route
list, a reference route count. Scion should report:

- `bks`
- `gap_pct`
- `bks_routes`
- `routes`
- `route_gap`
- `benchmark_feasible`

These belong in final quality reports. They should not be the primary campaign
promotion objective because route-count mismatches can create misleading
negative gaps.

### D4: CVRP Adapter Must Fail Closed

The CVRP adapter must not trust solver-reported feasibility or cost. It must
recompute and validate:

- every customer appears exactly once;
- depot is not duplicated inside customer routes;
- no unknown customer id appears;
- each route respects capacity;
- total distance is recomputed from the instance distance function;
- objective fields are finite and complete;
- benchmark comparability is explicit and separate from CVRP feasibility.

### D5: `campaign.py` Decomposition Is Part Of v0.4

CVRP should not be integrated by adding more responsibilities to
`CampaignManager`. v0.4 should split behavior while preserving existing tests
and public campaign semantics.

Initial target services:

| Component | Responsibility |
|---|---|
| `CampaignLoop` | round loop, scheduler, termination |
| `ProposalPipeline` | hypothesis, code generation, fix prompt, tainted context |
| `EvaluationPipeline` | contract, verification, canary, protocol, feature extraction |
| `DecisionCoordinator` | DecisionEngine invocation and rule provenance |
| `PromotionService` | champion snapshot, registry commit, stale marking |
| `EvidenceRecorder` | status, lineage, metrics refs, final evidence artifacts |
| `FailureLifecycleService` | reject, archive, retry/block, failure memory |
| `WeightOptCommitter` | background result drain and revision commit |

This is a staged extraction, not a rewrite.

## Architecture Workstreams

### A. Performance-Aware Framework

Tasks:

- Extend promotion evidence with runtime guard summaries.
- Ensure `DecisionFeatures` can represent runtime/pass-fail facts without
  exposing free text.
- Add runtime feedback to proposal context:
  - slow cases;
  - timeout cases;
  - candidate/champion elapsed ratio;
  - failed operator file;
  - bounded-neighborhood guidance.
- Add regression tests for old slow operator families.
- Make final quality harness count timeouts and crashes.

Acceptance:

- A candidate that improves objective but times out cannot promote.
- A candidate with incomplete validation/frozen evidence cannot promote.
- Perf guard details are sufficient to diagnose the slow case without log
  archaeology.

### B. Campaign Framework Cleanup

Tasks:

- Extract evaluation and promotion paths first.
- Move raw metrics/status/summary writing into an evidence recorder.
- Keep `DecisionEngine` pure and deterministic.
- Separate protocol rules, budget rules, and heuristic hints in reason codes.
- Keep stagnation and failure memory as proposal guidance, not Decision input.

Acceptance:

- Existing v0.3 campaign tests remain green after each extraction.
- `CampaignManager` no longer owns protocol execution, promotion commit, and
  evidence artifact writing directly.
- New components have focused unit tests.

### C. CVRP Problem Adapter

Tasks:

- Add `scion/problems/cvrp/problem-v1.yaml`.
- Add `scion/problems/cvrp/adapter.py`.
- Add tiny synthetic CVRP fixtures for tests.
- Add a thin workspace `solver.py` wrapper over the existing `vrp/src` baseline.
- Define CVRP objective metrics:
  - `fleet_violation`, minimize, priority 1;
  - `total_distance`, minimize, priority 2.
- Define CVRP LLM operator interface for bounded route-local operators.

Acceptance:

- Adapter unit tests cover valid, duplicate, missing, over-capacity, unknown-id,
  and recomputed-cost mismatch cases.
- Strict adapter-backed runtime verification passes on tiny fixtures.
- CVRP can run through canary/screening in a smoke campaign without warehouse
  fallback behavior.

### D. Final Evidence Harness

Tasks:

- Implement fixed-set final evaluation:
  - original baseline;
  - final champion;
  - optional reference/BKS.
- Emit:
  - `final_quality.json`;
  - `final_quality.csv`;
  - `per_case_quality.csv`;
  - `runtime_summary.json`;
  - `failure_summary.json`;
  - `evidence_manifest.json`.
- Share the top-level schema between warehouse delivery and CVRP.

Acceptance:

- Every formal campaign has final quality/runtime evidence.
- Promotion evidence and final quality evidence are reported separately.
- Timeout, crash, infeasible output, benchmark-incomparable output, and runtime
  regression are distinguishable.

### E. Legacy Risk Closure

Tasks:

- Remove or isolate warehouse-specific fallback paths that would make CVRP
  appear to work without adapter-native checks.
- Make V5/state-shape checks adapter-native for routes.
- Decide the bridge plan for old `ProblemSpec` and new `ProblemSpecV1`.
- Keep MILP and BKS references report-only unless the adapter defines a sound
  lower-bound/reference provenance.
- Ensure context exposure does not leak validation/frozen case-level feedback.

Acceptance:

- CVRP cannot run through warehouse-shaped feasibility or state-mutation checks.
- Production runs require metric specs and adapter-backed objective comparison.
- Context artifacts are clearly marked as proposal guidance, not Decision facts.

## Development Phases

### Phase 0: Design Lock

- Land this design document.
- Convert it into a task manifest.
- Identify tests that must stay green throughout the extraction.

Exit:

- v0.4 scope, non-goals, objective policy, and acceptance gates are fixed.

### Phase 1: Framework Extraction

- Extract `EvaluationPipeline`.
- Extract `PromotionService`.
- Extract `EvidenceRecorder`.
- Keep behavior unchanged.

Exit:

- Current campaign, protocol, lineage, and adapter tests pass.

### Phase 2: Performance Hardening + CVRP Smoke

- Add runtime guard evidence fields.
- Add timeout/slow-case feedback to proposal context.
- Add old-slow-operator regression tests.
- Add CVRP package under `scion/problems/cvrp`.
- Add tiny fixtures and adapter tests.
- Add solver wrapper.
- Run strict runtime verification.

Exit:

- Objective-improving but slow candidates fail before promotion.
- CVRP can pass adapter, verification, and screening smoke without reading raw
  CVRPLIB benchmark files in tests.

### Phase 3: Evidence Harness + CVRP Benchmark Reporting

- Implement shared final quality/runtime harness.
- Convert existing CVRP baseline result tables into the v0.4 evidence package.
- Prepare final champion evaluation wiring.

Exit:

- Final reports include quality, runtime, timeout, failure, and benchmark
  comparability fields.

### Phase 4: Formal CVRP Campaign

- Run smoke campaign first.
- Then run a small formal matrix only after evidence harness and performance
  gates are stable.

Candidate matrix:

```text
models = sonnet, gpt-mini
seeds = 11, 29, 47
rounds = 100
problem = cvrp
```

Exit:

- At least one complete CVRP campaign matrix produces promotion evidence and
  final quality evidence under the same schema.

## v0.4 Exit Criteria

v0.4 can close when:

- Warehouse v0.3 production rerun remains free of incomplete-evidence
  promotions.
- CVRP is integrated as a strict `ProblemAdapter`.
- CVRP promotion uses `fleet_violation -> total_distance`.
- Multi-route-count benchmark-incomparable solutions cannot promote as wins.
- Every validation/frozen promotion has complete evidence.
- Slow or unbounded generated operators are rejected before promotion.
- Every formal campaign has final quality/runtime comparison against baseline.
- `campaign.py` has clear extracted services for evaluation, promotion,
  evidence, and failure lifecycle.
- Documentation distinguishes framework closure, campaign dynamics, final
  quality, runtime acceptability, and benchmark-reference claims.

## Open Questions

1. Should `allowed_routes` be mandatory for all CVRP promotion cases, or may
   screening include cases without reference route counts using
   `fleet_violation = 0` and report-only benchmark comparability?
2. Should CVRP formal campaigns optimize only route-local operators first, or
   allow bounded construction changes after the first smoke run?
3. Should weight optimization be disabled for the first CVRP structural smoke
   campaign to reduce runtime/confounding?
4. Should `ProblemSpec` and `ProblemSpecV1` be unified in v0.4, or should v0.4
   define a strict bridge and defer full unification?
