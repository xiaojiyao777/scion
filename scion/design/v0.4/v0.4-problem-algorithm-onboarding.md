# Scion v0.4 Problem And Algorithm Onboarding

*Date: 2026-05-08*
*Status: design source for making research objects Scion-native*

## Purpose

The recent CVRP work exposed a structural distinction that must become explicit
in Scion:

```text
Scion core can govern autoresearch only after a problem/algorithm has been made
into a Scion-native research object.
```

Giving Scion a solver repository is not enough. A baseline solver can be run as
a black box, but Scion cannot safely or productively research it unless the
problem package declares:

- the problem semantics;
- the adapter/oracle boundary;
- the algorithm components that are safe to edit;
- the runtime evidence proving those components loaded, executed, and mattered.

This document defines **Problem And Algorithm Onboarding** as a first-class
Scion module. Its job is to turn a human-written solver into a governed
problem package before the autoresearch loop begins.

## Design Thesis

Scion has two different product layers:

```text
1. Onboard
   Convert a solver/problem into a Scion-native research object.

2. Autoresearch
   Run LLM proposal, Contract, Verification, Protocol, and Decision on that
   governed research object.
```

The v3 Scion core mostly solves layer 2. CVRP shows that layer 1 is at least as
important and often harder.

## Why A Solver Is Not Enough

A solver binary or Python package can answer:

```text
What solution did this algorithm produce?
```

It does not by itself answer the questions Scion needs:

```text
Which algorithm component changed?
Did that component actually execute?
Did it attempt moves?
Did it accept moves?
Did it change the objective?
Did it fail safely?
Was the change allowed to see this data?
Did the candidate preserve the problem semantics?
```

If Scion simply lets an LLM edit arbitrary solver code, the framework loses the
properties it is designed to protect:

- objective correctness;
- feasibility integrity;
- holdout isolation;
- deterministic decision inputs;
- auditable lineage;
- runtime failure semantics;
- protection against case-specific overfitting.

Therefore the research object is not merely "the algorithm source code." It is:

```text
baseline solver
+ ProblemAdapter
+ ProblemSpecV1
+ declared research surfaces
+ solver hooks
+ runtime audit
+ tests
+ benchmark/protocol assets
```

## What Onboarding Produces

Problem And Algorithm Onboarding should produce a problem package with these
artifacts:

```text
problem-v1.yaml
ProblemAdapter implementation
solver wrapper / policy hooks
research_surfaces declarations
objective metric specs and policy
runtime audit field contract
formal/controlled protocol assets
adapter-backed verification tests
surface contract tests
focused runtime smoke tests
engineering map entry
```

For a mature onboarding flow, most of this should be generated as a draft and
then reviewed by a human.

## Minimum Admission Criteria

A problem/algorithm should not enter Scion autoresearch until it satisfies the
following criteria.

### Problem Semantics

- Inputs, solution representation, constraints, and objective are stable.
- The adapter can load instances without relying on proposal-generated code.
- The adapter can normalize solver output into a problem-native artifact.

### Objective And Feasibility

- Objective values can be recomputed independently of the solver.
- Feasibility can be checked independently of the solver.
- Missing, malformed, infeasible, or inconsistent output fails closed.

### Experimental Protocol

- Screening, validation, and frozen splits are fixed before the run.
- Seeds are fixed in a seed ledger.
- Final reporting fields such as BKS/gap are report-only unless explicitly
  part of the objective policy.

### Runtime Boundary

- The solver can run through the generic runner boundary.
- Inputs and outputs are structured.
- Runtime failure categories are auditable.
- The solver can emit bounded runtime evidence for selected surfaces.

### Research Surfaces

The package must declare safe algorithm surfaces. Examples:

- construction policy;
- neighborhood portfolio;
- destroy/repair policy;
- acceptance policy;
- restart/perturbation policy;
- budget schedule;
- parameter space;
- top-level main-search strategy.

Each surface must define:

```text
algorithm role
invocation point
target files
required functions or return contract
bounds
runtime evidence fields
novelty identity
prompt/context metadata
focused tests
```

If these cannot be named, the problem is not implementation-ready for Scion
autoresearch.

## Research Surfaces Are Not Prompt Text

A research surface is an API and evidence contract, not a prompt hint.

Bad onboarding:

```text
Tell the LLM to improve the algorithm.
Let it edit a solver file.
Trust the solver output.
```

Scion-native onboarding:

```text
Expose a bounded destroy_repair_policy.
Whitelist package-owned destroy and repair components.
Validate returned component names and numeric ranges.
Emit attempts, accepted moves, objective deltas, errors, and stop reason.
Fail selected-surface audit if required evidence is missing.
```

This distinction matters because prompt text does not create a trustworthy
research object. Solver hooks and runtime audit do.

## Component Libraries Are Part Of The Research Object

For hard combinatorial optimization problems, a useful research surface usually
needs a problem-owned component library. Examples for route-based heuristics:

```text
construction:
  nearest neighbor, savings, sweep, demand-biased seed

destroy:
  random removal, worst removal, Shaw/related removal, route removal

repair:
  cheapest insertion, regret-2, regret-3, capacity-aware insertion

neighborhood:
  relocate, swap, 2-opt*, cross-exchange, Or-opt

acceptance:
  greedy, threshold, simulated annealing

restart/perturbation:
  ruin/rebuild, route-pair perturbation, incumbent reset
```

The LLM should not need to invent all of these from scratch inside one patch.
Scion can then research:

```text
which components
in which order
with which budgets
with which acceptance/restart policy
with which parameter schedule
```

This gives the autoresearch loop real, auditable levers.

## Algorithm-Family Templates

The onboarding module should not try to understand every combinatorial
optimization problem at once. It should begin with algorithm-family templates.

### Route-Based Metaheuristics

Examples:

- TSP;
- CVRP;
- VRP variants.

Likely surfaces:

- construction;
- neighborhood portfolio;
- destroy/repair;
- acceptance/restart;
- budget schedule;
- parameter space.

### Assignment And Local-Search Solvers

Examples:

- warehouse delivery assignment;
- facility assignment;
- matching-style heuristics.

Likely surfaces:

- assign/move/swap/repair;
- penalty policy;
- neighborhood portfolio;
- saturation/overflow repair;
- parameter space.

### Scheduling Heuristics

Examples:

- job shop;
- machine scheduling;
- dispatching-rule heuristics.

Likely surfaces:

- dispatching rules;
- insertion/swap neighborhoods;
- critical-path repair;
- idle-time repair;
- restart policy.

### MILP/CP Hybrid Heuristics

Examples:

- relax-and-fix;
- fix-and-optimize;
- rounding/repair loops.

Likely surfaces:

- relaxation policy;
- variable-fixing policy;
- rounding policy;
- repair neighborhood;
- subproblem budget schedule.

## Automation Boundary

Onboarding can be partially automated, but it should not be fully trusted.

Automation can draft:

- solver entrypoint map;
- parameter inventory;
- candidate surface manifest;
- wrapper hook locations;
- runtime audit field names;
- adapter/test checklist;
- initial problem-v1.yaml.

Humans must review:

- objective and feasibility semantics;
- allowed data exposure;
- which surfaces are safe;
- whether runtime audit proves behavior;
- benchmark split validity;
- whether any surface can leak holdout or case identity.

This keeps the human in the setup/review loop, not in every autoresearch
iteration.

## Relationship To Scion Core

Scion core remains problem-agnostic. It should own:

- lifecycle and scheduling;
- Contract / Verification / Protocol / Decision orchestration;
- generic surface schema handling;
- generic proposal tools;
- exposure policy;
- evidence refs and lineage.

Problem packages own:

- domain semantics;
- algorithm component libraries;
- adapter rendering;
- solver hooks;
- runtime audit field meanings;
- final evidence interpretation.

The onboarding module sits between them:

```text
human solver/problem
      |
      v
Problem And Algorithm Onboarding
      |
      v
Scion-native problem package
      |
      v
Scion autoresearch core
```

## CVRP Lesson

CVRP demonstrated the gap:

```text
post-baseline generated operators
```

were not enough because the real algorithmic strength lived in the repo-local
ALNS+VNS main search. Adding `main_search_strategy` created a whole-algorithm
orchestration surface, but later diagnostics clarified its limit: it can
arrange and parameterize declared main-search components, not invent new ALNS,
VNS, destroy/repair, or route-pair mechanisms by editing the original solver.

This is the right v3 boundary for an onboarded solver, because Contract,
Verification, Protocol, and runtime audit can reason about the declared
surface. It also caps the search ceiling when the exposed component library is
weak. The next CVRP work should treat route-pair swap, destroy/repair,
acceptance/restart, adaptive weights, neighborhood schedules, and parameter
schedules as problem-owned component libraries and deeper surfaces, not
prompt-only suggestions and not arbitrary `vrp/src` edits.

## Acceptance Criteria For This Module

The onboarding module is successful when:

- a new solver can be inspected and mapped to a draft problem package;
- the draft names candidate surfaces, invocation points, bounds, and runtime
  evidence;
- generated wrapper hooks are small and reviewable;
- adapter tests prove objective and feasibility integrity;
- selected-surface audit fails closed;
- Scion core does not learn problem-specific semantics;
- a human can approve or reject the generated research object before
  autoresearch starts.

## Non-Goals

Problem And Algorithm Onboarding should not:

- let LLMs directly edit arbitrary solver internals during campaigns;
- auto-approve problem semantics;
- define objectives from benchmark answers;
- weaken frozen holdout exposure;
- make Decision read proposal text or domain-specific runtime fields;
- claim a solver-quality improvement before Protocol evidence supports it.

## Near-Term Implementation Direction

For v0.4, this remains a design direction rather than a completed product
module. The practical next steps are:

1. Extract the CVRP manual onboarding lessons into reusable templates.
2. Define a route-based metaheuristic onboarding template.
3. Generate a draft `research_surfaces` manifest from solver structure and
   human-provided problem metadata.
4. Generate runtime-audit checklist stubs for each surface.
5. Keep human review mandatory before running autoresearch.
