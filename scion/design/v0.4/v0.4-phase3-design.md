# Scion v0.4 Phase 3 Design

*Date: 2026-05-01*
*Branch: v0.4-dev*

## Position

Phase 3 turns the CVRP smoke path from Phase 2 into a research-usable evidence
path.

Phase 2 proved that CVRP can enter Scion through the adapter boundary and run a
one-step campaign on tiny synthetic fixtures. Phase 3 should not jump directly
to formal campaigns. It should first implement the final quality/runtime
reporting harness that will judge those campaigns.

The controlling rule is:

```text
promotion evidence != final quality evidence
```

Promotion evidence explains why a branch was promoted at the time. Final
quality evidence compares the final champion against the original baseline on a
fixed evaluation set with runtime, failures, and benchmark comparability
accounted for.

## Scope

Phase 3 owns three concrete outcomes:

1. Shared final quality/runtime/failure evidence package writer.
2. CVRP benchmark-result reporting path that consumes existing result tables and
   later campaign outputs without reading raw CVRPLIB instances in tests.
3. Design preparation for real CVRP campaign evaluation subsets and runner
   integration.

## Non-Goals

- No formal CVRP campaign matrix yet.
- No raw `vrp/cvrplib/**` instance parsing in Scion tests.
- No BKS/gap-based promotion logic.
- No attempt to beat state-of-the-art CVRP solvers.
- No rewrite of `CampaignManager` in this phase.

## Evidence Contract

Every final quality package should contain:

```text
evidence_manifest.json
final_quality.json
final_quality.csv
per_case_quality.csv
runtime_summary.json
failure_summary.json
```

The writer should be problem-aware but not problem-coupled:

- Common fields are always present.
- CVRP-specific fields are optional extensions.
- Missing BKS or benchmark route counts are explicit, not inferred.
- Timeouts, crashes, infeasible outputs, and benchmark-incomparable outputs are
  counted instead of dropped.

## Common Case Fields

Per evaluated case:

```text
case_id
subset
seed
baseline_status
candidate_status
comparison
decisive_metric
baseline_objective
candidate_objective
metric_deltas
baseline_elapsed_ms
candidate_elapsed_ms
runtime_ratio
runtime_regression
error_category
```

`comparison` is computed from adapter metric specs and objective policy when
both sides have valid comparable objectives. If either side fails, comparison is
classified from status instead of invented from missing metrics.

## CVRP Extension Fields

CVRP final reports may include:

```text
baseline_cost
candidate_cost
bks
baseline_gap_pct
candidate_gap_pct
baseline_routes
candidate_routes
bks_routes
baseline_route_gap
candidate_route_gap
baseline_feasible
candidate_feasible
baseline_benchmark_feasible
candidate_benchmark_feasible
```

BKS/gap fields are report-only. They must not feed campaign promotion.

Benchmark comparability is stricter than CVRP feasibility:

```text
benchmark_feasible = cvrp_feasible and routes <= bks_routes
```

when `bks_routes` is known. When `bks_routes` is unknown, the field should be
`null` or `unknown`, not silently `true`.

## Aggregates

`final_quality.json` should include:

```text
problem_id
campaign_id
baseline_label
candidate_label
n_cases
n_ok
n_timeout
n_error
n_infeasible
n_benchmark_incomparable
better_vs_baseline
equal_vs_baseline
worse_vs_baseline
primary_delta_sum
primary_delta_median
wall_time_total_ms
wall_time_median_ms
runtime_regressions
```

CVRP aggregate extension:

```text
n_with_bks
n_with_bks_routes
mean_candidate_gap_pct
median_candidate_gap_pct
mean_baseline_gap_pct
median_baseline_gap_pct
candidate_benchmark_feasible
baseline_benchmark_feasible
```

## Relationship To Existing VRP Results

The repository already has baseline CVRP result tables under `vrp/results/`.
Phase 3 may read those CSV artifacts, such as:

```text
vrp/results/full_experiment_seed0_final.csv
vrp/results/analysis_full_seed0_final/summary_by_subset.csv
vrp/results/reference_validation_bad.csv
```

It must not read raw files under:

```text
vrp/cvrplib/**
```

The first implementation slice should use synthetic rows in Scion tests. A
later slice can add an importer for the existing `vrp/results` CSVs.

## Implementation Boundary

The first code slice should be pure and deterministic:

- no subprocess solver runs;
- no LLM calls;
- no campaign mutation;
- no database writes;
- standard-library JSON/CSV output is sufficient.

Recommended module:

```text
scion/scion/evidence/final_quality.py
```

Recommended tests:

```text
scion/scion/tests/unit/evidence/test_final_quality.py
```

The writer should accept in-memory case records and write the evidence package.
Runner integration and CVRP CSV import should be separate later slices.

## Acceptance

Phase 3 first-slice acceptance:

- Synthetic baseline-vs-candidate records produce all six evidence files.
- Timeout/crash/infeasible rows appear in `failure_summary.json` and
  `per_case_quality.csv`.
- Runtime regression counts are computed from a configurable ratio threshold.
- CVRP BKS/gap/route comparability fields are emitted but never affect the
  comparison outcome.
- Tests do not read `vrp/cvrplib/**`.

Phase 3 phase-level acceptance:

- Existing CVRP baseline result CSVs can be converted into a v0.4 evidence
  package.
- A future campaign final champion can use the same writer for final evidence.
- Documentation clearly separates campaign promotion evidence from final
  quality/runtime evidence.
