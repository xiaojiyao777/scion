# Scion v0.4 P0-C Promotion Integrity Design

*Date: 2026-05-04*
*Branch: v0.4-dev*

## Position

P0-C closes the remaining promotion-integrity gap found after the Phase 1-4
extraction work.

The v3 architecture treats promotion as a deterministic framework decision over
complete evidence. It also defines a champion as an immutable pool-level
snapshot, not a mutable workspace. v0.4 adds runtime and evidence completeness
as first-class promotion constraints.

Therefore a promotion is not complete until the framework can persist and
reconstruct the promoted champion snapshot, operator registry, branch state
effects, and lineage evidence consistently.

## Problem

The current service extraction made promotion easier to inspect, but several
side effects still happen in an unsafe order:

- `PromotionService.commit()` can install the in-memory champion and mark other
  branches stale before champion store persistence succeeds.
- `PromotionService.prepare()` can fall back to the previous operator pool when
  a snapshot `registry.yaml` exists but cannot be read.
- `WeightOptCommitter` can replace the in-memory champion before persisting the
  optimized champion revision.
- The direct `_on_promote()` compatibility entry can bypass the full branch
  state and hypothesis lifecycle expected by the normal frozen promotion path.
- Evaluation-only paths can spend protocol budget before detecting missing
  canonical hypothesis state.

These are not CVRP-specific bugs. They are framework integrity issues that
matter more once generated CVRP operators and registry-backed solver behavior
become part of the promoted champion.

## Non-Negotiable Invariants

P0-C inherits these invariants from v3 and the v0.4 design:

- A champion is a pool-level immutable snapshot.
- Promotion requires complete validation/frozen evidence and no critical
  verification/runtime failure.
- Screening evidence is not promotion evidence.
- Persisted facts must be sufficient to rebuild campaign views.
- Mutable summaries and in-memory state must not become a second source of
  truth.
- Registry/operator-pool mismatches must fail closed when they affect promoted
  code.

## Design Decisions

### D1: Promotion Prepare Fails Closed On Registry Read Failure

If a candidate champion snapshot contains `registry.yaml`, the registry is part
of the promoted solver contract. `PromotionService.prepare()` must read it
successfully.

Allowed fallback:

- If `registry.yaml` is absent, use `previous_operator_pool` only for legacy
  workspaces that genuinely have no registry file.

Disallowed fallback:

- If `registry.yaml` exists but cannot be parsed or validated, do not promote.
- Do not silently keep the previous operator pool while promoting new snapshot
  code.

### D2: Champion Persistence Happens Before In-Memory Promotion

The promotion commit order should be:

1. Persist the new champion snapshot record.
2. Run promotion-cycle bookkeeping that is valid only after persistence
   succeeds.
3. Install the in-memory champion and operator pool.
4. Transition the promoted branch to `PROMOTED`.
5. Mark other eligible branches stale and persist branch states.
6. Record promoted-branch search memory, lineage, and other audit side effects.
7. Launch weight optimization after the champion commit has succeeded.

If champion persistence fails, the commit aborts before in-memory champion
replacement, stale marking, branch-state persistence, search-memory mutation,
lineage promotion rows, or weight optimization launch.

### D3: Weight Optimization Uses The Same Persistence Rule

An optimized champion revision is a champion update. It must follow the same
ordering:

1. Validate that the completion event still matches the current champion
   version and base weight revision.
2. Build the optimized champion value object.
3. Persist the optimized champion.
4. Replace the in-memory champion.
5. Mark non-frozen active branches stale for weight-update reconciliation.

If persistence fails, keep the current in-memory champion and do not stale
branches for the failed weight update.

### D4: `_on_promote()` Is Not A Semantic Shortcut

The direct `_on_promote()` entry is a compatibility/test helper, not a second
promotion semantic path.

It must either:

- call the same branch transition and hypothesis lifecycle path used by normal
  frozen promotion; or
- be renamed/restricted so tests call a lower-level helper explicitly and do not
  imply a complete promotion lifecycle.

The preferred v0.4 cleanup is to keep `_commit_promote_plan()` low-level and make
all semantic promotion callers transition the branch to `PROMOTED` before stale
marking.

### D5: Missing Canonical Hypothesis State Fails Before Evaluation

An eval-only step without the canonical `HypothesisRecord` needed for evidence
and status mutation is an internal state error. Detect it before running canary
or protocol evaluation so budget is not consumed by an unreconstructable event.

## Implementation Scope

Files expected to change:

- `scion/scion/core/promotion_service.py`
- `scion/scion/core/promotion_lifecycle.py`
- `scion/scion/core/weight_opt_committer.py`
- `scion/scion/core/campaign.py`
- `scion/scion/tests/unit/core/test_promotion_service.py`
- `scion/scion/tests/unit/core/test_weight_opt_committer.py`
- `scion/scion/tests/test_campaign.py`

This is not a full `CampaignLoop` extraction. P0-C should keep the change
focused on promotion integrity and immediate campaign readability.

The promotion lifecycle may be extracted as a focused campaign boundary so
`CampaignManager` no longer owns champion prepare, commit, branch transition,
search-memory promotion notes, and weight-opt launch directly.

## Required Tests

Add or update tests for:

- registry read failure blocks promotion prepare when `registry.yaml` exists;
- absent registry preserves legacy previous-pool fallback;
- champion store failure aborts promotion commit side effects;
- promotion commit success still marks only eligible branches stale;
- weight-opt champion store failure does not replace in-memory champion;
- weight-opt champion store failure does not mark branches stale;
- `_on_promote()` cannot stale the promoted branch or bypass promoted state
  semantics;
- missing canonical hypothesis state is detected before protocol budget is
  consumed.

## Acceptance

P0-C is complete when:

- promotion prepare cannot create a snapshot whose code and operator pool are
  inconsistent due to registry fallback;
- champion persistence failure leaves in-memory champion, branch stale state,
  and weight optimization unchanged;
- weight-opt persistence failure leaves in-memory champion and branch stale
  state unchanged;
- the direct promotion helper no longer represents a divergent lifecycle;
- targeted promotion/weight/campaign tests pass;
- the full Scion suite remains green.
