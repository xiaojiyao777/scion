# Scion v0.4 Phase 4 Design

*Date: 2026-05-01*
*Branch: v0.4-dev*

## Position

Phase 4 turns the Phase 2/3 CVRP smoke and evidence infrastructure into a
controlled CVRP campaign path.

The first objective is not a formal multi-model matrix. The first objective is
to make one controlled CVRP run auditable end to end:

```text
fixed case manifest
  -> CVRP adapter/runtime
  -> short campaign
  -> manifest-driven final evaluation
  -> final evidence refs in campaign summary
```

## Current Gap

Phase 3 can build fixed manifests from existing `vrp/results` CSV artifacts.
Those manifests contain `.vrp` paths from the CVRPLIB-style baseline results.

Current `scion.problems.cvrp` runtime still loads only the tiny JSON fixture
format. Therefore Phase 4 must first make the Scion CVRP adapter/runtime accept
CVRPLIB-style `.vrp` input before using real benchmark manifests.

Tests must continue to use synthetic fixtures and must not read raw
`vrp/cvrplib/**` files.

## Non-Negotiable Boundaries

- BKS/gap remains final report evidence, not promotion logic.
- Route-count benchmark comparability remains separate from CVRP feasibility.
- Multi-route fake wins must stay blocked in final evidence.
- Final quality evaluation remains a post-campaign step.
- Campaign loop must not auto-run final evaluation.
- Tests must not depend on raw `vrp/cvrplib/**`.

## Phase 4 Slices

### P4-01: CVRPLIB Input Support

Add synthetic-tested `.vrp` parsing to the Scion CVRP problem package.

Acceptance:

- `CvrpAdapter.load_instance()` loads `.json` and `.vrp`.
- Parser supports EUC_2D CVRPLIB-style files.
- Optional sibling `.sol` parsing can populate `bks` and `bks_routes`.
- Unsupported edge-weight types fail closed.
- Tests use temp synthetic `.vrp/.sol` files only.

### P4-02: CVRP Solver Wrapper For `.vrp`

Make `scion.problems.cvrp.solver` accept `.vrp` paths through the adapter and
emit the same route/objective JSON contract.

Acceptance:

- `LocalSubprocessRunner` can run Scion CVRP solver on a synthetic `.vrp`.
- Output passes adapter consistency, feasibility, and recomputed objective.
- No raw CVRPLIB files in tests.

### P4-03: Controlled Manifest Fixtures

Generate small checked-in Scion-side Phase4 manifests from synthetic fixtures
or from CSV rows with opaque paths.

Acceptance:

- Screening/validation/frozen/final manifest shapes are fixed.
- Runtime budgets and seeds are explicit.
- The first controlled run has a documented case plan.

### P4-04: One Short CVRP Campaign

Run one controlled short campaign with mock or configured model, then run
manifest-driven final evidence after the campaign.

Acceptance:

- Campaign reaches at least screening with adapter-backed runtime checks.
- Final evidence package is generated after campaign.
- `campaign_summary.json` contains final evidence refs.

### P4-05: Formal Matrix Readiness

Only after P4-04 passes, define and run the multi-model/multi-seed matrix.

Candidate matrix remains:

```text
models = sonnet, gpt-mini
seeds = 11, 29, 47
rounds = 100
```

## Exit Criteria

Phase 4 can close when:

- one controlled CVRP campaign run is evidence-complete;
- final evaluation uses a fixed manifest;
- final evidence refs are attached to campaign summary;
- no promotion uses BKS/gap;
- runtime failures and benchmark-incomparable rows are counted, not skipped.
