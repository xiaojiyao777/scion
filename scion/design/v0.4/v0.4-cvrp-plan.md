# Scion v0.4 CVRP Generalization Plan

*Date: 2026-04-28*

## Decision

v0.4 will use **CVRP (Capacitated Vehicle Routing Problem)** as the next real
problem class for Scion.

The earlier v1.0 plan considered FCMCNF + Benders as the second problem. That
remains valuable for a later network-design/decomposition track, but CVRP is a
better immediate generalization probe because it is standard, benchmark-rich,
and structurally different from the current warehouse-delivery assignment
problem.

## Why CVRP Now

Warehouse delivery in v0.3 is a multi-attribute assignment/bin-packing problem:

```text
orders -> vehicles
objective = lexicographic(subcategory_splits, total_cost)
operators = move/swap/merge/split/destroy-rebuild over assignment state
```

CVRP is a routing problem:

```text
customers -> ordered vehicle routes
constraints = visit once + capacity + depot route structure
objective = usually total travel distance/cost
operators = relocate/swap/2-opt/2-opt*/Or-opt/cross-exchange/ruin-recreate
```

This makes CVRP a stronger adapter test than another assignment-style problem.
It forces Scion to handle route sequence semantics, distance matrices, route
capacity, and local-search operators whose runtime can easily become
quadratic/cubic without explicit bounds.

## Scope

v0.4 has two linked goals:

1. Make performance-awareness a first-class promotion constraint.
2. Prove that the v0.3 adapter boundary can support CVRP as a second real
   problem class.

These goals are linked because CVRP operators are a natural stress test for
runtime governance. A route operator can look promising on small cases while
being unusable on larger benchmark instances.

## Non-Goals

v0.4 does not try to:

- beat state-of-the-art CVRP solvers;
- implement every VRP variant;
- introduce VRPTW or pickup-and-delivery constraints;
- perform full paper-scale ablations;
- complete FCMCNF/Benders integration;
- do production deployment or shadow traffic.

The target claim is narrower:

```text
Scion can take a strong, modular CVRP heuristic baseline and run the same
hypothesis -> code -> verification -> protocol -> promotion loop on it,
with complete evidence and runtime-aware gates.
```

## CVRP Problem Adapter Requirements

The CVRP adapter should expose:

- instance parsing for depot, customer coordinates/demands, vehicle capacity,
  and optional vehicle count;
- solution parsing for ordered routes;
- objective recomputation from the distance matrix;
- feasibility checks:
  - every customer appears exactly once;
  - each route starts/ends at depot or uses the canonical implicit-depot format;
  - every route respects capacity;
  - no unknown customer ids appear;
- route-level summary for LLM context;
- operator interface rendering for route-local operators;
- benchmark split metadata for screening/validation/frozen.

## Initial Operator Library

The baseline should start with a small but serious operator set:

| Family | Operators |
|---|---|
| intra-route | 2-opt, Or-opt, segment relocate |
| inter-route | relocate, swap, 2-opt*, cross-exchange |
| large-neighborhood | ruin-recreate, route split/merge repair |
| route balancing | capacity-aware relocate, route load smoothing |

Scion-generated operators should be allowed to add new bounded variants of
these families, not rewrite the solver core in v0.4.

## Performance-Aware Contracts

CVRP must use the v0.4 performance rules from
[v0.4-performance-aware-plan.md](v0.4-performance-aware-plan.md):

- incomplete evidence cannot promote;
- timeout/crash pairs are visible failures;
- static contract rejects unbounded high-order enumeration;
- perf guard runs on representative larger CVRP cases;
- promotion requires objective improvement and runtime acceptability.

CVRP-specific prompt guidance should require:

- route-pair candidate caps;
- nearest-neighbor or top-k preselection when scanning route pairs;
- early no-op exits;
- explicit complexity notes in hypotheses;
- no full all-subset or all-permutation route enumeration.

## Evidence Plan

Minimum v0.4 CVRP validation:

| Evidence | Acceptance |
|---|---|
| Baseline validation | baseline produces feasible solutions on all selected cases |
| Adapter smoke | CVRP `ProblemAdapter` passes unit and integration tests |
| Operator verification | all base operators pass feasibility/objective/state checks |
| Campaign smoke | one short campaign reaches protocol evaluation without infra failure |
| Formal CVRP run | 2 models x 3 seeds, all complete |
| Promotion evidence | every validation/frozen promotion has complete evidence |
| Quality/runtime table | final champion vs baseline per case, with timeout counted |

Recommended first matrix:

```text
models = claude-sonnet-4-6, gpt-5.4-mini
variant = cvrp
seeds = 11, 29, 47
rounds = 100
weight optimization = sync unless runtime budget requires structure-only first
```

## Exit Criteria

v0.4 can close when:

- production warehouse rerun remains free of incomplete-evidence promotions;
- CVRP adapter is committed and strict;
- CVRP baseline validation artifacts are in the evidence manifest;
- at least one full CVRP campaign matrix completes;
- final CVRP report includes objective quality, runtime, timeout status, and
  failure breakdown;
- docs clearly distinguish:
  - v0.3 warehouse evidence;
  - v0.4 performance-aware hardening;
  - v0.4 CVRP generalization evidence;
  - later FCMCNF/Benders work.

## Relationship To FCMCNF

FCMCNF + Benders is deferred, not rejected.

It remains a strong later test for:

- mathematical lower-bound integration;
- decomposition-aware problem adapters;
- network-design operator families;
- single-objective cost minimization with meaningful optimum gaps.

CVRP should come first because it tests the most visible framework question:

```text
Can Scion leave the warehouse-delivery assignment world and operate on a
standard routing problem with mature benchmarks and different operator
semantics?
```
