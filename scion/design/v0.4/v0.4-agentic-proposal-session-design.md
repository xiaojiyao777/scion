# Scion v0.4 Agentic Proposal Session Design

*Date: 2026-05-05*
*Status: APS-1 skeleton, APS-2 read-only tool slice, APS-3 drafting/static
preview minimum slice, and May 13 code-phase inspection/preview-repair slice
implemented; later APS-4+ remains design source*

## Purpose

Scion v0.4 is expanding from operator design space optimization to heuristic
algorithm design space optimization. The `research_surfaces` model gives problem
packages a way to expose algorithm components beyond operators. That is
necessary but not sufficient.

The current LLM interaction model is still mostly single-shot:

```text
Round 1: one call -> HypothesisProposal
Round 2: one call -> PatchProposal
```

This is acceptable for small local operator edits. It is too weak for algorithm
surface design, where a model must inspect the declared surface, understand the
solver invocation point, compare alternatives, reason about runtime and
verification risk, and only then produce a bounded patch.

This document defines the next design step:

```text
single-shot proposal calls
-> bounded Agentic Proposal Session
```

The session makes the Creative Layer more capable. It does not weaken the v3
control model.

## Non-Negotiable Boundaries

The v3 architecture remains the controlling source.

- LLM output is tainted.
- The proposal agent belongs only to the Creative Layer.
- Contract, Verification, Protocol, and Decision remain separate control
  surfaces.
- Decision reads only deterministic `DecisionFeatures`.
- Proposal memory, rationale, rejected alternatives, and agent transcripts never
  enter `DecisionFeatures`.
- The proposal agent cannot directly promote, abandon, validate, or mutate
  champion state.
- The proposal agent cannot directly write the campaign workspace.
- Candidate code enters the framework only as a `PatchProposal` that must pass
  `ContractGate` before materialization.
- Pre-contract hypothesis context may include the current champion baseline
  code and summaries permitted by v3, but target-file code context for patch
  drafting is forbidden until an external `ContractGate` has approved the
  hypothesis.
- Validation and frozen raw metrics remain hidden from proposal tools.
- Problem-specific algorithm knowledge belongs in problem packages, adapters,
  and declared research surfaces, not in Scion core.

## Design Position

Scion should not become a general coding agent. It should have a bounded
proposal agent specialized for combinatorial-optimization research proposals.

The important shift is:

```text
LLM as function
  input context -> output JSON

LLM as proposal agent
  observe allowed context -> call allowed tools -> draft alternatives
  -> self-check -> output structured proposal
```

The role split is intentional. Scion controls boundaries, protocol, taint,
permissioning, replay, audit, and deterministic decisions. The inner research
work should still be done by an agent capable of inspecting the declared
problem object and allowed evidence. Exposing more component knobs is not a
substitute for letting the agent study and implement the algorithm body inside
the approved boundary.

The final proposal boundary remains structured:

```text
AgenticProposalSession
-> AgenticProposalOutput
-> HypothesisProposal / PatchProposal
-> ContractGate
-> WorkspaceLifecycle
-> VerificationGate
-> ExperimentProtocol
-> SafeFeatureExtractor
-> DecisionEngine
```

## Session Lifecycle

Minimum session phases:

```text
orient
diagnose
choose_surface
draft_hypothesis
inspect_interface
draft_patch
self_check
finalize
```

### 1. Orient

Load only exposure-controlled context:

- problem summary rendered by the adapter;
- objective policy and metric specs;
- declared research surfaces;
- current champion summary;
- current branch state;
- allowed proposal memory.

### 2. Diagnose

Inspect recent allowed evidence:

- screening feedback;
- runtime feedback;
- surface-level accepted/no-op rates;
- contract and verification failure summaries;
- rejected hypotheses and branch memory.

Validation information is aggregate-only. Frozen information is pass/fail,
budget, or campaign-level aggregate only.

### 3. Choose Surface

Select exactly one declared research surface and one action for this proposal.

For CVRP, this may be an operator surface such as `route_local`, but the agent
should also be able to choose policy or algorithm surfaces such as
`search_policy`, future `construction_policy`, or future
`neighborhood_portfolio`.

The selected surface must come from `ProblemSpecV1.research_surfaces` or the
legacy operator-category bridge.

### 4. Draft Hypothesis

Produce a typed hypothesis candidate naming:

- selected surface;
- action;
- target file or target artifact;
- expected behavior change;
- expected objective effect;
- runtime and complexity risk;
- protected invariants;
- reason for not choosing obvious alternatives.

The rationale is tainted. Only the structured fields proceed to contract.

### 5. Inspect Interface

Read only the target surface interface and current champion artifact permitted by
the problem package.

For operator surfaces, this may expose the operator execute signature. For
policy/config/portfolio surfaces, this should expose required module-level
functions, allowed values, bounds, and solver invocation point.

### 6. Draft Patch

Produce a complete file-content patch proposal. The proposal agent may write
draft artifacts under a proposal-session artifact directory, but it must not
write the candidate workspace.

Current implementation note: after an approved hypothesis, APS now runs a
bounded code-phase inspection loop before the final `PatchProposal` call. This
loop can request allowed context tools such as selected-surface reads,
problem/objective context, champion summary, branch state, memory, and
screening/runtime feedback. The selected surface can be read at full
code-preview budget during this phase. The LLM still returns only tool plans;
APS executes tools and enforces exposure policy.

May 13 smoke evidence: a balance-restored 2-round CVRP run confirmed that this
loop works end to end for the full `solver_design` boundary. Code-phase traces
requested `context.read_surface(detail="full", max_code_chars=12000,
target_file="policies/solver_algorithm.py")`, patch generation completed,
Contract-preview repair passed, and both candidates reached screening. The
second round then consumed the first round's screening/runtime feedback and
changed from a fast low-quality replacement solver to a baseline-plus-ILS
solver with 1 win, 15 ties, and 0 losses. This is a governance and feedback
success, not yet a solver-quality success.

Follow-up implementation note: code-phase tool access is not enough if the
final patch prompt repeats full target code through both `Target File` and
tool-observation payloads. APS now compacts code-generation observations:
selected-surface reads keep audited metadata and artifact size fields, but
omit `current_artifact.content_preview` because the target file is already
provided once. Code-phase agentic JSON has a tighter cap than hypothesis
context, the planner stops after one successful full selected-surface read,
and model-facing planner specs filter the holdout-summary tool name so strict
sanitization cannot create empty allowed-tool entries. Deterministic callers
can still invoke the direct holdout-summary tool under policy.

### 7. Self Check

Run deterministic preview tools only:

- schema validation;
- target file and action permission preview;
- surface interface preview;
- AST syntax preview;
- import whitelist preview;
- static complexity guard preview where available.

The agent must not run `VerificationGate`, `ExperimentProtocol`, canary,
screening, validation, frozen, or final evidence as tools.

Current implementation note: a failed Contract preview is not immediately
materialized or sent to evaluation. APS can feed the compact preview failure
back into one bounded patch-regeneration attempt, then re-run Contract preview.
If the final preview still fails or is skipped, the session fails closed.

### 8. Finalize

Emit one `AgenticProposalOutput`. A completed session means "ready for normal
ContractGate", not "accepted".

Partial sessions may be stored as tainted proposal memory but cannot enter
evaluation unless converted into a valid hypothesis and patch.

## Output Schema

Recommended final output:

```text
AgenticProposalOutput
- status:
    completed
    partial_hypothesis_only
    partial_patch_unchecked
    failed
- session_id
- campaign_id
- branch_id
- champion_version
- champion_weight_revision
- problem_id
- problem_spec_hash
- selected_surface
- action
- hypothesis: HypothesisProposal?
- patch: PatchProposal?
- rationale_summary
- evidence_used:
    - observation_id
    - exposure_level
    - summary
- rejected_alternatives[]
- self_check:
    schema_valid: bool
    contract_preview_passed: bool | null
    contract_preview_codes: list[str]
- tainted_artifact_refs[]
- termination_reason
```

`rationale_summary`, `evidence_used`, `rejected_alternatives`, and
`tainted_artifact_refs` are for audit and future proposal guidance. They must not
be read by `SafeFeatureExtractor` or `DecisionEngine`.

## Tool Model

The proposal agent needs tools because algorithm design requires inspection and
iteration. Tools must be deterministic framework functions with input schemas,
permission checks, and typed observations.

Suggested tool groups:

### Context Tools

```text
context.read_problem
context.list_surfaces
context.read_solver_mechanics
context.read_objective_policy
context.read_champion_summary
context.read_surface
```

These expose adapter-rendered problem context, declared research surfaces, and
current champion surface artifacts.

### Memory And Feedback Tools

```text
memory.query
feedback.query_screening
feedback.query_runtime
feedback.query_holdout_summary
```

These expose tainted proposal/search memory and exposure-controlled evaluation
summaries.

Rules:

- screening can include detailed feedback when the protocol allows it;
- validation is aggregate-only;
- frozen is pass/fail, aggregate, and budget-only;
- validation/frozen raw metrics cannot be opened by tools.

### Drafting Tools

```text
proposal.draft_hypothesis
proposal.draft_patch
proposal.draft_policy_patch
proposal.draft_registry_change
proposal.finalize
```

Drafting tools produce tainted artifacts. They do not materialize candidates.

### Static Preview Tools

```text
proposal.schema_preview
proposal.target_permission_preview
proposal.interface_preview
proposal.contract_preview
```

Preview tools are allowed because they are deterministic and side-effect-free.
They are not substitutes for `ContractGate`.

## Permission Classes

Use explicit permission classes rather than a broad "agent can use tools" flag.

| Permission | Meaning |
|---|---|
| `read_public_context` | Read adapter/spec-rendered proposal context. |
| `read_tainted_memory` | Read proposal/search memory and allowed screening feedback. |
| `read_champion_artifact` | Read current champion surface files through surface metadata. |
| `contract_preview` | Run side-effect-free static previews. |
| `draft_patch` | Produce tainted hypothesis/patch artifacts. |
| `write_scratch` | Write only transcript, summaries, and draft artifacts. |
| `forbidden` | Never exposed to proposal sessions. |

Forbidden capabilities:

- direct candidate workspace writes;
- champion snapshot writes;
- registry writes outside materialization;
- lineage mutation;
- protocol/final evidence writes;
- verification/protocol/decision execution;
- validation/frozen raw metrics reads.

## Tool Observation Model

Every tool result should become a typed proposal observation:

```text
ProposalObservation
- observation_id
- session_id
- tool_name
- tool_call_id
- observation_type
- summary
- structured_payload
- artifact_ref?
- taint: proposal
- exposure_level:
    public_spec
    champion_code
    screening_detail
    validation_aggregate
    frozen_aggregate
    scratch
- created_at
```

Large outputs should be persisted as artifacts and summarized into context:

```text
full output -> artifact_ref + checksum
model context -> short preview + structured summary
```

No code path may exist from `ProposalObservation` to `DecisionFeatures`.

## Context Exposure Policy

Agent tools must enforce exposure policy in code, not only through prompt
instructions.

Suggested policy:

```text
ContextExposurePolicy
- allow_screening_case_detail: bool
- validation_exposure: none | aggregate
- frozen_exposure: none | pass_fail | aggregate
- allow_raw_metrics_refs: bool
- allow_raw_metrics_read: bool
- allow_champion_code_read: bool
- allow_candidate_workspace_read: bool
- allow_final_evidence_read: bool
- context_policy_id: str
```

Default v0.4 policy:

- screening detail allowed;
- validation aggregate only;
- frozen pass/fail and budget only;
- no validation/frozen raw metrics read;
- champion surface artifacts readable only through declared target files;
- no candidate workspace read except for session scratch artifacts.

## Proposal Memory

Proposal memory is tainted and useful only for future Creative Layer guidance.

Suggested record:

```text
ProposalMemoryEntry
- entry_id
- campaign_id
- branch_id?
- champion_version
- champion_weight_revision
- problem_id
- problem_spec_hash
- surface_schema_hash
- context_policy_id
- surface
- source_session_id
- entry_kind:
    surface_diagnosis
    failed_plan
    rejected_hypothesis
    draft_patch
    contract_preview
    runtime_feedback_summary
    compaction_summary
- text_summary
- structured_tags
- artifact_ref?
- exposure_level
- created_at
```

Proposal memory can influence future proposal prompts. It cannot influence
promotion except indirectly through a future candidate that passes the normal
gates and protocol.

## Failure Handling

Agent session failure should be typed and auditable, not an unstructured crash.

Minimum failure and termination reasons:

```text
completed
partial_hypothesis_only
partial_patch_unchecked
proposal_fail
schema_fail
permission_fail
context_exposure_fail
max_turns
max_tool_calls
max_wall_time
max_tokens
repeated_tool_loop
structured_output_failed
compaction_failed
stale_champion
campaign_abort
```

Campaign handling:

- no valid hypothesis: record proposal failure and route through failure
  lifecycle;
- valid hypothesis but no patch: store tainted partial memory, do not evaluate;
- patch produced but preview failed: allow a bounded repair loop or return a
  normal contract failure path;
- repeated failures on one surface: reduce that surface's proposal priority
  through tainted guidance, not through Decision.

Suggested circuit breakers:

- 3 identical tool calls without progress;
- 3 failed compactions;
- 5 malformed final-output attempts;
- session wall-time budget;
- session token budget.

## Compaction And Resume

The session should keep two histories:

```text
full transcript for audit
compact API-visible summary for model continuation
```

Compaction must preserve:

- active branch and champion version;
- selected research surface;
- current hypothesis draft;
- current patch draft;
- tool observations used;
- rejected alternatives;
- exposure denials;
- next step.

Compaction must not introduce validation/frozen per-case detail into a lower
exposure summary.

Resume is allowed only when these anchors still match:

```text
campaign_id
branch_id
champion_version
champion_weight_revision
problem_spec_hash
surface_schema_hash
context_policy_id
```

If champion or problem spec changed, the session becomes historical memory and
must re-orient before producing a new patch.

## CVRP Workflow Example

The current CVRP bottleneck is that most generated candidates are post-baseline
operators applied after a strong ALNS+VNS baseline. Agentic proposal should make
surface diagnosis explicit.

Recommended CVRP session flow:

```text
1. Read CVRP surfaces:
   route_local, route_pair, ruin_recreate, search_policy,
   future construction_policy, future neighborhood_portfolio.

2. Read recent surface feedback:
   accepted moves, no-op rate, runtime ratio, invalid outputs,
   policy load errors, stop reasons.

3. Decide whether another operator is justified.
   If post-baseline accepted_move_rate is near zero, prefer a main-algorithm
   surface such as policy, construction, or portfolio.

4. Inspect selected surface:
   required functions, allowed modes/components, bounds, target files,
   solver invocation point, runtime audit fields.

5. Draft hypothesis:
   expected behavior change, runtime cap, feasibility invariant,
   route-count comparability protection, and no-op fallback.

6. Draft patch proposal.

7. Run static preview.

8. Finalize structured proposal.
```

This changes the default CVRP behavior from:

```text
write another post-baseline local operator
```

to:

```text
surface diagnosis
-> choose meaningful algorithm lever
-> bounded patch proposal
```

## Framework Extension Points

Keep changes generic.

### Proposal Layer

- `AgenticProposalSession`
- `ProposalToolRegistry`
- `ProposalObservation`
- `AgenticProposalOutput`
- transcript and scratch artifact persistence
- `ProposalPipeline.generate_agentic_proposal()`

### Context Layer

- `ContextExposurePolicy`
- `ContextManager.build_proposal_agent_view()`
- adapter-rendered surface/interface views
- proposal memory query and compaction views

### Contract Layer

- side-effect-free contract preview entry point;
- surface-aware target/action/interface preview driven by
  `ProblemSpecV1.research_surfaces`.

### Evidence Layer

- `StepRecord.proposal_session_ref`;
- proposal-session artifact refs in summaries;
- proposal failure reason summaries;
- proposal memory refs for future Creative Layer prompts.

### Problem Package Layer

- adapter renders surface interface and solver invocation point;
- solver emits surface-loaded/executed/behavior-change runtime audit fields;
- problem package owns algorithm surface files and allowed components.

Avoid adding problem-specific logic to:

- `DecisionEngine`;
- `SafeFeatureExtractor`;
- `ExperimentProtocol`;
- `VerificationGate`;
- `CampaignLoop`;
- `BranchStepRunner`;
- `LineageRegistry`.

## Implementation Phases

### APS-0: Document And Freeze Boundaries

- Land this design.
- Treat `agentic-proposal-reference/` as supporting engineering reference.
- Keep current single-shot proposal path as compatibility fallback.

### APS-1: Session Skeleton

Status: implemented as the RS2-5 minimum slice.

- `AgenticProposalSession` is available behind an explicit
  `ProposalPipeline` opt-in path (`use_agentic_proposal` or injected session).
- The default single-shot proposal path remains unchanged.
- The skeleton wraps the existing two-step hypothesis/code generation path, or
  accepts injected deterministic outputs for tests.
- Final completed sessions return only existing `HypothesisProposal` and
  `PatchProposal` shapes; ContractGate, WorkspaceLifecycle, Verification,
  Protocol, and Decision remain unchanged.
- Transcript and final output artifacts can be persisted under a caller-provided
  proposal-session artifact directory.
- `StepRecord.proposal_session_ref` and campaign-summary refs are deferred to
  APS-3; session rationale and memory are not added to `DecisionFeatures`.

### APS-2: Tool Registry And Exposure Policy

Status: implemented as the read-only MVP slice. The registry now also carries
APS-3 side-effect-free drafting/static preview tools, but the APS-2 read-only
exposure tests remain the guardrail for context tools.

- `ProposalToolRegistry`, `ProposalToolContext`, `ProposalObservation`, and
  minimal permission/exposure enums live in
  `scion/scion/proposal/tools.py`.
- `ContextExposurePolicy` defaults to strict proposal exposure: screening detail
  and screening-derived runtime guidance are allowed, validation/frozen
  holdout rows are hidden, raw metrics refs are not exposed, and champion code
  is readable only through declared surface targets.
- Read-only tools cover `context.list_surfaces`, `context.read_problem`,
  `context.read_surface`, `context.read_objective_policy`,
  `context.read_champion_summary`, `memory.query`,
  `feedback.query_screening`, `feedback.query_holdout_summary`, and
  `feedback.query_runtime`.
- Tests prove validation/frozen raw metric refs are not returned by read-only
  tools, memory/champion summaries do not leak promotion path signals, tools do
  not write workspace files, and proposal observations do not enter
  `DecisionFeatures`.
- Verification/Protocol/Decision execution as tools and candidate-workspace
  writes remain forbidden.

### APS-3: Contract Preview And Failure Lifecycle

Status: implemented through the production-agent minimum path.
`AgenticProposalSession` now runs a configurable bounded tool loop when the
Creative Layer exposes a tool-selection interface, falls back deterministically
to the fixed read-only plan otherwise, writes replayable compact tool-step
records without raw metric or holdout details, and emits tainted session audit
events to lineage without feeding rationale, memory, or observations into
`DecisionFeatures`.

- `proposal.draft_hypothesis` and `proposal.draft_patch` return tainted typed
  draft artifacts through `ProposalObservation`; they do not materialize
  candidate workspaces.
- `proposal.schema_preview`, `proposal.target_permission_preview`,
  `proposal.interface_preview`, and `proposal.contract_preview` run
  side-effect-free static checks over in-memory hypothesis/patch content.
- Preview uses existing `ContractGate.validate_hypothesis()` and
  `ContractGate.validate_patch()` paths where practical, including surface
  action/target permissions and required-function interface checks.
- Preview observations are tainted, bounded, and do not enter
  `DecisionFeatures`.
- Proposal-session refs are now stored as compact references in step/summary
  artifacts. They include session/artifact/digest metadata, not transcript,
  observations, rationale, or memory.
- Failure-lifecycle integration remains typed and fail-closed: timeout,
  repeated tool-call fuse, malformed model tool selection, forbidden tools, and
  preview failures terminate or degrade through explicit session reasons rather
  than bypassing ContractGate.

### APS-4: CVRP Algorithm-Surface Workflow

- Use the session to choose among CVRP surfaces.
- Add workflow guidance to prefer policy/portfolio/construction surfaces when
  post-baseline operator evidence is no-op dominated.
- Keep CVRP-specific guidance inside CVRP adapter/spec rendering.

### APS-5: Compaction And Resume

Status: implemented as a production minimum slice.

- Versioned session artifacts carry schema version, session/request ids,
  idempotency key, termination reason, tool-loop config, tool-budget usage,
  transcript digest, and taint metadata.
- Replay validation checks schema support, required fields, step uniqueness,
  budget bounds, digest consistency, and raw-ref marker absence.
- Resume support produces a bounded sanitized context summary from prior
  artifacts; it does not re-execute tools or reuse patches automatically.
- `AgenticSessionStore` maintains a file-backed recovery index under the
  session artifact directory and supports lookup by session id, idempotency key,
  latest request, and session listing.
- `ProposalPipeline` uses the store to find prior valid artifacts with the same
  idempotency key and injects only sanitized resume context.
- CLI inspection can validate one agentic artifact and list session summaries
  from an artifact directory without printing transcript, observations,
  rationale, raw metrics refs, or workspace code.

Remaining production hardening:

- Replace the single-file JSON recovery index with a multi-writer-safe store if
  concurrent campaign processes write the same artifact directory.
- Add lower-level timeout enforcement around any LLM/tool call that can block
  longer than the APS wall-time budget.
- Decide whether completed sessions should ever reuse compact
  hypothesis/patch content directly; the current behavior intentionally uses
  resume context only so ContractGate remains the mandatory path.

## Acceptance Criteria

Framework:

- Existing single-shot proposal path remains available.
- Agentic session can produce valid `HypothesisProposal` and `PatchProposal`
  outputs.
- Agentic session cannot directly write candidate workspaces.
- Agentic tools cannot read validation/frozen raw metrics.
- Proposal transcript and memory are persisted as tainted artifacts.
- `DecisionFeatures` remains free of proposal text and agent memory.
- Contract preview catches common surface/action/interface errors before final
  output.

CVRP:

- Agentic proposal can choose `search_policy` instead of another operator when
  repeated operator no-op evidence is present.
- Future `construction_policy` and `neighborhood_portfolio` surfaces can be
  handled without changes to campaign core.
- Runtime audit proves selected surfaces loaded and executed.

Regression:

- Warehouse operator-surface proposal path continues to work.
- Adapter-backed problems remain problem-owned.
- No CVRP or warehouse logic is added to Decision, Protocol, or campaign loop.
