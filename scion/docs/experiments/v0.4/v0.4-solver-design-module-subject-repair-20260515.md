# Solver-Design Module Subject Repair

*Date: 2026-05-15*

## Reason

The previous direct `solver_design` path fixed a larger mismatch: Scion should
let the research agent study and modify the branch-owned algorithm, not call a
baseline and then add a small post-processing operator. But the implementation
still had one granularity problem: every candidate patch to the preferred
target had to regenerate the whole `policies/baseline_algorithm.py` file.

That pushed the code phase toward broad one-shot rewrites and made small
operator or scheduler changes look like full solver replacements. It also made
review harder because the meaningful change was buried inside a large
replacement file.

## Repair

The CVRP solver-design subject is now a branch-owned package:

- `policies/baseline_algorithm.py` remains the stable
  `solve(instance, rng, time_limit_sec, context)` entrypoint;
- algorithm internals live under `policies/baseline_modules/`:
  config, state, construction, destroy/repair, local search, acceptance, and
  scheduler modules;
- `problem-v1.yaml` declares `policies/baseline_modules/*.py` as editable
  `solver_design` targets with create/delete/modify allowed, while
  `policies/baseline_modules/__init__.py` is frozen;
- `context.read_surface` for `solver_design` returns bounded support-artifact
  previews for the branch-owned module package, so hypothesis and code phases
  can inspect the actual algorithm internals instead of only the tiny
  entrypoint;
- `solver.py` loads policy files as package modules when they live under
  `policies/`, so relative imports inside the solver-design package work in
  copied branch workspaces;
- C7 interface checks are deferred for solver-design support modules because
  only the package entrypoint must expose `solve(...)`;
- CVRP problem preview imports module patches safely and delegates full
  entrypoint validation to workspace smoke;
- `proposal.algorithm_smoke` now applies support-module patches in a temporary
  workspace and runs the stable entrypoint on the configured canary.

The original `vrp/` source tree remains untouched. Candidate changes happen
inside Scion branch snapshots and are still governed by Contract,
Verification, Protocol, and Decision.

## First Gate

Before longer validation, run a 1-2 round Sonnet smoke. The gate is not solver
quality yet. The gate is that APS can legally choose module targets, generated
module patches pass static preview, and canary algorithm smoke validates the
full branch-owned solver through `baseline_algorithm.py::solve(...)`.

## Multi-File Patch Repair

Repair direction:

- Keep the existing single-file `PatchProposal` fields for compatibility.
- Add optional `additional_changes`, each with complete `file_path`, `action`,
  and `code_content`.
- Contract validates the primary target against the approved hypothesis, then
  validates every additional file independently against editable/frozen paths,
  selected-surface ownership, interface/import/sensitive-API/randomness, and
  C9c complexity controls.
- Additional changes are allowed only through the selected research-surface
  boundary. For `solver_design`, this lets the agent create a focused module
  and modify `baseline_algorithm.py` or `baseline_modules/scheduler.py` to
  wire it into the executable algorithm.
- Workspace materialization applies every file change before Verification.
- `proposal.algorithm_smoke` materializes all file changes in the same tainted
  runtime workspace, then runs the stable `solver_design` entrypoint.
- Agentic artifacts omit all code bodies, including additional changes, while
  retaining path/action/body-size metadata for audit.

## Code-Context and Smoke-Feedback Repair

The next 6-round Sonnet smoke completed without framework crashes:

```text

```

Experiment details:

- both official screening candidates targeted
  `policies/baseline_modules/scheduler.py`;
- both had `total_pairs=4`, `attempted=4`, `valid=4`, `failed=0`,
  `candidate_failed=0`, and `champion_failed=0`;
- both tied champion objective on all tiny cases and were abandoned for
  `SCREENING_FAIL_WIN_RATE`;
- one hypothesis failed early because a scalar
  `novelty_signature.acceptance_strategy` exceeded the 120-character semantic
  identity limit;
- multiple code attempts failed in `proposal.algorithm_smoke` with
  `solver_algorithm_errors=1` before official screening.

The important framework finding was not the tied screening result. The failed
code attempts showed a remaining code-context mismatch: when the approved
target was a module such as `scheduler.py` or `local_search.py`, code phase
could read the target file but not the sibling object model. Candidates then
treated `solution.routes` as `list[list[int]]`, while the branch-owned package
actually uses `_Solution.routes` as `_Route` objects with `.customers`,
`.load`, `.cost`, `.insert(...)`, `.remove(...)`, `.recalculate()`, and
`_Solution.rebuild_index()`. Algorithm smoke correctly stopped these patches,
but its compact retry feedback only said `solver_algorithm_errors=1`, so the
repair model did not reliably see the concrete object-model error.

Repair:

- `context.read_surface` now returns bounded solver-design support artifacts
  even when the primary target is itself under `policies/baseline_modules/`.
  `state.py` and the stable entrypoint are prioritized before sibling modules.
- Each Python support artifact includes a compact `python_api_summary`, so the
  code agent can see `_Route`, `_Solution`, scheduler, construction, and local
  search signatures without needing a full sibling-file dump.
- Full `target_preview` reads now honor the requested section instead of
  returning the full surface prompt/config, keeping the tool result below the
  proposal observation cap while still exposing the target and support module
  APIs.
- Code-phase observations now preserve support-artifact previews and API
  summaries. Previously they retained only current-target metadata, so sibling
  context was stripped before the LLM saw it.
- Solver-design code prompts now distinguish entrypoint targets from
  `baseline_modules` targets: module targets should return the complete
  module contents integrated into the existing entrypoint, not invent a
  top-level `solve(...)`.
- The prompt explicitly states the `_Route/_Solution` object model and warns
  not to slice or overwrite `solution.routes` as customer-list routes.
- `proposal.algorithm_smoke` compact and minimal retry observations now retain
  runtime-smoke details: failing case, audit detail, stderr/stdout summary,
  `solver_algorithm_errors`, and compact `solver_algorithm_events`.
- Semantic-signature guidance now gives a compact solver-design
  `novelty_signature` example and reports overlong scalar fields as overlong
  scalar fields, not as array-shape errors.
