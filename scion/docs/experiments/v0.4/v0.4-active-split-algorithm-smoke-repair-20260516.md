## Diagnosis

The main framework issue was not branch abandonment. It was pre-screen
coverage.

`proposal.algorithm_smoke` built its smoke cases from `split_manifest.yaml`
and `seed_ledger.yaml` inside the runtime workspace or champion snapshot.
Those files were the CVRP tiny smoke split:

- `data/tiny_canary.json`
- `data/tiny_5.json`
- `data/tiny_6.json`

The actual campaign was launched with the CVRP formal split and seeds. As a
result, algorithm smoke could pass on tiny JSON cases while formal screening
later found larger-case failures. Two rounds were also too weak to validate
this repair class; the 6-round run showed failures that the 2-round smoke did
not.

Two repeated API mistakes also deserved earlier feedback:

- integration edits to `policies/baseline_algorithm.py` imported nonexistent
  scheduler entrypoints such as `solve`/`run` instead of using
  `_ALNSVNSSolver(...).solve(instance, rng)`;
- code called `context.nearest_neighbor(rng)`, but the API takes no arguments
  and returns a public `CvrpSolution`.

## Repair

Changes made:

- `ProposalPipeline` now passes the active campaign `split_manifest` and
  `seed_ledger` into `ProposalToolContext`.
- `proposal.algorithm_smoke` prefers those active objects over workspace YAML.
  The workspace YAML remains only as a fallback for minimal local tests.
- Algorithm smoke now runs one canary plus four deterministic, evenly spaced
  screening cases instead of canary plus the first two screening cases.
- Smoke case resolution also checks `SCION_PROBLEM_DATA_ROOT`, matching formal
  CVRP launches where CVRPLIB data lives outside the branch workspace.
- CVRP problem preview now fails early when `baseline_algorithm.py` imports
  scheduler `solve`/`run`/`main`, or when solver-design code calls
  `context.nearest_neighbor(...)` with arguments.
- Solver-design prompt guidance now states the same stable scheduler and
  context-helper API contracts.

This keeps Scion's boundary unchanged: candidate branches may modify the
branch-owned solver algorithm modules, but objective semantics, constraints,
parsing, seeds, protocol splits, and Decision remain controlled.
