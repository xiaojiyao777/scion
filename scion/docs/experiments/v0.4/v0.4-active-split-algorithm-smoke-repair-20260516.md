## Diagnosis

The main framework issue was not branch abandonment. It was pre-screen
coverage.

`proposal.algorithm_smoke` built its smoke cases from `split_manifest.yaml`
and `seed_ledger.yaml` inside the runtime workspace or champion snapshot.
Those files were the CVRP tiny smoke split:

- `data/tiny_canary.json`
- `data/tiny_5.json`
- `data/tiny_6.json`

The actual campaign was launched with the CVRP formal split and seeds. As a
result, algorithm smoke could pass on tiny JSON cases while formal screening
later found larger-case failures. Two rounds were also too weak to validate
this repair class; the 6-round run showed failures that the 2-round smoke did
not.

Two repeated API mistakes also deserved earlier feedback:

- integration edits to `policies/baseline_algorithm.py` imported nonexistent
  scheduler entrypoints such as `solve`/`run` instead of using
  `_ALNSVNSSolver(...).solve(instance, rng)`;
- code called `context.nearest_neighbor(rng)`, but the API takes no arguments
  and returns a public `CvrpSolution`.

## Repair

Changes made:

- `ProposalPipeline` now passes the active campaign `split_manifest` and
  `seed_ledger` into `ProposalToolContext`.
- `proposal.algorithm_smoke` prefers those active objects over workspace YAML.
  The workspace YAML remains only as a fallback for minimal local tests.
- Algorithm smoke now runs one canary plus four deterministic, evenly spaced
  screening cases instead of canary plus the first two screening cases.
- Smoke case resolution also checks `SCION_PROBLEM_DATA_ROOT`, matching formal
  CVRP launches where CVRPLIB data lives outside the branch workspace.
- CVRP problem preview now fails early when `baseline_algorithm.py` imports
  scheduler `solve`/`run`/`main`, or when solver-design code calls
  `context.nearest_neighbor(...)` with arguments.
- Solver-design prompt guidance now states the same stable scheduler and
  context-helper API contracts.

This keeps Scion's boundary unchanged: candidate branches may modify the
branch-owned solver algorithm modules, but objective semantics, constraints,
parsing, seeds, protocol splits, and Decision remain controlled.

## Agent Behavior Analysis

The run produced three completed code proposals plus three hypothesis-only APS
records paused before code approval. That is expected for the current
hypothesis-approval split, but it means the per-round trace count is roughly
double the number of code candidates.

Input size was not the bottleneck. The hypothesis calls received roughly
122k, 136k, and 144k input characters. The code calls were smaller but still
substantial: about 31k, 43k, and 38k input characters. The issue is not that
the algorithm object was absent from context; the issue is that the code phase
is still too one-shot and too weak at rejecting inert or behavior-erasing
patches.

Patch scale:

| Round | Target | Source change size | Behavior quality |
|---|---|---:|---|
| 1 | `acceptance.py` | 50 insertions, 5 deletions | Added reheat state and `notify_segment_end`, but scheduler never calls the new hook. The patch is effectively inert. |
| 2 | `local_search.py` | 87 insertions | Added and wired `_or_opt_star` into `_default_vns_operators`. This was a real integrated algorithm change, but it tied objective quality and slightly worsened runtime. |
| 3 | `construction.py` plus `baseline_algorithm.py` and `scheduler.py` | 169 insertions, 239 deletions | Rewrote scheduler, removed the ALNS loop, called `_vns` with the wrong signature inside a swallowed exception path, and produced zero search iterations/moves. It was much faster because it mostly stopped searching, not because it found a better algorithm. |

Tool use was only partially effective. The hypothesis phase used reasonable
tools: `context.read_surface`, `context.read_problem`,
`context.read_champion_summary`, `memory.query`, and screening/runtime feedback.
The code phase, however, selected only `context.read_surface` for the approved
target file before generating a complete patch. Contract preview and algorithm
smoke happened after generation, but the checks did not force an iterative
debug/repair loop when the candidate was behaviorally weak.

Follow-up framework repair:

1. C9e now treats newly added class methods as solver-design helpers, not only
   module-level functions. A method such as `notify_segment_end()` must be
   reachable from the active solver path or the patch fails Contract preview.
2. C9e now checks the stable runtime contract for scheduler/entrypoint
   integration edits instead of rejecting by complete-file line-count size.
   Multi-module solver-design patches may use `scheduler.py` or
   `baseline_algorithm.py` in `additional_changes` when they preserve
   `_ALNSVNSSolver(...).solve(instance, rng)` and do not add detached top-level
   scheduler `solve`/`run`/`main` entrypoints.
3. `proposal.algorithm_smoke` now checks active search telemetry. A candidate
   that claims or touches search-bearing solver code and records
   `solver_algorithm_search_iterations=0` plus
   `solver_algorithm_move_attempts=0` on every successful smoke case fails before
   official screening.
