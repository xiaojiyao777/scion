## Summary

The run is a control success, not a solver-quality success.

What improved:

- The forced `main_search_strategy` constraint held for all five rounds.
- All five candidates passed Contract, Verification, and canary.
- All five candidates reached screening and then failed deterministically at
  `SCREENING_FAIL_WIN_RATE`.
- Selected-surface runtime audit was complete: all required
  `main_search_strategy` runtime fields were present in all candidate-side
  pairs, with no active/evidence-contract errors.
- APS ran both phases in every round: one hypothesis/research session and one
  code/implementation session.
- Feedback, runtime, memory, surface, schema, and target-permission tools were
  used without observation-budget collapse.

What did not improve:

- No candidate passed screening; no validation/frozen/promotion/final-evidence
  path was reached.
- Phase-level improvement remained absent. Accepted component moves were still
  recovery-only rather than improvements over phase best.
- Candidate changes stayed in the problem-owned
  `policies/main_search_strategy.py` surface and mostly changed bounded
  schedules/parameters/component selection.
- The deeper issue exposed by this run is in Scion's proposal layer: APS tool
  observations were available to the session, but the final
  hypothesis/code-generation calls did not render those observations into the
  prompt context.

## Core Interpretation

This does not contradict the earlier claim that the v3 governance core is
sound. The Contract, Verification, Protocol, Decision, and safe-feature
boundary behaved correctly.

The newly exposed gap is narrower and more actionable: the Agentic Proposal
Session tool loop was not connected strongly enough to final generation.
`AgenticProposalSession` placed tool observations into context dictionaries,
but `CreativeLayer` did not render `agentic_tool_observations` into the final
hypothesis/code prompts. As a result, the planner/tool loop could gather
screening/runtime evidence, while the final LLM call that produced the
hypothesis or patch could still behave as if it only had ordinary surface text.

That is a Scion core proposal-layer problem, not a CVRP-specific prompt issue.
The fix must remain generic: render bounded screening-derived research
diagnosis and tool observations into final generation prompts without adding
CVRP/VRP semantics to core.
