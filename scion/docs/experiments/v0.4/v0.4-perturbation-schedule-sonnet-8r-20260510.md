## Summary

The run is a useful control and diagnosis run, not solver-quality evidence.

What worked:

- The campaign completed normally: 8 steps, 5 screened experiments, stop reason
  `max_rounds_exhausted`, champion remained `v1`, and frozen budget remained
  unused.
- Forced-surface control held for all hypotheses and code sessions:
  `modify/main_search_strategy -> policies/main_search_strategy.py`.
- APS retrieved selected-surface context, memory, and same-campaign
  screening/runtime feedback. R2 onward had non-empty feedback, and both
  hypothesis and code prompts rendered tool observations/diagnosis.
- The new `perturbation.schedule` lever was used: candidates tested
  `after_no_improvement`, `before_first_round`, `before_each_round`, and
  disabled perturbation.
- All five candidates that reached screening had complete selected-surface
  runtime audit: 16/16 candidate-side pairs observed all 55 required
  `main_search_strategy` fields.

What did not work:

- No candidate passed screening. Every screened candidate failed
  `SCREENING_FAIL_WIN_RATE`, with median delta `0.0`.
- Component-local accepted moves still did not refresh phase best. In all
  screened candidate pairs, `main_search_component_phase_delta_sum`,
  `main_search_component_phase_improvement_counts`, and
  `main_search_objective_delta_by_phase.improvement_loop` stayed zero.
- R4 failed hypothesis Contract because the generated hypothesis omitted a
  usable structured `novelty_signature`.
- R5 and R7 generated shallow-only strategies that passed proposal Contract but
  failed Verification at `V5_solution_consistency` because
  `main_search_deep_components_selected` was empty.
- Observation budget pressure remains. Several schema/contract preview
  observations exceeded the session observation budget, so preview findings
  were not consistently actionable.

No validation, frozen, promotion, or final-evidence signal was produced.
`formal_ready=false` because `final_evidence_refs` were missing.

## Interpretation

The Scion governance path behaved correctly:

- forced-surface constraints stayed proposal-side and fail-closed;
- proposal observations remained tainted and did not enter `DecisionFeatures`;
- Contract, Verification, Protocol, and Decision stayed separate;
- selected-surface runtime audit failed closed for missing required evidence;
- deterministic Decision abandoned all screened candidates for screening
  quality, not proposal text.

Design interpretation: this run should not be read as evidence that Scion core
can only rearrange components. The limitation is in the current CVRP research
object. `main_search_strategy` is an orchestration-level whole-algorithm
surface: it can choose and parameterize declared main-search components, but it
cannot design new ALNS/VNS/destroy-repair/route-pair operators unless those are
exposed as deeper problem-owned surfaces. That boundary follows the v3 model
better than giving the agent unrestricted access to the original solver, but it
sets a lower ceiling until the problem package exposes stronger mechanisms.

The run exposes three concrete follow-up issues.

First, CVRP `main_search_strategy` efficacy remains the blocker. Perturbation
timing gives the proposal agent a real lever, but no schedule produced nonzero
phase-best improvement or median-distance benefit.

Second, proposal validation and runtime evidence are misaligned for shallow-only
plans. If forced diagnostics require at least one deep component, that should
be enforced by schema/contract/preview before code generation. If shallow-only
plans are intended to be legal `main_search_strategy` hypotheses, the runtime
contract must encode an explicit valid no-deep-components state instead of
failing later with an empty required field.

Third, APS compactness still needs work. The sessions completed, but
schema/contract preview observations often exceeded the budget, and the agent
could not reliably use preview feedback to avoid R4/R5/R7 failures.

## Next Actions

Do not launch long CVRP validation.

Recommended repairs before the next forced smoke:

1. Add a hard proposal-time guard or problem-owned preview for
   `main_search_strategy` plans that omit all deep components during forced
   diagnostics, or explicitly redesign the runtime evidence contract to allow
   shallow-only strategies.
2. Make `novelty_signature` required and preview-visible enough that R4-style
   `{}` hypotheses are rejected with a compact, actionable message before the
   final hypothesis is accepted.
3. Compact schema/contract preview observations so fatal/nonfatal issues remain
   visible under the 48k APS observation budget.
4. Add proposal feedback reason tags for `MAIN_SEARCH_ZERO_PHASE_DELTA` and
   `RECOVERY_ONLY_ACCEPTED_MOVES` so later hypotheses stop optimizing accepted
   counts when phase-best movement remains zero.
5. Continue CVRP problem-package work on component semantics or controlled
   diagnostic cases that can produce positive
   `main_search_component_phase_delta_sum` before any long formal validation.
