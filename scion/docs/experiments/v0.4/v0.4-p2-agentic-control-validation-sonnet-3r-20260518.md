# v0.4 P2 Agentic Control Validation, Sonnet 3R, 2026-05-18

## Purpose

Validate the v0.4 P2 agentic proposal control-loop repairs against the v3
architecture boundary:

- Creative Layer remains tainted and only proposes.
- Contract, APS preview, algorithm smoke, Verification, Protocol, and Decision
  remain deterministic control layers.
- Solver-design candidates edit the branch-owned algorithm subject, not legacy
  component surfaces or post-process wrappers.
- Every real-cost run must be reviewed at trace level: hypothesis stage, code
  stage, every LLM call, every proposal-tool call/result, and branch/gate
  behavior.

## Code Repairs

This repair set addressed the control gaps exposed by the previous
solver-design APS smokes:

- Added stronger `expected_telemetry` schema/prompt guidance. Top-level keys are
  categories only: `activity`, `activation`, `effect`, and `budget`. Runtime
  paths such as `best_delta`, `improvement_counts`, or `phase_runtime` must live
  under those categories.
- Split more APS control logic out of `agentic_session.py` into focused modules:
  grounding, planner policy, and failure classification. The session class
  remains the orchestrator, but new helper logic should not be added back into
  the monolith.
- Reworked APS session-timeout handling. Wall-time timeout now records
  `session_timeout`/`agentic_budget_control` skip evidence instead of
  `runtime_exception/tool_error`.
- Added wall-time reserve checks before code repair attempts. If there is not
  enough time left to run mandatory Contract/smoke previews, APS fails closed
  using the latest preview failure instead of generating an unpreviewable patch.
- Routed APS control timeouts as deterministic framework-control failures, not
  as proposal streaks or provider infra.
- Routed `algorithm_smoke_failure` as candidate-scoped
  `agent_quality_blocked`, not as generic `PROPOSAL`. Smoke failures no longer
  trigger `infra_suspected`, circuit breaker, or cross-branch proposal streaks.
- Added step/status heartbeats around long campaign steps and failure lifecycle
  transitions.
- Updated `AGENT_ONBOARDING.md` to make trace-level experiment review mandatory
  before declaring a short run healthy.

## Remaining Work

P1:

- The code-phase planner still often reads the compatibility hook
  `policies/solver_algorithm.py`; it should prefer the active entrypoint
  `policies/baseline_algorithm.py` and the owning module/call graph.
- `create_new` code sessions can still read too many full support files. Prefer
  compact API/symbol reads for integration files and keep full reads focused on
  the approved target.
- Exact APS smoke failure summaries should be easier for the next hypothesis
  stage to reuse, rather than depending on empty runtime/screening feedback.

P2:

- Dashboard/status surfaces should expose `ALGORITHM_SMOKE_FAILURE` or the
  quality-block summary directly, even when final branch `failure_codes` are
  dominated by campaign-end reasons such as `MAX_ROUNDS_EXHAUSTED`.
- `agentic_session.py` is still too large. Continue extracting orchestration
  submodules without moving problem-specific semantics into Scion core.
