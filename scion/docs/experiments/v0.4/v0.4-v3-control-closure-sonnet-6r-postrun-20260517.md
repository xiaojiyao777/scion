## Executive Conclusion

This run is a useful framework-control validation but not solver-quality
progress.

The important positive signal is that successful candidates now modify the
branch-owned baseline algorithm modules directly. They are no longer merely
postprocessing a baseline solution. The two-stage APS flow, Contract preview,
algorithm smoke, formal screening, Decision, metrics, DB events, and trace
files are present and mostly coherent.

The important negative signal is that the agent still repeatedly proposes
hypotheses grounded in an incomplete or misleading understanding of the active
baseline algorithm. Rounds 1, 2, and 3 are not meaningful algorithm research
failures; they are mostly grounding failures. Round 4 exposes a framework bug
in multi-module patch validation. Rounds 5 and 6 are closer to valid algorithm
research attempts, but the mechanisms were too local or unhelpful to beat the
champion.

The next framework repair should not simply add more exposed component knobs.
Scion needs a stronger controlled algorithm-reading and grounding path:
hypothesis generation must understand the active solver call graph and the
specific implementation it claims to improve. Code generation must also be
allowed to reject or revise a stale hypothesis when the full target file shows
that the premise is false.

### 2. ALNS Stagnation Restart Candidate

Target:

- `policies/baseline_modules/scheduler.py`

Outcome:

- Contract passed.
- Algorithm smoke passed.
- Formal screening ran.
- `16/16` pairs tied.
- `win_rate=0.0`
- Decision: abandon.

Primary diagnosis:

This round absorbed prior feedback and targeted the ALNS scheduler, but the
hypothesis misdescribed `_AdaptiveWeights`.

The hypothesis claimed the destroy/repair operators were selected by
"uniform random-weighted selection throughout the run" and lacked late
exploitation. The actual `_AdaptiveWeights` implementation is already
adaptive:

- all weights start at `1.0`;
- `choose()` samples by current weights;
- `record()` accumulates scores/usages;
- `update()` applies `(1-r) * old + r * observed`;
- the scheduler records every ALNS attempt and updates at segment boundaries.

The proposed restart expands destroy intensity and resets adaptive weights
after stagnation. That is a plausible algorithmic idea, but the stated reason
was wrong, and resetting weights may discard the very learning signal the
hypothesis claimed was missing.

This is a grounding failure in both phases:

- hypothesis did not have or did not use the full `acceptance.py` and
  `scheduler.py` call-chain details;
- code phase did not invalidate the hypothesis even though more code was
  available.

### 3. `_or_opt_cross` Local Search Candidate

Target:

- `policies/baseline_modules/local_search.py`

Outcome:

- Contract passed.
- Algorithm smoke passed.
- Formal screening ran.
- `16/16` pairs tied.
- `win_rate=0.0`
- Decision: abandon.

Primary diagnosis:

The hypothesis claimed the baseline lacked cross-route Or-opt segment
relocation for segments of length 2 or 3. That is false.

The champion already has:

- `_default_vns_operators()` with `_or_opt_2` and `_or_opt_3`;
- `_or_opt(solution, seg_len, ...)` that iterates source routes and destination
  routes;
- `if dst_idx == src_idx: continue`;
- segment extraction from the source route;
- capacity check against the destination route;
- insertion into a different route;
- optional segment reversal for `seg_len > 1`.

The failure mode is similar to round 1 but clearer:

- hypothesis phase saw only compact `local_search.py` preview and API summary;
- the API summary listed `_or_opt_1`, `_or_opt_2`, `_or_opt_3`, but did not
  state that they are cross-route segment relocations;
- the agent guessed they were intra-route Or-opt;
- code phase later saw the full target file, but still implemented the stale
  hypothesis.

This exposes the need for a novelty/duplication gate. If the hypothesis claims
an absent capability, Scion should require evidence that the active target file
does not already implement an isomorphic mechanism.

### 4. Recombination Module Candidate

Target:

- `policies/baseline_modules/recombination.py`
- integration edit in `scheduler.py`

Outcome:

- Did not enter formal screening.
- First generated patch was structurally close to the intended multi-file
  change.
- Contract preview rejected the integration import.
- Retry later produced malformed `additional_changes`.

Primary diagnosis:

This was the most algorithmically interesting attempt in the run. It tried to
add a population/recombination primitive and wire it into the ALNS segment
boundary. That is closer to real algorithm research than another small local
operator.

The failure chain:

1. The first code output created:

   ```text
   policies/baseline_modules/recombination.py
   ```

   and added a scheduler integration import:

   ```python
   from .recombination import _ElitePool, _try_recombination, _MAX_CALLS
   ```

2. Contract preview rejected the scheduler `additional_changes` with:

   ```text
   additional_changes[0].C8_import_whitelist:
   non-whitelisted imports: ['recombination']
   ```

   This is a framework bug. A patch set that creates
   `policies/baseline_modules/recombination.py` should be allowed to import
   `.recombination` from another file in the same patch set, subject to the
   normal package-boundary checks.

3. During repair, a later model output serialized `additional_changes` as a
   string containing JSON instead of as a JSON array. Pydantic rejected it:

   ```text
   additional_changes must be a JSON array
   ```

   That is a model-output/schema problem, but the repair loop made it harder
   to diagnose by mixing it with the earlier import-whitelist failure.

Additional observations:

- code phase repeatedly called `context.read_surface`, triggering the repeated
  tool fuse;
- `context.read_branch_state` was skipped to preserve self-check budget;
- repair feedback told the model that `recombination` was not whitelisted, but
  did not make clear that the likely problem was patch-set-level import
  validation rather than algorithm code.

This should be fixed at the framework level before judging recombination as an
algorithm idea.

## Framework Design Compliance

The following parts behaved well:

- Two-stage APS flow is visible: hypothesis-only session, then code patch.
- Tool observations are traceable:
  `context.read_surface`, `feedback.query_screening`,
  `feedback.query_runtime`, `context.read_problem`, `memory.query`,
  `proposal.contract_preview`, and `proposal.algorithm_smoke`.
- Code phase includes the target file and branch-current integration context.
- Successful candidates modify branch-owned baseline algorithm modules rather
  than postprocessing a completed baseline solution.
- Contract, smoke, screening, Decision, DB events, metrics, archives, and LLM
  traces exist for audit.
- Round 6 demonstrates a useful code-repair loop: Contract caught an
  integration problem, the agent retried, and the repaired patch passed smoke.

The following parts still need repair:

- Hypothesis phase does not reliably expose the active algorithm call graph.
- Compact support previews and legacy/inactive surface text can mislead the
  agent about the current solver.
- Code phase does not have a clear duty to reject an approved hypothesis when
  full target code contradicts its premise.
- C8 import whitelist does not account for same-patch created modules.
- Retry observability conflates schema failures and prior Contract failures.
- Repeated `context.read_surface` calls can consume budget and skip
  `context.read_branch_state`.
- Novelty checks are too semantic/identity-based and not concrete enough to
  catch duplicate mechanisms such as existing cross-route Or-opt or regret
  repair.

## Required Repairs

### P0: Active Algorithm Grounding for Hypothesis

Add an explicit controlled view of the current active algorithm, for example:

- `context.read_solver_call_graph`
- `context.list_algorithm_files`
- `context.read_algorithm_file(path)`
- `context.read_active_solver_design`

For CVRP `solver_design`, the hypothesis phase should be able to inspect or
receive a precise summary of:

```text
baseline_algorithm.solve
  -> _ALNSVNSSolver.solve
  -> _initial_solution
       -> sweep / Clarke-Wright / capacity-balanced / nearest-neighbor fallback
       -> optional initial VNS
  -> ALNS destroy/repair loop
  -> adaptive destroy/repair weights
  -> embedded VNS
  -> acceptance / annealing / stop reason
```

This should be generated by Scion from active branch/champion files, not left
for the model to infer from 800-character previews.

### P0: Hypothesis Grounding Gate

Before accepting a hypothesis, require a short grounding block:

- active entrypoint;
- relevant current functions/classes;
- current mechanism behavior;
- claimed weakness;
- why the proposed change is not already implemented;
- target module and integration path.

If a hypothesis says "baseline only uses nearest-neighbor" while the active
call graph shows sweep/CW/capacity-balanced, the proposal should fail closed or
be repaired before code generation.

If a hypothesis says "there is no cross-route Or-opt 2/3" while
`local_search.py` contains `_or_opt_2/_or_opt_3` calling cross-route
`_or_opt()`, the proposal should be rejected as duplicate.

### P0: Patch-Set Import Graph Validation

Contract should validate imports against the whole patch set:

1. parse primary `code_content`;
2. parse all `additional_changes`;
3. build the candidate file graph;
4. allow relative imports of modules created in the same patch set when they
   remain inside editable solver-design paths;
5. then run normal import whitelist and boundary checks.

This would have allowed `from .recombination import ...` when
`recombination.py` was created by the same candidate.

### P1: Code-Phase Hypothesis Revalidation

Code phase should not be a blind implementation step. After full target-file
context is available, the model or a deterministic checker should be able to
flag:

- "the approved hypothesis contradicts current code";
- "the proposed missing capability already exists";
- "this patch only duplicates an existing mechanism";
- "the target file is not the right mechanism owner."

This is especially important because hypothesis context is intentionally
compressed.

### P1: Retry and Error Attribution

Separate retry causes in session output and campaign summaries:

- schema-output failure;
- Contract-preview boundary failure;
- static import/patch-graph failure;
- code-generation timeout;
- model repair failed;
- framework policy bug.

Do not let a schema error in a later retry erase the fact that the previous
patch was blocked by a framework-side import whitelist issue.

### P1: Tool Strategy Improvements

The current agent has controlled context tools, not a free-form filesystem
reader. That is appropriate for Scion, but the tool set is too coarse for
algorithm research.

Current relevant tools:

- `context.read_surface`
- `context.read_problem`
- `context.read_branch_state`
- `context.read_champion_summary`
- `context.read_objective_policy`

Needed additions or policy changes:

- allow hypothesis-stage full reads for specific algorithm files under the
  active surface;
- make `context.read_branch_state` harder to skip during code-phase
  integration;
- avoid repeated identical `context.read_surface` calls when a more specific
  file/call-graph tool is needed;
- mark inactive/legacy surfaces as such and remove them from the main
  solver-design evidence path unless explicitly requested.

## What This Run Does Not Prove

This run does not prove that the LLM cannot improve the CVRP algorithm. The
first three candidates were under-grounded, and the fourth was blocked by a
framework bug before evaluation.

It also does not prove that the current champion algorithm is saturated. Round
5 produced limited positive movement, and round 4 attempted a more meaningful
population/recombination direction that should have been screened.

The reliable conclusion is narrower:

Scion's current governance path is much closer to the v3 design than previous
postprocessing/component-policy experiments, but the proposal layer still lacks
enough controlled algorithm understanding to make consistently valid research
hypotheses.

## Recommended Next Step

Implement the grounding and patch-set validation repairs before another long
run:

1. Add active solver call-graph/file-read context for hypothesis.
2. Add hypothesis grounding and duplicate-capability gates.
3. Fix same-patch module import whitelist.
4. Improve retry error attribution.
5. Rerun a small monitored 2-3 round smoke.
6. Only after the small smoke shows correct hypothesis grounding and
   multi-module patch handling, launch another 6-8 round background validation.
