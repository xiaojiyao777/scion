## Purpose

This run checked whether algorithm runtime can act as a governed positive
optimization signal when the lexicographic CVRP objective ties. The preceding
8R smoke showed five valid `solver_design` branches tying champion objective
on tiny screening cases while running orders of magnitude faster, but Decision
still abandoned them on `SCREENING_FAIL_WIN_RATE`.

The repair adds a deterministic `*_PASS_RUNTIME_TIE_IMPROVEMENT` path:

- candidate has no runtime failures;
- lexicographic objective is non-regressive/tie;
- runtime evidence has enough paired samples;
- median candidate/champion runtime ratio is at or below
  `runtime.tie_speedup_ratio`;
- the candidate still proceeds through Scion's screening, validation, and
  frozen stages rather than bypassing protocol.

The same repair also clarifies CVRP solver-design time-budget units:
`context.remaining_time()` returns seconds, while
`context.remaining_time_ms()` returns milliseconds. Contract preview now
rejects preferred `policies/baseline_algorithm.py` patches that compare
second-valued `remaining_time()` to millisecond-derived variables.

## Follow-Up Fix

The live run also exposed a protocol-layer ordering bug: `ExperimentProtocol`
computed the gate before runtime stats were inserted into `EvalStats`. As a
result, `ProtocolResult.gate_outcome` still showed
`SCREENING_FAIL_WIN_RATE` while Decision correctly emitted
`SCREENING_PASS_RUNTIME_TIE_IMPROVEMENT`.

After the run, `ExperimentProtocol` was fixed to compute protocol gates after
runtime stats are attached. Unit coverage now checks that screening gates see
runtime tie-speedup evidence inside `run_experiment`, not only in direct
`gates.py` calls.

## Interpretation

Positive:

- The generated code targeted the branch algorithm body,
  `policies/baseline_algorithm.py`, not a post-baseline operator.
- Code LLM timeout control remained stable; no SDK retry storm recurred.
- Decision now treats compute time as a first-class optimization signal under
  objective non-regression and runtime-completeness constraints.
- The branch was not promoted from screening; it was only queued for
  validation, preserving the Scion protocol boundary.

Still unresolved:

- The default smoke split is too tiny to validate solver-quality movement.
  It is useful for governance mechanics, but quality validation should use
  controlled/formal CVRP cases with nontrivial objective headroom.
- The candidate that passed did so by tying tiny objectives very quickly; this
  is valid runtime evidence, not solver-quality evidence.
- Generated solver-design code can still contain semantic mistakes that
  synthetic preview catches only after one or more expensive code calls.
