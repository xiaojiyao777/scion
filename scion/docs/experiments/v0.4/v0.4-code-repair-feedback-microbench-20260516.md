## Interpretation

The failures are not evidence that Scion should return to component-policy
exposure. The agent is now allowed to edit the branch-owned solver-design
package, and the formal gate correctly rejected objective regressions. The
remaining issue is the code-stage feedback loop:

- Schema brittleness turned a recoverable `additional_changes` formatting
  mistake into a proposal failure.
- C9e integration failures were correct, but the message did not tell the
  code agent which helpers were inert, which files changed, or which roots were
  recognized.
- Algorithm smoke caught real object-model mistakes, but repair prompts did
  not preserve enough concrete guidance.
- Screening feedback after repeated `solver_design` win-rate-zero candidates
  needed to force a materially different algorithm-body hypothesis rather than
  another shallow scheduler/budget variant.
- Candidates could pass canary runtime audit while being obviously worse than
  the current champion on the same canary case. Official promotion must remain
  protocol-owned, but code-stage repair needs a tainted early warning before
  expensive formal screening.

## Repair

The repair keeps Scion's hard boundary intact: objective semantics,
constraints, parser behavior, protocol splits, seeds, and Decision thresholds
remain adapter/protocol owned. The new evidence is proposal-stage debug
evidence only.

Changes:

- `PatchProposalInput.additional_changes` now accepts a JSON string that
  decodes to an array, while still rejecting malformed strings or non-array
  JSON.
- Solver-design code prompts explicitly require `additional_changes` to be a
  JSON array of objects, not a string containing JSON text.
- C9e now reports inert helper names, changed solver-design files, recognized
  integration roots, and file-local inert helpers. Its guidance tells the agent
  to integrate helper-only module changes through `additional_changes` edits to
  the scheduler or stable entrypoint.
- `proposal.algorithm_smoke` now runs a champion copy on the same smoke case
  after the candidate passes runtime audit. This tainted micro-benchmark is
  non-promotional and cannot promote a candidate. It only fails code-stage
  repair when every comparable smoke case is an objective loss against the
  current champion.
- Algorithm-smoke failure payloads now carry bounded `repair_guidance` for
  common branch solver object-model mistakes, including `_Solution.instance`,
  `_Solution.routes` containing `_Route` objects, and integer node ids for
  `CvrpInstance.distance(i, j)`.
- Algorithm-smoke preview gets a separate 36 second timeout from Contract
  preview's 12 second timeout, because it now also runs a champion canary.
- Compact preview feedback preserves longer Contract/C9e/runtime details,
  repair guidance, and the micro-benchmark summary for the bounded code repair
  attempt.
- Code-stage preview repair now supports up to two bounded repair attempts in
  one APS session. The repair loop re-runs Contract preview after each repair
  and can therefore handle sequential issues such as "fix inert helper" followed
  by "remove newly introduced `instance.name`".
- Solver-design code constraints now explicitly forbid adding new
  `instance.name`, `getattr(instance, 'name')`, or `hasattr(instance, 'name')`
  uses, including inside error messages. Generic error text is required because
  case identity remains outside the research boundary.
- Proposal context now emits solver-design plateau guidance after repeated
  recent `solver_design` screening failures with `win_rate=0`: the next
  hypothesis must name the failed pattern and propose a materially different
  algorithm body or explicit entrypoint integration.
