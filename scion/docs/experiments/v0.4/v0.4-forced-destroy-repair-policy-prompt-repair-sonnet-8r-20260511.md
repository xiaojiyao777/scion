The experiment also exposed a problem-package semantic gap. The policy surface
allowed `destroy_selectors` and `repair_selectors`, and runtime audit recorded
the generated values, but the solver's bounded destroy/repair implementation
still used fixed worst-removal ranking and regret-2 repair. As a result,
generated attempts to switch to `route_diverse_worst` or `cheapest` were mostly
surface-level configuration changes, not actual repair mechanism changes.

## Follow-Up Repair

Implemented after this analysis:

- `bounded_destroy_repair` now honors `destroy_selectors` by dispatching
  `worst_removal` versus `route_diverse_worst` removal ranking.
- `bounded_destroy_repair` now honors `repair_selectors`; `cheapest` performs
  low-budget cheapest insertion instead of always using regret-2 scoring.
- Regression coverage asserts route-diverse ranking changes the selected
  removal route order and that cheapest repair reinserts more customers under a
  tight repair budget than regret-2.
