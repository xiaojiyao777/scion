## Summary

Only three of the six candidates reached official screening. The screened
candidates were controlled, valid solver-design algorithm edits: all had 16/16
valid screening pairs and zero candidate runtime failures. They were correctly
abandoned by `SCREENING_FAIL_WIN_RATE` because none beat the champion.

The other three rounds failed before screening:

- rounds 1 and 6 were blocked by `C9e_solver_design_integration`;
- round 2 was blocked by generated Python code with a repeated keyword
  argument.

This is a framework-control finding, not evidence that the direct
`solver_design` direction is wrong. The agent stayed on branch-owned
solver-design modules, consumed feedback, and escalated from small scheduler
tweaks to local-search changes and then a larger PBIG-style solver restructure.

## Why Only Three Candidates Reached Screening

Round 1 added construction/or-opt helper logic in
`policies/baseline_modules/scheduler.py`. The new helpers were called from
`_ALNSVNSSolver` methods, but C9e only considered module-level entrypoints and
existing module functions. It ignored solver class methods, so the candidate
was falsely rejected before workspace materialization.

Round 2 retried the same direction. Contract first caught a legitimate
case-specific identity issue, then the generated code produced
`attempted=1, attempted=0` in one call. `ast.parse(...)` accepts that shape, but
Python compilation rejects it. The rejection was correct, but it should be
surfaced by C6 syntax/compile checking rather than later import/preview logic.

Round 6 attempted the most interesting architecture change: a
population-based iterated-greedy style solver with helper functions and a
runtime `_ALNSVNSSolver = _PBIGSolver` alias. Those helpers were called from
`_PBIGSolver.solve(...)`, but C9e again ignored class-method reachability and
falsely rejected the candidate before screening.

## Screened Candidate Behavior

Round 3 changed scheduler acceptance into a two-phase simulated-annealing
schedule. It passed Contract, algorithm smoke, Verification, and screening.
Result: 16/16 valid pairs, `win_rate=0.0`, `median_delta=0.0`, slight runtime
regression. Abandoning it was reasonable.

Round 4 used screening/runtime feedback and saw that VNS contributed much more
than ALNS. It reallocated budget toward VNS and added construction ensemble
logic. Result: 16/16 valid pairs, `win_rate=0.0`, `median_delta=-4.5`, slightly
faster median runtime. This was a reasonable feedback-driven attempt, but it
reduced quality.

Round 5 pivoted from scheduler logic to `local_search.py`, adding per-operator
stagnation counters for VNS. Result: 16/16 valid pairs, `win_rate=0.0`,
`median_delta=-10.25`, median runtime about 17.8% slower. The agent's target
pivot was reasonable, but the mechanism increased work without producing
objective improvement.

## Interpretation

The Scion agent behavior was materially better than the earlier
post-processing-operator pattern. It modified branch-owned algorithm modules
under `solver_design`, used prior feedback, and eventually tried a larger
solver restructure.

The weak part was not screening or branch abandonment. The three screened
candidates had valid evidence and were correctly rejected. The weak part was
pre-screening control:

- C9e was too narrow for class-based solver algorithms.
- C6 did not compile parsed code, so compile-time syntax errors surfaced too
  late.

## Repair

Implemented repair:

- C6 now parses and compiles patch code, so compile-time `SyntaxError` cases
  such as repeated keyword arguments fail in `C6_ast_syntax`.
- C9e now treats solver class `solve(...)` call chains as integration roots.
  It recognizes helpers reached from existing runtime solver classes and from
  runtime aliases such as `_ALNSVNSSolver = _PBIGSolver`.
- Detached classes still do not satisfy C9e; helper functions must be reachable
  from the actual algorithm path.
- C9e was extracted from `contract/gate.py` into
  `contract/checks/solver_design_integration.py` so future solver-design
  boundary fixes can be made and reviewed in a smaller module.

Regression coverage:

- repeated-keyword compile failure is caught by C6;
- helper called from a solver class method passes C9e;
- helper called from a runtime class alias passes C9e;
- helper called only from a detached unused class still fails C9e.
