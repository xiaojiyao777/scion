## Retry Finding

The first Sonnet code-generation request timed out after two Scion-visible
attempts. With provider SDK retries disabled, there was no hidden retry
multiplication. APS then performed one compact semantic timeout retry, and the
retry returned a valid patch.

This confirms the intended retry layering:

- provider SDK retry layer: disabled by default (`SCION_SDK_MAX_RETRIES=0`);
- Scion LLM retry layer: visible in trace files (`SCION_LLM_MAX_RETRIES=1` for
  this smoke);
- APS compact timeout retry: one semantic retry that changes code-generation
  scope instead of repeating the same request.

## Interpretation

This is a framework-stability success, not a solver-quality success. The
candidate was a controlled branch algorithm-body change and made it through
Contract, Verification, and formal screening. It was correctly rejected by
Decision because it was weaker than the champion.

The important validated behavior is that Scion now evaluates branch-owned
algorithm changes in `policies/baseline_algorithm.py`, rather than treating the
original CVRP solver as a reference object and asking the agent to write
post-baseline cleanup code.
