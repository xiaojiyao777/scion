## Trace Finding

The experiment exposed an APS architecture gap. The hypothesis/planning stage
used proposal tools such as `context.read_surface`, `context.read_problem`,
`memory.query`, `feedback.query_screening`, and `feedback.query_runtime`.

The code stage did not. Code traces were single `generate_patch` calls with
large static prompts, growing from roughly 49k to 58k characters. They received
the observations gathered earlier, but could not interactively query memory,
branch state, runtime feedback, or the full problem object while drafting code.
The later timeout is a direct symptom of this static prompt pressure.

## Interpretation

The failure is not explained by tool access alone.

The framework-side failure was that Scion treated the second round as a
single-shot structured code generator. That is too weak for whole-algorithm
research. Scion should control boundaries, permissions, taint, audit trail, and
protocol state; the research core still needs to be an agent that can inspect
the problem object, history, and feedback while implementing the algorithm.

The algorithm-side failure was also real. The first candidate was too far below
champion despite being fast. The later candidates leaned heavily on the
repo-local baseline and only had limited post-baseline budget, so they mostly
tied the champion rather than escaping its local optimum. Runtime is part of
the optimization target: spending most of the budget on baseline leaves too
little room for a genuinely different search body, but from-scratch search
quality is currently too weak to replace it.

## Repair

The APS repair adds a code-phase tool loop after ContractGate-approved
hypothesis context and before final patch generation:

- code phase can call bounded proposal tools such as `context.read_surface`,
  `context.read_problem`, `context.read_objective_policy`,
  `context.read_champion_summary`, `context.read_branch_state`,
  `memory.query`, `feedback.query_screening`, and `feedback.query_runtime`;
- code phase receives a full selected-surface read within budget instead of
  relying only on an 800-character compact preview;
- final patch generation still emits a normal `PatchProposal`;
- deterministic Contract preview remains the gate before workspace
  materialization;
- a failed Contract preview is fed back into one bounded patch-regeneration
  attempt before the session fails closed.

The repair keeps validation/frozen/raw metrics hidden and does not allow the
proposal agent to mutate workspaces, promote, abandon, or change objective and
constraint semantics.
