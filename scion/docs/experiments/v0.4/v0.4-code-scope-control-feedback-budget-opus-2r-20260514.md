## Root Cause

The failure was not a final code-generation timeout and not an argument for
returning to component-policy exposure.

The root cause was APS observation-budget accounting. Screening/runtime
feedback was stored in the session as large structured payloads. In the second
round, the hypothesis session used `63232/64000` observation characters, and
the approved-hypothesis code session used `63831/64000` before Contract
preview. Code phase correctly skipped optional reads to reserve self-check
budget, but only about 169 observation characters remained, so even the compact
Contract preview result was replaced with `result_too_large`.

This showed that Scion had the right conceptual boundary, but the APS evidence
channel was too expensive for repeated whole-solver learning. The useful
feedback fields are counts, recent screening summaries, reason codes,
runtime-signal rows, and movement telemetry; raw-sized feedback payloads should
not consume the same session budget as terminal self-check evidence.

## Repair

The follow-up framework repair changes APS control, not CVRP objective or
solver semantics:

- `feedback.query_screening` and `feedback.query_runtime` observations are
  compacted before they are charged to the APS observation budget.
- The compact feedback preserves screening rows, reason codes, stats,
  runtime attribution highlights, research diagnosis, and next-hypothesis
  requirements while dropping bulky case feedback and raw-sized value lists.
- Diagnosis loops stop feedback pulls earlier when the remaining observation
  budget is needed for schema/target/Contract preview.
- Code-phase self-check reservation now uses the full self-check observation
  reserve, not only the smaller code-phase floor.
- A successful feedback observation can satisfy repeated planner requests in
  the same session instead of re-querying the same large channel.
- Solver-design code prompts now require a compact complete algorithm slice:
  one construction/seeding path, one bounded improvement loop, no more than
  two move families, and a hard target around 180 lines/six helpers.

Unit coverage now exercises the APS feedback compaction path and confirms that
compact screening/runtime observations stay under the internal budget while
retaining `solver_design` evidence and research diagnosis.

The APS repair worked as intended. Both rounds completed code sessions, both
retained terminal Contract-preview evidence, and no APS session failed with
`result_too_large`. Round 2 also exercised the repair path: its first patch was
rejected by Contract preview for `C9c_complexity_bound`, the session generated
a repair patch, and the second preview passed.

The solver-quality result remained weak:

```text
round 1: solver_design, Contract/Verification passed, win_rate=0.0, median_delta=0.0
round 2: solver_design, Contract/Verification passed, win_rate=0.125, median_delta=0.0
decision: abandon both branches, SCREENING_FAIL_WIN_RATE
```

The important finding is not just that these candidates failed. The first
round exposed a deeper research-object mismatch. In the hypothesis phase, the
agent saw the CVRP problem object, the `solver_design` interface, the inactive
`policies/solver_algorithm.py` hook, and the existence of
`context.baseline(...)`, but it did not study the repo-local ALNS+VNS baseline
algorithm as the algorithm under research. The hypothesis therefore became a
generic "activate solver_design, call baseline for a seed, then add local
search" plan. In code phase, the agent reread the target file and produced a
simple post-baseline ILS/relabel/swap/2-opt style solver.

That is not the intended research loop. Scion's core purpose is to let an
agent study and improve the algorithm under boundary, protocol, and audit
control. For CVRP, the agent should be able to study the baseline algorithmic
body, propose an algorithmic change to that body, implement it in a controlled
candidate branch, and then let Contract/Verification/Protocol decide whether
the new algorithm becomes champion. If promoted, that candidate should become
the new baseline/champion algorithm. The current `solver_design` behavior
still encourages "baseline as oracle plus post-processing code", which can be
worse than the original baseline's full ALNS followed by VNS.

Two repair implications follow:

- Hypothesis-stage context must be strong enough to support algorithm research,
  not just surface selection. It should expose the relevant baseline algorithm
  structure and evidence at the level needed to form a concrete algorithmic
  hypothesis.
- Code-stage output should implement the approved hypothesis as part of the
  full candidate algorithm path, not as a shallow wrapper around
  `context.baseline(...)` plus small cleanup operators. A baseline helper can
  remain available, but it must not be the main research object.

This does not invalidate the APS feedback-budget repair. It validates the
framework evidence path while showing that the next optimization target is the
CVRP research-object adapter: expose the algorithm body that actually matters
to the research agent, while still keeping original `vrp/` files frozen and
evaluating candidate changes only inside Scion-controlled branches.

## Follow-up Repair Direction

The implementation now changes the CVRP research object rather than adding
another component surface:

- `solver_design` prefers `policies/baseline_algorithm.py`, a
  Scion-controlled inactive-by-default ALNS/VNS-style algorithm subject. The
  original `vrp/` files remain frozen.
- Runtime tries `baseline_algorithm.py` first and falls back to the older
  `policies/solver_algorithm.py` hook only for compatibility.
- CVRP problem-owned preview rejects preferred-target candidates that call
  `context.baseline(...)`; they must modify the controlled algorithm body
  itself.
- APS code self-check now runs `proposal.algorithm_smoke` after static
  Contract preview passes. The smoke is tainted and non-promotional, calls the
  candidate `solve(...)` on synthetic CVRP instances, and can feed one bounded
  repair attempt before official evaluation.
- Final evidence and promotion remain unchanged: Contract, Verification,
  Protocol, and Decision are still the only official gates.

The next experiment should be intentionally short. The first question is not
promotion quality; it is whether new candidates actually target
`baseline_algorithm.py`, pass Contract plus algorithm smoke, and enter
Verification as controlled algorithm-body changes rather than baseline-wrapper
post-processors.
