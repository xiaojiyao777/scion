## Summary

The repair worked for Scion's proposal loop, but did not produce
solver-quality evidence.

What improved:

- All five hypothesis traces contained `Agentic Research Diagnosis` and
  `Agentic Proposal Tool Observations`.
- All five code traces contained `Evidence Diagnosis Behind This Hypothesis`
  and `Agentic Proposal Tool Observations`.
- Later hypotheses explicitly used prior screening/runtime signals. R2/R3
  reacted to zero phase-best improvement; R4/R5 cited recovery-only deltas.
- All five rounds stayed on forced `main_search_strategy`, targeted
  `policies/main_search_strategy.py`, passed Contract/Verification/canary, and
  reached screening.

What did not improve:

- All candidates still failed deterministic screening/Decision gates.
- Accepted component deltas remained recovery-only. Across successful
  candidate-side pairs, `main_search_component_phase_delta_sum`,
  `main_search_component_phase_improvement_counts`, and
  `main_search_objective_delta_by_phase.improvement_loop` were zero.
- All generated patches only changed static `main_search_plan()` configuration
  in the CVRP problem package. No Scion core, solver implementation, operator
  implementation, registry portfolio, or weight logic was changed.
- R4/R5 claimed a perturbation-first mechanism, but the existing surface could
  not express that mechanism. It could only set perturbation enabled/strength/
  count; solver perturbation still occurred after a failed improvement round.
- R4/R5 also dropped `bounded_destroy_repair` and introduced candidate solver
  timeouts.

## APS Behavior

The run validates the previous core fix:

- APS tool observations reached final generation, not only the planner.
- The agent changed hypotheses based on feedback rather than repeating surface
  text.
- The v3 governance boundary remained intact: APS artifacts were tainted
  proposal context, while Contract/Verification/Protocol/Decision controlled
  candidate outcomes.

Remaining APS issue:

- `max_tool_calls=9` was too tight. Later code sessions often skipped target
  or contract preview after reading context/feedback/schema.
- R4/R5 had useful observations, but the structured `latest_runtime_diagnosis`
  wrapper could be empty, forcing the model to mine larger observations or
  memory instead of reading a stable summary.

## Interpretation

The current blocker is no longer "Scion cannot feed APS evidence into final
generation." That is repaired.

The blocker is now two-part:

1. Generic APS diagnosis should remain structured and preview-safe under later
   rounds. Empty latest diagnoses should not mask useful prior runtime signal,
   and tool budget should leave room for target/contract preview.
2. The CVRP `main_search_strategy` surface must expose the mechanism the agent
   is trying to study. If perturbation timing is a real hypothesis dimension,
   it needs a declared plan field and runtime evidence, not only prose.

Follow-up implementation addresses both: APS diagnosis aggregates the latest
non-empty runtime signal and the default tool-call budget is increased; CVRP
`main_search_strategy` adds `perturbation.schedule` with
`after_no_improvement`, `before_first_round`, and `before_each_round`, plus
runtime audit evidence for the selected schedule.
