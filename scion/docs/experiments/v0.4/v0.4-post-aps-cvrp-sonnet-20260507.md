## Policy Surface Findings

The core post-run question was whether the policy-surface, APS compactness, and
V8 auditability repairs moved CVRP policy surfaces through governed evaluation.

Policy surfaces now clear Contract and Verification in this smoke:

- `construction_policy`, `neighborhood_portfolio`, and `search_policy` all had
  `contract_passed=true` and `verification_passed=true`.
- Delegated DB inspection found V1-V9 passing for these policy candidates.
- V8 metadata recorded `adapter_backed=true`,
  `comparison_mode=adapter_canonical_signature`, and
  `comparison_equal=true` for the selected policy surfaces.
- The earlier `instance.customers` failure mode did not recur; generated policy
  code used the repaired safe instance API, and raw metrics reported
  `policy_errors=0`, `construction_errors=0`, and `portfolio_errors=0`.

Policy surfaces did not yet produce enough objective signal:

- `construction_policy` reached screening with 16/16 valid pairs and switched
  construction modes, but all objective comparisons tied.
- `neighborhood_portfolio` reached screening with 16/16 valid pairs, but all
  objective comparisons tied. The candidate had no generated registry
  operators to schedule, so the portfolio surface was mostly unexercised and
  stopped as `no_registry_operators`.
- `search_policy` produced the strongest nontrivial signal: 16/16 valid pairs,
  runtime median ratio about `0.842`, median runtime delta about `-1312.5ms`,
  and mixed objective evidence of 2 wins, 1 loss, and 13 ties. It still failed
  the screening win-rate gate.

The interpretation is therefore:

```text
policy surfaces now clear Verification: yes, for the tested policy surfaces
policy surfaces produce nontrivial screening signal: partially, but still weak
```

## Evidence Boundary

This run maintained the expected holdout boundary: validation and frozen stages
were never reached, frozen budget remained unused, and no holdout detail was
observed in APS artifacts.

Raw metrics refs appear in campaign summary and lineage artifacts as evidence
references. Those refs are appropriate for docs and audits, but raw pair
payloads should remain out of proposal context and summary prose.

One delegated audit found a naming ambiguity: SQLite `experiment_events`
`event_kind=experiment` rows have a `decision_features_json` column that
contains raw metrics refs, while `event_kind=decision` rows did not. If those
experiment rows are post-decision evidence envelopes, the behavior is
acceptable but the naming is confusing. If that JSON is ever passed into
DecisionEngine input, it should be treated as a raw-ref contamination bug.

## Conclusions

- The run completed cleanly and exercised the real CVRP formal-path campaign
  stack after the V2-V8, policy API, APS compactness, and V8 metadata repairs.
- Verification is no longer the blocking point for the tested policy surfaces.
- Screening still rejects all candidates on quality, not infrastructure.
- Search policy has the first useful runtime signal, but objective evidence is
  still below the screening gate.
- Neighborhood portfolio needs either generated registry operators or a
  different invocation design before its efficacy can be judged.
- The next CVRP work should focus on policy-surface efficacy and APS surface
  context compactness before adding more post-baseline route operators.

## Residual Uncertainty

- The round-1 rejected patch body was omitted from compact artifacts, so the
  exact code line for the uncapped loop was not independently rechecked by the
  delegated audit.
- Screening raw pair runtime fields appear less complete than verification/V8
  artifacts for some policy surfaces. This may be protocol serialization
  pruning or a required-runtime-field coverage gap outside verification/canary.
- The SQLite `decision_features_json` column name should be reviewed so audit
  refs are not mistaken for actual DecisionEngine input.
