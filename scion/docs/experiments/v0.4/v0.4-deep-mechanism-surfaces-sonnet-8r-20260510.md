The run is useful governance and surface-discovery evidence, but still not
solver-quality evidence.

What worked:

- The campaign completed normally: 8 steps, 7 screened experiments, stop
  reason `max_rounds_exhausted`, champion remained `v1`, and frozen budget was
  unused.
- All generated code candidates passed Contract and Verification; no
  Verification failure breakdown was recorded.
- APS did not require a forced surface to run end-to-end. It selected several
  declared surfaces and reached screening on `main_search_strategy`,
  `baseline_policy`, `route_local`, `algorithm_blueprint`,
  `construction_policy`, and `alns_vns_policy`.
- The newly exposed `alns_vns_policy` surface reached code generation,
  Verification, screening, and selected-surface runtime audit in R8.

  that reached screening; candidate-side runtime had 16/16 observed pairs for
  `main_search_strategy`, `baseline_policy`, `algorithm_blueprint`,
  `construction_policy`, and `alns_vns_policy`.

  `SCREENING_FAIL_WIN_RATE`, with median delta `0.0`.
- The free-surface run mostly revisited older surfaces. Only one of the four
  newly added deep mechanism surfaces was selected: `alns_vns_policy` in R8.
  `destroy_repair_policy`, `route_pair_candidate_policy`, and
  `acceptance_restart_policy` were not selected.
- R1 failed hypothesis Contract at `C10_novelty` because the
  `main_search_strategy` hypothesis omitted usable structured
  `novelty_signature` fields.
- `main_search_strategy` candidates still showed recovery-only accepted moves:
  route-pair moves were accepted, but `main_search_component_phase_delta_sum`,
  `main_search_component_phase_improvement_counts`, and
  `main_search_objective_delta_by_phase.improvement_loop` stayed zero.
- `bounded_destroy_repair` never accepted a move in the two screened
  `main_search_strategy` candidates.
- R8 activated ALNS/VNS policy controls and produced some seed-level wins, but
  `alns_vns_phase_delta_sum` was zero for all 16 pairs. The surface is active,
  but its current attribution fields do not yet show phase-level improvement.
- Observation-budget pressure remains under the 48k default. Most APS sessions
  still hit about 46k-48k observed characters; many tool observations exceeded
  budget, and every completed code session skipped Contract preview because the
  tool loop limit was reached.

No validation, frozen, promotion, or final-evidence signal was produced.
`formal_ready=false` because `final_evidence_refs` were missing.

## Interpretation

The Scion governance path behaved correctly:

- proposal observations remained tainted and did not directly decide outcomes;
- Contract stopped an invalid structured-identity hypothesis before code;
- Verification accepted only candidates that satisfied the declared code and
  runtime contracts;
- Protocol produced deterministic screening metrics;
- Decision abandoned every screened candidate for screening quality, not for
  proposal text.

The main finding is that the next bottleneck is now surface-directed
optimization and runtime attribution, not basic deep-surface plumbing.

`force_surface=none` allowed APS to discover the expanded surface set, but it
mostly selected familiar orchestration or policy surfaces. The run therefore
does not yet validate the full deep mechanism family. It validates only that
`alns_vns_policy` can be selected, patched, audited, and screened.

For efficacy, the result remains negative. `main_search_strategy` still turns
some local accepted moves into recovery-only movement, not phase-best movement.
`algorithm_blueprint` and construction changes were inert at the screening
level. `alns_vns_policy` produced a small number of case/seed wins, but the
runtime attribution fields still report zero phase delta, so the mechanism is
not yet explainable enough to promote or broaden.

Observation budget is also still a control-plane issue. The 48k default is
better than the old 24k budget, but the planner reads many large surface
interfaces before previewing schema/target/contract results. The next repair
should compact surface-specific context and keep fatal previews visible.

## Next Actions

Do not launch long CVRP solver-quality validation.

Run another short diagnostic only after the next optimization slice. The
highest-value slice is:

1. Make deep-surface selection explicit for diagnostics: either force one of
   `alns_vns_policy`, `destroy_repair_policy`, `route_pair_candidate_policy`,
   or `acceptance_restart_policy` per short run, or add a diagnostic selection
   prior that suppresses repeated `main_search_strategy`/legacy surfaces until
   each new deep surface has been exercised.
2. Compact APS context and previews so selected-surface schema, target/action,
   and Contract preview can fit under the 48k observation budget. The current
   sessions often skip Contract preview exactly when it would be most useful.
3. Improve deep-surface runtime attribution, starting with `alns_vns_policy`:
   clarify whether `alns_vns_accepted` means policy activation or accepted
   improving moves, and add before/after objective attribution that can explain
   R8's seed-level wins instead of reporting zero phase delta.
4. Add selected-surface feedback tags for `ZERO_PHASE_DELTA`,
   `RECOVERY_ONLY_ACCEPTED_MOVES`, and `DEEP_SURFACE_NOT_SELECTED` so later APS
   hypotheses stop optimizing accepted counts or old orchestration surfaces
   when the desired evidence is missing.
5. Consider a route-local runtime-summary bridge. R4 had useful generic
   operator telemetry, but `candidate_surface_runtime_summary` was empty for
   the selected `route_local` surface, which makes feedback less uniform than
   the policy surfaces.

After those repairs, run a forced short smoke on one deep surface at a time.
The next run should be judged first on selected-surface coverage, complete
preview visibility, and nonzero mechanism-level attribution; screening win rate
is secondary until attribution moves.

## Post-Analysis Optimization

Implemented on 2026-05-11 after this analysis:

- APS now reserves terminal tool-call, step, and observation-char budget for
  schema, target/action, interface, and Contract previews after required
  diagnosis context is collected.
- Compact selected-surface reads now use 800-character code previews, and
  `context.list_surfaces` exposes generic diagnostic surface priorities.
- Runtime diagnosis now tags unexercised declared mechanism surfaces, all-zero
  phase/objective-delta fields, and accepted/recovery movement without
  phase-level benefit.
- `alns_vns_policy` runtime audit now records construction-start distance,
  returned baseline distance, and objective deltas in addition to phase delta.

The next campaign should validate whether those changes make previews visible
and make ALNS/VNS or other deep-surface attribution explainable before judging
solver efficacy.
