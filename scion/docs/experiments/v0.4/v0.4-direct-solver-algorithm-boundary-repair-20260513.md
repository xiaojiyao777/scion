# Direct Solver-Algorithm Boundary Repair, 2026-05-13

## Trigger

The latest CVRP diagnostics showed a small positive signal inside
`solver_design`, but the optimization path was still too indirect. Scion was
being asked to fill or tune componentized lifecycle tables instead of studying
the algorithm itself.

## Change

`solver_design` now targets `policies/solver_algorithm.py`.

Candidates implement:

```python
def solve(instance, rng, time_limit_sec, context):
    ...
```

The candidate may construct and improve CVRP routes directly, use bounded
helper methods on `context`, and record phase runtime evidence. The adapter and
solver still own feasibility, objective recomputation, timeout governance,
seeds, protocol splits, and Decision rules.

The former `policies/main_search_strategy.py` path remains declared as
`main_search_strategy`, a legacy config surface for regression coverage. It is
not the preferred solver-design research object.

## Early Validation Outcome

The launch produced a useful boundary signal but no evaluated experiments.
APS stayed on `solver_design` and generated direct
`policies/solver_algorithm.py` patches, but the run blocked before screening
after repeated proposal/code-stage failures:

```text
status=blocked_infra
n_experiments=0
failure_mode=proposal/code_generation_failed_before_screening
```

Observed failures:

- `C8_import_whitelist` rejected `time`, even though generated full algorithms
  naturally used monotonic timing.
- `C9c_complexity_bound` rejected natural finite algorithm-body `while` loops
  such as shrinking `unvisited` construction loops, capped ILS rounds, and
  local-search loops with bounded break conditions.
- Synthetic preview rejected `context.baseline(..., time_limit_sec=...)`
  because preview and runtime context helpers were not API-compatible.
- Candidate code also compared `context.objective(...)` values directly or
  used tuple-style objective indexing, which was a predictable friction point
  for a lexicographic CVRP objective.

This was framework friction, not solver-quality evidence.

## Follow-Up Repair

The direct algorithm boundary was widened where it should be flexible and kept
fixed where it must remain fixed:

- `context.baseline` now accepts an optional seed solution plus
  `time_budget_sec` or compatibility alias `time_limit_sec`.
- `context.objective` still returns a mapping with `fleet_violation` and
  `total_distance`, but also supports lexicographic comparison and `[0]`/`[1]`
  access as `(fleet_violation, total_distance)`.
- `context.objective_key` and `context.is_better` are exposed in preview and
  runtime.
- `time` is whitelisted for monotonic timing; the guidance still forbids
  sleeps and keeps runtime budget decisions under solver timeout governance.
- `instance.depot` is documented as part of the direct solver API.
- C9c now recognizes bounded full-algorithm `while` patterns: shrinking
  collections, incrementing counters against finite caps, and loops with
  bounded-break or `context.remaining_time()` guards.

Framework issue found: successful active runs still reported

not overwritten on valid active return. This misled later hypotheses into
trying to "activate" an already active solve path. Follow-up repair now writes
`completed` for valid active direct-algorithm returns.

Additional C9c/preview friction found: three code-generation attempts still
failed on runtime-guarded algorithm loops such as
`while context.remaining_time() > 0.5` and `while True` with internal
`context.remaining_time()`/stagnation breaks. Follow-up repair accepts those
runtime-guarded loops and makes synthetic preview decrement its remaining-time
budget, preventing preview hangs while preserving fail-closed behavior for
plain unguarded `while True`.
