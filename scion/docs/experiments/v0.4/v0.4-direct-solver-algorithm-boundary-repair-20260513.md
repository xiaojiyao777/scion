# Direct Solver-Algorithm Boundary Repair, 2026-05-13

## Trigger

The latest CVRP diagnostics showed a small positive signal inside
`solver_design`, but the optimization path was still too indirect. Scion was
being asked to fill or tune componentized lifecycle tables instead of studying
the algorithm itself.

## Change

`solver_design` now targets `policies/solver_algorithm.py`.

Candidates implement:

```python
def solve(instance, rng, time_limit_sec, context):
    ...
```

The candidate may construct and improve CVRP routes directly, use bounded
helper methods on `context`, and record phase runtime evidence. The adapter and
solver still own feasibility, objective recomputation, timeout governance,
seeds, protocol splits, and Decision rules.

The former `policies/main_search_strategy.py` path remains declared as
`main_search_strategy`, a legacy config surface for regression coverage. It is
not the preferred solver-design research object.

## Interpretation

This is an engineering boundary repair, not solver-quality evidence. It fixes
the research object: Scion can now propose a complete CVRP heuristic under a
fixed problem contract instead of being steered through one exposed component
or lifecycle policy.

The next short experiment should verify that APS actually selects
`solver_design`, edits `policies/solver_algorithm.py`, passes Contract preview
with a valid `solve`, and produces useful `solver_algorithm_*` runtime evidence
