## Interpretation

This run validates the control path that was missing before:

- `--force-surface main_search_strategy` held across proposal rounds.
- The selected whole-algorithm surface loaded, activated, and emitted complete
  runtime audit fields.
- The candidate disabled post-baseline registry operators by default and ran
  inside the problem-owned main-search path rather than relying on generated
  post-baseline operators.

It does not validate solver-quality improvement. Round 1 still failed
screening. The useful algorithm signal is that owned local-search components
accepted some moves and runtime improved, but the accepted moves did not
translate into enough final objective improvement. The run also failed to test
the newly exposed `route_pair_swap` and `bounded_destroy_repair` components.

The dominant near-term blockers are now proposal/control quality rather than
runtime audit:

- APS `context.read_surface(main_search_strategy)` produced repeated
  `result_too_large` observations and near-exhausted the 24k observation
  budget.
- Only one of four APS sessions completed; three were
  `partial_hypothesis_only`.
- Rounds 2 and 3 were blocked by `C10_novelty` duplicates even though the
  forced surface remained correct.

## Recommendations

Do not expand directly to 5-10 rounds yet. First:

- Make `context.read_surface(main_search_strategy)` compact enough to return a
  usable observation under the APS budget.
- Tune novelty for singleton semantic surfaces so distinct
  `main_search_strategy` hypotheses do not collapse into `C10_novelty`
  duplicates before patch/evaluation.
- Update `main_search_strategy` guidance to encourage evaluation of
  `route_pair_swap` and `bounded_destroy_repair`, not only
  `intra_route_2opt` / `inter_route_relocate`.

After those fixes, run a new tightly forced `main_search_strategy` diagnostic.
