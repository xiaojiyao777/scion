The selector-semantics repair was real but not enough. The two screened

The failed workspaces exposed a sharper interface problem. Several candidates
put subset-strategy names into `destroy_selectors`:

```text
destroy_selectors=["route_diverse"]
destroy_selectors=["single_worst"]
```

Those are not valid destroy selector values. `route_diverse` and
`single_worst` are valid only in `subset_strategy`; valid `destroy_selectors`
are `worst_removal` and `route_diverse_worst`. The existing
`destroy_repair_policy` interface text said "return bounded destroy selectors,
repair selectors, subset strategy..." without listing the disjoint enum sets,
which made this failure predictable.

## Follow-Up Repair

Implemented after this analysis:

- The CVRP adapter-rendered `destroy_repair_policy` interface now lists the
  valid `destroy_selectors`, `repair_selectors`, and `subset_strategy` values
  explicitly.
- The interface now warns that `single_worst` and `route_diverse` belong in
  `subset_strategy`, not `destroy_selectors`.
- Regression coverage asserts that the rendered interface contains the
  disjoint enum sets and that problem preview rejects
  `destroy_selectors=['route_diverse', 'single_worst']`.
