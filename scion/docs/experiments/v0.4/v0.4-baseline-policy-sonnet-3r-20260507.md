## Interpretation

This run validates implementation/control readiness for `baseline_policy`:
the selected surface can be proposed, patched, loaded, audited, and screened
through the formal CVRP path. It does not validate solver-quality improvement.

The main negative finding is that `--force-surface baseline_policy` did not
hold the surface for all three rounds. The only `baseline_policy` candidate
showed weak pair-level movement, but still failed case-level screening. The
other two rounds reverted to already-known tie/no-op dominated surfaces.

## Recommendations

- Fix or clarify `--force-surface` semantics before treating a multi-round run
  as a forced-surface diagnostic. For a true `baseline_policy` diagnostic, all
  proposal rounds should stay on `baseline_policy` unless the command is
  intentionally documented as first-round-only.
- Run another short, tightly forced `baseline_policy` diagnostic after that
  control issue is resolved. Keep runtime audit fields and compare small versus
  larger formal instances explicitly.
- Steer the next `baseline_policy` candidate toward conservative
  instance-size schedules rather than only increasing destroy intensity; the
  first candidate showed some positive pair-level movement but also objective
  losses.
- Reduce APS context/surface payload pressure later. `result_too_large` and
  maxed tool loops did not cause this screening failure, but they are likely to
  degrade proposal quality and diagnostic stability.
