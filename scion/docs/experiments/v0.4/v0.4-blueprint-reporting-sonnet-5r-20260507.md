## Solver-Quality Interpretation

This run is control-path evidence, not solver-quality evidence.

The round-1 blueprint candidate activated the top-level lifecycle path and
made the candidate faster:

```text
runtime_ratio_median=0.821
runtime_delta_median_ms=-1497.5
runtime_regression_rate=0.0
```

However, target-quality evidence stayed screening-negative. Pair-level
objective results were 1 win, 3 losses, and 12 ties, which collapsed to
case-level `win_rate=0.0` and `median_delta=0.0`. The generated plan lowered
baseline time fraction to `0.65`, ran the construction ensemble, and executed
the package-owned local-search phase, but local search accepted no moves:

```text
algorithm_local_search_attempts=192 per pair
algorithm_local_search_accepted=0 per pair
algorithm_stop_reason=max_local_search_rounds
```

The useful diagnosis is therefore not a framework/gate problem. The current
CVRP algorithm surface can be selected, patched, loaded, audited, screened,
and reported, but the exposed package-owned local-search primitives are not
strong enough to compensate for a reduced baseline budget or to beat the
repo-local `vrp_alns_vns` baseline on formal screening cases.

The remaining non-blueprint rounds confirm the same efficacy bottleneck:

- `route_local` produced two winning pairs but no winning cases; it attempted
  26 operator applications and accepted only two moves.
- `neighborhood_portfolio` and `construction_policy` were runtime-valid but
  objective-tie dominated.
- `route_pair` attempted 24 operator applications, accepted zero moves, and
  produced one losing pair.

## Recommendation

Do not start a long formal solver-quality validation yet. The reporting and
control path is healthy enough to unblock research work, but a longer run is
likely to spend LLM/API and solver budget exploring surfaces that already show
tie/no-op dominated screening behavior.

Recommended next step is a focused CVRP surface-efficacy slice before any long
run:

- Improve or add problem-owned algorithm surfaces that can change solution
  quality before/inside the strong baseline, not only after it.
- Add a real destroy/repair or acceptance/perturbation/restart hook if CVRP can
  define bounded interfaces and runtime audit fields for it.
- Tighten `algorithm_blueprint` guidance so candidates do not reduce baseline
  budget unless the compensating package-owned phase has evidence of accepted
  moves.
- Review the current restart semantics: the exposed restart knobs currently
  record stagnation/stop behavior rather than performing a true perturbing
  restart, so they should not be treated as an escape mechanism.
- After a small efficacy repair, run another 5-10 round diagnostic. Move to a
  longer Sonnet run only after at least one candidate reaches nontrivial
  screening quality, such as positive case-level win-rate with nonzero accepted
  algorithm moves or a policy change that improves objective evidence without
  relying only on speed.
