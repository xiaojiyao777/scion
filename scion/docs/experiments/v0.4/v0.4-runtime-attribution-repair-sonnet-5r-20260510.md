## Summary

The deterministic runtime attribution was recorded, but it still did not reach
the proposal agent in a usable form. R1-R4 had complete selected-surface runtime
audit, including the new `main_search_component_accepted_delta_sum`,
`main_search_component_accepted_best_delta`,
`main_search_component_accepted_positive_counts`, and
`main_search_objective_trace` fields. Those values appeared in raw metrics and
campaign summaries with numeric summaries.

However, APS feedback shown to later hypothesis sessions still emphasized
compact representative/modal values and omitted the key accepted-delta and
objective-phase attribution. The agent repeatedly interpreted prior rounds as
"zero or near-zero accepted moves" instead of diagnosing the actual failure:
some rounds accepted positive component moves, but those moves did not become
`improvement_loop` phase delta, returned-best improvement, or case-level wins.

Forced-surface governance remained healthy. All five hypotheses and patches
stayed on `modify/main_search_strategy -> policies/main_search_strategy.py`.
R1-R4 passed Contract, Verification, and canary, then failed screening by
`SCREENING_FAIL_WIN_RATE`. R5 passed Contract but failed Verification because
its plan selected only `intra_route_2opt`; selected-surface runtime evidence
failed closed with `empty=main_search_deep_components_selected`.

No validation, frozen, promotion, or final-evidence signal was produced.

## Interpretation

The core campaign/protocol controls are not the immediate blocker:

- forced `main_search_strategy` control held for all five hypotheses;
- C10/Contract no longer blocked valid structured singleton proposals;
- R1-R4 passed Verification, canary, and selected-surface runtime audit;
- R5 was correctly stopped by selected-surface runtime evidence;
- feedback retrieval worked;
- deterministic runtime attribution was recorded;
- `DecisionFeatures` did not consume tainted attribution payloads.

The remaining blocker has two linked parts:

1. Proposal feedback propagation is still too lossy. The new attribution exists
   in protocol summaries, but APS did not receive the accepted-delta and
   objective-phase facts needed to diagnose accepted-but-zero-phase-delta.
2. CVRP `main_search_strategy` component semantics are not producing net
   phase/case benefit. Route-pair and bounded destroy/repair can accept moves,
   but accepted local deltas are not becoming phase-best or returned-best
   improvement under formal screening.

## Next Actions

Do not launch long CVRP validation.

Recommended next repair:

1. Fix proposal feedback attribution selection so `feedback.query_runtime` and
   `feedback.query_screening` expose compact accepted-delta totals, nonzero
   pair counts, objective-phase deltas, and accepted-but-zero-phase-delta
   diagnostics for the selected surface.
2. Repair CVRP `main_search_strategy` component semantics: inspect acceptance,
   incumbent/current solution updates, phase-best updates, and returned-best
   accounting for route-pair and bounded destroy/repair moves.
3. Add a problem-owned forced diagnostic guard or preview signal so a forced
   `main_search_strategy` diagnostic cannot silently drop both deep components
   and reach runtime before failing `main_search_deep_components_selected`.
4. Re-run a five-round forced `main_search_strategy` smoke only after those
   repairs; judge first on APS attribution use and phase/case propagation, not
   on long-validation quality.

Follow-up implementation in the current slice addresses items 1-3: proposal
feedback now prioritizes objective-phase and accepted-delta fields, and CVRP
`main_search_strategy` now separates component-local accepted moves from
phase-best improvements before gating bounded destroy/repair or resetting
stagnation.
