## Root Causes

The `C9c` failures were mixed. One generated patch still had a true unbounded
improvement-flag loop (`while improved:`) and should fail closed. Several other
patches used explicitly bounded collection-size loops such as
`while len(neighbors) < cap` and `while len(removed) < q`, where `cap` or `q`
had been bounded by `min(...)`/`max(...)` and the loop body visibly grew the
collection. Those are finite algorithm-body patterns and should not be rejected.

The `proposal.algorithm_smoke` failures were a framework bug. Champion
snapshots are intentionally read-only. The smoke helper copied the read-only
snapshot with preserved `444/555` permissions, then attempted to write the
candidate patch into the copied workspace. The temporary workspace is tainted
debug state, so it is allowed to be writable even when the source champion
snapshot remains immutable.

## Repair

- `ContractGate` now recognizes bounded collection-size `while` loops when:
  - the condition compares `len(collection)` to an effectively bounded limit;
  - the loop body visibly grows or shrinks that same collection in the
    direction that makes the condition false;
  - lower-case limits such as `q` are accepted only when a prior assignment in
    the same block bounds them through `min(...)`, `max(...)`, a declared scale
    term, or a runtime guard.
- The existing rejection of true unbounded improvement-flag loops remains.
- `proposal.algorithm_smoke` now restores owner write permission on the copied
  temporary target path before applying the candidate patch. It does not change
  the read-only champion snapshot.

## Research-Object Design Debt

The current `PatchProposal` protocol still requires `code_content` to be the
complete contents of one file. For `solver_design`, the prompt explicitly asks
for a compact complete replacement file. Therefore a small algorithmic change,
such as modifying one repair operator, still forces the LLM to regenerate the
whole `policies/baseline_algorithm.py` file.

This is acceptable for boundary bring-up but not ideal for serious algorithm
research. The next object-model repair should split the branch-owned algorithm
subject into controlled modules inspired by the original `vrp/` implementation:
construction, destroy, repair, local search, acceptance, scheduler/runtime
allocation, and telemetry.

The intended boundary is module-flexible: the agent may add, delete, or modify
algorithm modules inside the declared solver-design package. Scion should
constrain the problem boundary, not the number of internal files. The fixed
adapter continues to own parsing, feasibility, objective recomputation, seeds,
protocol splits, and promotion decisions; Scion audits the resulting complete
candidate workspace through the stable solve entrypoint.
