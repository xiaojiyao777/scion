# v0.4 v3 Control-Closure Repair Before Live Validation

*Date: 2026-05-17*

## Purpose

The previous v3-aligned repair fixed the largest Contract and APS context
gaps. This follow-up closes the remaining framework-control issues that were
still risky before another live Sonnet smoke:

- scheduler rewrites through `additional_changes`;
- pending code retries consuming apparent exploration rounds;
- inconsistent `solver_algorithm` compatibility handling;
- incomplete solver-design telemetry in generic protocol summaries;
- background status lag on champion-side runtime failures;
- `agentic_session.py` remaining too monolithic.

## Repairs

### Scheduler Ownership Gate

For `solver_design`, if the approved primary target is not
`policies/baseline_modules/scheduler.py`, a scheduler `additional_changes`
entry is now treated as wiring-only. Contract compares the champion and
candidate `_ALNSVNSSolver.solve` loop structure and rejects added/removed
`while` loops, changed `while` conditions, or added/removed/changed `for`
loops. Import and operator-registration wiring remains allowed.

This directly targets the previous failure mode where
`destroy_repair.py`/`construction.py` candidates repeatedly rewrote scheduler
and hit C9c uncapped-loop failures.

### Retry and Round Accounting

Pending code retries are no longer counted as fresh exploration rounds.
Retry attempts:

- reuse the current campaign round number in `StepRecord`;
- set `proposal_session_ref.retry_attempt=true`;
- keep the original prior failure in `retry_prior_failure`;
- do not increment `total_rounds` or idle-round counters;
- do not consume the outer `max_rounds` budget.

`CampaignLoop` now counts only `StepResult.counts_toward_max_rounds=True`
steps toward the max-round budget, with a hard attempt cap to prevent
non-counting retry loops from hanging a run.

### Surface Alias and Runtime Telemetry

`solver_algorithm` is normalized to `solver_design` for runtime audit,
protocol execution, and APS algorithm smoke. This keeps the compatibility hook
inside the declared solver-design boundary instead of letting each layer handle
the alias differently.

Generic protocol runtime observation now includes solver-design telemetry:

- `solver_algorithm_stop_reason`;
- search iterations and move attempts;
- accepted, improving, and neutral accepted moves;
- baseline calls/errors;
- solver-algorithm errors and events.

Champion process failures, missing outputs, and champion runtime-audit failures
now emit progress callbacks so `status.json` does not lag during those rare
branches.

Algorithm smoke payloads also record resolved case path and data-root
provenance (`data_root`, `data_root_source`, `data_root_status`) so formal case
resolution is auditable.

### APS Module Split

`agentic_session.py` was reduced from 4011 to 3242 lines without moving the
main session class or rewriting the state machine. The first split extracts:

- `agentic_session_budget.py`: tool/observation/self-check budget helpers;
- `agentic_session_tools.py`: tool allowlists, selection, and read-argument
  shaping;
- `agentic_session_feedback.py`: feedback query filters and compact feedback
  observation shaping.

This keeps APS behavior stable while making the module easier to extend.

## Live Validation Plan

Run a monitored 3-round Sonnet formal CVRP smoke first. Inspect each round's
APS traces and campaign summary for:

- mandatory code-phase `context.read_surface`;
- branch-current integration context;
- scheduler `additional_changes` limited to wiring unless scheduler is the
  approved target;
- retry attempts marked without consuming exploration rounds;
- normalized `solver_design` selected surface and `solver_algorithm_*`
  telemetry;
- progress updates on protocol failure paths.

If the 3-round smoke does not show framework-control regressions, launch a
6-round background Sonnet validation with the same configuration.
