# v0.4 P0/P1 Telemetry Closure Validation, Sonnet 6R

Date: 2026-05-18

## Scope

This round repaired Scion v3 framework control gaps found in the previous
`v0.4-v3-control-closure-sonnet-6r` experiments. The goal was framework
closure, not CVRP solver quality.

The fixes intentionally keep Scion problem-neutral. CVRP-specific algorithm
meaning remains in the CVRP problem adapter and declared `solver_design`
surface; Scion core only enforces protocol, boundary, telemetry, evidence, and
audit contracts.

## Framework Changes Validated

- Active solver tool hardening:
  - `context.read_algorithm_file` and `context.read_algorithm_symbol` reject
    surface ids such as `solver_design`, absolute paths, traversal paths, and
    non-allowlisted files.
  - Tool schemas now expose allowed path guidance and make clear that
    `solver_design` is a surface id, not a file path.
- Premise failure classification:
  - `premise_contradicted` is recorded as `agent_grounding_failure` /
    `agent_quality_blocked`, not infra.
- Patch and contract control:
  - Same-patch module imports are supported.
  - Sensitive import/API alias checks were strengthened.
  - `expected_telemetry` is checked by contract against adapter/surface
    declared telemetry keys.
- Public evidence and audit refs:
  - Public status, summary, final refs, and DB audit payloads use relative or
    opaque artifact refs, not host absolute paths.
- Prompt manifest:
  - API-visible prompt manifests contain `section_statuses` matching prompt
    sections and keep `raw_prompt_saved=false`.
- Non-formal final evidence closure:
  - Short experiments that finish normally without a formal final evidence
    package now write a machine-readable non-formal closure instead of
    reporting a missing package as if formal readiness were expected.
- Telemetry guard:
  - Added a generic `scion.telemetry_guard.v1` over adapter/surface declared
    telemetry fields. It does not hard-code CVRP, ALNS, VNS, or route semantics.

## Remaining Gap

Telemetry guard is now present and bounded by adapter/surface declarations, but
it should not yet be treated as a complete solution for mechanism activation or
budget starvation.

The 6R analysis found that the guard proved the solver ran and emitted declared
generic telemetry, but did not prove that a newly proposed mechanism actually
activated. For example, a candidate can introduce a new mechanism while existing
generic fields such as solver iterations and elapsed time remain positive.

Follow-up direction:

- Bind hypothesis `expected_telemetry` to the mechanism or phase introduced by
  the patch.
- Require adapter-declared per-mechanism activation/effect counters where the
  problem surface can provide them.
- Emit hard warnings or fail closed when a declared mechanism has no activation
  evidence after smoke/screening.

## Conclusion

The framework-control fixes are ready to commit as Scion v3 boundary and
evidence improvements. The result should be described narrowly:

- tool guidance, evidence closure, manifest completeness, audit ref hygiene,
  premise failure classification, and generic telemetry guard plumbing are
  validated;
- solver quality did not improve in this validation;
- mechanism-level activation/effect attribution remains a follow-up item for
  the problem adapter and surface telemetry contract.
