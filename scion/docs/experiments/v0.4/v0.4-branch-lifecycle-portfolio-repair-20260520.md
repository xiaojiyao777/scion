direction and may evolve through multiple hypotheses.

## Repair

- Added generic `core.branch_lifecycle_policy`.
- `DecisionFeatures` now carries structured `wins`, `losses`, and `ties`.
- Low-win screening is classified as:
  - weak-positive continue;
  - neutral/mostly-tie continue;
  - zero-win streak exhausted;
  - clear regression via no-win losses, negative median delta, runtime slowdown,
    or high runtime-regression rate.
- Weak-positive and neutral continue paths preserve branch workspace and patch
  so the next proposal can iterate on the same branch.
- Telemetry-repairable formal failures remain non-counting repair attempts and
  do not pollute zero-win streaks.
- Stale reconcile keeps the old fail-closed behavior: re-screen low-signal
  reconcile results still abandon rather than silently restoring stale code.
- Scheduler now supports a branch portfolio:
  - protocol continuations, validation/frozen work, stale reconciliation, and
    pending retries remain higher priority;
  - an established ordinary explore branch can coexist with newly created
    sibling branches until `max_active_branches` is reached;
  - at capacity, research branches rotate by oldest `updated_at`.

## Telemetry Noise Fix

`expected_telemetry` normalization no longer treats runtime field container
names such as `solver_algorithm_phase_runtime_ms` as mechanism ids. A path such
as `solver_algorithm_phase_runtime_ms.<mechanism_id>` remains a valid
mechanism-specific runtime attribution field when the mechanism itself is
declared separately.
