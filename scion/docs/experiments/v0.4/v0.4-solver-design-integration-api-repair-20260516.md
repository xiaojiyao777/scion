## Findings

- The low-effort smoke gate was not validated by this run. Candidates failed
  earlier on runtime API incompatibilities.
- Construction-target candidates tried to integrate a new seed by modifying
  `policies/baseline_algorithm.py` and calling `_ALNSVNSSolver` with positional
  arguments or by calling `solver.solve(instance, rng, initial_solution=...)`.
  The stable entrypoint does not support either protocol.
- Acceptance-target candidates rewrote `scheduler.py` as an integration edit
  with a different `_ALNSVNSSolver.__init__` API, or imported scheduler symbols
  that do not exist in the branch-owned modules.
- Algorithm smoke correctly caught the runtime failures, but the failure should
  be surfaced earlier and more explicitly by the solver-design integration
  contract.

## Repair

- C9e now rejects `baseline_algorithm.py` integration edits that instantiate
  `_ALNSVNSSolver` without the explicit stable constructor keywords:
  `time_limit`, `destroy_ratio`, `segment_length`, `reaction_factor`,
  `vns_max_no_improve`, `use_vns`, `cw_threshold`, `vns_threshold`,
  `alns_threshold`, `max_destroy_customers`, `max_routes`, and `context`.
- C9e now rejects `baseline_algorithm.py` integration edits that call
  `solver.solve(...)` with extra positional or keyword arguments.
- C9e now rejects `scheduler.py` integration edits that no longer accept the
  stable constructor keywords or no longer expose
  `_ALNSVNSSolver.solve(self, instance, rng)`.
- Code prompts and the CVRP solver-design surface now state that new seed or
  initial-state hooks belong inside scheduler methods, not in a changed
  `baseline_algorithm.py` call protocol.

The repair keeps the intended flexibility: a candidate may still make broad
algorithmic changes when the scheduler or entrypoint is the approved primary
target, and module-level candidates may still include scheduler/entrypoint
integration edits when those edits preserve the auditable runtime call chain.

## Object-Model Repair

- Solver-design prompt scope now states the internal state API explicitly:
  `_Solution` has no `from_public`, `from_routes`, `from_cvrp_solution`, or
  `to_public` bridge methods, and candidates must not add those methods to
  `baseline_modules/state.py`.
- CVRP surface/interface rendering and support-artifact `python_api_summary`
  now show the correct construction recipe: create `_Route(instance, route)`
  objects, build `_Solution(instance, routes)`, and return through
  `context.make_solution(solution.routes_as_tuples())`.
- C9e now rejects invented `_Solution` bridge calls and bridge method
  definitions on `baseline_modules/state.py` before algorithm smoke.
- APS final failure detail now uses the latest evaluative Contract/smoke
  preview and ignores non-evaluative skipped/permission/fuse preview
  observations when a real preview result exists.
- Stagnation now classifies repeated object-model/API failure streaks as
  `object_model_loop` with `inspect_agent_trace` guidance, instead of a generic
  infra-loop recommendation.
- APS maintainability was improved by splitting `agentic_session.py` into
  focused modules for models, artifacts, diagnostics, code-context shaping,
  preview/self-check helpers, and shared utilities.

## Destroy/Repair API-Manifest Repair

The repair keeps the solver-design boundary flexible but makes the branch
algorithm object explicit enough for the code stage to modify the right file:

- `ContextManager.build_code_context(...)` now accepts a `branch_workspace`.
  When a branch already has verified code, the code stage reads the current
  branch target file and API manifest instead of unconditionally reading the
  champion snapshot.
- Solver-design code context now includes a bounded module API manifest for
  `baseline_algorithm.py` and the active `baseline_modules` package. It lists
  exact importable function/class signatures and current sibling imports, so
  the model sees real names such as `_clarke_wright_savings` and
  `_nearest_neighbor` rather than guessing `_clarke_wright_solution` or
  `_savings_construction`.
- Destroy/repair targets now have an explicit ownership rule in code prompts
  and APS scope control: implement the destroy/repair mechanism in
  `destroy_repair.py`; use `scheduler.py` only as minimal operator-pool wiring
  for exact symbols imported from `.destroy_repair`.
- C9e missing-import detail now includes `primary_target` and
  `available_exports` for the imported module, so repair generations can
  correct to an existing symbol or define the exact symbol in the same patch.
