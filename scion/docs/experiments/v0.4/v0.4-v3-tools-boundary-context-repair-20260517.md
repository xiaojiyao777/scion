# v0.4 v3-Aligned Tools, Boundary, and Branch-Context Repair

*Date: 2026-05-17*

## Inputs

This repair was based on three evidence sources:

- [`scion-architecture-v3.md`](../../../design/scion-architecture-v3.md): Scion's baseline architecture. LLM proposal output is tainted; deterministic Contract, Verification, Protocol, and Decision layers own control.
- [`scion-code-audit-20260517`](../../reviews/scion-code-audit-20260517/README.md): code audit findings around stale branch context, dynamic API bypasses, Contract/APS drift, and maintainability.
- `/home/clawd/research/scion-experiments/v04-api-manifest-sonnet-6r-20260517T042338Z`: six-round Sonnet run. Rounds 1-3 reached screening and were abandoned for solver quality; rounds 4-6 failed in code generation because repeated `additional_changes` rewrote `scheduler.py` with C9c-violating uncapped `while` loops.

## Design Conclusion

The issue was not that Scion needed more component knobs. The useful direction
remains the v3 model: Scion controls boundary, protocol, lineage, and audit,
while a real proposal agent studies and modifies the declared problem object.
For CVRP v0.4 that object is the branch-owned `solver_design` algorithm:
`policies/baseline_algorithm.py` plus modules under
`policies/baseline_modules/`.

The failed run showed two concrete implementation gaps:

- Code generation saw the approved target file, but not enough branch-current
  content for `additional_changes`; scheduler wiring was therefore generated
  from incomplete context.
- Several solver-design invariants existed in prompt text or APS preview, but
  not in the core Contract layer, so the framework was not fail-closed.

## Repairs

### Proposal Tools Package

`scion.proposal.tools` is now a package instead of one long file:

- `registry.py`: `ProposalToolRegistry` and call boundary.
- `context.py`: public context readers.
- `surface.py`: surface discovery and `context.read_surface`.
- `feedback.py`: screening/runtime/holdout feedback and memory queries.
- `preview.py`: draft/schema/interface/contract/smoke preview tools.
- `models.py`, `base.py`, `utils.py`: shared tool contracts and helpers.

The public import path is preserved:

```python
from scion.proposal.tools import ProposalToolRegistry, ProposalToolContext
```

This makes tools a real extensible module while keeping proposal observations
tainted and outside Decision inputs.

### Contract Fail-Closed Boundary

Contract now rejects boundary bypasses that were previously prompt-only or
adapter-preview-only:

- Dynamic sensitive APIs such as `__import__`, `importlib.import_module`,
  reflective dynamic calls, `os.environ`, and file/path read helpers.
- Preferred `policies/baseline_algorithm.py` wrappers that call
  `context.baseline(...)`.
- Instance identity leaks such as `repr(instance)`, `str(instance)`,
  `vars(instance)`, `instance.__dict__`, computed `getattr(instance, ...)`,
  and obvious dataclass reflection.
- Dead solver-design helper references such as `unused = helper`; C9e now
  recognizes real call edges or explicit active registration structures.
- C9c uncapped `while` messages now include the condition snippet and repair
  hint, improving code-stage repair traces.

### APS and Code Context

Code phase now treats `context.read_surface(detail="full")` as mandatory for
the selected surface. It is not skipped merely to reserve self-check budget.

Feedback queries default to the active single problem-boundary surface when no
forced surface is configured. This prevents `solver_design` runs from drifting
back toward legacy component-policy evidence.

Solver-design code context now includes branch-current integration files with
provenance. The prompt separates:

- approved target file full current content;
- branch-current integration files for entrypoint/scheduler/state;
- compact sibling module API summaries.

The instruction for `additional_changes` is now concrete: use the current
branch content and make the smallest necessary wiring edits.
