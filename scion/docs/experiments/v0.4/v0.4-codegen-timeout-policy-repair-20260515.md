# v0.4 Codegen Timeout Policy Repair

Date: 2026-05-15

## Trigger

The 8-round Sonnet validation showed many `code_generation` failures:

```text
run_root=/home/clawd/research/scion-experiments/v04-sdk-retry-control-sonnet-8r-20260514T181734Z
model=claude-sonnet-4-6
rounds=8
time_limit_sec=60
agentic_session_timeout_sec=1800
SCION_SDK_MAX_RETRIES=0
SCION_LLM_MAX_RETRIES=1
```

The initial interpretation that prompt size was the primary cause was
incorrect. Trace timing and backend observations point to a client timeout
policy mismatch for long non-streaming code-generation calls.

## Trace Finding

The run produced 77 LLM trace files:

- `tool_selection`: 57/57 succeeded.
- `hypothesis`: 5/5 succeeded.
- `code`: 2/15 succeeded, 13/15 failed.

All failed `code/generate_patch` traces ended with the same timeout error and
clustered tightly around `125.1s`. In this launch, `SCION_LLM_MAX_RETRIES=1`,
so each failed trace was:

```text
60s client timeout + 5s Scion backoff + 60s client timeout
```

Successful code traces were not smaller in any meaningful way. One succeeded
in about `50s`; another succeeded in about `114s`. A failed compact retry and a
successful compact retry had effectively the same prompt length. Therefore the
failure was not a simple context-window or prompt-size threshold.

Because Scion currently calls provider SDKs in non-streaming mode, a local
timeout means the client stopped waiting. It does not prove the provider
stopped generating. Backend records showing occasional 100s+ completions are
consistent with this diagnosis.

## Repair

`LLMClient` now resolves timeout/retry policy per request kind.

Default policy remains unchanged for ordinary calls:

```text
SCION_LLM_TIMEOUT_SEC=60
SCION_LLM_MAX_RETRIES=2
SCION_SDK_MAX_RETRIES=0
```

Code-like tool calls now have safer defaults:

```text
request_kind=code/fix
timeout_sec=max(SCION_LLM_TIMEOUT_SEC, 180)
max_retries=0
```

The code/fix policy is still explicitly tunable:

```text
SCION_LLM_CODE_TIMEOUT_SEC=240
SCION_LLM_CODE_MAX_RETRIES=1
SCION_LLM_FIX_TIMEOUT_SEC=240
SCION_LLM_FIX_MAX_RETRIES=1
```

This avoids the previous worst case where one APS code phase could trigger up
to four backend requests:

```text
2 LLMClient attempts for normal codegen
+ 2 LLMClient attempts for APS compact timeout retry
```

With the new default, APS still has one semantic compact retry, but each code
generation prompt is sent once and gets enough time to return.

`CreativeLayer` trace files now include `request_policy`, so future analysis
can see the effective `timeout_sec`, `max_retries`, `sdk_max_retries`, and
`max_tokens` used by each LLM request.

## Streaming Note

Streaming provider responses would help with long code-generation calls
because the client could receive progress instead of waiting silently for one
large final response. It is not a replacement for the timeout repair: tool-use
streaming requires reconstructing structured `tool_use` inputs from streamed
chunks, and it does not address algorithm quality or branch integration. Treat
streaming as a follow-up robustness improvement after the simpler policy fix
is validated.

## Claude Code Tool-Loop Comparison

Follow-up read-only analysis of `/home/clawd/research/claude-code-src` confirms
that Claude Code's core is a continuous native tool loop, not a two-stage
static generation pipeline:

- `analysis/00-summary.md` describes the loop as context assembly, streaming
  LLM call, parse `tool_use`, run tools, append `tool_result`, continue.
- `analysis/11-tool-system.md` and `src/Tool.ts` describe tool contracts with
  schema, validation, permission checks, execution, result mapping, large-output
  handling, and errors as returned observations.
- `src/query.ts` passes registered tools directly to the model and appends
  tool results back into the same message history.
- `StreamingToolExecutor.ts` can begin tool execution while tool-use blocks are
  still streaming, with safe tools parallelized and exclusive tools serialized.

Scion currently preserves the right boundary properties but does not yet expose
an equivalent native continuous agent kernel. APS asks the model to choose a
tool name through `plan_proposal_tool_call`; Scion executes that tool and later
forces final `generate_patch`. Code phase can inspect branch/surface/memory
through a bounded planner loop, but patch drafting, contract preview, and
algorithm smoke are not yet a model-driven "inspect, draft, preview, repair"
cycle.

Design direction:

- Keep Scion's permission, taint, exposure, transcript, and promotion gates.
- Do not expose shell or direct filesystem writes to the model.
- Convert Scion proposal tools into native API tools inside a sustained loop,
  returning every success, schema error, permission denial, timeout, preview
  failure, and truncation as auditable `tool_result`-style observations.
- Treat `generate_patch` as a finalizer tool, not the whole code phase.
- In code phase, allow controlled calls to `context.read_surface`,
  `context.read_branch_state`, memory/feedback tools, `proposal.draft_patch`,
  `proposal.contract_preview`, and `proposal.algorithm_smoke` before final
  patch submission.

This is the deeper framework direction after the timeout-policy smoke passes:
Scion should remain the boundary/protocol/audit controller, while the LLM acts
as a real research agent that can repeatedly inspect and repair a controlled
branch copy of the algorithm.

not an LLM timeout failure.
