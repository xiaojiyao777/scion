## 3. 验证实验结果

### 3.1 Campaign A — 首次 Promote（8 轮）

```
R1: SubcategorySplitRepair → verify ✅ → screening wr=0.33 → CONTINUE_EXPLORE
R2: SmartMergeVehicles → verify ✅ → screening wr=0.67 → QUEUE_VALIDATE
R3: (validation) wr=0.778 → EXPAND → QUEUE_FROZEN
R4: (frozen) → PROMOTE ✅ → champion v2
```

**SmartMergeVehicles** 在 medium_1 上将 subcategory_splits 从 10 降到 5-6。

### 3.2 Campaign B — 完整链路验证（10 轮）

```
R1: SubcategorySplitRepair → wr=0.125 → CONTINUE_EXPLORE
R2: RegionAwareMerge → wr=0.625 → expand → wr=0.609 → EXHAUSTED
R3: SubcategoryAwareMove → wr=0.50 → EXHAUSTED
R4: SubcategoryConsolidate → wr=0.375 → CONTINUE_EXPLORE
R5: CategoryAwareDestroyRebuild → wr=0.50 → EXHAUSTED
R6: SubcategoryGatherMove → wr=0.50 → EXHAUSTED
R7: SmartCostMerge → wr=0.625 → BORDERLINE → QUEUE_VALIDATE
```

**观察**：
- 全部 7 个算子通过 Verification（0 infeasible），feasibility constraint prompt 效果显著
- LLM 在连续 6 轮 splits 维度无突破后，自主转向 cost 维度（SmartCostMerge）
- medium_2 全 tie（退化 instance）稀释了 win_rate

### 3.3 框架机制触发统计

| 机制 | 触发次数 | 结果 |
|---|---|---|
| Contract C10 novelty 拦截 | 1 | 正确拦截重复 hypothesis |
| Verification V3 feasibility 拦截 | 5+（早期 campaign） | 正确拦截 infeasible 算子 |
| Verification V5 state_leak 拦截 | 5+（早期 campaign，后修复） | sys.modules bug 导致误报，已修 |

## 4. 超出 v0.1 Spec 的改进

| 改进 | 来源 | 价值 |
|---|---|---|
| tool_use 替代 JSON 文本解析 | CC 源码分析启发 | JSON 错误率从 ~10% 降到 0 |

| CC-inspired failure diagnosis | CC 源码分析 | 连续失败时注入诊断摘要 |
| Branch direction tracking | CC plan 模式启发 | 记录分支方向，引导迭代 |
| Feasibility constraints in prompt | 实验诊断 | V3 违规率从 100% 降到 0% |
| Workspace archive | 诊断需求 | ABANDON 时归档代码供复盘 |

## 5. 已知限制与 v0.2 方向

### 已知限制
1. **Instance 退化**：medium_2 全 tie，需替换
2. **分支迭代方式**：LLM 倾向于"换方向"而非"深化改进"，框架层面已支持但 LLM 行为需要更好的 prompt 引导
3. **PoolManager 未深度集成**：registry 操作在 WorkspaceMaterializer 中内联
4. **Frozen holdout 统计力弱**：当前 frozen split 只有 1 个 instance（xlarge_1），容易 trivially pass

### v0.2 方向（按 architecture v3 §19）
- 参数层搜索（外层 LLM 结构 + 内层贝叶斯优化权重）
- 更强的 Verification（unit tests / regression tests 的 surrogate 集成）
- 多问题泛化
- 论文级 ablation 实验
