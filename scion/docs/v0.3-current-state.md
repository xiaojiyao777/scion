## Current Engineering State

### Framework Boundary

- `surrogate/` remains the research object: warehouse-delivery VNS solver and operators.
- `scion/scion/` owns the framework: campaign loop, state machine, decision gates, lineage, context assembly, protocol execution, and adapter boundary.
- `run_validation_campaign.py` now fails fast on synthetic/production protocol mismatch.
- Production validation uses `protocol_prod.yaml`; synthetic validation uses `protocol.yaml`.

### Objective Handling

- Objective semantics are generic and adapter-driven.
- Lexicographic multi-objective evaluation is the current warehouse protocol:
  - priority 1: `subcategory_splits`
  - priority 2: `total_cost`
- Weighted-sum objective mode is implemented as a generic single aggregate objective for future problems.
- Prompt context includes:
  - objective policy
  - objective opportunity profile
  - exploration/exploitation guidance
  - objective steering when higher-priority dimensions are saturated/stable
  - LLM trace persistence

### Weight Optimization

- `parameter_search.execution` supports:
  - `async`: retained for machines with enough cores;
  - `sync`: used for the current 2-core server and formal v0.3 validation.
- `run_validation_campaign.py` defaults to sync weight optimization.
- Sync mode blocks after promotion until the weight optimizer completes and records lineage.
- `status.json` now publishes weight-optimization status during sync/async runs:
  - phase
  - active runs
  - cases/seeds/operators
  - completed evaluations
  - estimated solver runs
  - final improvement result

### Campaign Observability

- `status.json` includes stage-level progress for long screening/validation/frozen phases.
- `experiment_events` records `raw_metrics_ref`, `case_ids`, and `seed_set`.
- `campaign_summary.json` is written by `CampaignManager`; `validation_runner_summary.json` is runner-specific.
- Branch lifecycle persistence is synchronized for abandon/promote/reconcile paths.
- Weight-only updates record invalidation and weight revision state.

## Final Experiment Operating Notes

Use `setsid -f` or an equivalent queue runner for detached long runs. In this environment, plain `nohup ... &` does not reliably survive the tool process group cleanup.

## v0.4 Planning Input

Production validation after the formal 12-campaign run exposed a general framework requirement: optimization must be performance-aware. A generated production operator can improve objective values while being unusably slow on larger production cases, and timeout/failed pairs must never be omitted from promotion evidence.

The v0.4 planning input is now tracked in:

- [v0.4-performance-aware-plan.md](v0.4-performance-aware-plan.md)
- [v0.4-cvrp-plan.md](v0.4-cvrp-plan.md)

The scope has expanded from performance hardening only to:

```text
v0.4 = performance-aware optimization + CVRP second-problem onboarding
```

Initial fixes landed before v0.4 planning starts:

```text
74c6869 fix(v0.3): fail incomplete evidence and bound operators
e8e3673 fix(surrogate): tolerate stale move-order assignments
```

Open v0.4 theme:

```text
performance-aware operator optimization = objective improvement
                                      + complete evidence
                                      + runtime/complexity acceptability

CVRP generalization = adapter boundary
                   + route-sequence operator semantics
                   + standard benchmark evidence
                   + shared quality/runtime reporting
```
