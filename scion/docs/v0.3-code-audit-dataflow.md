# Scion v0.3 Code Audit: Logic and Data Flow

Date: 2026-04-25

This note consolidates the three read-only code audits run after the F3 closure
changes. It is intended as a maintenance map for v0.3 finalization.

## 1. Main Loop

```text
CampaignManager.run(max_rounds)
  -> should_stop()
       -> EarlyStopController
       -> queue-aware delay for validation/frozen/stale work
       -> TerminationChecker
  -> run_one_step()
       -> Scheduler.select_next(active_branches)
          P1 READY_FROZEN
          P2 READY_VALIDATE
          P3 STALE / STALE_WEIGHT_UPDATE
          P4 EXPLORE / EXPLORE_EXPAND / VALIDATING / VALIDATING_EXPAND / FROZEN_TESTING
          P5 create_new
       -> _run_explore_step | _run_eval_step | _run_reconcile_step
  -> _run_stagnation_check()
  -> _check_soft_stagnation()
  -> wait async weight optimization on shutdown
```

Promotion path:

```text
ExperimentProtocol
  -> SafeFeatureExtractor
  -> DecisionEngine
  -> _apply_decision_and_finalize(PROMOTE)
  -> _on_promote()
       -> champion version +1
       -> reset stagnation/idle counters
       -> mark other active branches STALE
       -> spawn async weight optimization worker
       -> worker prepares WeightOptCompletionEvent
       -> CampaignManager main loop drains event and commits optimized revision
```

Abandon path:

```text
Decision.ABANDON
  -> cleanup branch workspace / patch / hypothesis
  -> mark hypothesis rejected
  -> branch ABANDONED
  -> hard abandon counter increments only on finalizer path
```

Stale path:

```text
champion update
  -> mark_all_stale() or mark_stale_for_weight_update()
  -> Scheduler prioritizes stale branches
  -> _run_reconcile_step()
       -> reapply patch against new champion
       -> Contract -> Verification
       -> reconcile_stale(success=True) restores EXPLORE on the new champion
       -> _evaluate() -> DecisionEngine -> _apply_decision_and_finalize()
```

## 2. Objective and Protocol Data Flow

```text
solver.py JSON output
  -> LocalSubprocessRunner -> SolverOutput.objective
  -> ExperimentProtocol.run_experiment()
       for each case x seed:
         champion objective vs candidate objective
         -> compare_lexicographic(metric_specs)
         -> PairwiseCaseFeedback(objective_comparison, scalar delta)
  -> _aggregate_pairs_to_case_level()
       majority vote per case
       median scalar delta
       median per-metric deltas
  -> compute_eval_stats()
       legacy scalar CI when no metric_specs
       priority-aware hierarchical CI when metric_specs exist
  -> gates.py
       screening: win_rate + median_delta
       validation/frozen: statistical_status first, legacy ci fallback
  -> SafeFeatureExtractor
  -> DecisionEngine
```

Current supported objective modes:

- `single`: supported as one objective metric.
- `lexicographic`: supported through `ObjectiveMetricSpec.priority`.
- `weighted_sum`: rejected for now. Use a single adapter-computed `weighted_score`
  objective with `mode: single` until weighted execution semantics exist.

MILP / lower bounds:

- MILP should remain report-only by default.
- `objective_lower_bounds` no longer affects early-stop unless
  `use_objective_lower_bounds_for_early_stop=True` is explicitly passed.

## 3. Creative to Deterministic Boundary

```text
ContextManager
  -> CreativeLayer.generate_hypothesis()
  -> HypothesisProposal (tainted)
  -> ContractGate.validate_hypothesis()
  -> HypothesisStore
  -> CreativeLayer.generate_code()
  -> PatchProposal (tainted)
  -> ContractGate.validate_patch()
  -> WorkspaceMaterializer.apply_patch()
  -> VerificationGate
  -> ExperimentProtocol
  -> SafeFeatureExtractor
  -> DecisionEngine
```

The intended boundary is clear: LLM text/code is tainted; `DecisionFeatures`
only carries enums, booleans, numeric stats, and closed-set reason codes.

## 4. Highest-Priority Risks

P0 already addressed in this pass:

- Last legal evaluation could be vetoed by `budget_remaining_ratio == 0`.
  Budget termination is now left to `should_stop()` before scheduling the next
  experiment, so a completed frozen pass can still promote.
- `weighted_sum` was accepted by schema but executed as lexicographic. It is now
  rejected until explicit weighted comparator/gate semantics are implemented.
- Queue-aware early-stop ignored stale branches. `STALE` and
  `STALE_WEIGHT_UPDATE` now delay early-stop.
- Lower bounds could affect early-stop by default. They are now opt-in for
  early-stop and otherwise remain report-only.
- Patch paths are not fully normalized/resolved before whitelist and write.
  `normalize_relative_patch_path()` now rejects absolute paths, traversal,
  empty path segments, `.` segments, and backslash paths; workspace writes also
  use resolved containment checks.
- Interface verification imports tainted candidate code in the orchestrator
  process. V2 is now AST-only and does not execute top-level candidate code.
- Verification can fail open when runner/spec/canary are missing.
  CLI and `run_v3_campaign.py` now construct `VerificationGate` with
  `strict_runtime_checks=True`, which fails closed when runner/spec/canary or
  champion workspace are unavailable.
- V5 solution consistency failures were reported as light and top-level solver
  outputs were not checked. V5 now reports heavy severity and supports both
  nested `solution` and top-level `vehicles`/`assignment` JSON.
- Operator interface is still warehouse-shaped in contract, verification, and
  prompt code. ContractGate, VerificationGate, CLI, v3 campaign, and
  CampaignManager now accept `ProblemSpecV1.operator_interface.execute_signature`.
  Prompt/tool text no longer hardcodes `execute(self, solution, rng)` or
  warehouse-specific move rules in the framework layer; problem-specific detail
  comes from adapter `render_operator_interface()`.
- Promote is still not an atomic commit. Snapshot/freeze should be completed
  successfully before branch/hypothesis/champion state is committed. Promotion
  now uses a two-stage prepare/commit flow: build and freeze the champion
  snapshot first; only then record promote lineage, mark branch/hypothesis
  promoted, update champion, mark siblings stale, and spawn weight optimization.
  Snapshot preparation failure is treated as infra failure and does not advance
  champion version or mark the hypothesis promoted.
- Weight optimization did not pass `metric_specs`, so non-warehouse parameter
  scoring could fall back to legacy delta semantics. Async and manual weight
  optimization now pass objective metric specs into `evaluate_weights()`.
- Optimized champion revisions were not persisted as champion rows. The
  champions table now supports `(version, weight_revision)` as the identity,
  and improved background weight optimization snapshots are persisted as
  `champion_vN_rM` rows.
- `stale` reconcile bypassed the normal DecisionEngine/finalizer and recorded
  limited lineage. After patch reapply, Contract, and Verification pass,
  reconcile now restores the branch to `EXPLORE` and uses the canonical
  `_evaluate()` plus `_apply_decision_and_finalize()` path. Screening `expand`,
  soft abandon, hard abandon, lineage, and `StepRecord` behavior now match
  normal explore/eval execution.
- Async weight optimization mutated champion and branch state directly from the
  worker thread. The worker now only runs optimization and prepares immutable
  optimized snapshot artifacts, then enqueues `WeightOptCompletionEvent`.
  `CampaignManager` drains those events on the main loop/shutdown boundary to
  record lineage, update the champion pointer, persist the champion row, and
  mark screening branches `STALE_WEIGHT_UPDATE`.
- Branch persistence omitted runtime fields needed for restartable campaigns.
  `BranchStore` now round-trips screening/validation expand counters,
  direction, weight revision, pending retry, blocked rounds, consecutive LLM
  retries, and infra block counters, with SQLite migration for existing DBs.
- Early abandon paths counted hard stagnation inconsistently. Non-T4 abandon
  paths now route through a once-per-branch hard-abandon counter helper,
  including finalizer ABANDON, eval metadata loss, stale reconcile failure,
  permanent infra failure, and `abandon_fast`. Soft abandon remains excluded.
- Hypothesis family classification used warehouse-specific fallback taxonomy by
  default. Campaign construction now passes the full
  `ProblemSpecV1.family_taxonomy` object, including problem-owned `aliases`,
  into classifier/search/context/evidence paths. Framework classification is
  now generic normalization plus family/alias matching only.
- Production runtime verification could still silently use legacy direct-oracle
  reconstruction when no adapter was configured. `VerificationGate` now supports
  `require_adapter_for_runtime=True`; CLI, `run_v3_campaign.py`, and
  `run_validation_campaign.py` set it together with `strict_runtime_checks=True`.
  Legacy direct-oracle checks remain available only for explicit
  compatibility/unit-test paths.
- Long-running campaigns lacked a cheap machine-readable progress file.
  `StatusReporter` now writes atomic `status.json` snapshots with campaign id,
  rounds, branch states, champion revision, budget, and last step result.
- `fix_code` light-retry entry/exit is now logged at INFO level.
- Soft/hard stagnation locus diversification no longer defaults to
  `vehicle_level/order_level` when the problem spec has no operator categories.
- Production `ExperimentProtocol` entrypoints now pass
  `require_metric_specs=True`; missing objective metric specs fail at
  construction time instead of silently falling back to the warehouse
  `compare_with_breakdown` / `compute_delta` path. Legacy no-`metric_specs`
  comparison remains available only for compatibility tests and emits a warning.
- New `CaseAggregateFeedback` generated by `_aggregate_case_feedback()` no
  longer populates deprecated warehouse alias fields; generic
  `decisive_metric` and `median_deltas` are the canonical data path.
- `ObjectiveBreakdown` and `compare_with_breakdown()` have been removed. The
  only remaining legacy objective compatibility path is scalar
  `compute_delta()` for explicit no-`metric_specs` tests/tools; production
  entrypoints cannot reach it because they set `require_metric_specs=True`.

P0 still open:

- No known P0 remains after the current pass. Continue treating production
  campaign runs as adapter-backed and strict-verification only.

P1 open:

- No known P1 remains in the production v0.3 path. Legacy direct-oracle
  verification fallbacks are still present as explicit compatibility helpers,
  but strict production entrypoints require an adapter.

## 5. Recommended Next Work Order

1. Keep legacy direct-oracle verification fallbacks out of production entrypoints.
   Remove them in v1.0 only after compatibility tests have migrated.
2. Keep scalar `compute_delta()` and `CaseAggregateFeedback` aliases out of
   production data paths. Remove the remaining compatibility surface in v1.0
   after historical tests/tools migrate.
