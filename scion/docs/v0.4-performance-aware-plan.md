# Scion v0.4 Performance-Aware Optimization Plan

*Date: 2026-04-28*

## Scope Update

v0.4 now has two connected tracks:

1. harden Scion's promotion logic so objective improvement is never accepted
   without complete evidence and acceptable runtime;
2. use CVRP as the next real problem class to test whether the v0.3 adapter
   boundary generalizes beyond warehouse delivery.

The CVRP-specific integration plan is tracked in:

- [v0.4-cvrp-plan.md](v0.4-cvrp-plan.md)

## Why v0.4 Exists

The v0.3 final validation proved the Scion campaign loop can optimize synthetic cases and can produce production candidates. It also exposed a framework-level gap: objective improvement alone is not enough. Generated operators must also satisfy runtime and evidence-completeness constraints.

Two concrete failures drove this plan:

1. A production champion generated by Scion used high-order vehicle combination enumeration and became too slow on production-scale cases.
2. The old protocol skipped failed/timeout pairs, allowing a frozen gate to promote from incomplete evidence.

These are not warehouse-only problems. Any combinatorial-optimization framework that lets an LLM generate neighborhood operators must treat runtime complexity as a first-class constraint.

## v0.4 Goal

Make Scion **performance-aware**:

```text
An operator is valid only if it improves objectives under complete evidence
and remains within accepted runtime/complexity bounds.
```

v0.4 should harden this while onboarding CVRP. CVRP is a good stress test for
the same requirement because route operators can easily become too slow when
they scan all route pairs, customer triples, or route segments without caps.

## P0: Evidence Completeness Gate

Status: initial v0.3 patch landed in `74c6869`.

Required semantics:

- Candidate timeout/crash is a loss or stage failure, never invisible.
- Champion timeout/crash is an invalid/infrastructure signal.
- Validation/frozen cannot pass when `valid_pairs < total_pairs`.
- Raw metrics must record:
  - `failed_pairs`
  - candidate/champion failure counts
  - case
  - seed
  - side
  - error category
  - elapsed time
  - stderr tail
- LLM context must expose timeout/failure summaries after screening failures.

Acceptance:

- Unit tests cover candidate timeout, champion failure, and incomplete validation/frozen evidence.
- A reproduction using the old slow production champion fails instead of promoting.
- No production promotion has `valid_pairs < total_pairs`.

## P1: Runtime And Complexity Contracts

Status: first static contract landed in `74c6869`.

Rules:

- Reject unbounded high-order enumeration such as:
  - `combinations(..., 3)`
  - `combinations(..., 4)`
  - `combinations(..., variable_k)`
- Allow pairwise combinations only when bounded by candidate preselection.
- Prompt must require:
  - top-k candidate caps
  - sampling or early exit
  - explicit no-op condition
  - runtime comparable to champion

Acceptance:

- The old `region_consolidate_cost.py` style operator is rejected by ContractGate.
- LLM traces show candidates describing their runtime cap.
- Contract failures are recorded as search guidance, not hidden infra errors.

## P2: Production-Scale Perf Guard

Status: first production perf case wiring landed in `74c6869`.

Required behavior:

- `V9_perf_guard` should run on a representative production-scale case when production validation is active.
- Candidate runtime must satisfy both:
  - relative bound: `candidate <= champion * N`
  - absolute bound: case timeout
- Timeout must be reported as performance failure.

Acceptance:

- A generated operator that is fast on canary but slow on medium production cases is rejected before screening.
- Perf guard logs include case name, candidate time, champion time, ratio, and timeout.

## P3: Solver Baseline Stability

Status: `MoveOrder` stale-assignment crash fixed in `e8e3673`.

The production sanity run exposed a baseline crash:

```text
MoveOrder -> ValueError: list.remove(x): x not in list
```

Root cause: `assignment` and `vehicle.order_ids` could drift in VNS state. `MoveOrder` now repairs assignment from vehicle contents and no-ops when the source is unsafe.

Acceptance:

- `instance_prod_scr_ms01.json seed=42` runs successfully under the baseline registry.
- Production screening no longer shows repeated champion-side invalid pairs from `MoveOrder`.

## P4: Performance Feedback To LLM

Add runtime feedback to hypothesis context:

- slow case names
- timeout case names
- failed operator file
- candidate/champion elapsed ratio
- guidance to reduce neighborhood size

This should be separate from objective feedback. A candidate can have promising objective direction but still be rejected due to runtime.

Acceptance:

- LLM retries after timeout mention the timeout and propose bounded variants.
- Prompt contains both objective opportunity and performance risk.

## P5: Quality Evaluation Harness

The final quality comparison should also be checkpointed per case.

The shared output contract is tracked in:

- [v0.4-evidence-harness.md](v0.4-evidence-harness.md)

Required:

- per-case checkpoint rows
- per-case timeout
- slowest-case summary
- objective summary vs baseline and CPLEX
- timeout cases counted in final quality result

Acceptance:

- A slow production champion can be diagnosed without losing completed rows.
- Final report distinguishes:
  - objective improvement
  - runtime regression
  - infeasible output
  - timeout

## v0.4 Exit Criteria

v0.4 can close when:

- Production-only rerun completes with no invalid-pair promotions.
- Every validation/frozen promotion has complete evidence.
- Slow/unbounded generated operators are rejected before promotion.
- Final champion quality comparison includes runtime and timeout status.
- Documentation clearly distinguishes:
  - framework closure
  - objective quality
  - runtime acceptability

## Relationship To v1.0

The earlier roadmap placed the second real problem in v1.0. That boundary has
changed: CVRP will be introduced in v0.4 because it is the most direct way to
test the adapter boundary and performance-aware gates together.

v1.0 should build on the v0.4 CVRP evidence and focus on larger cross-problem
validation, cleaner operational tooling, and mechanism studies. FCMCNF/Benders
remains useful for a later v1.x track where lower-bound and decomposition
interfaces become the main research target.
