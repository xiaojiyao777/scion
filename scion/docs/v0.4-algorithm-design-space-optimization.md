# Scion v0.4 Algorithm Design Space Optimization

*Date: 2026-05-05*
*Status: design source for the next v0.4 development slice*

## Purpose

This document fixes a repeated planning failure in the v0.4 work: "algorithm
design space optimization" was discussed several times, but implementation kept
collapsing back to a narrower surface:

```text
generate an operator file -> append it after the incumbent heuristic -> evaluate
```

That was enough for the warehouse-delivery research object in v0.3 because its
operator pool is a large part of the heuristic algorithm. It is not enough for
CVRP because the current CVRP package runs a strong repo-local ALNS+VNS baseline
first, then lets Scion-generated operators polish the already-improved solution.

The design target is therefore:

```text
operator design optimization
is a subset of
heuristic algorithm design space optimization
```

Scion must remain the generic research-governance framework. Problem packages
must expose the algorithm surfaces that are safe for Scion to govern.

## Git History Audit

The current `v0.4-dev` history shows why the implementation is still too narrow.

- `bc63efb feat(v0.4): harden campaign framework and cvrp runtime`
  added the CVRP package, formal/controlled fixtures, campaign decomposition,
  runtime evidence, and the solver wrapper. The CVRP wrapper architecture is:
  `vrp/src ALNS+VNS baseline -> bounded post-baseline Scion operators`.
- `e6e779d docs(v0.4): design cvrp research surfaces`
  correctly identified that post-baseline operators are too narrow, but scoped
  the implementation to a first slice.
- `c8bbb9c feat(v0.4): add research surface policy support`
  implemented the first non-operator research surface, `search_policy`, but that
  surface only controls baseline time fraction, post-baseline operator rounds,
  and whether post-baseline operators run.
- `64f6c97`, `1aa9bac`, `75e4310`, and `6b8a043` mainly hardened governance,
  evidence, context, novelty, runtime, and campaign composition.
- The current uncommitted P0 repair slice improves policy novelty, summary
  reason codes, promotion lineage, and max-round closeout. It does not add
  algorithm-level CVRP surfaces.

The conclusion is specific:

```text
Scion core can now govern declared research surfaces,
but the CVRP problem package currently declares only one narrow policy surface
plus post-baseline operator surfaces.
```

## Architectural Principle

Scion core owns governance:

- branch lifecycle;
- contract gate;
- verification gate;
- protocol evaluation;
- runtime/evidence completeness;
- decision finalization;
- promotion/abandon lineage;
- stale/reconcile/broadcast control.

Problem packages own algorithm exposure:

- editable surfaces;
- surface-specific contracts;
- prompt semantics;
- solver integration;
- runtime audit fields;
- evidence interpretation;
- feasibility/objective checks.

Scion must not learn CVRP-specific ALNS, VNS, destroy/repair, or routing logic.
CVRP must expose those algorithmic levers through declared, bounded,
auditable surfaces.

## Current Surface Model

`ProblemSpecV1.research_surfaces` currently supports:

```text
name
kind: operator | policy | config
description
target_files
prompt_hint
required_functions
create_new_allowed
modify_allowed
remove_allowed
```

This is a good generic foundation. The problem is not the schema. The problem is
the CVRP package's declared surfaces:

```text
route_local      -> post-baseline operator file
route_pair       -> post-baseline operator file
ruin_recreate    -> post-baseline operator file
search_policy    -> baseline/operator budget and post-baseline enablement
```

These surfaces do not control the main CVRP algorithm.

## Required Design Shift

For algorithm design space optimization, a research surface should correspond
to a stable algorithm component or policy boundary, not just to a Python file
pattern.

A surface must answer:

- What part of the heuristic algorithm does this surface control?
- When is it invoked in the solver pipeline?
- What inputs and outputs are allowed?
- How is it bounded for runtime?
- How are invalid outputs detected?
- Which runtime audit fields prove it actually executed?
- Which evidence fields let future prompts learn from it?

Operator files remain valid surfaces, but they should no longer be treated as
the universal default for every combinatorial optimization problem.

## Target Surface Types

### Operator Surface

Purpose: local or component-level transformation.

Examples:

- warehouse order move/swap/merge operators;
- CVRP route-local or route-pair polishing operators;
- controlled fixtures where the operator itself is the algorithm component.

Governance:

- class/function signature contract;
- feasibility and determinism verification;
- runtime budget and no-op audit;
- registry integration.

### Policy Surface

Purpose: bounded solver behavior decisions without arbitrary solver edits.

Examples:

- baseline time fraction;
- operator round limit;
- component enable/disable switches;
- instance-size-specific schedule selection.

Governance:

- required module-level functions;
- return-type checks;
- clamp or fail-closed rules;
- runtime audit of loaded policy and decisions.

### Portfolio Surface

Purpose: choose or weight predeclared algorithm components.

Examples:

- CVRP neighborhood portfolio;
- destroy/repair component mix;
- local-search schedule;
- restart/perturbation policy.

Governance:

- LLM edits a bounded config/policy file, not solver internals;
- solver consumes only whitelisted component names and numeric ranges;
- invalid component names or out-of-range weights are runtime audit failures;
- evidence records component attempts, accepted moves, and stop reasons.

### Construction Surface

Purpose: alter initial solution construction before the main search.

Examples:

- nearest-neighbor tie-breaking;
- sweep/savings seed construction;
- capacity-aware customer ordering;
- route count protection heuristics.

Governance:

- bounded function interface;
- must produce a feasible `CvrpSolution`;
- no benchmark answer reads;
- runtime audit includes construction mode, failures, and elapsed time.

### Acceptance/Restart Surface

Purpose: control how the heuristic escapes local optima.

Examples:

- acceptance thresholds;
- restart cadence;
- perturbation intensity;
- stagnation-triggered diversification.

Governance:

- deterministic policy functions;
- bounded numeric ranges;
- cannot override objective correctness;
- promotion still depends on protocol evidence, not policy claims.

## CVRP Next-Slice Design

The next CVRP slice should add problem-owned algorithm surfaces while leaving
Scion core generic.

### New Files

Suggested first files:

```text
scion/scion/problems/cvrp/policies/neighborhood_portfolio.py
scion/scion/problems/cvrp/policies/destroy_repair_policy.py
scion/scion/problems/cvrp/policies/construction_policy.py
scion/scion/problems/cvrp/policies/acceptance_restart_policy.py
```

The first implementation does not need all four. The recommended minimum is:

```text
neighborhood_portfolio.py
construction_policy.py
```

because they expose algorithm levers before and inside the main search rather
than only after the baseline.

### `neighborhood_portfolio.py`

Required functions:

```python
def enabled_components(instance, time_limit_sec):
    return ["relocate", "swap", "two_opt_star", "or_opt", "ruin_repair"]

def component_weights(instance, time_limit_sec):
    return {
        "relocate": 1.0,
        "swap": 0.8,
        "two_opt_star": 0.7,
        "or_opt": 0.5,
        "ruin_repair": 0.4,
    }

def candidate_limits(instance, time_limit_sec):
    return {
        "max_routes": 40,
        "max_customers_per_route": 120,
        "top_k_neighbors": 12,
    }
```

Solver contract:

- ignore unknown components only if recorded as a policy error, otherwise fail
  closed;
- clamp numeric values to declared safe ranges and count the clamp;
- record component attempts, accepted moves, runtime, and stop reason;
- expose component-level evidence in raw metrics.

### `construction_policy.py`

Required functions:

```python
def construction_mode(instance, time_limit_sec):
    return "baseline_default"

def construction_bias(instance, time_limit_sec):
    return {
        "savings_weight": 1.0,
        "nearest_weight": 1.0,
        "demand_weight": 0.2,
    }
```

Allowed modes should be predeclared by the problem package, for example:

```text
baseline_default
nearest_capacity
sweep_seed
savings_seed
cluster_then_route
```

The LLM must not write arbitrary construction code in the first slice. It tunes
a bounded policy consumed by problem-owned construction implementations.

### `acceptance_restart_policy.py`

This can be deferred until after the first portfolio/construction slice. It is
more dangerous because acceptance policy can create subtle overfitting and
runtime regressions.

When implemented, it must not change promotion logic. It may only affect solver
search dynamics inside a candidate run.

## Scion Core Changes Required

Keep core changes small and generic.

Required:

- Surface schema should already support `kind: policy` and reserved
  `kind: config`.
- ContractGate should validate policy/config required functions and target file
  permissions generically.
- ContextManager should render each surface's interface through the adapter.
- Proposal guidance should say "research surface" rather than "operator
  category".
- EvidenceRecorder should keep exposing runtime and decision reason codes.

Potential addition:

- Add `kind: portfolio` if `policy/config` is not expressive enough.

Avoid:

- CVRP-specific checks in Scion core.
- Allowing LLM to edit `solver.py`.
- Letting policy text override deterministic gates.

## Context Requirements

Hypothesis context for algorithm surfaces must include:

- solver pipeline stage where the surface is invoked;
- current champion algorithm configuration;
- recent failures grouped by surface and mechanism;
- accepted-move rate and no-op rate;
- runtime ratio/regression rate;
- case patterns where the surface helped or failed;
- current frozen/validation budget state;
- explicit instruction that weak/no-op surfaces should be abandoned or shifted.

The current context already includes some runtime feedback, but it needs to be
surface-specific. For CVRP, "operator_attempts=20, accepted=0" should route the
LLM away from another post-baseline operator and toward a portfolio or
construction policy surface.

## Evidence Requirements

Every algorithm surface must prove three things in raw metrics:

1. It loaded.
2. It executed.
3. It changed either search behavior or objective/runtime evidence.

Minimum fields:

```json
{
  "surface_name": "neighborhood_portfolio",
  "surface_loaded": true,
  "surface_errors": 0,
  "surface_config": {},
  "component_attempts": {},
  "component_accepted": {},
  "component_runtime_ms": {},
  "surface_stop_reason": "no_improvement_round"
}
```

For construction surfaces:

```json
{
  "construction_surface_loaded": true,
  "construction_mode": "savings_seed",
  "construction_elapsed_ms": 123.4,
  "construction_routes": 12,
  "construction_distance": 12345.6
}
```

This evidence must be available to:

- protocol summaries;
- campaign summaries;
- prompt runtime feedback;
- post-run audit scripts.

## Promotion Semantics

Promotion remains unchanged:

```text
contract pass
-> verification pass
-> screening pass
-> validation pass
-> frozen pass
-> promote
```

Algorithm surfaces do not get special promotion shortcuts. They only give the
LLM meaningful levers to generate candidates.

Runtime remains a first-class promotion constraint:

- incomplete evidence cannot promote;
- candidate runtime failures cannot be treated as ties;
- runtime regressions can veto or downgrade decisions through the existing
  decision layer.

## Plateau Semantics

The VRP run shows a need for a plateau rule that is generic but surface-aware.

Candidate generic rule:

```text
if N consecutive candidates on a surface have:
  accepted_move_rate == 0
  and win_rate < screening_min
  and no validation advancement
then:
  lower priority for that surface
  force exploration of a different declared surface
```

This belongs in Scion governance because it is surface-generic. The metric names
must come from problem evidence, not hardcoded CVRP logic.

## Acceptance Criteria

Design acceptance:

- Docs clearly state that operator design is a subset of algorithm design.
- Current git history no longer obscures what was implemented versus deferred.

Framework acceptance:

- Problem packages can declare multiple surface kinds.
- Scion core remains problem-agnostic.
- Surface-aware novelty, context, and contract checks work for singleton policy
  files.

CVRP acceptance:

- At least one new algorithm surface affects solver runtime audit fields.
- A controlled CVRP campaign proves non-operator surface edits can change search
  behavior.
- Formal CVRP screening shows nontrivial behavior: not all tie/no-op, and
  accepted-move rate is visible.

Regression acceptance:

- Full Scion tests pass.
- Warehouse campaign path remains functional.
- Raw CVRPLIB case contents are not copied into docs or tests.

## Non-Goals

- Do not make Scion a CVRP solver.
- Do not move `vrp/src` internals into Scion core.
- Do not let the LLM edit arbitrary solver internals.
- Do not use BKS/gap for promotion.
- Do not weaken frozen holdout discipline to get promotions.

## Development Rule Going Forward

Every future "algorithm design space" task must name:

```text
surface
problem package owner
target file(s)
solver invocation point
contract checks
runtime audit fields
prompt/context fields
tests
```

If a task cannot fill these fields, it is not implementation-ready and should
remain a design note.
