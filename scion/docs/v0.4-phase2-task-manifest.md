# Scion v0.4 Phase 2 Task Manifest

*Date: 2026-05-01*
*Parent: v0.4-design.md*
*Branch: v0.4-dev*

## Phase 2 Goal

Phase 2 makes Scion's runtime verification and feedback boundaries
adapter-native before CVRP is added.

The goal is not to integrate CVRP yet. The goal is to remove the remaining
warehouse-shaped verification assumptions that would make CVRP appear to work
without route-native solution checks.

## Scope

Phase 2 owns four framework tasks:

1. Make `V5_solution_consistency` use `ProblemAdapter.check_solution_consistency`
   when an adapter is available.
2. Promote runtime guard facts from free-text detail into structured evidence
   metadata.
3. Add bounded runtime feedback to hypothesis context as proposal guidance only.
4. Decide the lifecycle of `EvaluationPipeline` after Phase 1 extraction.

## Non-Goals

- No CVRPLIB raw instance parsing in Scion tests.
- No change to `DecisionEngine` promotion thresholds.
- No BKS/gap-based promotion logic.
- No final quality harness implementation in this phase.

## Scope Amendment

After P2-01 through P2-05 passed, Phase 2 was extended with two small
readiness slices:

1. Add a tiny synthetic CVRP adapter/protocol/campaign smoke path, without
   reading raw CVRPLIB benchmark files.
2. Add a thin `ProblemSpecV1 -> ProblemSpec` bridge so CVRP campaign wiring is
   derived from one authoritative problem schema instead of hand-built dual
   specs.

## P2-01: Adapter-Native V5

**Priority**: P0

**Write scope**

- `scion/scion/verification/state_mutation.py`
- `scion/scion/verification/gate.py`
- `scion/scion/tests/test_verification.py`

**Purpose**

`V5_solution_consistency` must be problem-native. When a `ProblemAdapter` is
available, V5 must call:

```text
adapter.load_instance
adapter.deserialize_solver_output
adapter.check_solution_consistency
```

Legacy warehouse-shaped assignment/vehicle consistency remains only as a
fallback for old non-adapter runtime verification.

**Acceptance**

- Adapter consistency failure fails closed.
- Strict adapter-backed runtime verification includes V5.
- Legacy V5 tests continue passing.

## P2-02: Runtime Guard Metadata

**Priority**: P0

**Write scope**

- `scion/scion/core/models.py`
- `scion/scion/core/features.py`
- `scion/scion/verification/perf_guard.py`
- `scion/scion/core/evidence_recorder.py`
- focused tests

**Purpose**

`V9_perf_guard` detail text is useful for humans but insufficient for audit and
final evidence. It must also emit structured metadata:

```text
case_id
candidate_ms
champion_ms
ratio
limit_ratio
candidate_timeout
champion_timeout
candidate_error_category
champion_error_category
timeout_sec
```

`DecisionFeatures` may carry only bounded structured runtime facts:

```text
runtime_guard_passed
runtime_guard_ratio
runtime_guard_timeout
```

No free-text runtime detail may enter Decision.

**Acceptance**

- Perf guard tests assert metadata.
- `SafeFeatureExtractor` extracts runtime facts without violating
  `_validate_no_free_text`.
- Evidence lineage payloads include runtime guard metadata.

## P2-03: Runtime Feedback Context

**Priority**: P1

**Write scope**

- `scion/scion/proposal/context_manager.py`
- `scion/scion/proposal/engine.py`
- context/prompt tests

**Purpose**

Runtime failures should guide future hypotheses, but they must stay outside the
Decision Layer.

Hypothesis context may include bounded runtime feedback from prior verification
failures:

```text
Recent runtime guard failures:
- round
- target file or locus
- slow case / ratio / timeout detail
- bounded-neighborhood guidance
```

**Acceptance**

- Runtime feedback appears in hypothesis system blocks when present.
- Feedback is absent when there are no V9 failures.
- Feedback is derived from verification history, not validation/frozen
  case-level results.

## P2-04: EvaluationPipeline Lifecycle Decision

**Priority**: P1

**Decision**

Keep `EvaluationPipeline` constructed per evaluation for now.

Reason:

- existing tests and campaign harnesses still mutate `_experiment_protocol`
  directly;
- `CampaignManager` still owns protocol progress callbacks and budget state;
- making the pipeline long-lived before `CampaignLoop`/protocol ownership is
  extracted would add lifecycle coupling without reducing current risk.

Revisit this after `CampaignLoop` or `DecisionCoordinator` extraction.
