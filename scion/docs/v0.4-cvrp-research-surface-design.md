# Scion v0.4 CVRP Research Surface Design

*Date: 2026-05-05*
*Branch: v0.4-dev*

## Context

The 2026-05-05 short CVRP Sonnet diagnostic showed that the framework boundary
is now mostly clean, but the CVRP research surface is too narrow:

- 4/4 generated candidates reached screening and failed closed.
- Contract, verification, canary, and runtime evidence were complete.
- Prompt traces no longer showed warehouse taxonomy contamination.
- The generated CVRP code was safe but ineffective: 96 operator attempts, 0
  accepted moves, all stopping with `operator_stop_reason=no_improvement_round`.

This is not only "weak LLM code". The deeper issue is that Scion currently
exposes only post-baseline registry operators for CVRP. The real `.vrp` path
first runs the repo-local `vrp/src` ALNS+VNS baseline, then generated operators
try to polish the already-improved incumbent.

For warehouse delivery in v0.3, the operator pool was effectively the heuristic
algorithm. For CVRP, that equivalence does not hold. A strong CVRP heuristic is
a coupled search system: construction, destroy/repair, local neighborhoods,
adaptive weights, acceptance policy, restart/perturbation, and budget allocation
interact. Appending one more local operator after ALNS+VNS is a much narrower
intervention.

## Design Position

Scion's research object is the heuristic algorithm exposed by a problem package,
not "operator files" as a universal abstraction.

Operators remain one valid research surface, but they are not the only one.
The framework should search over problem-owned **research surfaces**:

```text
problem hypothesis
-> branch
-> edit one declared research surface
-> contract / verification / protocol
-> screening / validation / frozen
-> promote / abandon
```

The Scion core stays problem-agnostic. It should not learn CVRP-specific ALNS,
VNS, or routing logic. The problem package declares which surfaces are editable,
how they are described to the LLM, and how the solver consumes them.

## Non-Negotiable Boundaries

- Do not allow arbitrary LLM edits to `solver.py`.
- Do not move CVRP-specific search policy into `scion.core`.
- Do not make Decision read free-text policy explanations.
- Runtime and complete-evidence governance remains default.
- Final BKS/gap remains report-only; promotion still uses adapter objectives.
- Frozen holdout promotion discipline is unchanged.
- Backward compatibility: problems without `research_surfaces` continue using
  the existing operator-category behavior.

## Research Surface Model

Add an optional `research_surfaces` field to `ProblemSpecV1`.

```yaml
research_surfaces:
  - name: route_local
    kind: operator
    description: Intra-route bounded improvement operators.
    target_files:
      - operators/*.py
    prompt_hint: Use only bounded same-route neighborhoods.

  - name: search_policy
    kind: policy
    description: Baseline/operator budget and scheduling policy.
    target_files:
      - policies/search_policy.py
    prompt_hint: Tune baseline time fraction and post-baseline operator budget.
    required_functions:
      - baseline_time_fraction
      - max_operator_rounds
      - enable_post_baseline_operators
```

Initial schema:

```text
ResearchSurfaceSpec
- name: str
- kind: "operator" | "policy" | "config"
- description: str
- target_files: list[str]
- prompt_hint: str
- required_functions: list[str] = []
- create_new_allowed: bool = true
- modify_allowed: bool = true
- remove_allowed: bool = false
```

Only `operator` and `policy` need implementation in the first slice. `config`
is reserved so the schema does not need to change again when budget/config
surfaces are introduced.

## Compatibility Mapping

`CampaignManager`, `ContractGate`, and proposal schemas still use the legacy
field name `change_locus`. For v0.4, treat `change_locus` as "research locus",
not necessarily an operator category.

Bridge behavior:

- If `ProblemSpecV1.research_surfaces` is present, legacy
  `ProblemSpec.operator_categories` becomes the list of surface names.
- If absent, keep the old mapping from `operator_interface.categories`.
- Attach `research_surfaces` to the legacy spec for prompt/context helpers.

This keeps old tests and old problem packages working while allowing CVRP to
declare non-operator surfaces.

## CVRP First Slice: Search Policy Surface

Add this checked-in file:

```text
scion/scion/problems/cvrp/policies/search_policy.py
```

Default behavior must match current solver behavior:

```python
def baseline_time_fraction(instance, time_limit_sec):
    return 0.8

def max_operator_rounds(instance, time_limit_sec):
    return 20

def enable_post_baseline_operators(instance, time_limit_sec):
    return True
```

The CVRP solver auto-loads `policies/search_policy.py` from the active
workspace if it exists. A candidate can then modify this policy without editing
`solver.py`.

The policy may control only bounded, auditable levers:

- baseline time fraction, clamped to a safe range such as `[0.2, 0.95]`;
- max post-baseline operator rounds, clamped to `[0, 20]`;
- whether to run post-baseline operators at all.

Policy load failures, invalid return types, and out-of-range values should be
runtime audit failures, not silent ties.

Runtime JSON should include:

```json
{
  "policy_path": "policies/search_policy.py",
  "policy_loaded": true,
  "policy_errors": 0,
  "baseline_time_fraction": 0.8,
  "operator_round_limit": 20,
  "post_baseline_operators_enabled": true
}
```

`runtime_audit_failure_from_runtime()` should fail closed when
`policy_errors > 0`.

## Prompt and Context Changes

Prompt language should stop saying "operator category" as the universal
abstraction. Use:

- `research surface`;
- `change_locus`;
- `operator surface` only when the surface kind is `operator`.

Hypothesis prompt should include a `Research Surfaces` block:

```text
## Research Surfaces
- route_local [operator]: Intra-route bounded improvement operators.
  targets: operators/*.py
- route_pair [operator]: Inter-route bounded relocate/swap/exchange operators.
  targets: operators/*.py
- ruin_recreate [operator]: Bounded destroy/repair operators.
  targets: operators/*.py
- search_policy [policy]: Baseline/operator budget and scheduling policy.
  targets: policies/search_policy.py
```

Action guidance must be surface-aware:

- `modify` is valid for `search_policy` when `policies/search_policy.py`
  exists, even if the champion has no operator files.
- `remove` should not be suggested for required singleton policy files.
- `create_new` remains valid for operator surfaces that allow new files.

Code context should render the active surface interface:

- operator surfaces: existing `CvrpOperator.execute(...)` interface;
- `search_policy`: module-level policy functions, no class required.

Contract validation should be surface-aware:

- declared operator surfaces must still define an operator class with the
  problem-specific `execute(...)` signature;
- declared policy surfaces must satisfy their problem-declared
  `required_functions` contract and use module-level functions;
- action and target checks should enforce each surface's `create_new_allowed`,
  `modify_allowed`, `remove_allowed`, and `target_files` metadata.

## Acceptance Criteria

Framework/schema:

- `ProblemSpecV1` accepts optional `research_surfaces`.
- Existing problem specs without `research_surfaces` still load unchanged.
- CVRP bridge exposes `route_local`, `route_pair`, `ruin_recreate`, and
  `search_policy` as valid `change_locus` values.

CVRP runtime:

- Default `policies/search_policy.py` preserves current behavior.
- A modified policy can change `baseline_time_fraction` and
  `max_operator_rounds` without editing `solver.py`.
- Runtime output records policy fields.
- Invalid policy output is a runtime audit failure.

Prompt/context:

- Hypothesis context includes a research-surface block.
- The hypothesis prompt says "research surface" rather than only "operator
  category".
- Empty operator pool does not block `modify` guidance for singleton policy
  files.

Tests:

- ProblemSpec/bridge tests for `research_surfaces`.
- CVRP solver test proving policy changes baseline budget metadata.
- Runtime audit test for `policy_errors`.
- Context test proving `search_policy` appears and `modify` is allowed when
  the policy file exists.
- Full Scion test suite remains green.

## Out Of Scope For First Slice

- Editing `vrp/src` directly.
- Arbitrary YAML config generation.
- Full ALNS destroy/repair policy replacement.
- LLM-controlled acceptance criteria that bypass adapter objective comparison.
- Any claim that CVRP quality improves before a new controlled campaign proves
  it.

## Next Design Step After First Slice

If search-policy modification still produces mostly no-op results, add a
second problem-owned surface for the internal CVRP heuristic portfolio:

```text
policies/neighborhood_portfolio.py
```

That surface should control a bounded menu of destroy/repair/local-search
components rather than letting the LLM write arbitrary solver logic.
