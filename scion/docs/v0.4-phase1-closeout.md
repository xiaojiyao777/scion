## What Was Extracted

### Evaluation

`CampaignManager._evaluate()` now delegates canary/protocol execution and
structured feature extraction to `EvaluationPipeline`.

Preserved behavior:

- screening, validation, and frozen stage selection;
- screening/validation expand counters;
- experiment budget accounting;
- DecisionEngine still receives only structured `DecisionFeatures`;
- validation/frozen per-case feedback stays hidden from proposal context while
  `case_ids`, `seed_set`, and `raw_metrics_ref` remain available for evidence
  reconstruction.

### Promotion

`CampaignManager._prepare_promoted_champion()` now delegates snapshot copying,
freezing, registry reading, weight reading, and hash computation to
`PromotionService.prepare()`.

Preserved behavior:

- promotion prepare and commit remain separate;
- failed prepare does not commit branch state or lineage;
- `_commit_promote_plan()` still owns stagnation reset, search memory, champion
  store persistence, stale marking, and weight-opt trigger.

### Evidence

`CampaignManager` now delegates these artifact boundaries to
`EvidenceRecorder`:

- step append/search-memory update boundary;
- `status.json` writing;
- protocol progress status updates;
- lineage event and decision payload construction;
- `campaign_summary.json` writing.

Preserved behavior:

- `status.json` retains `last_result` and `current_progress`;
- `campaign_summary.json` retains existing top-level and step fields;
- lineage writes continue through `LineageRegistry`;
- final evidence refs are supported as a future top-level summary extension
  without changing step schema.

## Behavior Changes

No intentional campaign behavior change was introduced.

One test was updated to match the current weight-optimization architecture:

- `TestPromoteWeightOptimizationHook.test_on_promote_runs_weight_optimization`

It now asserts that promotion triggers `AsyncWeightOptCoordinator` rather than
the older direct `_run_weight_optimization()` call. This aligns the test with
the existing v0.3 coordinator-based implementation.

## Remaining `campaign.py` Responsibilities

`campaign.py` still owns substantial orchestration:

- LLM hypothesis/code/fix generation;
- workspace setup and patch application;
- failure lifecycle;
- branch state transitions;
- decision finalization;
- promotion commit side effects;
- weight optimization event drain;
- stagnation and early-stop handling;
- branch/hypothesis store coordination.

The file is smaller after Phase 1 but remains the integration hub.

## Phase 2 Prerequisites

Before CVRP integration, the next framework tasks should be:

1. Make V5/state-shape checks adapter-native.
2. Add runtime guard evidence fields to promotion records.
3. Add runtime feedback summaries to proposal context without exposing them to
   Decision.
4. Resolve or explicitly classify the stale weight-update full-suite failures.
5. Decide whether `EvaluationPipeline` should become long-lived on
   `CampaignManager` or remain constructed per evaluation until more dependencies
   are extracted.
