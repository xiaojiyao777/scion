# Scion v0.4 Phase 1 Task Manifest

*Date: 2026-05-01*
*Parent: v0.4-design.md*
*Branch: v0.4-dev*

## Phase 1 Goal

Phase 1 extracts the first stable service boundaries from `CampaignManager`
without changing campaign behavior.

The goal is not to add CVRP yet. The goal is to make the campaign core easier
to reason about before adding CVRP, final evidence harnesses, and stronger
runtime governance.

Phase 1 target extractions:

1. `EvaluationPipeline`
2. `PromotionService`
3. `EvidenceRecorder`

`campaign.py` is the integration point and should be edited by only one owner at
a time.

## P1-01: EvaluationPipeline Service Shell

**Priority**: P0

**Suggested worker ownership**

- Owns new files:
  - `scion/scion/core/evaluation_pipeline.py`
  - `scion/scion/tests/unit/core/test_evaluation_pipeline.py`
- Does not edit `campaign.py` during parallel service-shell work.

**Purpose**

Move evaluation orchestration behind a focused interface so `CampaignManager`
does not directly own contract, verification, protocol execution, and
`DecisionFeatures` assembly.

**Initial interface**

```python
@dataclass(frozen=True)
class EvaluationRequest:
    branch_id: str
    branch_state: BranchState
    candidate_workspace: str
    champion_workspace: str
    hypothesis_action: str
    expand: bool = False
    expand_round: int = 0


@dataclass(frozen=True)
class EvaluationOutcome:
    protocol_result: ProtocolResult | None
    decision_features: DecisionFeatures
    verification_detail: str | None
    raw_metrics_ref: str | None
```

This interface may be refined during implementation, but it must preserve the
separation:

```text
evaluation facts -> DecisionFeatures -> DecisionEngine
```

**Extraction target**

Start from the logic currently concentrated around:

- `CampaignManager._evaluate`
- `_build_verification_detail`
- `scion.core.features.build_decision_features`
- `ExperimentProtocol.run_canary`
- `ExperimentProtocol.run_experiment`

**Rules**

- No free-text LLM context may enter `DecisionFeatures`.
- Validation/frozen per-case feedback exposure remains blocked.
- Existing protocol raw metrics references must be preserved.
- Existing incomplete-evidence fail-closed behavior must not change.

**Tests**

- Unit test a mocked protocol pass from screening to `DecisionFeatures`.
- Unit test a verification failure produces failed features and detail.
- Unit test validation/frozen outcome exposes only aggregate protocol summary.

**Acceptance**

- Service tests pass independently.
- The service can be integrated without changing public campaign semantics.

## P1-02: PromotionService Service Shell

**Priority**: P0

**Suggested worker ownership**

- Owns new files:
  - `scion/scion/core/promotion_service.py`
  - `scion/scion/tests/unit/core/test_promotion_service.py`
- Does not edit `campaign.py` during parallel service-shell work.

**Purpose**

Move champion snapshot preparation and commit logic out of `CampaignManager`.

**Initial interface**

```python
@dataclass(frozen=True)
class PromotionRequest:
    branch_id: str
    candidate_workspace: str
    champion_version: int
    champion_weight_revision: int


@dataclass(frozen=True)
class PromotionPlan:
    candidate_snapshot_ref: str
    new_champion_version: int
    registry_hash: str | None
    weight_revision: int
```

**Extraction target**

Start from:

- `CampaignManager._prepare_promoted_champion`
- `CampaignManager._commit_promote_plan`
- pool registry sync paths used during promotion
- champion snapshot persistence
- stale marking after promotion

**Rules**

- Prepare and commit remain separate phases.
- If snapshot preparation fails, no branch state or lineage commit is written.
- Promotion commits must preserve current lineage and hypothesis status
  semantics.
- Weight optimization trigger remains outside the immutable champion commit
  itself, or is represented as a post-commit event.

**Tests**

- Successful prepare returns an immutable promotion plan.
- Failed prepare does not mutate branch/champion state.
- Successful commit updates champion version and registry reference.
- Stale marking skips frozen branches as current behavior requires.

**Acceptance**

- Promotion service tests pass independently.
- Existing full promote path tests pass after integration.

## P1-03: EvidenceRecorder Service Shell

**Priority**: P0

**Suggested worker ownership**

- Owns new files:
  - `scion/scion/core/evidence_recorder.py`
  - `scion/scion/tests/unit/core/test_evidence_recorder.py`
- May coordinate with existing:
  - `scion/scion/core/status_reporter.py`
- Does not edit `campaign.py` during parallel service-shell work.

**Purpose**

Move status, step recording, lineage event construction, and campaign summary
artifact writing behind one evidence boundary.

Phase 1 does not implement the v0.4 final quality harness. It prepares the
place where final evidence artifacts will later be attached.

**Extraction target**

Start from:

- `CampaignManager._record_step`
- `CampaignManager._record_step_lineage`
- `CampaignManager._write_status`
- `CampaignManager._write_campaign_summary`
- protocol progress status updates

**Rules**

- Raw protocol metrics files remain the source for stage evidence.
- `status.json` and `campaign_summary.json` must remain backward compatible.
- New recorder must preserve append-only lineage event semantics where the
  existing stores support append-only facts.
- Mutable views must be rebuildable from persisted facts and artifact refs.

**Tests**

- Step record writes preserve current summary fields.
- Protocol progress updates preserve raw metrics refs.
- Promotion event records include branch id, decision, reason codes, champion
  version, and metrics refs.
- Recorder can attach future final evidence refs without changing step schema.

**Acceptance**

- Evidence recorder tests pass independently.
- Existing lineage/status/summary tests pass after integration.

## P1-04: Campaign Integration Pass 1

**Priority**: P0

**Owner**: main session or one integration worker

**Write scope**

- May edit:
  - `scion/scion/core/campaign.py`
  - tests that must be adjusted for moved boundaries
- Must not refactor unrelated behavior.

**Purpose**

Wire the three service shells into `CampaignManager` while preserving current
behavior.

**Rules**

- This is a behavior-preserving extraction.
- No CVRP files are added in this pass.
- No protocol threshold changes are made in this pass.
- No final evidence harness implementation is added in this pass.
- If a behavior change appears necessary, stop and record it as Phase 2+.

**Acceptance**

- Existing campaign promote path still passes:
  - screening pass -> ready validation;
  - validation pass -> ready frozen;
  - frozen pass -> promote.
- Existing early-stop queue-delay behavior still passes.
- Existing incomplete-evidence protocol tests still pass.
- Existing status/lineage tests still pass.

## Subagent Development Model

Main session responsibilities:

- Own design decisions and acceptance gates.
- Own task decomposition and worker write scopes.
- Integrate or review changes before marking tasks complete.
- Run or delegate final test gates.
- Keep docs and closeout notes current.

Worker agent responsibilities:

- Work only inside assigned file/module scope.
- Do not revert unrelated edits.
- Do not edit `campaign.py` unless explicitly assigned as integration owner.
- Add focused tests for the new boundary.
- Report changed files and test commands in final response.

Recommended parallel assignment:

| Agent | Role | Write scope |
|---|---|---|
| Worker A | `EvaluationPipeline` | `core/evaluation_pipeline.py`, its unit tests |
| Worker B | `PromotionService` | `core/promotion_service.py`, its unit tests |
| Worker C | `EvidenceRecorder` | `core/evidence_recorder.py`, its unit tests |
| Verifier | regression tests and risk review | read-only unless asked |
| Main / Integration Worker | `campaign.py` wiring | `core/campaign.py` and adjusted tests |

This keeps development parallel where it is safe and serial where shared state
is risky.
