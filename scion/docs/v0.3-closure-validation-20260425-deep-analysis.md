## 1. Executive Conclusion

This experiment is a **successful v0.3 framework closure validation**, but it is
**not a successful restoration of W16 promotion cadence**.

The system behaved normally as an engineered Scion campaign stack:

- all 6 campaigns completed with exit code 0;
- no launcher crash, campaign traceback, or missing terminal artifact was found;
- `status.json`, `campaign_summary.json`, `validation_runner_summary.json`,
  `scion.db`, and `metrics/*.json` were present and internally auditable;
- contract / verification / protocol / decision / promotion / weight revision
  events were recorded in lineage;
- CPLEX-infeasible generated instances were absent from the actual closure
  metrics.

The search did produce real algorithmic improvements:

- 7 structural promotions passed frozen holdout;
- every frozen promotion was a 12/12 pair win on the frozen gate set;
- a post-run fixed frozen evaluation found every final champion beating the
  baseline on all 12 case/seed pairs;
- median final improvement over baseline ranged from **+9 to +41
  subcategory_splits** on the frozen gate set.

However, the search did not meet the pre-registered cadence target:

- Sonnet averaged **1.0** promotions vs target **>= 2.0**.
- GPT-mini averaged **1.33** promotions vs target **>= 1.5**.
- Most campaigns stopped shortly after the first promotion, leaving the system
  with a valid but shallow final champion.

The most important new finding is that **CPLEX cannot currently be used as a
strict exact-gap certificate for these reports**. The post-run sanity check found
baseline and final Scion solutions that are feasible under the current oracle but
better than several CPLEX `optimal` f1 values. This must be treated as a
MILP/model/oracle/data-version consistency issue, not as "Scion beat the exact
optimum."

## 2. Experiment Scope vs Design Scope

This run is **not** the full 18-campaign W16 matrix from
`design/validation-prereg.md`.

It is the synthetic 6-campaign closure matrix introduced by
`docs/v0.3-closure-validation-prereg.md` after the post-opt regression:

| Dimension | Value |
|---|---|
| Models | `claude-sonnet-4-6`, `gpt-5.4-mini` |
| Variant | `warehouse_synthetic` only |
| Seeds | 11, 29, 47 |
| Max rounds | 100, early-stop enabled |
| Concurrency | 2 |

Therefore this experiment can answer:

- whether the v0.3 campaign stack is stable after closure hardening;
- whether the A1-A4 / B1-B3 fixes avoid the known post-opt crashes/regressions;
- whether synthetic full-system search still produces confirmed improvements;
- whether promotion cadence recovered to the pre-registered threshold.

It cannot answer:

- synthetic vs production behavior;
- model sensitivity beyond Sonnet vs GPT-mini;
- structure-only vs full-system contribution;
- parameter-only contribution;
- causal early-stop benefit;
- classifier causal utility.

Those remain W16/W17 or v0.4/v1 questions.

## 12. CPLEX / MILP Anchor Audit

The CPLEX package was successfully imported and infeasible instances were
excluded from compact bounds. That part is correct.

But the new post-run frozen evaluation exposed a more serious issue:

| Case | Baseline median f1 | CPLEX f1 | Baseline minus CPLEX |
|---|---:|---:|---:|
| `instance_v3_fro_l01.json` | 64 | 67 | -3 |
| `instance_v3_fro_l02.json` | 80 | 77 | +3 |
| `instance_v3_fro_x01.json` | 159 | 160 | -1 |
| `instance_v3_fro_x02.json` | 198 | 199 | -1 |

For three of four frozen reference cases, the current baseline VNS produced
solutions that are feasible under the runtime oracle and have lower f1 than the
CPLEX-reported reference value. Final champions improve much more:

- median final f1 gap to CPLEX ranges from -8.5 to -42.0;
- all final champions have feasible solver outputs under the current oracle.

This cannot be interpreted as "Scion beats exact optimum." It means the CPLEX
package should be treated as report-only reference data until the consistency
question is resolved. Possible explanations include:

1. CPLEX used a different data version than the server-side run;
2. the MILP model is stricter or otherwise not equivalent to the runtime oracle;
3. the runtime oracle is missing a constraint present in the MILP model;
4. the objective mapping between MILP and VNS is inconsistent;
5. the CPLEX package needs data-hash and solution replay provenance before
   exact-gap claims.

Until this is resolved:

- CPLEX may remain report-only reference data;
- CPLEX values must not be called strict lower bounds in final claims;
- `optimality_gap` against CPLEX must be labeled "reference gap" or omitted;
- impossible-better-than-exact sanity checks should be automated in the report
  pipeline.

This is the main v0.3 reporting risk discovered by the deep analysis.

Production adds one useful nuance. In the CPLEX final production package, the
v1 baseline warm start ties the CPLEX reference on 26/31 comparable instances;
all five CPLEX-better rows are small/medium screening cases with modest cost
improvements. Production frozen and validation cases tie the v1 baseline
throughout. This supports the hypothesis that the production v1 baseline is
already strong, especially on the primary objective and on larger cases. W16
production still found meaningful first-promotion gains: Sonnet production
campaigns reported frozen md of +32.3k to +46.8k, and the frozen cost reference
gap for Sonnet final champions was about -15% to -23%. The correct conclusion is
not "production has no room," but "production starts closer to the reference, so
Scion's gains are mostly cost/fleet-compaction gains and later promotions are
harder."

## 14. Is the Experiment "Normal"?

Yes, at the framework level:

- no crash;
- no stuck launcher;
- no missing terminal summaries;
- no infeasible CPLEX-bad instances in actual metrics;
- all final champions are reproducibly runnable;
- all final champions beat baseline on fixed frozen evaluation.

No, if "normal" means "restored W16 search productivity":

- Sonnet is far below W16 synthetic cadence;
- GPT-mini is slightly below target;
- semantic search diversity is narrow;
- early-stop likely truncates some post-promotion exploration.

## 15. Did Scion Truly Optimize the Algorithm?

Yes, with a precise scope.

The evidence is strong that Scion generated useful operators and changed the
solver's behavior:

- promoted candidates passed frozen holdout;
- final champion snapshots independently beat baseline on every frozen case/seed
  pair in the post-run evaluation;
- improvements are in the primary lexicographic objective, not only runtime or
  noise;
- the improvements are encoded in actual operator files and registry snapshots.

The evidence is not enough to claim:

- generalization beyond the synthetic frozen gate;
- parameter-search marginal contribution;
- superiority to exact MILP;
- causal benefit from early-stop or classifier;
- restored W16-level search depth.

## 16. Recommended Follow-Ups

Before using CPLEX in final comparative claims:

1. Add data hashes to every CPLEX result and compact bound.
2. Replay the CPLEX incumbent solution through the current server oracle.
3. Replay any VNS solution that beats CPLEX through the MILP-side checker.
4. Add an automatic "candidate better than exact" sanity warning in reporting.

Before claiming search-mechanism effectiveness:

1. Run W17-A or a smaller structure-only paired subset.
2. Run W17-B offline parameter-only analysis on final/promoted snapshots.
3. Replay early-stop thresholds on the full lineage.
4. Analyze family fragmentation and duplicate mechanisms offline.

For v0.3 release wording:

> v0.3 is a framework-engineering milestone: the autonomous optimization loop is
> stable, auditable, and capable of producing real operator improvements. The
> current search policy is not yet a mature, high-cadence optimizer, and MILP
> exact-gap claims need one more consistency layer before they can be trusted.
