## 1. 关键决策记录

以下决策在实施讨论中确定，作为后续所有 sprint 的约束：

### 1.1 GPT-5.4Pro Review P0 意见：全部采纳

| P0 | 意见 | 采纳理由 |
|----|------|---------|
| P0-1 | W15 边界必须覆盖 11+ 耦合点，不是 Draft v2 的 5 点 | Detail Plan 的 ProblemAdapter Protocol 已覆盖 |
| P0-2 | Objective 不转 `dict[str, float]`，用泛型比较层 | lexicographic 比较是问题特定的，不该硬编码在核心 |
| P0-3 | `lower_bound_rule` 不用 string DSL，用 adapter hook 或常量 | 字符串 DSL 脆弱且难以验证 |
| P0-4 | W2 用 immutable snapshot + atomic pointer | 并发状态管理的标准做法 |
| P0-5 | stage-aware stale invalidation | 粗粒度 stale 导致不必要重跑 |
| P0-6 | hard/soft saturation 分离 | 概念正确，实现在 N2 做 W3 时细化 |
| P0-L1 | W16 矩阵不够回答所有 RQ，需要 W17 | Final Design 已吸收 |
| P0-L2 | W6 先于 W7（family persistence 是 classifier 前提） | 逻辑依赖明确 |
| P0-L3 | memory 用 "persist facts, rebuild views" 原则 | 避免多套 mutable 状态互相打架 |

### 1.2 MILP 集成方式：离线预计算

- MILP 对所有测试实例**离线运行一次**，结果存为静态文件（per instance）
- 中小规模实例：精确解（optimal）
- 中大规模实例：1 小时 MILP 的 lower bound
- Scion 的 W4 仅负责**读取预计算结果**，计算 `optimum_gap`
- MILP 结果是 report-only，**不参与决策层**，不参与 termination
- 结果出现在 metrics 和 research log 里供 LLM 参考

### 1.3 toy_tsp 范围：最小可行示例

- 10-20 个点，nearest-neighbor 初始解 + 2-opt 算子
- 目的是验证 ProblemAdapter 接口完整性，不是做 TSP 研究
- 不需要 MILP solver，只有 heuristic
- 不需要生产级质量，只需过 adapter 全部 Protocol 方法

## 3. Sprint N0 详细计划

Sprint N0 是 v0.3 的地基。以下是具体任务分解。

### 3.1 目标

建立 `ProblemAdapter` 抽象边界，使 Scion core 不再直接依赖 `warehouse_delivery` 的任何具体类型/路径。
用 `toy_tsp` MWE 证明该边界是真解耦。

### 3.2 任务分解

#### T1: ProblemAdapter Protocol + 数据类型

**文件**: `scion/scion/problem/contracts.py`（新建）

- `CheckReport(frozen dataclass)`: passed, reasons
- `LowerBoundEstimate(frozen dataclass)`: metric_name, value, kind, note
- `SolverArtifact(frozen dataclass)`: raw_output, objective, feasible, normalized_solution
- `ProblemAdapter(Protocol, runtime_checkable)`: 9 个方法
  - `render_problem_summary() -> str`
  - `render_operator_interface() -> str`
  - `load_instance(path) -> Any`
  - `deserialize_solver_output(raw, instance) -> SolverArtifact`
  - `check_solution_consistency(artifact, instance) -> CheckReport`
  - `check_feasibility(artifact, instance) -> CheckReport`
  - `recompute_objective(artifact, instance) -> Mapping[str, int|float]`
  - `estimate_lower_bound(metric, paths) -> LowerBoundEstimate | None`（可选）
  - `compare_objectives(candidate, champion) -> ObjectiveComparison`

**验收**: import 成功；Protocol 的 runtime_checkable 可用

#### T2: ProblemSpecV1 strict schema

**文件**: `scion/scion/problem/spec.py`（新建）

- Pydantic `StrictModel` with `extra="forbid"`
- 字段: id, display_name, root_dir, description, search_space, solver, parameter_search, operator_interface, objectives (list[ObjectiveMetricSpec]), llm_hints, family_taxonomy, adapter (ProblemAdapterRef)
- `ObjectiveMetricSpec`: name, direction ("minimize"/"maximize"), priority, tie_tolerance
- `@model_validator`: objectives 唯一性 + priority 连续性 + adapter import_path 前缀校验

**验收**: 现有 warehouse problem.yaml 可加载为 ProblemSpecV1（可能需要 migration）

#### T3: Generic objective comparator

**文件**: `scion/scion/problem/objectives.py`（新建）

- `MetricComparison(frozen)`: name, candidate_value, champion_value, signed_delta, relation, decisive
- `ObjectiveComparison(frozen)`: outcome, decisive_metric, scalar_delta, metrics
- `compare_lexicographic(metric_specs, candidate, champion) -> ObjectiveComparison`

**验收**: 单元测试覆盖 win/loss/tie 各种组合；与现有 `protocol/evaluation.py` 的 `lexicographic_compare` 结果一致

#### T4: Adapter loader

**文件**: `scion/scion/problem/loader.py`（新建）

- `load_problem_adapter(spec: ProblemSpecV1) -> ProblemAdapter`
- 路径校验（pinned to `scion.problems.*`）
- 动态 import + Protocol isinstance 检查
- `ProblemAdapterLoadError` 异常

**验收**: 能加载 warehouse adapter；路径不合法时正确抛错

#### T5: toy_tsp MWE

**目录**: `scion/problems/toy_tsp/`

- `adapter.py`: `ToyTspAdapter` 实现全部 ProblemAdapter 方法
- `models.py`: TspInstance, TspSolution（极简）
- `oracle.py`: tour 有效性 + 距离计算
- `solver.py`: nearest-neighbor + 2-opt
- `operators/`: 至少 1 个算子（2-opt 或 swap）
- `data/`: 2-3 个小实例（10-20 点）
- `problem.yaml`: ProblemSpecV1 格式
- `tests/`: 基本冒烟测试

**验收**: `load_problem_adapter` 可加载 toy_tsp adapter；所有 Protocol 方法可调用且返回正确类型

#### T6: problem 模块入口

**文件**: `scion/scion/problem/__init__.py`（新建）

- 导出 ProblemAdapter, ProblemSpecV1, load_problem_adapter, compare_lexicographic 等公共 API
- 确保 core 只通过 `scion.problem` 访问问题抽象

**验收**: core 模块可 `from scion.problem import ProblemAdapter`

#### T7: 测试

**文件**: `scion/scion/tests/test_problem_adapter.py`（新建）

- ProblemSpecV1 schema 验证测试（合法/非法 YAML）
- adapter loader 测试（正常加载 / 路径非法 / Protocol 不满足）
- generic comparator 单元测试
- toy_tsp adapter 集成测试

**验收**: pytest 全绿

### 3.3 N0 不做的事

- **不动 campaign.py**：N0 只建立新模块，不改 core 的调用方式。core 切换到 adapter 在 N1 做。
- **不迁移 warehouse**：warehouse adapter 封装在 N1 做。N0 的 warehouse 保持原样运行。
- **不改 protocol/evaluation.py**：现有 lexicographic_compare 保留，N1 再切换到 generic comparator。
- **不加 MILP 集成**：W4 在 N1 做。

### 3.4 依赖与风险

| 风险 | 缓解 |
|------|------|
| ProblemAdapter Protocol 设计不够用，N1 迁移时发现缺方法 | toy_tsp MWE 提前暴露遗漏 |
| ProblemSpecV1 与现有 ProblemSpec 字段不兼容 | T2 阶段做字段映射分析，必要时做 migration helper |
| generic comparator 与现有 lexicographic_compare 行为不一致 | T3 用 property-based test 对比两套实现 |

## 4. 工作流约定

### 4.1 每个 Sprint 的固定流程

1. 开始前：更新本文档的对应 Sprint 状态
2. 实施中：每个 task 完成后跑 pytest 确认不回退
3. Sprint 结束：
   - pytest 全绿
   - smoke / replay 通过
   - 更新本文档状态
   - 写 sprint 小结（如有必要）

- 在 `v0.3-dev` 分支上开发
