### 1.2 Post-opt 实验发现的回归

Sonnet synthetic seed=11 从 W16 的 2 promotions 回退到 0 promotions。根因不是随机：

1. **SPND（`SCREENING_PASS_NEGATIVE_DELTA`）缺 expand_count 上限**：为修 deep-fix sprint 的死循环 bug 加的规则把 `wr≥0.6 md<0` 候选从 screening_fail 直接送到 validation，这些 cost-regressive 候选通过 validation 但在 frozen 因 ci_low<0 挂掉。post-opt R3/R4 消耗 10/15 轮在这条 fail pipeline 上。
2. **`_rounds_since_last_promote` 计数器把 frozen_fail/val_fail 也算 idle**：同一 branch 从 screen → val → frozen 走完失败的多轮都被计为 "没 promote"，把 `budget_efficiency` 早停门槛（stagnation_window=15）提前触发。
3. **stagnation_window=15 对 100 轮 budget 太紧**：W16 数据显示 Sonnet 第 2 promote 平均在 round 20，post-opt 在 round 15 就停了。

详细分析见 memory `project_v03_postopt_regression.md`。

## 二、设计原则：不得破坏的基石

以下约束在本次优化中**不得动摇**（直接引自 v3 蓝图 / v0.3 设计）：

1. **Creative Layer tainted / Decision Layer deterministic**：LLM 输出不进 Decision
2. **Decision Input Guard**：`DecisionFeatures` 只含数值 + 闭集枚举；不读自由文本
3. **Contract / Verification / Protocol 三层独立**：边界不合并
4. **Champion = 池级别**，A/B 评估语义不变
5. **三级 split（screening / validation / frozen）+ exposure control**：暴露规则不变
6. **多目标 = lexicographic**（本次不改评估层口径，只改上下文层倾向语言 — 这部分 Phase 2 已做）
7. **Problem adapter 边界**：core 代码不直接 import warehouse-specific 逻辑（v0.3 已做，本次不回退）

以及本次优化**额外明确**的两条约束：

8. **最低限度服务抽离**：campaign.py 职责搬家到协调器类；不改外部接口、不改主循环签名
9. **所有 974 测试继续通过**；新增服务类必须有对应 unit test

## 三、工作项

### A. 回归修复（3 项，P0）

#### A1. `SCREENING_PASS_NEGATIVE_DELTA` 加 expand_count 上限

**文件**：`scion/scion/core/decision.py:61-65`

**问题**：SPND 无条件 → `QUEUE_VALIDATE`，把 cost-regressive 候选推到 frozen 耗资源。

**修复方案**：两种等价方案，择一实现

方案 a（推荐，简单）：添加 `expand_count ≤ 1` 上限，超过后走 `CONTINUE_EXPLORE`

```python
elif wr >= threshold and md is not None and md < 0:
    # 首次遇 wr>=threshold md<0：validation bootstrap CI 判别
    # 再次遇：该候选 cost-regressive 已稳定，不值得烧 val/frozen budget
    if features.expand_count >= 1:
        return self._out(features, Decision.CONTINUE_EXPLORE,
                         ["SCREENING_PASS_NEGATIVE_DELTA_EXHAUSTED"])
    return self._out(features, Decision.QUEUE_VALIDATE,
                     ["SCREENING_PASS_NEGATIVE_DELTA"])
```

方案 b（更精细）：要求 `md > -SPLITS_WEIGHT / 2`（即 cost regression 不超过半个 split 价值），否则 `CONTINUE_EXPLORE`

方案 a 简单且足够，推荐。

**测试**：`scion/tests/test_decision.py` 加一条 `test_decision_screening_pass_negative_delta_exhausted`。

#### A2. `_rounds_since_last_promote` 不累加 frozen_fail / val_fail 的 eval step

**文件**：`scion/scion/core/campaign.py:866`

**问题**：当前所有 eval step 都 `+=1`，包括 branch 在 screen → val → frozen 通过但最后 frozen_fail 的多轮路径。这让 idle ratio 假性升高。

**修复方案**：区分两类 eval step：

- **idle 类**（计入）：expand_screening, expand_validation 的 self-loop（候选反复扩样本说明信号不足）
- **生产类**（不计入）：validation 首次、frozen 首次（说明候选确实进了 pipeline，即使最终 fail 也是 "有产出"）

实现：`_run_eval_step` 在 `_round_num += 1` 同时，只有当决策是 `CONTINUE_EXPLORE / EXPAND_SCREENING` 时才 `_rounds_since_last_promote += 1`；`QUEUE_FROZEN / PROMOTE / QUEUE_VALIDATE / EXPAND_VALIDATION / ABANDON (from frozen)` 不计。

更干净的做法：**任何触达 frozen 的 branch（无论 pass/fail）都不算 idle**。具体：在 `_apply_decision_and_finalize` 或 `_run_eval_step` 结尾，如果当前 step 的 `stage == "frozen"`，重置或不递增 idle 计数。

**测试**：`scion/tests/test_early_stop_integration.py` 加一条新测试验证"15 轮 screen_fail + 1 轮 frozen_fail → idle ratio 不触发 early_stop"。

#### A3. `stagnation_window` 15 → 25

**文件**：`scion/scion/core/early_stop.py`（或 config）

**问题**：W16 Sonnet 第 2 promote 在 round 20+，窗口 15 太紧。

**修复方案**：默认值 15 → 25；保留 per-campaign override。

**测试**：`test_early_stop.py` 已有 stagnation_window 参数化测试，更新默认断言。

#### B.2 协调器边界

依据 `reviews/v0.3-design-detail-plan.md §6.1`，抽出 3 个协调器（v0.3 最低限度版）：

**① `ProblemRuntime`** — 包 problem 的 "如何启动一个 campaign 一次" 的准备工作。

承担职责（当前在 CampaignManager 的方法）：
- adapter 加载 + 传递（当前 `__init__` 的一部分）
- problem_spec / metric_specs 分发给下游（experiment_protocol / context_manager）
- 初始 workspace materialization（`_setup_workspace` 的 champion snapshot 路径）
- code_hash 计算

不承担：
- 主循环
- 单步执行逻辑
- 失败路由

接口：
```python
class ProblemRuntime:
    def __init__(self, problem_spec, adapter, champion, campaign_dir, ...)
    def materialize_champion(self) -> str  # returns champion workspace path
    def render_problem_context(self, branch, champion, step_history) -> dict
    def render_code_context(self, branch, hypothesis, champion) -> dict
    def render_fix_context(self, branch, patch, verification_result) -> dict
```

**② `AsyncWeightOptCoordinator`** — 权重优化线程生命周期。

承担职责：
- `_on_promote` 的 spawn weight opt
- `_bg_weight_opt_task` 线程体
- `_run_weight_optimization` 核心
- STALE_WEIGHT_UPDATE 状态标记（trigger 端）
- weight opt 完成后的 champion snapshot 原子切换

不承担：
- 决定是否 promote（仍在 CampaignManager 主循环里）
- Decision Engine 相关

接口：
```python
class AsyncWeightOptCoordinator:
    def on_promote(self, new_champion) -> None  # spawn bg thread
    def wait_all(self, timeout: Optional[float]) -> None  # shutdown hook
    def get_pending_count(self) -> int
```

**③ `PlateauController`** — 早停 + 停滞检测。

承担职责：
- `EarlyStopController` 实例持有（已存在）
- `_run_stagnation_check` / `_check_soft_stagnation`
- `_rounds_since_last_promote` 计数（A2 的 idle 语义改动在这里落地）
- `_consume_forced_locus` / `_get_diversification_locus`

不承担：
- Decision Engine
- branch state 转换

接口：
```python
class PlateauController:
    def on_step(self, branch, decision, step) -> None  # update counters
    def on_promote(self) -> None  # reset idle counter
    def should_stop(self) -> EarlyStopSignal
    def consume_forced_locus(self) -> Optional[str]
    def compute_stagnation_signal(self, recent_steps) -> Optional[StagnationSignal]
```

#### B.3 剩余在 CampaignManager 的内容

拆分后 CampaignManager 剩下：
- `__init__`（依赖注入 3 个协调器）
- `run / run_one_step / should_stop / get_state`（主循环骨架）
- `_run_explore_step / _run_eval_step / _run_reconcile_step`（三种 step 的 orchestration，但内部 delegate 给协调器）

目标：**CampaignManager 大约 1000-1200 行**（从 2451 降），<800 留给 v1.0。

### D. 状态机 P5 minor gap（P2）

**问题**：`VALIDATING_EXPAND + EXPAND_VALIDATION` 未映射。

**方案 a**：添加明确映射 `VALIDATING_EXPAND: VALIDATING_EXPAND`（self-loop，与 EXPLORE_EXPAND 对称）。但这会允许无限 expand，与 `expand_count>=1` guard 的意图冲突。

**方案 b**（推荐）：添加明确映射但同时把 guard 加强为契约：`_decide_validation` 永远不对 `state=VALIDATING_EXPAND` 返回 `EXPAND_VALIDATION`。加 assertion。

**方案 c**（最省事）：保留 brittle by guard 现状 + 在 `_DECISION_TRANSITIONS` 加一个注释说明为什么该组合不映射。

方案 c 满足"穷举 + 文档化"的 P5 要求，改动最小，推荐。

### F. 验证 rerun（P0）

#### F.1 目的

验证 A 修完后：
1. Sonnet synthetic × 3 seeds 能恢复到 W16 水平的 promotion cadence
2. GPT synthetic × 3 seeds 与 W16 相当或更好
3. budget_efficiency 早停在合理轮次触发（不提前，不遗漏）
4. SPND 的 expand_count 上限生效，不再重复 burn val/frozen budget

#### F.2 实验矩阵

最小验证集：
- model × 2（claude-sonnet-4-6, gpt-5.4-mini）
- variant × 1（synthetic，production 留 v1.0 验证）
- seed × 3（11, 29, 47）
- 合计 **6 campaigns**，100 rounds each

不用跑 production，因为 post-opt 发现的回归是 synthetic 上更清晰；production 留给 v1.0 Phase 2 FCMCNF 验证前的完整 re-benchmark。

#### F.3 成功标准

| 指标 | 目标 |
|---|---|
| Sonnet synthetic × 3 seeds 平均 promotions | ≥ 2.0（W16 水平 2.3） |
| GPT synthetic × 3 seeds 平均 promotions | ≥ 1.5（W16 水平 1.7） |
| 早停触发轮次（若触发）| ≥ 25（不再 round 15 早停） |
