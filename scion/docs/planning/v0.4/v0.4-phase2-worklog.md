# Scion v0.4 Phase 2 Worklog

*Date: 2026-05-01*
*Branch: v0.4-dev*

## Summary

Phase 2 has started with the framework hardening slice required before CVRP:

- `V5_solution_consistency` now uses the `ProblemAdapter` consistency hook when
  available.
- `V9_perf_guard` emits structured runtime metadata in addition to human detail
  text.
- `DecisionFeatures` exposes only bounded runtime facts, not free-text runtime
  explanations.
- Lineage evidence records now include verification check metadata and runtime
  guard evidence.
- Hypothesis context now includes bounded runtime feedback from prior V9
  verification failures.

## Implementation Notes

### Adapter-Native V5

`scion.verification.state_mutation.check_state_mutation()` now accepts an
optional adapter.

When an adapter is provided, V5 executes:

```text
adapter.load_instance
adapter.deserialize_solver_output
adapter.check_solution_consistency
```

This prevents CVRP from falling through to the warehouse assignment/vehicle
shape checker.

Legacy behavior remains as fallback for old non-adapter runtime verification.

### Runtime Guard Metadata

`CheckResult` now has a `metadata` field. `V9_perf_guard` populates it with:

```text
case_id
timeout_sec
candidate_ms
champion_ms
ratio
limit_ratio
candidate_timeout
champion_timeout
candidate_error_category
champion_error_category
```

`SafeFeatureExtractor` extracts only these Decision-safe facts:

```text
runtime_guard_passed
runtime_guard_ratio
runtime_guard_timeout
```

The Decision Layer still does not receive free text.

### Evidence Recording

`EvidenceRecorder` now includes:

```text
verification_checks[]
runtime_guard
```

inside lineage metadata. This prepares promotion evidence for the v0.4 evidence
harness without changing the step summary schema.

### Runtime Feedback Context

`ContextManager` builds `runtime_feedback` from previous V9 verification
failures. `CreativeLayer` injects it into hypothesis system blocks as proposal
guidance.

This feedback is intentionally advisory. It does not affect promotion decisions
directly.

### EvaluationPipeline Lifecycle

`EvaluationPipeline` remains constructed per evaluation.

This is deliberate for now because existing campaign tests still mutate
`_experiment_protocol` directly, and `CampaignManager` still owns budget and
protocol-progress lifecycle. A long-lived pipeline should wait until
`CampaignLoop` or protocol ownership is extracted.

## Phase 2 Closeout Slice

### ProblemSpecV1 Bridge

Added `scion.problem.bridge` as the single compatibility path from
`ProblemSpecV1` into legacy campaign/runtime objects.

The bridge returns:

- legacy `ProblemSpec` for current `CampaignManager` and `VerificationGate`
  call sites;
- `metric_specs` and `objective_policy` for `ExperimentProtocol`;
- `operator_execute_signature` for contract/runtime interface gates.

The bridge intentionally keeps objective metadata and execute-signature metadata
outside the legacy `ProblemSpec`; those remain explicit runtime fields derived
from the authoritative v1 schema.

CVRP `problem-v1.yaml` now declares
`canary_case_path: "data/tiny_canary.json"`, and the CVRP one-step campaign
smoke now uses the bridge instead of a hand-built legacy spec.
