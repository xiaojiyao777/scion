# Scion v0.4 Phase 3 Task Manifest

*Date: 2026-05-01*
*Parent: v0.4-phase3-design.md*
*Branch: v0.4-dev*

## Phase 3 Goal

Build the evidence/reporting layer needed before formal CVRP campaigns.

Phase 3 begins with a pure final-quality package writer. It deliberately does
not run raw CVRPLIB instances or formal Scion campaigns.

## Dependency Graph

```text
P3-01 Final Quality Package Writer
  -> P3-02 CVRP Baseline CSV Importer
  -> P3-03 CVRP Final Champion Evaluation Runner
  -> P3-04 Formal CVRP Campaign Readiness Gate

P3-01 Final Quality Package Writer
  -> P3-05 EvidenceRecorder Attachment

P3-03 CVRP Final Champion Evaluation Runner
  -> P3-06 Campaign Matrix Plan

P3-07 Performance Contract Hardening
  can run after P3-01 or in parallel with P3-02
```

## P3-01: Final Quality Package Writer

**Priority**: P0

**Owner**: single worker subagent

**Write scope**

- `scion/scion/evidence/final_quality.py`
- `scion/scion/evidence/__init__.py`
- `scion/scion/tests/unit/evidence/test_final_quality.py`
- docs updates only if needed

**Purpose**

Implement a pure, deterministic evidence package writer from in-memory case
records.

It should write:

```text
evidence_manifest.json
final_quality.json
final_quality.csv
per_case_quality.csv
runtime_summary.json
failure_summary.json
```

**Design requirements**

- Do not run solvers.
- Do not call LLMs.
- Do not touch `CampaignManager`.
- Do not read `vrp/cvrplib/**`.
- Use standard-library JSON/CSV unless a repo-local helper already exists.
- Keep the module problem-generic with optional CVRP extension fields.
- BKS/gap fields are report-only and must not affect baseline-vs-candidate
  comparison.

**Suggested API**

```python
QualityCaseRecord
FinalQualityConfig
FinalQualityPackage
build_final_quality_package(records, config)
write_final_quality_package(package, output_dir)
```

The exact names may change if the local style suggests a better shape, but the
module must keep a small public API.

**Required semantics**

- `comparison` may be supplied by caller or computed from objective metrics
  when both sides are valid.
- Timeout/crash/infeasible cases are counted and preserved.
- Runtime regression is computed from configurable ratio threshold.
- CVRP benchmark comparability is represented separately from CVRP feasibility.
- CSV and JSON outputs are stable enough for tests to assert.

**Tests**

Cover at least:

- all six files are written;
- one win/tie/loss aggregate;
- timeout and crash rows enter failure summary;
- infeasible row enters failure summary;
- runtime regression threshold;
- CVRP fields with BKS/gap/route-gap are emitted as report-only fields;
- missing `bks_routes` does not silently mark benchmark comparability true.

## P3-02: CVRP Baseline CSV Importer

**Priority**: P1

**Write scope**

- `scion/scion/evidence/cvrp_baseline_import.py` or equivalent
- tests using synthetic temporary CSV files

**Purpose**

Convert existing `vrp/results/full_experiment_seed0_final.csv`-style rows into
the final quality package schema.

The importer is a translation layer only. It should not write evidence files
directly; it should produce `QualityCaseRecord` rows consumed by P3-01.

**Expected source columns**

The existing VRP baseline CSV has this shape:

```text
instance,subset,path,dimension,bks,bks_routes,cost,gap_pct,routes,route_gap,
iterations,time,time_limit,seed,feasible,benchmark_feasible,mode,status,error,
wall_time
```

Tests should use synthetic CSV files with this same column style.

**Suggested API**

```python
CvrpCsvResultRow
load_cvrp_result_rows(csv_path)
build_cvrp_quality_records(baseline_rows, candidate_rows=None)
load_cvrp_quality_records(baseline_csv, candidate_csv=None)
```

Exact names may change, but keep the public API small.

**Mapping rules**

- Treat `path` as an opaque source artifact string. Do not open it.
- `instance` maps to `case_id`.
- `subset` maps to `subset`.
- `seed` maps to `seed`.
- `status`, `error`, `feasible`, and `benchmark_feasible` map to side-specific
  final-quality status fields.
- `cost` maps to objective/cost.
- `gap_pct`, `routes`, `bks`, `bks_routes`, and `route_gap` map to CVRP
  report-only fields.
- `wall_time` or `time` should map to elapsed milliseconds. Prefer
  `wall_time` when present.
- Missing numeric fields should become `None`, not `0`.
- Missing `bks_routes` must not silently imply benchmark comparability.

**Comparison behavior**

For `candidate_rows=None`, produce a baseline self-comparison package:

- baseline and candidate values come from the same row;
- `comparison="equal"` for rows that are otherwise valid;
- failed rows remain `not_comparable` through the P3-01 writer.

For paired baseline/candidate rows:

- align rows by `(case_id, seed)`;
- missing candidate rows become candidate error rows;
- missing baseline rows should fail closed with a clear exception;
- comparison may be left to P3-01 objective comparison when both sides have
  valid cost values.

**Constraints**

- May read CSV result artifacts.
- Must not read raw `vrp/cvrplib/**` instances in tests.
- Should preserve source paths as opaque artifact identifiers.

## P3-03: CVRP Final Champion Evaluation Runner

**Priority**: P1/P2 after P3-01 and P3-02

**Purpose**

Evaluate original baseline and final champion snapshots on the same fixed CVRP
case list and write final evidence through P3-01.

This is where solver subprocess execution belongs, not in P3-01.

### P3-03a: No-Run CVRP CSV Evidence Package Builder

**Priority**: P0 before runner-backed evaluation

**Write scope**

- `scion/scion/evidence/cvrp_package.py` or equivalent
- `scion/scion/evidence/__init__.py`
- `scion/scion/tests/unit/evidence/test_cvrp_package.py`

**Purpose**

Wire P3-01 and P3-02 together without running solvers:

```text
CVRP result CSV(s)
  -> CvrpCsvResultRow
  -> QualityCaseRecord
  -> FinalQualityPackage
  -> six evidence artifacts
```

This is the first end-to-end evidence package path, but it is still no-run and
does not touch campaign state.

**Suggested API**

```python
CvrpEvidencePackageConfig
build_cvrp_evidence_package_from_csv(...)
write_cvrp_evidence_package_from_csv(...)
```

**Required behavior**

- Accept a baseline CSV and optional candidate CSV.
- Build `FinalQualityConfig` with `problem_id="cvrp"` by default.
- Return both the in-memory package and written artifact refs, or a small
  dataclass containing both.
- Do not read or open opaque instance paths from CSV rows.
- Do not run any solver.
- Do not mutate campaign summaries yet.
- Allow caller to set labels, campaign id, runtime threshold, and output dir.

### P3-03b: Runner-Backed CVRP Final Evaluation Service

**Priority**: P1 after P3-03a

**Write scope**

- `scion/scion/evidence/cvrp_final_evaluation.py` or equivalent
- `scion/scion/evidence/__init__.py`
- `scion/scion/tests/unit/evidence/test_cvrp_final_evaluation.py`

**Purpose**

Run baseline and final champion workspaces on a fixed case/seed set and produce
`QualityCaseRecord` rows for P3-01.

This is the first runner-backed final evaluation layer. It still must not
mutate campaign state or influence promotion decisions.

**Suggested API**

```python
CvrpFinalEvaluationConfig
CvrpSideResult
evaluate_cvrp_final_quality_records(...)
build_cvrp_final_evidence_package(...)
write_cvrp_final_evidence_package(...)
```

**Required behavior**

- Inputs include baseline workspace, candidate workspace, case paths, seeds,
  `Runner`, and CVRP `ProblemAdapter`.
- For each `(case, seed)`, run both sides with the same time limit.
- Load instances through the adapter.
- Deserialize solver outputs through the adapter.
- Check solution consistency and feasibility through the adapter.
- Recompute objective through the adapter; do not trust solver-reported
  objective.
- Map successful side result into:
  - status `ok`;
  - elapsed ms;
  - cost/total_distance;
  - routes;
  - bks;
  - bks_routes;
  - route_gap;
  - gap_pct;
  - CVRP feasibility;
  - benchmark feasibility.
- Map failed runs into status `timeout`, `crash`, `oom`, `error`, or
  `infeasible` as appropriate.
- Missing or invalid output must become non-comparable final evidence, not a
  dropped row.
- Comparison may be left to P3-01 writer when both sides are valid and
  benchmark-comparable.
- Do not open raw `vrp/cvrplib/**` files in tests; tests may use tiny synthetic
  JSON fixtures or fake adapter/runner objects.

## P3-04: Formal CVRP Campaign Readiness Gate

**Priority**: P1

**Purpose**

Define the exact screening/validation/frozen/final case lists and runtime
budgets for the first real CVRP campaign.

Initial recommendation:

- quick regression: A/B/P/E synthetic or small selected result rows;
- first optimization target: X comparable cases;
- final report: fixed subset with route-count comparability explicit.

### P3-04a: CVRP Case Manifest Builder From Result CSV

**Priority**: P1

**Write scope**

- `scion/scion/evidence/cvrp_case_manifest.py` or equivalent
- `scion/scion/evidence/__init__.py`
- `scion/scion/tests/unit/evidence/test_cvrp_case_manifest.py`

**Purpose**

Build deterministic fixed evaluation case manifests from existing CVRP result
CSV artifacts. This prepares runner-backed final evaluation and formal campaign
setup without reading raw CVRPLIB instance files.

**Suggested API**

```python
CvrpCaseSelectionConfig
CvrpCaseEntry
CvrpCaseManifest
build_cvrp_case_manifest_from_rows(...)
build_cvrp_case_manifest_from_csv(...)
write_cvrp_case_manifest(...)
```

**Required behavior**

- Consume `CvrpCsvResultRow` rows or a `full_experiment_seed0_final.csv`-style
  CSV path.
- Treat row `path` as an opaque case path string and never open it.
- Filter to rows with usable status/feasibility according to config.
- Support subset filtering, e.g. `("A", "B", "P", "E")` for quick regression
  or `("X",)` for first optimization target.
- Support `require_bks`, `require_bks_routes`, and
  `require_benchmark_feasible`.
- Select deterministically with optional `max_cases_total` and
  `max_cases_per_subset`.
- Emit enough metadata to reconstruct selection criteria:
  - source label/path;
  - subset filters;
  - seed list;
  - selected/eligible/rejected counts;
  - rejection counts by reason.
- JSON output should be stable and contain selected opaque case paths.

### P3-04b: CVRP Manifest-Driven Final Evidence Builder

**Priority**: P1

**Write scope**

- `scion/scion/evidence/cvrp_manifest_evaluation.py` or equivalent
- `scion/scion/evidence/__init__.py`
- `scion/scion/tests/unit/evidence/test_cvrp_manifest_evaluation.py`

**Purpose**

Connect the fixed case manifest from P3-04a to the runner-backed final
evaluation service from P3-03b.

The service should not change campaign execution. It turns an auditable case
manifest into final evidence:

```text
CvrpCaseManifest + workspaces + runner + adapter
  -> CvrpFinalEvaluationConfig
  -> final evidence package
```

**Suggested API**

```python
CvrpManifestEvaluationConfig
build_cvrp_final_evaluation_config_from_manifest(...)
build_cvrp_manifest_final_evidence_package(...)
write_cvrp_manifest_final_evidence_package(...)
```

**Required behavior**

- Use `manifest.cases[*].source_path` as case paths.
- Use `manifest.config["seeds"]` or `manifest.metadata["seed_list"]` as seeds,
  with an explicit override allowed.
- Propagate campaign id, labels, workspaces, time limit, runtime threshold, and
  output dir.
- Reuse P3-03b for runner-backed evaluation.
- Fail closed when manifest contains no cases or no seeds.
- Do not open case paths.
- Do not mutate campaign summaries.

## P3-05: EvidenceRecorder Attachment

**Priority**: P2

**Purpose**

Attach final quality package refs to `campaign_summary.json` through
`EvidenceRecorder.attach_final_evidence_refs()` after the runner exists.

### P3-05a: Final Evidence Ref Payload Helper

**Priority**: P1

**Write scope**

- `scion/scion/evidence/final_evidence_refs.py` or equivalent
- `scion/scion/evidence/__init__.py`
- `scion/scion/tests/unit/evidence/test_final_evidence_refs.py`
- focused `EvidenceRecorder` test update only if needed

**Purpose**

Provide a small stable payload builder for attaching generated final evidence
packages to campaign summaries.

This slice should not make `CampaignManager` run final evaluation. It only
normalizes already-generated artifact refs into a summary payload.

**Suggested API**

```python
build_final_evidence_refs(package_result, label="final_quality")
attach_final_evidence_package(recorder, package_result, label="final_quality")
```

**Required behavior**

- Accept a result shape containing `package` and `artifacts`, such as
  `CvrpEvidencePackageResult`.
- Convert artifact `Path` values to strings.
- Include manifest-level metadata:
  - problem id;
  - campaign id;
  - baseline label;
  - candidate label;
  - number of cases;
  - schema/package type if available.
- Preserve all artifact keys from P3-01:
  - manifest;
  - final_quality_json;
  - final_quality_csv;
  - per_case_quality_csv;
  - runtime_summary;
  - failure_summary.
- Attach through `EvidenceRecorder.attach_final_evidence_refs()`.
- Do not mutate step schema.
- Do not run solvers or read artifact file contents.

## P3-07: Performance Contract Hardening

**Priority**: P1/P2

**Purpose**

Extend static contract tests for CVRP-style unbounded route operators:

- all route-pair x all segment x all insertion enumeration without caps;
- unbounded permutations/subsets;
- nested customer triples/quads;
- missing top-k or early-exit guard.

This can be implemented after P3-01 because P3-01 gives final reporting a place
to count runtime failures consistently.
