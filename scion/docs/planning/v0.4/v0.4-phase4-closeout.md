# Scion v0.4 Phase 4 Closeout

*Date: 2026-05-01*
*Branch: v0.4-dev*

## Summary

Phase 4 completed the first controlled CVRP campaign path.

The phase started with an input/runtime gap: Phase 3 could build fixed CVRP
case manifests, but the Scion CVRP runtime only loaded tiny JSON fixtures.
Phase 4 closes that gap and proves a synthetic controlled CVRP campaign can run
through Scion's branch-governed path:

```text
controlled CVRP cases
  -> adapter/runtime checks
  -> screening
  -> validation
  -> frozen
  -> promote
  -> post-campaign final evidence package
  -> campaign_summary.json final_evidence_refs
```

## Completed Components

### CVRPLIB Input Boundary

Added:

```text
scion/scion/problems/cvrp/cvrplib.py
```

Responsibilities:

- parse synthetic-tested CVRPLIB-style `.vrp` files;
- support EUC_2D only;
- map CVRPLIB ids into Scion zero-based ids with depot `0`;
- parse sibling `.sol` files for report-only `bks` and `bks_routes`;
- fail closed for unsupported edge-weight types.

### CVRP Solver `.vrp` Runtime

scion/scion/problems/cvrp/solver.py

Responsibilities:

- load JSON and `.vrp` instances through `CvrpAdapter`;
- keep the route/objective JSON output contract stable;
- preserve promotion objectives as `fleet_violation`, `total_distance`, and
  `routes`.

### Controlled Synthetic Fixtures

### CVRP Operator Runtime Boundary

Added:

```text
scion/scion/problems/cvrp/registry.yaml
scion/scion/tests/test_cvrp_solver_operator_runtime.py
```

Updated:

```text
scion/scion/problems/cvrp/solver.py
```

Responsibilities:

- read registry operators when `--registry` exists;
- preserve baseline behavior when registry is missing or empty;
- load workspace-relative operator classes;
- skip and audit path escapes, missing classes, invalid returns, and infeasible
  outputs;
- apply operators in a bounded deterministic loop;
- accept only lexicographic improvements on `(fleet_violation, total_distance)`;
- keep BKS/gap out of runtime acceptance and promotion.

### Multi-Step Promotion Smoke

The controlled smoke campaign now exercises:

```text
screening -> QUEUE_VALIDATE
validation -> QUEUE_FROZEN
frozen -> PROMOTE
```

The promoted champion snapshot contains the generated operator and registry.
Post-campaign final evidence compares the original baseline workspace against
the promoted champion snapshot and attaches refs to `campaign_summary.json`.

## Safety Boundaries

Phase 4 did not:

- read raw `vrp/cvrplib/**` benchmark files in tests;
- import from `vrp/src`;
- use BKS/gap for promotion or runtime operator acceptance;
- make the campaign loop automatically run final evaluation;
- change per-step summary schema for final evidence refs.

## Exit Criteria Check

- One controlled CVRP campaign run is evidence-complete: yes.
- Final evaluation uses a fixed manifest: yes.
- Final evidence refs are attached to campaign summary: yes.
- No promotion uses BKS/gap: yes.
- Runtime failures and invalid operator outputs are counted or audited, not
  silently accepted: yes.
