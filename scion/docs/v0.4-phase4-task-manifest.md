## Phase 4 Goal

Prepare and execute the first controlled CVRP campaign path.

Phase 4 starts by closing the input/runtime gap between Phase 3 manifests and
the Scion CVRP adapter.

## P4-01: CVRPLIB Input Support

**Priority**: P0

**Owner**: single worker subagent

**Write scope**

- `scion/scion/problems/cvrp/cvrplib.py` or equivalent parser module
- `scion/scion/problems/cvrp/adapter.py`
- `scion/scion/tests/test_cvrp_cvrplib_adapter.py`
- docs update only if needed

**Purpose**

Allow Scion CVRP runtime to load CVRPLIB-style `.vrp` files into the existing
`CvrpInstance` model.

**Constraints**

- Do not read or depend on raw `vrp/cvrplib/**` files.
- Tests must use temporary synthetic `.vrp` and `.sol` files.
- Do not import from `vrp/src`; Scion CVRP package should own its runtime
  parser boundary.
- Support EUC_2D only.
- Unsupported edge weight types must fail closed.

**Required behavior**

- `CvrpAdapter.load_instance(path)` dispatches:
  - `.json` -> existing `CvrpInstance.from_json`;
  - `.vrp` -> new CVRPLIB parser.
- Parser reads:
  - `NAME`;
  - `DIMENSION`;
  - `CAPACITY`;
  - `EDGE_WEIGHT_TYPE`;
  - `NODE_COORD_SECTION`;
  - `DEMAND_SECTION`;
  - `DEPOT_SECTION`.
- Convert CVRPLIB ids to Scion ids consistently. Prefer Scion's existing
  zero-based depot/customer ids for runtime.
- Preserve integer-distance behavior for standard integer EUC_2D coordinates.
- If a sibling `.sol` exists, parse:
  - reference cost as `bks`;
  - route count as `bks_routes`.
- If `.sol` is missing, leave `bks` and `bks_routes` as `None`.
- Adapter consistency/feasibility/objective checks must pass on synthetic
  `.vrp` solutions.

**Tests**

Cover at least:

- `.vrp` loads into `CvrpInstance`;
- `.vrp + .sol` populates `bks` and `bks_routes`;
- missing `.sol` leaves reference fields `None`;
- unsupported `EDGE_WEIGHT_TYPE` raises clear error;
- adapter validates a route solution for the parsed `.vrp`;
- depot/customer id conversion is unambiguous.

### P4-04b: CVRP Operator Runtime Boundary

**Priority**: P1

**Write scope**

- `scion/scion/problems/cvrp/solver.py`
- `scion/scion/problems/cvrp/registry.yaml`
- `scion/scion/tests/test_cvrp_solver_operator_runtime.py`
- existing CVRP solver/protocol tests as needed
- docs update only if needed

**Purpose**

Make generated CVRP operators observable by the runtime. Without this boundary,
campaign-generated operator files can pass contract/verification but cannot
change solver output, so controlled runs plateau at screening ties.

**Required behavior**

- `scion.problems.cvrp.solver` reads `--registry` when present.
- Empty or missing registry preserves the existing nearest-neighbor baseline.
- Registry operators are loaded from workspace-relative Python files.
- Operator paths that escape the workspace fail closed or are skipped with
  recorded runtime metadata.
- Operators are applied in a deterministic bounded improvement loop after the
  initial construction.
- Only `CvrpSolution` outputs that pass adapter consistency and feasibility are
  eligible.
- Candidate outputs are accepted only if they improve the promotion objective:
  lexicographic `(fleet_violation, total_distance)`.
- Solver JSON keeps the existing objective contract and may add runtime/operator
  audit metadata.

**Constraints**

- Do not read or depend on raw `vrp/cvrplib/**` files.
- Do not use BKS/gap for runtime acceptance or promotion.
- Do not import from `vrp/src`.
- Keep the operator loop bounded by iteration count and wall-time budget.

### P4-04c: Controlled Multi-Step Promotion Smoke

**Priority**: P1

**Write scope**

- `scion/scion/problems/cvrp/controlled/data/*.vrp`
- `scion/scion/problems/cvrp/controlled/data/*.sol`
- `scion/scion/problems/cvrp/controlled/*.csv`
- `scion/scion/problems/cvrp/controlled/manifests/*.json`
- `scion/scion/tests/test_cvrp_controlled_campaign.py`
- controlled manifest tests as needed
- evidence final-evaluation tests only if registry defaults need tightening

**Purpose**

Use the operator-aware CVRP solver from P4-04b to prove a controlled mock-LLM
campaign can traverse the full branch-governed path:

```text
screening -> validation -> frozen -> promote -> post-campaign final evidence refs
```

**Required behavior**

- Controlled synthetic stage fixtures should include known routes that improve
  over the baseline nearest-neighbor construction.
- The improvement operator must be generated through `MockLLMClient` and pass
  normal contract/verification gates.
- Campaign steps should exercise:
  - screening decision `QUEUE_VALIDATE`;
  - validation decision `QUEUE_FROZEN`;
  - frozen decision `PROMOTE`.
- Promotion should create a champion snapshot with the generated operator and
  registry.
- Post-campaign final evidence should compare the original baseline workspace
  against the promoted champion snapshot.
- Final evidence should use explicit baseline/candidate registry paths or a
  documented safe default.
- Summary refs remain top-level `final_evidence_refs`; step schema remains
  unchanged.

**Constraints**

- The synthetic improvement is a smoke harness, not a benchmark claim.
- Do not read or depend on raw `vrp/cvrplib/**` files.
- Do not use BKS/gap for promotion or runtime acceptance.
- Do not make the campaign loop auto-run final evaluation.
