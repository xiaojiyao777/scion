# Scion v0.4 Core/CVRP Gap Audit

*Date: 2026-05-09*
*Status: read-only architecture and code audit*

## Scope

This audit separates four objects that should not be conflated:

1. **Architecture baseline**: `design/scion-architecture-v3.md`, treated as
   the controlling design source.
2. **Scion core framework**: `scion/scion/core`, `proposal`, `contract`,
   `verification`, `protocol`, `problem`, `runtime`, `evidence`, `lineage`,
   `config`, and `cli`.
3. **CVRP Scion problem package**: `scion/scion/problems/cvrp`.
4. **Original VRP baseline**: `vrp`, treated as the human-written solver that
   the CVRP problem package wraps.

Raw CVRPLIB instances, raw experiment artifacts, and raw metrics/log outputs
were not read. Source inspection was split by area so framework issues,
problem-package issues, and original-baseline issues remain distinct.

## Executive Judgment

The current blocker is real, but it is not simply "Scion core is wrong."

Scion's v3 governance model still holds at the main control boundary: LLM
outputs are tainted, Contract / Verification / Protocol / Decision remain
separate, and `DecisionEngine` does not directly read proposal free text.

The core gap is narrower and more precise:

```text
Scion core can govern declared research surfaces,
but v0.4 CVRP still exposes an incomplete Scion-native research object.
```

The CVRP package is integrated well enough to be governed. It has an adapter,
objective recomputation, feasibility checking, solver wrapper, declared
surfaces, selected-surface runtime audit, controlled/formal protocol assets,
and tests. It is not yet integrated deeply enough to give Scion strong research
levers over the real CVRP algorithm. The original `vrp` solver's high-leverage
components are construction, adaptive ALNS destroy/repair, VNS portfolio,
simulated-annealing acceptance, thresholds, and adaptive weights. The current
problem package only partially exposes that structure.

The v0.4 target should therefore be tightened to:

```text
Scion can run governed autoresearch on manually onboarded problem packages
without absorbing problem semantics into core.
```

v0.4 should not claim general solver onboarding, nor launch long CVRP
solver-quality validation until a short diagnostic shows nontrivial screening
quality through a deep problem-owned surface.

## Version Boundary

The v0.4 design has not violated v3, but its scope has expanded. The original
v0.4 goal was performance-aware governance, CVRP adapter integration, campaign
cleanup, and final evidence plumbing. It then grew to include
`research_surfaces v2`, `AgenticProposalSession`, multiple CVRP policy/config
surfaces, and the `main_search_strategy` whole-algorithm surface.

That expansion was justified as diagnosis and hardening. It should not continue
until v0.4 becomes a general solver-ingestion product.

Use this version boundary:

- **v0.4**: manually onboarded problem packages; prove the v3 loop governs
  warehouse and CVRP without core problem semantics; fix remaining governance
  audit gaps; improve CVRP's manually declared deep surface enough to run a
  meaningful short diagnostic.
- **v0.5**: make Problem and Algorithm Onboarding a first-class module:
  solver entrypoint map, component inventory, draft `ProblemSpecV1`, adapter
  checklist, surface manifest, runtime-audit plan, tests, and mandatory human
  review.

## Scion Core Findings

The core control chain still mostly satisfies v3: `DecisionEngine` reads
structured features, not proposal rationale or raw metrics. The main core
risks are in Contract/proposal governance around selected surfaces, not in the
promotion decision layer.

### P0: C10 Semantic Novelty Still Has Free-Text Fallback

`ContractGate` comments indicate that missing structured signature fields for
singleton semantic novelty should fall back to strict target-file identity.
The implementation instead falls back to normalized `hypothesis_text` for
singleton surfaces. That lets LLM wording influence duplicate detection.

Relevant code:

- `scion/scion/contract/gate.py`
- `scion/scion/problem/spec.py`

Impact: C10 no longer collapses every singleton attempt by target file, but the
repair is incomplete. A candidate can still bypass novelty by changing natural
language rather than bounded structured identity. This weakens APS budget
control and v3 taint discipline. Treat this as a v0.4 governance blocker.

v0.4 action: make missing or unsupported semantic signature fields fall back to
strict target-file identity, or require problem-declared structured signature
fields for semantic novelty.

### P1: Patch-Level Contract Checks Still Infer Surface By Path

`validate_patch()` receives the approved hypothesis, but key patch-level checks
still infer the selected surface from target file paths. `C7` interface checks,
`C9d` instance-identity checks, and `C9c` complexity bounds should all use the
authoritative selected surface from the approved hypothesis when present.

Relevant code:

- `scion/scion/contract/gate.py`
- `scion/scion/contract/surface_interface.py`

Impact: this is a core design gap for deeper algorithm surfaces. It can
misclassify or under-check shared target files, singleton strategy files, and
future multi-surface or multi-policy artifacts.

v0.4 action: thread the approved selected surface through patch ContractGate
checks and APS preview. Reject explicit surface/target mismatches instead of
silently selecting a surface by path.

### P1: Research-Surface Return Contracts Are Too Shallow For Deep Strategies

The current return-contract schema covers simple types, allowed literals,
top-level dict keys, and broad numeric ranges. That is enough for small policy
functions, but weak for nested strategy plans with phases, component lists,
conditional parameters, budget schedules, and resource semantics.

Relevant code:

- `scion/scion/problem/spec.py`
- `scion/scion/contract/surface_interface.py`

Impact: Contract remains safe, but it cannot give strong static feedback for
deep non-operator surfaces. Too much validation falls to runtime audit, which
currently checks field presence more than behavior predicates.

v0.4 action: if `main_search_strategy` remains the v0.4 closeout surface, add
the smallest generic extension needed for nested plan validation or
problem-owned static validators. Larger schema generalization belongs in v0.5.

### P1: Lineage Does Not Persist The Actual DecisionFeatures Snapshot

`DecisionCoordinator` and `DecisionEngine` carry the real `DecisionFeatures`
snapshot, but `EvidenceRecorder` reconstructs a partial metadata object for
`decision_features_json`. The persisted row can omit actual decision inputs
such as CI fields, gate outcomes, expand counters, and pair counts, while also
mixing in metadata that is not part of `DecisionFeatures`.

Relevant code:

- `scion/scion/core/decision_coordinator.py`
- `scion/scion/core/decision.py`
- `scion/scion/core/evidence_recorder.py`

Impact: lineage is append-only, but a reviewer cannot reliably replay or prove
that a recorded decision followed from the exact features used by
`DecisionEngine`.

v0.4 action: persist a schema-versioned serialization of the actual
`DecisionFeatures` snapshot returned with the decision, and keep raw refs /
runtime summaries in adjacent audit metadata rather than in the decision
feature payload.

### P1: Soft-Abandon Creates A Second Decision Path

`EvaluationOrchestrator` can override a `CONTINUE_EXPLORE` decision into
`ABANDON` for soft-abandon behavior after `DecisionEngine` has already produced
reason codes. `DecisionFinalizer` can then persist stale or misleading reason
codes.

Relevant code:

- `scion/scion/core/evaluation_orchestrator.py`
- `scion/scion/core/decision_finalizer.py`

Impact: this is not an LLM-taint issue, but it violates the v3 principle that
the deterministic Decision layer is the single source of promotion/abandon
outcomes.

v0.4 action: move soft-abandon into `DecisionEngine`, or represent it as an
explicit post-decision `CoordinatedDecision` outcome with its own reason-code
provenance.

### P1: Proposal Context Still Contains CVRP/ALNS Runtime Field Heuristics

`ContextManager` includes hardcoded "interesting" runtime suffixes such as
`_use_vns`, `_destroy_ratio`, and `_max_destroy_customers`.

Relevant code:

- `scion/scion/proposal/context_manager.py`

Impact: new problem packages or new algorithm surfaces need core edits to get
equivalent runtime feedback. This weakens the problem-agnostic boundary.

v0.4 action: move interesting runtime fields into problem/surface metadata, or
derive them from selected-surface `evidence.required_runtime_fields` plus
problem-declared feedback fields.

### P2 Core Follow-Ups

- `DecisionFeatures.statistical_metric` is an unchecked string and is not used
  by `DecisionEngine`.
- Protocol runtime summaries undercount inactive `*_active` fields compared
  with the runtime audit guard.
- APS selected-surface read can return an empty observation when tool budget is
  exhausted, instead of fail-closing the session.
- Legacy non-APS proposal context can still read whole champion surface files
  without the compact `surface-contract.v1` budget discipline used by APS.
- `scion.evidence` exports CVRP-specific final-evidence builders from a core
  namespace. This does not affect Decision today, but the boundary should be
  cleaned up in v0.5.
- Legacy no-adapter verification fallbacks still carry old problem-shape
  assumptions. Adapter-required packages fail closed, so this is compatibility
  debt rather than the current blocker.
- Research-surface schema remains too shallow for multi-file atomic surfaces,
  pre/postconditions, problem-declared feedback fields, and resource semantics.

These are not current blockers for the CVRP diagnostic, but they are relevant
for v0.5 onboarding.

## CVRP Problem-Package Findings

The CVRP package is a valid Scion integration object. It is not yet a strong
research object.

### P0: `main_search_strategy` Has No Component-Coverage Contract

`problem-v1.yaml` allows `route_pair_swap` and `bounded_destroy_repair`, but
does not require a forced diagnostic to select either component. Runtime audit
checks that the selected surface loaded, activated, emitted required fields,
and had no errors. It does not require coverage of the deeper components.

Impact: a candidate can select only `intra_route_2opt` and
`inter_route_relocate`, pass the surface contract, reach screening, and still
tell us nothing about the intended deeper algorithm components.

v0.4 action: add problem-owned diagnostic coverage expectations for forced
`main_search_strategy` runs. Runtime should distinguish:

```text
selected components
attempted components
accepted components
skipped components
skip reasons
per-component objective deltas
```

### P0: Deep Components Are Weaker Than The Original VRP Baseline Components

The original `vrp` baseline already has a stronger component library:

- construction: Clarke-Wright, sweep, nearest-neighbor, capacity-balanced;
- VNS: two-opt, relocate, Or-opt, swap, two-opt-star;
- ALNS destroy: random, worst, Shaw, route removal;
- ALNS repair: greedy, regret-2, regret-3;
- adaptive weights and simulated-annealing acceptance.

The CVRP package's `route_pair_swap` is a bounded scan with limited candidate
ranking, and `bounded_destroy_repair` behaves closer to single-customer
remove/reinsert than true bounded ALNS destroy/repair.

Impact: even if Sonnet selects the deeper components, they may not provide
enough leverage after a strong repo-local ALNS+VNS baseline.

v0.4 action: make the CVRP package's deep components reflect the original
baseline's real primitives, but keep them behind problem-owned bounded APIs and
runtime audit. Do not let proposals edit `vrp/src` directly.

### P1: `baseline_policy` Is Coarse Parameter Tuning

`baseline_policy` can pass bounded parameters into the repo-local baseline, but
it does not expose enough telemetry or structure to let proposals reason about
destroy/repair weights, acceptance rate, component history, or search phase
behavior.

Impact: candidates are effectively blind-tuning global parameters across mixed
screening cases.

v0.4 action: expose bounded baseline telemetry such as ALNS destroy/repair
selection, accepted ratios, iteration summaries, construction mode, VNS
improvements, and acceptance behavior.

### P1: `algorithm_blueprint` Cannot Exercise The Deeper Components

The blueprint path supports only shallow package-owned local-search components.
It does not exercise `route_pair_swap` or `bounded_destroy_repair`.

Impact: blueprint diagnostics cannot answer the current blocker.

v0.4 action: stop treating `algorithm_blueprint` as the next surface for
quality evidence. Use `main_search_strategy` or a narrower component-policy
surface that can actually invoke the intended components.

### P1: Tests Prove Plumbing More Than Component Effectiveness

Existing tests show surface loading, preview, selected-surface audit, and
runtime fields. They do not include controlled fixtures that force
`route_pair_swap` or bounded destroy/repair to produce a known improvement.

v0.4 action: add controlled fixtures:

- one where a route-pair swap must improve objective without violating
  capacity;
- one where bounded multi-customer destroy/repair must improve objective;
- assertions for selected, attempted, accepted, skipped reason, and objective
  delta per component.

### P2: Surface Bounds Can Drift From Solver Bounds

Some policy metadata and solver clamps are not identical. For example,
`search_policy` declares a broader `baseline_time_fraction` lower bound than
the solver actually accepts. `neighborhood_portfolio` also has declared caps
that can diverge from solver-side runtime caps.

Impact: small today, but it weakens the idea that the surface declaration is
the authoritative contract. New onboarding templates should treat metadata /
solver-bound consistency as a required test axis.

v0.4 action: add focused tests that compare declared numeric ranges and allowed
components against solver-side clamps for active CVRP surfaces.

## Original `vrp` Baseline Findings

The original baseline is not a Scion problem package; it is the human-written
solver being wrapped. Its algorithm is much richer than the surfaces currently
used by Sonnet.

Key components:

- Entry points: `main.py`, `benchmark.py`, `solve_instance.py`,
  `run_full_experiment.py`.
- Model: `src/models.py` with `Instance`, `Route`, and `Solution`.
- Construction: `src/construction.py`.
- Solver loop: `src/solver.py`.
- VNS: `src/local_search/vns.py` and `src/local_search/operators.py`.
- ALNS destroy/repair: `src/alns/destroy.py`, `src/alns/repair.py`.
- Adaptive weights and acceptance: `src/alns/weights.py`,
  `src/acceptance.py`.

For v0.4, the right move is not to let LLMs edit this baseline directly. The
right move is to hand-map the baseline's real component inventory into the CVRP
problem package:

| Baseline structure | Better Scion surface |
| --- | --- |
| Clarke-Wright / sweep / nearest / capacity-balanced construction | `construction_policy` or `main_search_strategy` |
| VNS operator order and budget | `neighborhood_portfolio` or `main_search_strategy` |
| random/worst/Shaw/route destroy + greedy/regret repair | `ruin_recreate` or `main_search_strategy` |
| SA acceptance and cooling | `search_policy` / future acceptance surface |
| adaptive weight scoring and reaction | `baseline_policy` / future portfolio policy |
| thresholds and construction-only mode | `baseline_policy` / `main_search_strategy` |

## v0.4 Recommendation

Do not launch a long CVRP solver-quality validation next.

First close the v0.4 governance issues in core:

1. Fix C10 semantic novelty fallback.
2. Thread approved selected surface through patch ContractGate checks and APS
   preview.
3. Persist actual `DecisionFeatures` snapshots in lineage.
4. Put soft-abandon under explicit Decision provenance.
5. Move CVRP/ALNS runtime-field heuristics out of core proposal context.
6. Add minimal nested-plan or problem-owned static validation only if needed to
   close `main_search_strategy` in v0.4; defer broad schema work to v0.5.

Then do one narrow CVRP surface-efficacy slice:

1. Add component-coverage expectations for forced `main_search_strategy`
   diagnostics.
2. Strengthen `route_pair_swap` with problem-owned candidate ranking or
   shortlist logic.
3. Replace single-customer `bounded_destroy_repair` with a genuinely bounded
   ALNS-style destroy/repair subset.
4. Add controlled fixtures proving the deep components can improve known cases.
5. Expose per-component runtime telemetry.
6. Run a short forced `main_search_strategy` diagnostic.

Proceed to longer validation only if that diagnostic shows both:

```text
deep components selected and attempted
nontrivial screening quality beyond tie/no-op evidence
```

## v0.5 Recommendation

v0.5 should not be framed as "better CVRP prompting." It should be framed as
Problem and Algorithm Onboarding.

The v0.5 onboarding assistant should draft, but not auto-approve:

- solver entrypoint map;
- problem model and objective/feasibility checklist;
- component-library inventory;
- candidate `ProblemSpecV1`;
- adapter checklist;
- research-surface manifest;
- target files and required functions;
- numeric bounds and resource semantics;
- runtime audit field plan;
- controlled fixture skeletons;
- protocol/split/seed checklist;
- human-review checklist.

For route-based metaheuristics, the first onboarding template should detect or
ask for construction, neighborhoods, destroy/repair, acceptance/restart,
budget schedule, random seed boundary, and per-component telemetry.

## Project Narrative Update

Recommended wording:

```text
v0.4 is Scion's v3-governed second-problem validation and boundary-hardening
release. It does not claim automatic solver ingestion or CVRP solver-quality
improvement. It aims to prove that, once a problem package has been manually
onboarded with adapter semantics, declared surfaces, runtime audit, and tests,
the same evidence-driven autoresearch loop can govern it without absorbing
problem semantics into Scion core.

v0.5 should make the missing onboarding layer explicit: converting a
human-written solver into a reviewed Scion-native research object before the
autoresearch loop begins.
```
