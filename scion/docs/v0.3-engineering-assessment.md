# Scion v0.3 工程价值评估

**日期**：2026-04-23
**作者**：Claude (Opus 4.7) × Xiao Jiyao
**定位**：读完 README / scion-architecture-v3 / scion-v0.3-design / sprint-w16-optimization / w16-results 等核心文档后的冷静评估，作为"当前 sanity validation 跑完后做优化"的输入
**范围**：工程价值优先，学术价值不在本文评估口径内

---

## 一、一句话定位

Scion 在"仓配协同 VNS + LLM 算子演化"这个具体问题上产出了真实价值；作为"LLM 驱动组合优化框架"的目标**只完成了一半**——架构模式成熟，实现仍是研究级代码。

---

## 二、工程上真正值钱的东西（可迁移）

### 2.1 Decision Input Guard 模式

LLM 输出永远是 tainted，Decision Layer 只读 `DecisionFeatures`（frozen dataclass，无自由文本字段），中间通过 Safe Feature Extractor 强制转换。这个边界在代码里是**硬的**，不是文档约束。

任何 LLM-driven codegen / 自动化项目都可以抄这个模式。

### 2.2 Contract / Verification / Protocol 三层独立结构

- **Contract**：结构/边界（文件白名单、import、AST）
- **Verification**：语义正确性（feasibility、objective 重算、state leak）
- **Protocol**：统计可靠性（三级 split + bootstrap CI）

很多 "LLM codegen with safety" 项目把这三件事糊成一个 `validate()` 函数，Scion 把它们彻底分开了。

### 2.3 Screening / Validation / Frozen 三级协议 + 暴露控制

- per-case 原始数据只在 screening 对 LLM 可见
- validation 只露 aggregate
- frozen 只露 pass/fail

有效防止 overfitting 到评估集。这是**最容易迁移到其他 LLM-driven 自动化场景**的设计。

### 2.4 Append-only SQLite lineage

`hypothesis → contract → code → verification → protocol → decision` 全链路事件可回放。MVP 阶段不搞 hash-chain 但接口预留，节奏感对。

### 2.5 工程闭环本身

W16 实验（33h）→ 写 9 个 P-code 问题清单（w16-results §7）→ Phase 1-5 系统化修复 → 端到端 sanity validation。**观察→归因→修复→验证**循环是这个项目最值钱的工程资产之一。

## 三、工程上真正脆弱的东西

### 3.1 `core/campaign.py` 是 2375 行 god object

v0.3 reviewer 点名要求拆成 `ProblemRuntime` / `AsyncWeightOptCoordinator` / `PlateauController`，v0.3 没做。W2/W3/W7 全部继续往 campaign.py 里塞。下一次 refactor 的必然入口。

### 3.2 Adapter 抽象的 silent fallback 是设计缺陷

`context_manager.py:313`：

```python
if adapter is not None and hasattr(adapter, 'render_problem_summary'):
    return adapter.render_problem_summary()
# legacy fallback
```

**忘记传 adapter 不会报错，会静默降级为 warehouse 硬编码逻辑**。真正的 adapter 抽象应该缺 adapter 直接 raise。legacy fallback 借口是"后向兼容"，实际是抽象不彻底的信号。

类似问题还有：
- `_MECHANISM_KEYWORDS` 硬编码 fallback 保留
- `extract_champion/candidate_metrics` 仍引用字符串 `"champion_splits"/"champion_cost"`
- `CaseAggregateFeedback` 的 deprecated aliases 保留
- `compare_with_breakdown()` legacy 函数保留

### 3.3 "多问题框架"的证据不足

除 warehouse_delivery 外只有 `toy_tsp`，且官方描述：**"Minimal Working Example，不追求质量，只需过 adapter 全部 Protocol 方法"**。

换句话说，**adapter 接口没在第二个真实问题上压测过**。CaseAggregateFeedback / FamilyTaxonomy / MechanismKeywords 仍有 warehouse 思维残留。

### 3.4 MILP 下界无效，但项目 RQ4 依赖它

w16-results §5 自己的结论："Large+ 规模上 Champion 超越 MILP 是因为 MILP 模型不能表达车辆重构"。

即：**MILP 不是下界，是另一个 heuristic 的最优**。RQ4（"距离最优解的 gap"）目前**物理上不可回答**。对核心机制（promote / V5 / 早停）影响有限，但对外宣称"gap to optimum"是空话。

### 3.5 Experiment orchestration 是手工作坊

W16 的 12 campaigns 是 6 批 × 2 并发手动 bash 启动。当前 sanity validation 也是 `bash -c ... | tee` 起的。

CLAUDE.md 明确要求"长期任务必须 systemd --user 或 nohup+setsid，绝不 bare tmux"，但 `run_validation_campaign.py` 启动脚本没走这层。**断开终端就完蛋**。

### 3.6 fix_code retry 在 INFO 级别完全静默

今天的 sanity check 通过文件 mtime 才推断出 light fix 通过。严重的可观测性漏洞。

W17 全矩阵起来后，GPT 上百次 V5 失败的修复路径 debug 会是灾难。

### 3.7 并发架构凑合

Python threads + 全局 `_champion_lock` + `sys.path` 修改。v0.3 修了 immutable snapshot 那个最坏的 race，但**根本性的"进程级共享状态"问题没解决**。多 campaign 并发靠路径隔离和运气。

### 3.8 4 个 pre-existing test failures 从 v0.2 带到 v0.3 没修

不是 block 问题，但是**测试红线不严格的信号**。

## 四、当前就可提取的独立模块（不需 refactor）

以下模块可以**单独提取**复用到其他项目：

| 模块 | 估算行数 | 独立价值 |
|---|---|---|
| `scion/problem/contracts.py` + `objectives.py` + `loader.py` | ~300 | ProblemAdapter Protocol + 通用 lexicographic 比较 |
| `scion/runtime/subprocess_runner.py` + `workspace.py` | ~500 | subprocess 隔离 + workspace materialization |
| `scion/verification/gate.py` | ~400 | V1-V8 checklist + light/heavy severity + fix retry |
| `scion/protocol/` | ~800 | 三级评估 + bootstrap CI + exposure control |
| `scion/core/features.py` (DecisionFeatures) | ~100 | Decision Input Guard 教科书实现 |

这些模块可以打包成 `llm-codegen-safety-kit` 或类似独立库，**比提取整个 Scion 容易 10 倍**。

## 五、优化路径：两条

### 路径 A：让 Scion 继续解决仓配协同，尽可能工程化

**目标**：别人接手不痛苦。约 3-4 周。

**P0（必做）**：
1. 拆 `campaign.py` → `ProblemRuntime` / `AsyncWeightOptCoordinator` / `PlateauController` 三个 service
2. 去掉所有 `if adapter is not None` fallback，adapter 缺失直接 raise
3. `run_validation_campaign.py` 接 `systemd --user` 或至少 `nohup+setsid`
4. fix_code / light retry / V5 路径全链路加 INFO log，能 grep

**P1（应做）**：
5. 删除 `ObjectiveBreakdown` / `_MECHANISM_KEYWORDS` 硬编码 / `compare_with_breakdown` 等 DEPRECATED 代码
6. 修 4 个 pre-existing test failures（至少搞清楚为什么还挂着）
7. 写一个 campaign launcher 脚本（queue + 并发控制 + 失败重启），替代手工 bash

**P2（可选）**：
8. `_champion_lock` + `sys.path` 的进程级共享状态改成 workspace-local
9. Observability：prometheus metrics / campaign 进度 JSON 实时输出

### 路径 B：让 Scion 变成真正的多问题框架

**目标**：第二个团队可以用它做第二个问题。约 3-4 个月。

**A 的所有 P0/P1 +**：
10. **接入第二个非平凡问题**（CVRP / JSSP / FlowShop 其一），跑通一次完整 W16-规模 campaign
11. 在此过程中暴露出来的每一处 warehouse 残留都修掉
12. MILP 接口扩展为"多下界来源"（MILP / LP relaxation / column generation / problem-specific bounds），由 adapter 决定
13. 并发架构升级：多进程 + queue，而非 thread + shared state
14. 文档：`design/scion-problem-interface-v1.md` 正式化（v0.3 plan 里列了但没写）

## 六、当前最大的单点风险

**你在一个 problem 上过拟合了 framework 设计。**

W17 是自救，但 W17 还没跑，而且设计只有 `1 model × 2 variants × 3 seeds`。在找到第二个真实问题并花 4-6 周接入之前，"Scion 是框架"是一个**尚未检验的 claim**。

warehouse_delivery 的假设在这些地方还有残留（今天已观察到）：
- `_MECHANISM_KEYWORDS` 关键词（subcategory / consolidation / merge 等）
- `CaseAggregateFeedback` 的 legacy aliases
- `extract_champion/candidate_metrics` 的字段名硬编码
- `context_manager` Locus 集合虽然改成 `spec.operator_categories` 驱动，但 stagnation handler 的 `order_level ↔ vehicle_level` 切换逻辑还是 warehouse 假设

## 七、学术 vs 工程价值分离的坦率评估

学术价值路径（**非本文评估口径**，但注明避免混淆）：
- 三级协议 + 暴露控制可以独立成一篇方法论论文
- W16 的"Champion vs MILP crossover at ~110 orders"是有意思的经验结论，可进 case study
- "LLM-driven operator evolution beats restricted-MILP on large instances" 是一个可 defend 的 claim

工程价值路径（本文口径）：
- 核心机制稳定（V5 light + fix retry、早停三规则、adapter 渲染、ObjectiveComparison 管线今日 sanity 已经确认工作）
- 实验基础设施脆弱（bash + tmux + 手工调度）
- 可复用单元清晰（第四节 5 个模块）
- 框架化 claim 未被第二个 problem 检验

---

## 八、最冷的结论

**Scion 目前是一个做得很认真的研究原型，有生产级的方法论意识，但实现距离生产级还差一层工程投入。**

它的价值不在代码本身被大规模复用，而在**架构决策作为参考样本**——任何想做 LLM 驱动自动化、又必须严肃对待评估可信度的项目，都应该先读一遍 `scion-architecture-v3.md` 再开工。

三层 gate、Decision Input Guard、三级协议 + 暴露控制这三样东西单独拆出来都有独立价值。整合在一起、绑死在 warehouse_delivery 上、用 bash + tmux 跑 campaign——是这个项目现在的样子。

路径 A（3-4 周）让它变成"能稳定运行的仓配算法优化工具"。
路径 B（3-4 个月）让它变成"真正的框架"。

两条路径的分叉点是一个问题：**你真的想让它成为框架吗？还是仓配协同是主战场，框架只是副产品？** 这不是技术问题，是优先级问题。

---

## 附：本评估的局限

- 未实际跑第二个问题验证 adapter 抽象
- 未做 `campaign.py` 拆分的可行性 spike
- 未 benchmark 并发架构在更大规模（>4 并发）下的行为
- MILP 扩展为 column generation 的复杂度未评估
- 成本估算（3-4 周 / 3-4 个月）是 rough bound，不是精确估算

任何一条结论在实际优化时都应先做小规模验证，不要把评估当 spec。
