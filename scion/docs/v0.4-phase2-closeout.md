# Scion v0.4 Phase 2 Closeout

*Date: 2026-05-01*
*Branch: v0.4-dev*

## Summary

Phase 2 completed the framework hardening needed before larger CVRP campaigns.

The main outcome is that runtime verification, evidence recording, proposal
guidance, and CVRP campaign wiring are now adapter-native and performance-aware
without pushing free-text runtime evidence into the Decision Layer.

## Completed Work

### Runtime Verification

- `V5_solution_consistency` uses `ProblemAdapter.check_solution_consistency`
  when an adapter is present.
- Strict runtime verification can require an adapter and fails closed when
  runtime prerequisites are missing.
- CVRP runtime checks no longer fall back to warehouse-shaped assignment
  consistency assumptions.

### Runtime Evidence

- `CheckResult` carries structured metadata.
- `V9_perf_guard` records bounded runtime facts:
  - case id;
  - timeout;
  - candidate/champion elapsed ms;
  - slowdown ratio;
  - timeout/error categories.
- `EvidenceRecorder` persists verification checks and runtime guard evidence in
  lineage metadata.

### Decision and Proposal Boundaries

- `DecisionFeatures` exposes only bounded runtime fields:
  - `runtime_guard_passed`;
  - `runtime_guard_ratio`;
  - `runtime_guard_timeout`.
- Runtime failure text is not passed to `DecisionEngine`.
- Proposal context can include bounded runtime feedback from prior V9 failures
  as guidance for future hypotheses.

### CVRP Smoke Path

- Added `scion.problems.cvrp` with route-native adapter, model, solver, base
  operator, protocol config, split manifest, seed ledger, and tiny synthetic
  fixtures.
- Promotion objective is lexicographic:
  1. minimize `fleet_violation`;
  2. minimize `total_distance`.
- BKS/gap remains a final reporting concept, not a promotion objective.
- Added CVRP protocol smoke and one-step `CampaignManager` smoke using real
  runner/protocol/verification and mock LLM.

### ProblemSpecV1 Bridge

- Added `scion.problem.bridge`.
- `ProblemSpecV1` is now the single source for deriving:
  - legacy `ProblemSpec`;
  - metric specs;
  - objective policy;
  - operator execute signature.
- CVRP campaign smoke no longer hand-builds a second legacy problem spec.

## Behavior Changes

- Adapter-backed V5 is authoritative when an adapter is present.
- Runtime guard failures now produce structured evidence fields.
- CVRP strict verification uses `data/tiny_canary.json` from
  `problem-v1.yaml`.
- No intentional change was made to promotion thresholds or accept/reject
  policy.

## Remaining Work For Phase 3

Phase 3 should move from smoke integration to research-useful CVRP execution:

1. Add a real CVRP benchmark/reporting harness that keeps route-count
   comparability explicit and reports BKS/gap only as final evidence.
2. Define the campaign protocol for CVRP validation/frozen reporting beyond the
   tiny synthetic fixtures.
3. Continue `campaign.py` decomposition around loop/control responsibilities,
   likely `CampaignLoop` or `DecisionCoordinator`.
4. Add stricter performance-aware contract rules for obviously unbounded
   generated operators.
5. Decide whether `EvaluationPipeline` should become long-lived after campaign
   loop ownership is separated.
