# Scion v0.3 Production Timeout-Fix Validation

*Date: 2026-04-28*

## Context

The formal 12-campaign run exposed two production-specific correctness issues:

1. A generated production operator used unbounded high-order vehicle combinations and became unusably slow on large production cases.
2. The old protocol skipped failed/timeout pairs, so a frozen gate could promote from incomplete evidence.

Fixes landed:

```text
74c6869 fix(v0.3): fail incomplete evidence and bound operators
e8e3673 fix(surrogate): tolerate stale move-order assignments
d29d843 docs(v0.4): add performance-aware plan
```

This validation reran production-only campaigns after those fixes.

Interpretation:

- The production gains are structural, not weight-driven.
- Sync weight optimization closure works correctly.
- For production, weight optimization currently adds runtime cost but no marginal gain in these three runs.

## Model Difference

Sonnet:

- 3/3 production campaigns promoted.
- All promotions had complete evidence.
- All promotions improved `total_cost` while preserving `subcategory_splits`.

GPT-mini:

- 0/3 production campaigns promoted.
- Main failure remains generated-code reliability:
  - `V5_solution_consistency` x8 / x7 / x2 for seeds 11 / 29 / 47.
  - seed47 also had 4 contract failures.

Interpretation:

- Production optimization is feasible under the fixed framework.
- Production success depends strongly on model capability.
- GPT-mini is not yet reliable enough for production-scale operator generation.

## Impact On v0.3 Conclusion

This rerun supersedes the production part of `v0.3-final-12campaign-analysis.md`.

Updated conclusion:

- Synthetic result remains valid and strong.
- Production result is now stronger for Sonnet than the first formal run suggested: 3/3 Sonnet production promotions under complete evidence.
- GPT-mini production remains 0/3.
- Framework correctness is improved: invalid evidence no longer supports promotion.
