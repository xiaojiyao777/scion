## 0. 总结论

这次 6-campaign closure validation 的结论需要分成两层看。

**工程框架层面：通过。** 6 个 campaign 全部正常结束，没有 launcher crash、campaign traceback、缺失 summary、缺失 metrics、DB 无法审计等问题。`status.json`、`campaign_summary.json`、`validation_runner_summary.json`、`scion.db`、`metrics/*.json` 都可用于后验追踪，branch lifecycle、protocol decision、promotion、weight revision、failure routing 都有证据链。

**搜索效率层面：未达预注册目标。** Sonnet synthetic 平均 promotion 为 1.0，低于目标 `>=2.0`；GPT-mini synthetic 平均 promotion 为 1.33，低于目标 `>=1.5`。这说明 v0.3 closure 已经修复严重执行/证据问题，但还没有恢复到 W16 预期的 promotion cadence。

**算法优化层面：有真实优化，但结论边界必须收紧。** 6 个最终 champion 在固定 frozen gate 后验评估中全部 12/12 pair 战胜 baseline；median f1 改善范围为 `+9` 到 `+41` 个 `subcategory_splits`。这足以说明 Scion v0.3 能自主生成、验证并推广有效算子。但该证据只覆盖本次 closure rerun 的 synthetic frozen gate；它本身不回答 production 泛化、weight-opt 单独贡献、early-stop 因果收益或 classifier 因果收益。这里不是否定 W16 的 production 证据，而是区分“本次实验能证明什么”。

**CPLEX / MILP 层面：只能作为 report-only reference。** CPLEX 结果包含 `optimal`、`feasible`、`timeout`、`infeasible` 多种状态；即使是 reported `optimal`，在当前项目里也仍要受 MILP 建模范围、数据版本和 runtime oracle 一致性的约束。后验评估发现当前 VNS baseline/final champion 在若干 frozen case 上给出比 CPLEX-reported reference 更低的 f1，且 runtime oracle 认为可行。这不能解释为“Scion 超过最优解”，而是说明这些 MILP 数字应作为离线参考对比，不应作为严格 exact-gap certificate。

## 1. 本次实验能回答什么

本次实验不是 `design/validation-prereg.md` 中的完整 W16 18-campaign matrix，而是 `docs/v0.3-closure-validation-prereg.md` 定义的 synthetic closure rerun。它的目标是验证 closure hardening 后：

- campaign 主循环是否稳定；
- A/T 系列修复后是否不再出现早期中断和证据丢失；
- B1-B3 service extraction 后是否没有引入行为回归；
- adapter-backed objective / hierarchical protocol / async weight revision 是否能在真实长实验中闭环；
- synthetic full-system search 是否仍能产生有效 promotion；
- promotion cadence 是否恢复到预注册阈值。

它不能单独回答：

- synthetic 与 production 的泛化差异，也不覆盖 W16 已有 production 结果；
- Sonnet/GPT-mini 之外的模型差异；
- structure-only、weight-only、early-stop、classifier 的因果贡献；
- classifier 是否提升搜索效率；
- CPLEX/MILP reference gap 的严格最优性结论。

这些问题仍属于 W16 完整矩阵、W17 mechanism study 或 v0.4/v1 的范围。

## 9. Weight Optimization 行为

Weight optimization 共运行 7 次，全部在内部 score 上有 improvement，并以 revision `r1` 落盘。

| Campaign | Campaign hours | Weight-opt cumulative hours | Runs | Best scores |
|---|---:|---:|---:|---|
| `gpt-54-mini_synthetic_seed11` | 4.31 | 4.34 | 2 | 3.0, 1.0 |
| `gpt-54-mini_synthetic_seed29` | 4.02 | 1.80 | 1 | 1.0 |
| `gpt-54-mini_synthetic_seed47` | 2.31 | 1.78 | 1 | 1.0 |
| `sonnet-4-6_synthetic_seed11` | 2.61 | 1.62 | 1 | 2.0 |
| `sonnet-4-6_synthetic_seed29` | 2.74 | 1.52 | 1 | 2.0 |
| `sonnet-4-6_synthetic_seed47` | 3.53 | 2.03 | 1 | 2.0 |

这里的 weight-opt hours 是 background worker 的累计耗时，不等同于额外串行 wall-clock；因此 GPT seed11 的累计 weight-opt hours 可接近 campaign wall-clock。

设计符合性：

- final champions 均有 `r1` weight revision；
- structural version 与 weight revision 分离；
- weight-only snapshot 单独持久化；
- obsolete weight update 会被丢弃；
- committed weight update 会触发 active branch stale/reconcile。

结论：async weight revision lifecycle 是 v0.3 的成功项。

但这次实验不能证明 weight opt 的边际贡献，因为没有 weight-opt-off paired run。这个问题仍需要 W17-B 或 offline paired analysis。

| Case | Baseline median f1 | CPLEX f1 | Baseline - CPLEX |
|---|---:|---:|---:|

这只能说明存在一致性问题，可能来源包括：

- CPLEX 使用的数据版本与当前 runtime 数据不同；
- MILP 模型比 runtime oracle 更严格；
- runtime oracle 漏掉了 MILP 中的某些约束；
- MILP/VNS objective mapping 不一致；
- CPLEX result 缺少足够的数据 hash 和 solution replay provenance。

Production 组还给了一个重要补充：在 CPLEX final production 的 31 个可比较实例中，v1 baseline warm start 与 CPLEX reference 打平 26 个；5 个 CPLEX 更优的 case 全部在 screening 小/中规模上，且主要是 cost 小幅改进。production frozen 和 validation 上，v1 baseline 与 CPLEX reference 全部打平。这说明 production v1 baseline 本身已经非常强，特别是在 frozen/validation 大实例上，MILP/CPLEX reference 很多时候没有比 v1 baseline 找到更好的解。

但 W16 production final champion 仍有可观改进：Sonnet production 三个 campaign 均有 1 次 promotion，frozen md 为 `+32,300` 到 `+46,800`；production frozen cost gap 对 CPLEX/HiGHS reference 的后验汇总为 Sonnet `-15.2%` 到 `-22.8%`，GPT seed29 为 `-4.2%`。因此更准确的解释不是“production 上没有优化空间”，而是：

- v1 baseline 已经把 production 的 primary objective 基本做到很强；
- 后续 Scion 的主要收益转向 cost / vehicle-type / fleet compaction；
- first promotion 之后再次改进变难，所以 production promotion cadence 低。

v0.3 最终报告中应采用以下表述：

- CPLEX 是 report-only reference；
- CPLEX `optimal` 暂不作为跨 runtime oracle 的严格下界证书；
- 不报告或谨慎标注 `optimality_gap`；
- 添加 “candidate better than exact reference” sanity warning。

## 13. 是否正常、是否符合预期、是否真的优化

**实验是否正常？**

正常，若“正常”指工程执行和证据闭环：6 个 campaign 都完整跑完，无 crash，可审计，最终 champion 可复跑。

不正常，若“正常”指恢复 W16 promotion cadence：Sonnet 低于目标，GPT-mini 也略低于目标。

**是否符合 v0.3 设计框架？**

大体符合。核心框架边界、三层 gate、hierarchical objective、adapter-backed runtime、async weight revision、lineage evidence 都在真实长实验中跑通。剩余不足主要是搜索策略效率、early-stop calibration 和 CPLEX reporting consistency，不是主循环工程闭环问题。

**是否真正实现算法优化？**

是，但范围要精确。证据支持：

- 7 个结构性算子/算子修改通过 frozen holdout；
- 每次 promotion frozen pair wins 都是 12/12；
- 6 个 final champion 后验固定 frozen eval 都 12/12 战胜 baseline；
- 改善主要发生在 primary objective `subcategory_splits`；
- promoted operator 被写入实际 champion snapshot，不是报告层虚构改进。

证据不支持：

- 泛化到 production；
- 优于 CPLEX exact optimum；
- weight opt 单独有效；
- early-stop 带来因果收益；
- classifier 带来因果收益；
- 当前 search policy 已经成熟到能稳定 100 rounds 深度优化。

## 14. 对是否进入后续正式实验的判断

可以把 v0.3 作为一个干净的工程基线进入后续正式实验，但需要在实验声明中保持以下边界：

1. v0.3 的主结论是“工程化 autonomous optimization loop 已闭环，可审计，可复跑，并能产生真实 operator improvement”。
2. promotion cadence 没有恢复到预注册目标，后续实验应把搜索效率作为研究问题，而不是假设已解决。
3. CPLEX 只作为 report-only reference，不能作为 strict exact gap 证书。
4. 若要证明泛化，需要独立 analysis set 或 production/mixed matrix。
5. 若要证明机制贡献，需要 W17 paired/offline study。

简短表述：

> v0.3 是 Scion 从原型走向工程化研究框架的里程碑：框架闭环、证据闭环、问题适配边界和多目标 protocol 都已在长实验中跑通，并能产生真实算法改进。但它还不是高 cadence 的成熟搜索器；search policy、early-stop calibration 和 MILP/oracle consistency 是下一阶段的核心研究与工程问题。
