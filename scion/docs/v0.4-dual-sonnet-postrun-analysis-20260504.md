## Executive Conclusion

The run shows framework defects, not only weak generated heuristics.

CVRP did not merely "fail to improve". It repeatedly evaluated post-baseline
operators in a way that made no-op operators expensive: every solver run can
apply each generated operator for up to 20 rounds even when the previous round
accepted no improvement. With a 5s solver budget and a real `vrp/src` ALNS+VNS
baseline consuming roughly 4s on large cases, this created avoidable candidate
timeouts and turned runtime pressure into search noise.

Warehouse did promote once, proving the old problem path still works. But it
also stopped at 25 rounds instead of the requested 50 because the generic idle
early-stop policy overrode the experiment intent. Two proposal-schema failures
were not represented as normal `campaign_summary.json` step rows, leaving an
audit gap.

The most important design deviation is from the v3/v0.4 evidence loop:

```text
hypothesis -> branch -> code -> contract -> verification -> protocol
-> structured decision -> evidence -> context feedback -> next hypothesis
```

The run produced evidence, but the proposal context compressed several distinct
CVRP causes into "scr=0.00" style summaries. The LLM then inferred a false
schema-mismatch diagnosis and spent many later rounds writing introspection
operators rather than addressing the actual causes: strong baseline, no accepted
operator moves, timeout pressure, and weak post-baseline search surface.

## Design Criteria Used

The audit is judged against:

- v3 blueprint: Contract, Verification, Protocol, Safe Feature Extractor, and
  Decision are independent control surfaces; Decision reads only structured
  features; Lineage is auditable and replayable; Context Manager controls LLM
  exposure.
- v3 branch model: one branch is one hypothesis direction that can iteratively
  use its own evidence and failure history.
- v3 holdout model: screening is not promotion evidence; validation/frozen are
  governed and scarce.
- v0.4 design: performance and evidence completeness are first-class promotion
  constraints; runtime feedback should enter proposal context as bounded
  guidance; CVRP promotion objective is `fleet_violation -> total_distance`;
  BKS/gap is final evidence only; formal campaigns require final quality/runtime
  evidence.
- v0.4 P1 runtime design: timeout/crash/incomplete runtime evidence cannot
  promote, and runtime feedback should help future hypotheses avoid repeating
  slow or failing mechanisms.

## CVRP Input/Context Quality

The first CVRP branch received a clean, problem-specific context:

- CVRP problem summary;
- lexicographic objective policy: `fleet_violation -> total_distance`;
- solver mechanics: `vrp/src` baseline, post-baseline generated registry
  operators, acceptance only on feasible lexicographic improvement;
- no champion operators.

By late rounds, dynamic context was present in `system_blocks[2]`:

- Campaign Search Memory;
- Campaign Research Journal;
- Exploration Coverage;
- Strategy Guidance;
- Objective Opportunity Profile;
- Runtime Feedback.

So the problem is not that context was absent. The problem is that causal
compression was wrong or too weak.

Examples:

- Research journal repeatedly summarized branches as `scr=0.00 -> abandon`.
- This hid whether the branch had objective ties, candidate timeouts, champion
  timeouts, accepted operator moves, or contract failures.
- Later hypotheses explicitly claimed that all operators probably failed
  because of "schema mismatch", "dict key mismatch", or "silent crash".
- That diagnosis is contradicted by the raw metrics: many candidates ran through
  the correct `solution.routes` API, loaded operators, and simply produced no
  accepted improvements or timed out.

The LLM therefore shifted from algorithm design to schema introspection:

- `schema_probe_two_opt.py`
- `adaptive_schema_two_opt.py`
- `two_opt_no_imports.py`
- `or_opt1_list_discovery.py`
- `two_opt_intra_introspect.py`
- `detour_relocate_introspect.py`

This is a context-governance failure. v0.4 says runtime feedback may enter
proposal context as bounded guidance. It entered, but it did not explicitly
separate:

- no accepted operator moves;
- candidate timeout;
- champion timeout;
- contract rejection;
- weak positive signal;
- wrong hypothesis family;
- wrong API assumption.

## CVRP Runtime Mechanism Failure

The CVRP solver applies generated operators after the baseline. The generated
operator loop currently does:

```text
for round in 0..19:
    for operator in registry:
        run operator
        accept only if lexicographic objective improves
        otherwise mark skipped
```

There is no break when a full operator round accepts no improvement.

Observed consequence:

- A no-op operator on a complete 24-pair screening stage often produced exactly
  `24 * 20 = 480` operator attempts.
- These attempts were mostly `skipped/not an improvement`.
- The ALNS+VNS baseline used about 4s budget on large cases inside a 5s solver
  budget, leaving little room for repeated post-baseline repair.
- Candidate timeouts became common on larger cases such as `E-n101-k8` and
  `P-n101-k4`.

This violates the v0.4 runtime design intent. Runtime should be a governance
signal and optimization target; it should not be damaged by the framework's own
post-baseline loop.

## CVRP Operator Quality

The generated operators were syntactically and structurally mostly valid after
contract/verification, but weak relative to the runtime surface:

- The first half is standard CVRP/TSP local search: 2-opt, Or-opt, 3-opt,
  2-opt*, node swap, route merge, ruin/recreate.
- These are reasonable against a nearest-neighbor baseline, but they are much
  less likely to beat a repo-local ALNS+VNS baseline that already spends most of
  the time budget.
- Many operators used repeated exhaustive neighborhoods or defensive wrappers,
  making them less suitable as a small post-baseline improvement layer.
- Later operators were shaped by false schema-diagnosis and became diagnostic
  probes rather than competitive heuristics.

The interface may be too weak for formal CVRP improvement: Scion currently lets
the LLM add post-baseline operators, not alter the baseline search schedule,
neighborhood portfolio, acceptance policy, destroy/repair mix, or runtime
budget allocation. For a strong ALNS+VNS baseline, this leaves little room.

## Root Causes

### R1: CVRP post-baseline loop repeats no-op operators

Severity: P0 for CVRP experiment validity.

Evidence:

- `scion/scion/problems/cvrp/solver.py` loops up to 20 operator rounds.
- There is no stop condition for "no operator accepted in this round".
- CVRP raw metrics show 480 attempts on many 24-pair no-signal stages.

Impact:

- Runtime budget is consumed by repeated no-op attempts.
- Candidate timeouts become framework-induced.
- Runtime-aware optimization becomes noisy: candidates are punished partly for
  framework loop policy.

Repair:

- Track per-round accepted count.
- Break after a full round with zero accepted moves.
- Record `operator_no_improvement_rounds` and `operator_stop_reason`.

### R2: CVRP context compresses failure causes incorrectly

Severity: P0/P1 for research-loop quality.

Evidence:

- Later hypotheses explicitly blame schema mismatch and wrong dict keys.
- Raw metrics show operators loaded and ran via `solution.routes`; accepted
  moves were rare and timeouts were common.
- Research journal and trajectory text overemphasize `scr=0.00` and understate
  operator attempts/accepted/timeouts.

Impact:

- Branch search collapses into diagnostic introspection instead of better
  algorithm design.
- The LLM repeats families with false causal explanations.

Repair:

- Render protocol reason codes, valid/failed pairs, candidate/champion failure
  counts, runtime ratio, operator attempts/accepted/skipped/errors when
  available.
- Render contract failures as contract failures, not as algorithmic failures.
- For screening with `candidate_failed_pairs > 0`, label root cause as runtime
  failure before objective no-signal.

### R3: Forced 50-round experiments are not possible through current CLI

Severity: P1 for validation methodology.

Evidence:

- Both campaigns requested 50 rounds and stopped at 25.
- Stop reasons were idle-ratio early-stop rules.

Impact:

- Experimental intent and framework behavior diverge.
- A "50-round validation" can actually be a 25-round early-stop run.

Repair:

- Add CLI option to disable early stop for formal diagnostics.
- Alternatively add explicit `--validation-mode` with `force_continue=True`.

### R4: Proposal-stage failures are not first-class summary steps

Severity: P1 for v3 auditability.

Evidence:

- Warehouse `total_rounds=25`, summary rows=23.
- The two missing rows correspond to LLM schema validation failures.

Impact:

- The summary cannot reconstruct every round without reading `run.log`.

Repair:

- When hypothesis/code generation fails, record a `StepRecord` with
  `failure_stage=proposal` or `code_generation` and a deterministic failure
  detail.
- Ensure summary row count matches campaign rounds, modulo explicitly skipped
  scheduler no-op rounds.

### R5: CVRP research surface is too narrow for a strong ALNS+VNS baseline

Severity: P1/P2 depending on v0.4 goal.

Evidence:

- Operators are only post-baseline registry operators.
- Baseline already uses ALNS+VNS for real `.vrp` cases.
- Textbook post-baseline local search barely improves and often times out.

Impact:

- The current CVRP adapter can test framework plumbing, but it may not offer a
  realistic search surface for discovering improvements over the baseline.

Repair options:

- Keep post-baseline operators for smoke/controlled fixtures.
- For formal CVRP, expose baseline neighborhood portfolio, destroy/repair
  policy, acceptance schedule, operator weights, iteration allocation, or
  runtime budget strategy as adapter-owned research surfaces.

## Recommended Repair Order

1. CVRP runtime loop: break after no accepted move in a full operator round and
   audit the stop reason.
2. Context/evidence causality: render structured protocol/runtime/operator
   causes into proposal context and research journal.
3. Architecture P0-1: formal `.vrp` baseline fail-closed.
4. CLI validation mode: allow requested rounds to run without idle early-stop.
5. Proposal failure evidence: add summary/lineage rows for LLM schema/code
   failures.
6. Architecture P0-2/P0-3: frozen-use budget and final evidence readiness.
7. P1 hardening: programmatic strict runtime default, route-native taxonomy, and
   complexity guard expansion.
8. CVRP research surface: decide whether formal CVRP should optimize
   post-baseline operators only or adapter-owned baseline search policy.

## Immediate Re-Run Criteria

A short rerun is meaningful only after at least items 1 and 2 are fixed.

Expected changes after those fixes:

- no-op CVRP operators should show roughly 24 attempts per screening stage, not
  480;
- candidate timeouts should drop materially;
- later CVRP prompts should distinguish "no improvement after N attempts" from
  "candidate timeout" and should stop claiming schema mismatch without evidence;
- a forced diagnostic run should complete the requested round count unless it
  hits an explicit hard failure or balance limit.
