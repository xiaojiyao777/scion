# Scion v0.4 P1 Runtime-Aware Optimization Design

*Date: 2026-05-04*
*Branch: v0.4-dev*

## Position

P1 makes algorithm efficiency a default Scion optimization capability.

The v3 blueprint defines objective comparison as lexicographic business or
problem metrics followed by runtime and efficiency. v0.4 applies that rule at
the framework layer: a candidate cannot promote by improving objective values
while producing incomplete evidence, timing out, crashing, or regressing runtime
beyond the accepted bound.

This is not a CVRP-only feature. CVRP exposes the need clearly because route
operators can become expensive quickly, but the same rule applies to every
problem adapter.

## Invariants

- LLM output remains tainted and cannot directly influence promotion.
- `DecisionEngine` reads only bounded `DecisionFeatures`.
- Runtime feedback may enter proposal context only as guidance for future
  hypotheses.
- Timeout/crash pairs are counted, never skipped.
- Solver-reported operator runtime errors are counted as runtime failures, even
  when the solver recovers by returning the unchanged incumbent.
- Validation and frozen decisions fail closed when evidence is incomplete.
- BKS/gap and other benchmark references remain final-quality evidence, not
  promotion or runtime acceptance inputs.

## Runtime Evidence Flow

```text
ExperimentProtocol
  -> EvalStats runtime/completeness fields
  -> SafeFeatureExtractor
  -> DecisionFeatures
  -> DecisionEngine runtime veto
  -> EvidenceRecorder
  -> proposal runtime_feedback for later hypotheses
```

### Protocol Stats

`EvalStats` now carries pair-level completeness and runtime failure facts:

- `total_pairs`
- `attempted_pairs`
- `valid_pairs`
- `failed_pairs`
- `candidate_failed_pairs`
- `champion_failed_pairs`
- `runtime_ratio_median`
- `runtime_delta_median_ms`
- `runtime_regression_rate`
- `runtime_pairs`

The protocol owns these counts because it is the first layer that sees
candidate/champion pair outcomes across a stage.

Solver outputs may also include a runtime audit block, for example
`runtime.operator_errors`. If this count is positive, the run is not treated as
an objective tie. Screening records it as a candidate runtime failure/loss;
validation and frozen treat it as incomplete evidence.

### Decision Features

`DecisionFeatures` exposes the protocol and runtime facts as structured values:

- `protocol_gate_outcome`
- pair completeness/failure counts
- runtime guard pass/ratio/timeout facts
- median runtime ratio and regression rate

Free-text runtime explanations are not admitted into Decision.

### Decision Runtime Veto

`DecisionEngine` now rejects or abandons candidates before stage-specific
objective decisions when any of these framework-level failures is present:

- runtime guard timeout;
- runtime guard failure;
- candidate runtime failure pairs;
- validation/frozen incomplete runtime evidence;
- median runtime ratio above the default framework bound.

The default runtime ratio bound is `2.0` and lives in protocol configuration as
`runtime.max_runtime_ratio`. Problem protocols can tighten or relax the bound
without changing Decision code.

### Evidence And Context

`EvidenceRecorder` persists runtime summary metadata and pair completeness
counts. Campaign summaries therefore distinguish objective failure, protocol
gate failure, incomplete evidence, timeout, crash, and runtime regression.

`ContextManager` can read screening raw metrics and summarize bounded runtime
feedback for later proposals:

- slow case ids;
- candidate/champion runtime ratios;
- candidate timeout or crash cases;
- stage and operator context.

This feedback is advisory. It helps the next hypothesis reduce neighborhood
size or add caps, but it is not a Decision input.

## Acceptance

- A candidate that improves objective but times out cannot promote.
- A candidate with candidate-side runtime failures cannot promote.
- Validation/frozen decisions fail closed on incomplete pair evidence.
- Runtime regressions are represented in structured evidence and summaries.
- Proposal context receives bounded runtime feedback without exposing free-text
  failure detail to Decision.
- Existing campaign/CVRP control tests remain green.
