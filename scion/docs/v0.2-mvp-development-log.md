## 0. 背景

v0.1 MVP 于 2026-04-08 完成，证明了 Scion 的核心机制可工作（双闸门 + 三级协议 + 分支治理）。v0.2 的核心目标是**加上第二层搜索维度——promote 后自动做参数（权重）搜索**，使结构贡献和参数贡献可分离、可比较、可论文化。

## 1. 开发决策

## 4. 开发方法论总结

### 工作流

```
BigBOSS 审核设计 → Cris 精炼 TASK.md → CC 执行开发 → Cris 验证 → 真实 campaign 验证
```

**每个 Sprint 的 TASK.md 是关键产物**——从 6 份 v0.2 设计文档中提取、去重、合并为一份精确的执行 spec。CC 拿到就能直接干活，不需要读全量源码或设计文档。

### 效率数据

| 环节 | 耗时 | 产出 |
|---|---|---|
| 设计文档阅读 + 理解 | ~20 min | 完整文档地图 |
| TASK.md 准备（3 份） | ~10 min/份 | 精炼执行 spec |
| CC 开发 + commit | 15-30 min/Sprint | 代码 + 测试 |
| Cris 验证 | 5-10 min/Sprint | 测试 + diff 审查 |
| 真实 campaign | 30-60 min | 端到端验证 |

**文档占总工作量 ~50%**——符合讨论中的判断。

### 教训

1. **CLI 和完整 pipeline 的差距**: `scion run` CLI 缺少 Runner/ExperimentProtocol 接入，真实 campaign 需要用 `run_v3_campaign.py`。Sprint E 或后续应修复
2. **CC timeout 问题**: 30min 对复杂 task（T16 修改 `_on_promote`）略紧，但代码已完成只差 commit，手动补上即可
3. **环境变量传递**: OpenClaw exec 不继承 bashrc，需要通过 `.env` 配置 `SCION_API_KEY`
4. **实验产物管理**: 第一次就确立了"不进 git + tar.gz 归档"规范，避免后续踩坑
