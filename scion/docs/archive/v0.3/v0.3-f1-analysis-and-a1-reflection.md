## 二、A1 的设计目的回顾

A1 在 v0.3 optimization design 里的原始描述（`docs/v0.3-optimization-design.md §三.A.A1`）：

> **问题**：SPND 无条件 → `QUEUE_VALIDATE`，把 cost-regressive 候选推到 frozen 耗资源。
> post-opt R3/R4 消耗 10/15 轮在这条 fail pipeline 上。
> **修复方案**：添加 `expand_count ≤ 1` 上限，超过后走 `CONTINUE_EXPLORE`。

从 `project_v03_postopt_regression.md` memory：
> 0 promote 根因：SPND 规则把 cost-regressive 候选送 frozen_fail + stagnation_window=15 太紧

**设计目的混合了三层顾虑**：

1. **避免 val/frozen 预算浪费**（工程顾虑）：一个 SPND 候选走完 val+val_expand+frozen 消耗 ~3 eval 轮
2. **避免 idle counter 假性累积触发早停**：A2 修之前，val/frozen eval 轮都被计为 idle
3. **"cost-regressive 候选在 frozen 必然 ci_low<0 失败"**：隐含统计假设

A1 的实现是 `if features.expand_count >= 1: CONTINUE_EXPLORE`。

---

## 三、三层顾虑逐个核对：A1 其实一层都站不住

### 3.1 顾虑 2（idle counter）已被 A2 完全解决

A2 把 `_run_eval_step` 的 idle +1 限制为 `action_label == "explore"`，即 `EXPLORE_EXPAND` 的 eval。`VALIDATING` / `VALIDATING_EXPAND` / `FROZEN_TESTING` 的 eval 轮不再进 idle 计数。

**A1 在 A2 之上完全冗余** —— A2 已经让"SPND→val+frozen→fail"不会污染早停信号。

### 3.2 顾虑 3（cost-regressive 必然 frozen_fail）被 F1 数据反驳

| 分支 | screening (wr, md) | 走到 frozen 时 md | 最终 |
|---|---|---|---|
| sonnet s29 SPND #1 | (0.667, -748) | **+15685** | **PROMOTE** ✅ |
| sonnet s47 SPND #1 | (0.7, -546) | **+15468** | **PROMOTE** ✅ |

screening 的 `md` 用 6-10 个**固定 deterministic cases**（`all_cases[:n]`），偏向小/中实例，cost 回归在这里被放大；validation/frozen 用 12-20 个**多样 cases**，splits 改善的价值在大实例被 MILP_gap 放大（W16 分析 XLarge 上 Champion 领先 MILP 100+ splits）。

**screening 的 `md<0` 不是 frozen 必然失败的可靠信号。**

### 3.3 顾虑 1（浪费 val/frozen 预算）被 v3 §11.5 硬上限已保护

v3 §11.5 明确：`frozen uses per campaign: 3`。frozen 本身就是 campaign 级限量。**A1 是在 screening 层重复 frozen 层已有的保护，且是用更弱的信号**（screening md vs frozen ci_low）。

**A1 作为"防浪费"的安全网没有增加实际保护，反而产生了误杀。**

### 3.4 结论

A1 是在 A2/A3 尚未完成时的"保险丝"思维产物。当 A2/A3 落地后，A1 不但冗余，其实现（用共享 counter）还引入新的过阻断。**A1 的问题不是"参数 tuning 不对"，而是"设计定位不对"——它把 validation 的职能错位地放在了 screening 层**。

这不是"A1 有 bug"，而是"A1 从设计意图就偏离了 v3 的分层责任"。

## 四、深层设计反思：这次 bug 揭示的三个架构问题

### 4.1 计数器语义错位（已由 T3 部分修复）

`branch.expand_count` 实现为**分支级字段**，但它服务的 v3 §11.5 约束是 **per-candidate**。当一个 branch 迭代多个 hypothesis 时，这个字段语义错位：
- screening expand 和 validation expand 混合增量（T3 前）
- 新 hypothesis 生成时不重置（T3 前）

**T3 的修法是拆分字段 + 在 `_run_explore_step` 的 `pending=None` 分支重置**。但更彻底的架构应该是：**计数器挂在 `HypothesisRecord`（per-candidate），不挂在 `Branch`（per-direction）**。

T3 是"把 Branch 字段用作 Candidate 语义"，仍然是一种语义和载体的错配。v1.0 Phase 1 / v0.4 应该考虑把 expand counter 迁到 HypothesisRecord。

### 4.2 Decision Engine 的职责定位模糊

v3 §4.1 明确 Decision Layer 是 deterministic、只读 DecisionFeatures。但 decision.py 的规则设计夹带了**多种职责**：

1. **Protocol gate 判决**（wr>=threshold, md>=min_delta 等）：本职
2. **Budget protection**（expand_count cap 防止无限 loop）：工程考虑
3. **基于经验的 pre-filtering**（如"认为 cost-regressive 必然失败"）：统计启发

A1 是 (2) 和 (3) 混合的产物。当它表现不对时，我们才发现 decision 规则里实际上混了三种不同性质的逻辑。

**v0.4 规划建议**：在 decision engine 层面把三种职责显式分离：
- 规则层（pure protocol gates，由 v3 蓝图定义）
- 预算层（统一的 expand budget manager，独立数据结构）
- 启发层（或去掉，或明确是 "advisory hint"，不放 hard cap）

### 4.3 Screening/Validation/Frozen 三层的信号质量没有显式建模

F1 失败的根子问题是："screening 的 md 被当作 validation 的 ci_low 用"。但在代码和文档里，这两个信号的**可靠性差异**从未显式建模。

- Screening md：6-10 个固定小实例，deterministic
- Validation ci_low：12 个多样实例，bootstrap
- Frozen ci_low：12-20 个 holdout 实例，bootstrap

A1 隐含假设"screening md 能预判 frozen ci_low"，这在 lex-order 下是错的。**v3 §8.3 只说了三层的用途**（粗筛/正式/最终），没有说三层信号在判决不同子问题（win rate 可靠性、delta 可靠性、gap 可靠性）的权重。

**v0.4 规划建议**：在 `design/` 下单独写一篇 `signal-reliability-taxonomy.md`，把每一层能可靠回答什么、不能回答什么、应该只被 decision 规则怎么使用，显式地列出来。这样下一次有人写新规则时不会再掉进 A1 的坑。

### 4.4 v0.3 原 optimization 的一个流程教训

v0.3 optimization 第一次确定 A1 方案时是 2026-04-23（commit `5e964fd`），**没有跑完整 F 实验就进入了下一个 Sprint**。A4 是 smoke 试出来的（40 分钟 pre-formal run）补丁，但 A1 的过阻断要 6-campaign F 才能显形（因为需要特定 seed 的 screening expand → SPND 时序）。

**v0.4 规划建议**：在引入新的 Decision 规则后，至少跑一次 mini-validation（2-3 seeds），而不是只 smoke 一个 campaign。`v0.3-design-review-report.md §5.1` 里 reviewer 就建议"Mini-Validation A/B"，这次没充分执行。

## 五、T1+T2+T3 的修复思路与 v0.4 的关系

### 5.1 T1+T2+T3 做了什么

| # | 目的 | 改动 |
|---|---|---|
| T1 | 撤销 A1，恢复 w16-optimization 的 "validation bootstrap CI 才是 SPND 判决层" 原则 | `decision.py` SPND 分支去掉 expand_count cap |
| T2 | 把"防浪费 frozen"从 screening 层迁到 validation 层（信号更可靠）| validation 分支加 md 三路分叉（FAIL / MARGINAL_PASS / EXPAND）|
| T3 | 修计数器语义错位的表层 | Branch 字段拆分 + 新 hypothesis 重置 |

### 5.2 T1+T2+T3 解决了什么，没解决什么

**解决了**：F1 sonnet s11 的具体阻断 bug；validation 的无条件升 frozen 漏洞（原 §七.1 open question）；counter 跨候选污染。

**没解决**（留给 v0.4 或 v1.0）：
- 计数器挂 Branch 而非 HypothesisRecord 的语义错位（T3 是缓解，不是根治）
- Decision Engine 里 protocol/budget/heuristic 三种职责混合（A1 只是其中一个样本；类似问题可能还存在）
- Screening/Validation/Frozen 的信号可靠性没有显式建模
- LLM proposal 分布漂移（倾向性 context 产生更多 cost-regressive 候选）是否需要上下文层干预

### 5.3 为什么说 v0.4 而不是 v1.0 Phase 1？

v1.0 Phase 1 的 scope 已经定义（`docs/v1.0-roadmap.md`）：
- `campaign.py` < 800 行重构
- adapter silent fallback 去除
- DEPRECATED 类删除
- systemd launcher
- Observability
- FCMCNF 第二 problem 接入

这些是**工程固化 + 多 problem 框架化**的 scope。而 F1 暴露的是**决策规则设计层**的系统性问题，不属于 v1.0 Phase 1 的工程范畴，但也不适合继续塞进 v0.3（v0.3 已是收尾阶段）。

**v0.4 的定位可能是**：
- 决策引擎的职责重新分层（protocol / budget / heuristic 分离）
- 计数器架构迁移（per-candidate 载体）
- Signal reliability taxonomy 文档
- 在引入新 decision 规则时的 mini-validation 流程化
- 可能还包括：LLM proposal 分布引导（是 context 层任务，但会影响 decision 层的负载）

**v0.4 的实际 scope 应等 rerun 结果出来后决定**：
- 如果 rerun 通过且数据干净 → v0.4 scope 可以聚焦于"文档 + 架构微调"，不一定要大改代码
- 如果 rerun 仍暴露新问题 → v0.4 scope 需要把"新问题根因 + 架构整理"一起做

## 七、Rerun 仍失败的预案

如果 `v03-closure-validation-t1t2t3/` rerun（启动于 2026-04-24 14:18）仍未达 Pre-reg 成功标准，分支情景：

### 情景 A：rerun sonnet 均值仍 < 2.0，但 sonnet s11 不再 0-promote
说明 T1+T2+T3 修了阻断 bug，但 cast-regressive 候选在 validation 层也难以通过（T2 的 md guard 在过多路径上触发 FAIL）。根因可能在 LLM proposal 分布（V0.4-E）或上下文层的倾向性引导过强。

### 情景 B：rerun sonnet s11 仍 0-promote
说明根因不止 A1 共享 counter。可能：
- screening gate 本身的 threshold 需要调
- 某个 seed 的 LLM 持续生成非 promotable 候选（V0.4-E replay 分析）
- 其他未发现的阻断（需要另一轮 decision.py audit，V0.4-C2）

### 情景 C：gpt s29 timing 巧合继续
说明"分支处于 val_expand 时早停打断"是系统性问题，V0.4-F 成为必须。

### 情景 D：出现新的未知失败模式
直接进入 v0.4 规划，把当前文档作为基础输入。

## 九、一句话总结

F1 失败不是"sonnet s11 运气差"，是 **A1 的设计从意图出发就偏离了 v3 分层责任**：它用 screening 层的弱信号做 validation 层的判决，又用 per-branch counter 承担 per-candidate 语义。T1+T2+T3 按 v3 原理校准了规则，但更深的架构问题（计数器载体、Decision 职责分层、信号可靠性建模）需要在 **v0.4** 做一次完整规划，而不是继续在 v0.3 补丁。
