## Purpose

Run the production complement of the completed synthetic closure validation.

The prior closure run (`v03-closure-validation-20260425`) validated the v0.3
campaign stack on `warehouse_synthetic`. This run uses the same hardened v0.3
campaign path on `warehouse_prod` so the final v0.3 audit has production evidence
instead of relying only on synthetic closure data.

This is not a synthetic-to-production transfer test. Each campaign starts from
the same v1 surrogate baseline and searches directly on production instances.

## Questions

1. Does the v0.3 closure campaign stack remain stable on production data?
2. Does Scion produce production champion improvements over the v1 baseline?
3. Is production promotion cadence still lower than synthetic after v0.3
   closure hardening?
4. Are production gains concentrated in `total_cost` / fleet compaction because
   v1 baseline is already strong on the primary objective?
5. Do GPT-mini V5 solution-consistency failures remain materially higher than
   Sonnet on production?

## Success Criteria

Framework success:

- all 6 campaigns complete without launcher or campaign crash;
- every campaign writes `status.json`, `campaign_summary.json`,
  `validation_runner_summary.json`, `scion.db`, and `metrics/*.json`;
- no known infeasible production instance appears in metrics;
- frozen promotion evidence, if any, is traceable through DB and raw metrics;
- failures are contained by contract / verification / protocol gates.

Optimization success:

- at least one Sonnet production campaign promotes;
- final production champions are compared against v1 production baseline;
- CPLEX is used only as report-only reference, not as a strict exact-gap
  certificate.

Expected behavior based on W16:

- Sonnet production may promote fewer times than synthetic because production v1
  baseline is already strong.
- GPT-mini may suffer elevated V5 solution-consistency failures.
- Improvements, when found, may be more visible in cost/fleet compaction than
  in primary split count.
