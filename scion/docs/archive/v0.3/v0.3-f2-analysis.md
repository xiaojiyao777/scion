## 四、设计框架严格审查：v3 §8.6 lex-order 与统计度量错配

这是 F2 最重要、最深的发现。

### 4.1 v3 §8.6 设计层（splits 主、cost 次）

`design/scion-architecture-v3.md §8.6` 规定：

```
Validation Gate（正式验证）
- win_rate >= 2/3
- median_delta >= δ_validate
- bootstrap_ci_low >= 0
- 结果：pass_to_frozen / expand_sample / fail

Frozen Gate（最终确认）
- bootstrap_ci_low >= 0
- canary_passed == True
- 无 critical verification/runtime failure
- 结果：confirmed / rejected
```

并且 v3 全文确立 lex-order：**splits 是主轴判决，cost 是次轴信息**。每对 cmp=win/loss/tie 由 lex-order 给出，`decisive=subcategory_splits` 表示这一对在 splits 维度上已分胜负。

### 4.2 实现层：`ci_low` 实际度量在 cost delta 上

`scion/protocol/stats.py:8-42`：

```python
def compute_eval_stats(comparisons, deltas, ...):
    win_rate = wins / n
    median_delta = median(deltas)
    ci_low, ci_high = bootstrap_ci(deltas, ...)  # ← 注意 deltas 是什么
```

`bootstrap_ci(deltas, ...)`：在 1000 次有放回重采样上对 median 求 95% CI。

**关键问题**：`deltas` 来自 protocol/experiment 每对 pair 的 `delta` 字段，对应**total_cost 差值**（验证：log 中 "cmp=win delta=-19890 cand(splits=194, cost=519800) champ(splits=204, cost=499900)" → champ.cost - cand.cost = 499900 - 519800 = -19900 ≈ -19890；不是 splits 差，也不是 lex-order 决胜）。

即：
- `win_rate` = lex-order 胜率（主轴 splits 主决，cost 次决，是 OK 的）
- `median_delta` = **cost** 差值的中位数（次轴信息）
- **`ci_low` = bootstrap CI on median(cost_delta)（次轴信息！）**

`scion/core/decision.py:126-134` `_decide_frozen`：

```python
if ci_low >= 0:
    return PROMOTE  # FROZEN_PASS
return ABANDON     # FROZEN_FAIL
```

**结果**：frozen gate 实际上是"**cost** 维度的 bootstrap CI 不显著为负"——这等价于"**单目标在 cost 上的统计显著性测试**"，**而不是 v3 §8.6 主张的 lex-order 判决**。

### 4.3 这一错配在 sonnet s11 c877f314 上的体现

候选 c877f314 在 frozen 阶段：
- 主轴 splits：12/12 cand 严格更少 splits（binomial CI [0.74, 1.0] 远 > 0.5）— 主轴 clean improvement
- 次轴 cost：median +4605，但有 3 个大负 outlier（-19890, -7492, -6295）使 bootstrap CI 跨 0

**v3 §8.6 lex-order 应当判决**：splits 主轴显著更优 → confirmed → PROMOTE
**当前实现实际判决**：cost 次轴 ci_low<0 → rejected → ABANDON

这不是单一 case 的 unlucky bootstrap，是**实现层 metric 选择**对 v3 设计意图的系统性偏离。

### 4.4 为什么 W16 / F2 GPT seeds / sonnet s29、s47 没暴露

那些成功 promote 的候选 frozen md 都很大（central tendency 远超方差能撼动的范围）：

| 案例 | val md | frozen md | 结果 |
|---|---|---|---|
| gpt s11 promote | +35568 | +81437 | PASS |
| gpt s47 promote | +12717 | +34986 | PASS |
| sonnet s29 promote | +2613 | +4409 | PASS |
| sonnet s47 promote | +2859 | +15661 | PASS |
| **sonnet s11 c877f314** | **+8659** | **+4605** | **FAIL** |

c877f314 的 cost 中位数（+4605）落在 sonnet s29 promote（+4409）的同一量级，但因方差更大被拒。**结论：当前 frozen gate 对"splits 清洁赢、cost 中位数赢但 cost 高方差"的候选系统性误杀。** v3 lex-order 下这类候选应 PROMOTE。

W16 没暴露是因为 W16 评估更接近 single-composite，未明确区分 lex-order 主次轴；F2 暴露是因为 v3 在判决层落地了 lex-order，但统计层（compute_eval_stats）没跟上。

### 4.5 Validation gate 同样问题

c877f314 在 validation R9：wr=1.0 md=+8659 ci_low<0 → 触发 EXPAND（同样基于 ci_low 判决）。
v3 §8.6 validation gate 三条件：`wr>=2/3 AND md>=δ AND ci_low>=0`。前两个满足，第三个失败。

如果 ci_low 应该 on lex-order 决胜（如 binomial CI on win-rate margin 或 splits delta CI），c877f314 早在 validation 就 PASS 了，不会走 EXPAND，更不会走 MARGINAL_PASS → frozen → FAIL 的迂回路径。

**T2 设计意图 vs 底层 metric**：T2 引入 MARGINAL_PASS 是为了"validation 层把握不准时让 frozen 用更大样本最终判决"。但 frozen 用**同一种** cost-CI 度量，自然继承同一种偏向。MARGINAL_PASS 实际只是把 ABANDON 推迟一层。**T2 自身没错，错在底层 metric 不兑现 lex-order**。

## 五、其他设计框架问题

### 5.1 早停的 queue-aware 缺失

如 §3.3，B10 的 SCREENING_PASS 候选被早停在同一秒打断。早停只看 `_rounds_since_last_promote`，不看 queue 中是否有"已通过 SCREENING / 排队 validation / 排队 frozen"的高质量候选。

`v0.3-f1-analysis-and-a1-reflection.md §六.V0.4-F` 已预测此问题；F2 命中。**优先级应从 V0.4-F 提升为必做项**。

### 5.2 Signal Reliability Taxonomy 缺失（再次印证）

c877f314 路径的 md 在三层信号中漂移：

| 层 | n_cases | md | ci_low |
|---|---|---|---|
| screening | 8（10 expand 后）| -246 | — |
| validation | 12 | +8659 | <0 |
| val_expand | 12+ | +8659（不变）| 仍 <0 |
| frozen | 12 holdout | +4605 | <0 |

每层信号的"统计权威性"差异不被显式建模，导致同一种 ci_low 阈值（>=0）被用在三种不同分布的样本上，且都用 cost-delta 一种 metric。这是 `v0.3-f1-analysis-and-a1-reflection.md §四.3` 提出的"Signal Reliability Taxonomy"问题在 F2 上的具体表现。

### 5.3 LLM proposal 分布漂移（部分确认）

sonnet s11 在 25 round 内生成 10 个独立 branch（远多于其他 campaign 的 4-6 个），其中 6 个是单轮 abandon（screening wr 极低）、2 个是 splits-winning 但 cost-regressive。这暗示 sonnet s11 的 seed 上下文导致 LLM 倾向产出"激进 splits 优化但 cost 不稳"的候选。

但这一问题的修复**应在统计判决层而非 context 层**——如果 frozen gate 正确兑现 lex-order，"splits 赢 + cost 高方差"的候选 promote 是合理的；context 层不需要"避免产生这类候选"。

## 七、对 v0.4 规划的输入更新

`v0.3-f1-analysis-and-a1-reflection.md §六` 列出了 V0.4-A 至 V0.4-F 的候选项。F2 触发以下更新：

### 7.1 新增最高优先级项：V0.4-G

**V0.4-G：v3 §8.6 lex-order 与统计判决度量的一致性修复**

- **G1**：明确 frozen / validation gate 的 `ci_low` 应 on 哪个 metric
  - 候选 1：lex-order 决胜（cmp 转化为 +1/0/-1，对该序列做 binomial 或 wilcoxon CI）
  - 候选 2：splits-delta median bootstrap CI（直接对 cand_splits - champ_splits 做 CI）
  - 候选 3：splits-weighted composite metric（与 W16 风格的兼容路径）
- **G2**：相应修改 `scion/protocol/stats.py` 让 `bootstrap_ci` 接受可配置 metric source；DecisionFeatures 可能需要拆出 `splits_ci_low` 和 `cost_ci_low`
- **G3**：补充 `scion-architecture-v3.md §8.6`，把 ci_low 的 metric specification 写明
- **G4**：F-replay：用同一份 c877f314 frozen pair 数据重算新 metric → 验证是否 promote
- **G5**：跑一次 mini-validation（2-3 seeds）验证修复后 sonnet s11 能否到 ≥2 promote

**优先级**：v0.4 的入口任务（G1-G3 决定后续所有规则审计的基线）

### 7.2 升优先级项：V0.4-F（早停 queue-aware）

原 V0.4-F 列为"如果 gpt s29 时序巧合继续则成为必做"。F2 中 sonnet s11 B10 命中同类问题（早停打断 SCREENING_PASS），**升级为 v0.4 必做项**。

- F2：N 取多少需 replay 校准（grant 多少 round 给候选完成 validation+frozen 后再 trigger 早停）
