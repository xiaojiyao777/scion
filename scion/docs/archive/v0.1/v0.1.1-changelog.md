*日期：2026-04-08*
*基于 v0.1 MVP (2026-04-05)*

## 一、本次修复的核心问题

### 1.1 ContextManager 上下文真空

**发现过程**：跑 full campaign 时观察到 LLM 的 API 调用输入只有 ~300 tokens。逐步排查发现：
- `ChampionState.operator_pool` 初始化为空 dict（没从 registry.yaml 加载）
- `ContextManager._summarise_pool` 只读元数据不读代码文件
- `_summarise_champion_code` 因 `code_snapshot_path` 路径无效返回错误字符串
- Problem spec 只传了名字，没有描述
- Operator interface spec 只有 `execute(self, solution, rng)` 签名

**根因**：v0.1 开发时把 ContextManager 当成"拼字符串"的活，快速写了骨架就过测试——但测试用 MockLLM 不需要理解上下文。

**修复**：全量重写 ContextManager，prompt 从 ~300 tokens 增长到 4500+ tokens。

### 1.2 Campaign 不存储实验历史

**发现**：`_apply_decision_and_finalize` 中 `ProtocolResult`（含 win_rate、delta）在决策后被垃圾回收，下一轮 LLM 什么都看不到。

**修复**：增加 `StepRecord` dataclass，在 campaign 主循环每个退出点 append 历史。

### 1.3 新算子不被 Solver 加载（全 tie 根因）

**发现**：LLM 生成的代码被写入 workspace，但 `registry.yaml` 没更新。Solver 只跑 registry 里注册的算子。

**修复**：`WorkspaceMaterializer.apply_patch` 中 `action=create` 时自动更新 registry.yaml。

### 1.4 EXPLORE_EXPAND 状态机 Bug

**发现**：`schedule_branch` 把 `EXPLORE_EXPAND → EXPLORE`，然后 refresh 后匹配到 `_run_explore_step` 而非 `_run_eval_step`。导致 expand 永远重新生成假设，不复用 workspace。

**修复**：从 `schedule_branch` 调用块中移除 `EXPLORE_EXPAND`，让它直接落到 `_run_eval_step`。

### 1.5 Decision Engine 逻辑缺陷

**发现**：`win_rate=0.75` 但 `median_delta=0`（小实例全 tie）时被判定为 `CONTINUE_EXPLORE` 丢弃。

**修复**：
- `median_delta >= 0` 时允许 QUEUE_VALIDATE（ties 是中性的，不是负面的）
- `win_rate >= threshold` 但 `median_delta < 0` 时 EXPAND（矛盾信号，需要更多数据）
- Expand 时 double seeds（2 → 4）增加统计功效

## 二、CC 源码上下文管理分析

详见 `~/research/claude-code-src/analysis/07-comprehensive-context-management.md`

核心结论：Scion 作为单轮推理系统，上下文信息密度要求比 CC（多轮对话）高得多。CC 的设计中可直接借鉴：prompt cache 分区、状态恢复附件、"永远不截断只外包"、熔断器。不适用：对话摘要、消息配对保护。

## 三、Benchmark 扩展

生成了 3 个大规模实例：
- `instance_large_1.json` — 50 订单, 7 危险品, 8 锁车
- `instance_large_2.json` — 50 订单, 4 危险品, 8 锁车, 5 子类别
- `instance_xlarge_1.json` — 100 订单, 18 危险品, 19 锁车, 6 子类别

Split manifest 更新：screening 用 small + medium + large。

## 四、Prompt Caching

配置 Anthropic 1h extended cache TTL：
- System prompt（静态：problem spec + champion 代码 + interface spec）~5800 tokens → 1h cache
- User prompt（动态：experiment history + task）~300-2500 tokens → 不缓存

实测 Round 2 hypothesis 调用完美 cache hit（created=0, read=5835）。

## 五、验证结果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| Prompt 大小 | ~300 tokens | 4500-5500 tokens |
| Screening 结果 | 全 tie | 3W/1L/2T (wr=0.75) |
| Decision 多样性 | 只有 continue_explore | expand_screening + abandon |
| Cache 命中 | 无 cache | 5835 tokens cache hit |
| 生成的算子 | 盲猜 RelocateOperator | 针对性 SubcategorySplitReducer |

## 六、遗留问题

### 6.1 JSON 格式频繁失败
100+ 行代码嵌入 JSON 字符串导致解析错误。需要架构级方案（XML tags / 分步调用 / prefill）。

### 6.2 缺 Case 级别 Feedback
LLM 只看到 aggregate win_rate，不知道在哪些 instance 上赢/输、输在哪个目标维度。需要在 experiment_history 里加 per-case 详情。

### 6.3 上下文增长管理
随 campaign 轮数增长，step_history 会膨胀。当前 8 轮窗口足够，但 100+ 轮需要裁剪策略。
