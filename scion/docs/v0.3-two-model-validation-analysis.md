# Scion v0.3 Two-Model Validation Analysis

Date: 2026-04-25

Base directory:
`~/research/scion-experiments/v03-two-model-validation-20260425/`

Campaigns:

- `claude-sonnet-4-6`, synthetic, seed 11, `max_rounds=100`
- `gpt-5.4-mini`, synthetic, seed 11, `max_rounds=100`

## 1. Executive Summary

Both campaigns completed normally and wrote `campaign_summary.json`.
There were no traceback / ERROR / CRITICAL failures in the campaign logs.

The validation result is positive for Scion's core search behavior:

- both models completed full `screening -> validation -> frozen -> promote`
  chains;
- both reached at least champion v3;
- Sonnet reached champion v4 plus weight revision 1;
- adapter-backed verification and generic objective comparison ran in
  production mode;
- hierarchical objective decisions consistently favored
  `subcategory_splits` as the primary objective while allowing `total_cost`
  to act as secondary evidence.

However, the run also exposed framework-quality issues that should be fixed
before treating the next 6-campaign rerun as final evidence. The biggest
problems are observability and auditability, not the search loop itself:

- `status.json` only updates at step boundaries and hides long in-progress
  validation/frozen work.
- `campaign_summary.json` from `run_validation_campaign.py` overwrites the
  richer `CampaignManager` summary and drops `stopped_reason`,
  `stagnation_signals`, diagnostics, failure breakdown, cache detail, and
  case feedback.
- `experiment_events.raw_metrics_ref` is empty for all experiment rows,
  so DB lineage cannot directly recover the metric evidence files.
- `branches.state` is not synchronized after many abandon/promote outcomes.
- expanded screening can rerun identical evidence repeatedly.
- weight-only updates can land while a structural branch is in validation /
  frozen; this path needs a stage-aware invalidation audit before full reruns.

## 2. Completion Evidence

| Model | Finished | Time | Rounds | Experiments | Final Champion | Weight Rev | Promotions |

Both runs stopped before 100 rounds because early-stop triggered on
diminishing returns:

- GPT: no promotion in 25 rounds with stagnation plateau.
- Sonnet: no promotion in 25 rounds with stagnation plateau.

This means `--max-rounds 100` is currently a hard cap, not a promise to run
exactly 100 rounds. If a later experiment requires exactly 100 active rounds,
early-stop must be disabled or explicitly configured.

### 2.1 Raw Artifact And Lineage Audit

The raw artifacts confirm that the experiments completed, but they also show
that the persistence layer is not yet sufficient for final evidence.

| Check | GPT | Sonnet | Assessment |
|---|---:|---:|---|
| `metrics/*.json` files | 37 | 41 | matches experiment counts |
| experiment rows with empty `raw_metrics_ref` | 37 / 37 | 41 / 41 | lineage gap |
| experiment rows with empty `case_ids` / `seed_set` | 37 / 37 | 41 / 41 | split/seed lineage gap |
| `status.json` final stop reason | `run_complete` | `run_complete` | masks early-stop reason |
| log stop reason | diminishing returns early-stop | diminishing returns early-stop | should be final status reason |

The metrics files contain the exact pair evidence, including stage, instance,
seed, comparison, scalar delta, decisive metric, and per-objective deltas. This
is good. The issue is that the database does not point to these files, so a
post-run audit currently has to reconstruct the mapping from file timestamps
and log order.

Branch state persistence is also inconsistent:

| Model | DB `abandoned` | DB `explore` | DB `ready_frozen` | Expected Interpretation |
|---|---:|---:|---:|---|
| GPT | 4 | 21 | 2 | many soft-abandoned/promoted branches stayed non-terminal |
| Sonnet | 1 | 24 | 3 | most terminal branches stayed non-terminal |

This does not appear to have broken the in-memory campaign loop during these
runs, because both campaigns advanced and stopped correctly. It is still a
serious framework issue: resuming, reporting, or deriving memory from SQLite
would read a misleading branch lifecycle.

### 2.2 Search Shape

The two models explored different hypothesis styles:

| Model | Hypothesis Actions | Locus Mix | Repeated Targets |
|---|---|---|---|
| GPT | 43 create, 3 modify | 34 vehicle, 12 order | `subcategory_shuttle.py`, `vehicle_type_coarsening.py` |
| Sonnet | 27 create, 20 modify, 2 remove | 34 vehicle, 15 order | `rebalance_vehicle_pair.py`, `subcategory_swap.py` |

GPT was heavily biased toward creating new vehicle-level consolidation
operators, then shifted to pickup/order-level variants only very late. Sonnet
used more modification actions and showed a clearer exploitation path:
`subcategory_swap.py` and `rebalance_vehicle_pair.py` were revisited and
improved inside branch directions. This difference is useful for model
comparison, but Scion's final reports should make action/locus/family coverage
first-class summary fields instead of requiring manual reconstruction.

## 4. Operator Optimization Behavior

The run shows a clear split between early useful improvements and late
low-yield exploration.

Useful operator families:

- order-level subcategory relocation / swap:
  `subcategory_shuttle.py`, `subcategory_swap.py`;
- vehicle-level cost or type rebalance:
  `rebalance_vehicle_pair.py`, `vehicle_type_coarsening.py`.

Low-yield late families:

- many vehicle-level forced consolidation variants;
- pickup-name / region-category bottleneck variants;
- broad FFD / destroy-and-rebuild variants;
- remove-operator or remove-vehicle variants.

This suggests the current prompt/context can discover useful operators, but
after the first 2-3 promotions it tends to cycle through increasingly brittle
structural transformations. Stagnation detection eventually reacts, but the
reaction comes late and does not yet strongly steer mechanism selection.

Two operator behaviors are important for later full reruns:

1. **Branch-internal learning works.** Sonnet's `subcategory_swap.py` first
   produced only weak screening evidence, then a modified version passed
   screening, validation, frozen, and promotion. GPT's `23449f8a` similarly
   moved from a weak evacuation-swap hypothesis to a stronger
   `vehicle_type_coarsening.py`.
2. **Late-stage proposal diversity is still weak.** After champion v3/v4, both
   campaigns repeatedly tried broad consolidation, FFD, forced vehicle
   elimination, or pickup-bottleneck variants. Plateau detection identified the
   pattern, but only after many low-yield screening runs.

## 5. Failure Handling

### 5.1 GPT Failure Profile

GPT failures were mostly heavy verification failures:

- 9 verification-failed steps in summary;
- repeated `V5_solution_consistency`;
- abandon-fast worked after repeated heavy failures;
- hypothesis statuses were blacklisted for failed verification paths.

The failure router behavior is mostly correct: unsafe candidates do not enter
experiment protocol, archives are created, and search continues.

Weakness: after many repeated low-signal screening failures, the model kept
generating variants of similar consolidation mechanisms. Stagnation signals
were logged, but their downstream effect was weak until soft stagnation forced
locus diversification late in the run.

### 5.2 Sonnet Failure Profile

Sonnet failures were more LLM/contract-side:

- duplicate hypothesis contract failures (`C10_novelty`);
- code generation timeouts;
- final empty `code_content` failures;
- screening-positive but validation-negative candidate rejected correctly.

This is useful behavior overall: contract checks and validation gates are
filtering bad candidates before promotion. The remaining issue is that these
facts are not exposed well enough in final summary because the validation
runner overwrites the richer campaign summary.

## 6. Weight Optimization Behavior

Weight optimization is active and partially aligned with the intended v0.3
architecture:

GPT:

- champion v2 weight optimization improved and committed `v2_r1`;
- champion v3 weight optimization completed with no improvement;
- final champion is v3_r0.

Sonnet:

- v2_r1 and v3_r1 were prepared but correctly discarded because structural
  champion had already advanced;
- v4_r1 was committed near the end;
- final champion is v4_r1.

This is a positive signal for async weight-opt lifecycle handling. However,
there is one important audit risk:

- GPT committed `v2_r1` while branch `23449f8a` was active in the
  validation/frozen path against champion v2.
- The branch later promoted to structural champion v3.

This may be correct if evaluation used the current champion after the weight
update, but the branch metadata still records weight revision 0. Before the
next full rerun, weight-update invalidation must be audited stage-by-stage:
an active validation/frozen branch should either be explicitly reconciled
against the new weight revision or record why the current comparison remains
valid.

## 7. Design Alignment

### 7.1 Aligned With v3/v0.3

The core search loop aligns with v3/v0.3 design:

- Branches can evolve within one direction instead of being one-shot.
- Screening, validation, and frozen holdout all executed.
- Promotion required frozen evidence.
- Heavy verification failures did not reach experiment protocol.
- Adapter-backed verification and generic objective comparison ran in
  production mode.
- Weight-only updates and structural promotions are represented separately in
  champion versioning.
- Early-stop delayed until no recent promotion plus stagnation plateau.

The experimental result is therefore strong enough to show that v0.3's
framework design is functioning at the search level.

### 7.2 Not Fully Aligned

Several implementation details are not yet aligned with the intended
auditable framework:

1. Final `status.json` says `stopped_reason=run_complete`, while the actual
   log reason is early-stop due to diminishing returns. This loses the real
   termination semantics.
2. `run_validation_campaign.py` overwrites `CampaignManager`'s richer
   `campaign_summary.json`, dropping `stopped_reason`, stagnation signals,
   diagnostics, failure breakdown, action/locus coverage, family coverage,
   verification detail, and case feedback.
3. `branches.state` is stale for many branches: promoted or abandoned branches
   remain `explore` or `ready_frozen` in SQLite.
4. `experiment_events.raw_metrics_ref` is empty for all experiment events,
   even though `metrics/*.json` exists.
5. GPT repeated identical expanded screening evidence for
   `subcategory_shuttle.py` three times before queueing validation.
6. Stagnation signals are visible but mostly advisory. They do not strongly
   constrain action/locus/mechanism until very late soft-stagnation handling.
7. `max_rounds=100` currently means "up to 100 rounds"; with early-stop on,
   the actual campaigns stopped at 46 and 49 rounds. This is acceptable only
   if pre-registration explicitly treats early-stop as part of the protocol.

### 7.3 Implementation Root Causes

The main gaps are traceable to concrete implementation points:

1. `CampaignManager.run()` writes the rich summary, waits for weight opt, then
   writes `status.json` with `stopped_reason=run_complete`. If `run_one_step()`
   stopped because of early-stop, that reason is not preserved in final status.
2. `run_validation_campaign.py` writes a second, simpler
   `campaign_summary.json` after `CampaignManager.run()` returns. This
   overwrites the richer manager summary containing stagnation signals,
   diagnostics, failure breakdown, coverage, verification detail, and screening
   case feedback.
3. `ProtocolResult.raw_metrics_ref` is produced by
   `protocol/experiment.py`, but `_record_step_lineage()` in `campaign.py` does
   not persist it, and does not persist the case/seed lists either.
4. `campaign.py` has early returns in `CONTINUE_EXPLORE`, `PROMOTE`, and the
   T4 soft-abandon path. Several of these paths change in-memory branch state
   but do not persist the final state to `BranchStore`, which explains the
   stale `branches.state` rows.
5. Expanded screening currently records aggregate counts but not enough split
   identity or evidence fingerprints to prevent repeated identical expanded
   evidence. GPT's `subcategory_shuttle.py` produced the same expanded
   screening stats three times before validation.
6. Weight-only update handling has the right high-level version/revision idea,
   but the branch metadata does not record the comparison revision used for
   validation/frozen when a weight update lands mid-stage.

## 8. Readiness For Follow-Up Experiments

### Search-Level Readiness

The system is ready from a search-loop standpoint:

- both models produced promotions;
- both completed without runtime crash;
- both exercised branch evolution, validation expansion, frozen promotion,
  failure routing, and weight optimization;
- the framework can run multi-hour full-system campaigns.

### Evidence/Framework Readiness

Before the validation hardening patch, the system was not evidence-ready for a
final 6-campaign rerun: post-run analysis would have depended too much on log
reconstruction. The hardening patch now closes those P0 auditability gaps in
code.

Recommended decision:

- Run one short confirmation campaign after the hardening patch.
- Verify status heartbeat, summary preservation, DB lineage, branch state,
  weight invalidation audit, and partial metrics checkpoint.
- Start the 6-campaign rerun once those artifacts are trustworthy without
  manual log archaeology.

## 9. Required Optimization Backlog

### P0 Before 6-Campaign Rerun

Implementation status after validation hardening patch: items 1-7 are handled
in code and should be verified by a short confirmation run before starting the
6-campaign rerun.

1. **Stage-level heartbeat in `status.json`** — implemented  
   Include current branch, stage, candidate target, current case/seed,
   completed pair count, step start time, last progress timestamp, and real
   stop reason.

2. **Do not overwrite rich campaign summaries** — implemented  
   Merge validation-runner metadata into the `CampaignManager` summary or
   write a separate `validation_runner_summary.json`. Preserve
   `stopped_reason`, stagnation signals, diagnostics, failure breakdown,
   action/locus coverage, family coverage, case feedback, and verification
   detail.

3. **Persist metrics lineage** — implemented  
   Populate `experiment_events.raw_metrics_ref`, `case_ids`, and `seed_set` for
   every experiment event. The DB must be sufficient to recover the exact
   metrics file and split/seed evidence without guessing from timestamps or
   logs.

4. **Synchronize branch lifecycle state** — implemented  
   On abandon/promote/blacklist/frozen completion, update `branches.state`
   consistently. Keep hypothesis status and branch state coherent.

5. **Fix expanded-screening de-dup/accounting** — implemented  
   Prevent identical expanded screening reruns for the same candidate and
   split set. If rerun is intentional, record the reason and budget charge.

6. **Audit weight-update invalidation** — implemented  
   When a weight-only revision lands while a branch is in validation/frozen,
   record whether the branch was evaluated against old or new champion
   weights, and reconcile according to a stage-aware invalidation matrix.

7. **Partial summary/checkpoint support** — implemented  
   Long validation/frozen work should be auditable while in progress and
   after interruption. At minimum, emit partial evidence per completed pair or
   per case.

### P1 After Rerun Gate

1. Stronger stagnation-to-scheduler effect:
   plateau/oscillation should bias mechanism family, locus, and action more
   directly.
2. Failure summary in prompt context:
   repeated `V5_solution_consistency`, duplicate hypotheses, and empty code
   content should be summarized compactly for the next proposal.
3. Explicit secondary-objective reporting:
   promoted candidates can be lexicographically valid while cost-regressive;
   reports should show that tradeoff clearly.
4. Clarify `max_rounds` semantics:
   rename/report as `max_rounds` and explicitly document early-stop as part
   of the protocol, or add a fixed-round validation mode.

## 10. Conclusion

The experiment validates the v0.3 search architecture: Scion can use LLMs to
generate, repair, test, validate, freeze, and promote meaningful OR heuristic
changes under the new adapter/objective boundaries.

The experiment also showed that v0.3 needed an evidence-layer hardening pass
before final validation. That pass is now implemented in code. The remaining
gate is a short confirmation run that verifies the new status, summary,
lineage, branch-state, weight-invalidation, and partial-metrics artifacts.
After that confirmation, the 6-campaign rerun can be treated as the v0.3 final
evidence run.
