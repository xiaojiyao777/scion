# Scion v0.4 Evidence Harness

*Date: 2026-04-28*

## Why This Exists

v0.3 showed that promotion counts are useful campaign diagnostics, but they are
not final quality evidence.

One synthetic campaign promoted four times while most promoted campaigns
promoted once. That is an important mechanism signal, but it does not by itself
prove that the four-promotion campaign is the best final solver. A campaign can
promote many times and still lose to another campaign's final champion on a
larger or more representative evaluation set.

This matters because Scion is not a simple single-chain accept/reject optimizer.
Its research process is branch-governed: each direction can become a branch with
state, evidence, failure history, stale detection, reconciliation, and broadcast
logic. That design is meant to reduce local-optimum risk in the operator design
space, but it also creates a sharper evidence requirement. Branch dynamics can
explain search behavior; they cannot replace direct final quality measurement.

v0.4 needs a standard evidence harness that separates:

- campaign search dynamics;
- branch trajectory and broadcast effects;
- promotion evidence;
- final champion quality;
- runtime acceptability;
- benchmark/reference comparison;
- mechanism interpretation.

## Core Rule

Every formal campaign must produce two evidence layers:

1. **Promotion evidence**: why a candidate was promoted at the time.
2. **Final quality evidence**: how the final champion compares with the original
   baseline and references on a fixed evaluation set.

Promotion count is never a substitute for final quality comparison.

## Required Outputs

Each formal run should write a machine-readable evidence package:

```text
evidence_manifest.json
campaign_summary.json
status.json
final_quality.json
final_quality.csv
per_case_quality.csv
runtime_summary.json
failure_summary.json
```

The repository-level [evidence-manifest.md](evidence-manifest.md) should point
to the final files for each version.

For v0.4, the first baseline evidence package is the CVRP seed0 baseline:

```text
vrp/docs/experiment_results_seed0.md
vrp/results/full_experiment_seed0_final.csv
vrp/results/reference_validation_bad.csv
vrp/results/analysis_full_seed0_final/summary_by_subset.csv
```

These files are baseline evidence only. They are not Scion campaign evidence
until the CVRP adapter and campaign runner produce final champion comparisons.

## Campaign-Level Summary

For each campaign:

| Field | Meaning |
|---|---|
| `problem_id` | `warehouse_delivery`, `cvrp`, ... |
| `model` | proposal model |
| `seed` | campaign seed |
| `rounds` | total rounds executed |
| `experiments` | protocol evaluations executed |
| `promotions` | structural promotions |
| `weight_revisions` | accepted weight-only revisions |
| `stop_reason` | early stop / budget / plateau / max rounds |
| `bad_metrics` | failed or incomplete evidence count |
| `final_champion_ref` | path to final champion snapshot |

This answers: what happened inside the search loop?

It does not answer: is the final solver best?

## Promotion Evidence

For every promoted candidate, record:

| Field | Meaning |
|---|---|
| `operator` | generated or modified operator |
| `family` | mechanism family |
| `screening` | win rate, median delta, cases, seeds, complete evidence |
| `validation` | win rate, median delta, cases, seeds, complete evidence |
| `frozen` | win rate, median delta, cases, seeds, complete evidence |
| `runtime_guard` | candidate/champion runtime ratio and timeout status |
| `raw_metrics_ref` | metrics JSON path |

Promotion evidence is local to the protocol stage and current champion.

## Final Quality Evidence

For every campaign final champion, run a fixed evaluation:

```text
baseline solver vs final champion vs optional reference
same cases
same seeds
same timeout
same objective comparator
timeout/crash counted as failure
```

Required aggregate fields:

| Field | Meaning |
|---|---|
| `n_cases` | attempted cases |
| `n_ok` | successful feasible outputs |
| `n_timeout` | timeout cases |
| `n_error` | crash / invalid output cases |
| `better_vs_baseline` | case-level wins |
| `equal_vs_baseline` | case-level ties |
| `worse_vs_baseline` | case-level losses |
| `primary_delta_sum` | sum of primary objective deltas |
| `primary_delta_median` | median primary objective delta |
| `secondary_delta_when_primary_tie` | secondary objective comparison |
| `wall_time_total` | total solver runtime |
| `wall_time_median` | median per-case runtime |
| `runtime_regressions` | cases exceeding accepted runtime ratio |

## CVRP-Specific Quality Fields

For CVRP:

| Field | Meaning |
|---|---|
| `cost` | route distance/cost |
| `bks` | reference best known solution cost when available |
| `gap_pct` | `(cost - bks) / bks * 100` |
| `routes` | number of routes used |
| `bks_routes` | reference route count |
| `route_gap` | `routes - bks_routes` |
| `feasible` | CVRP feasibility |
| `benchmark_feasible` | feasible and `routes <= bks_routes` when comparing with BKS |
| `status` | ok / timeout / error / unsupported |

CVRP final reports must distinguish:

- CVRP feasibility;
- benchmark-comparable feasibility;
- instances without BKS;
- unsupported reference formats;
- route-count differences that make BKS gap interpretation unsafe.

## Warehouse-Specific Quality Fields

For warehouse delivery:

| Field | Meaning |
|---|---|
| `subcategory_splits` | primary objective |
| `total_cost` | secondary objective |
| `solve_wall_s` | runtime |
| `milp_status` | report-only CPLEX/MILP status |
| `milp_exact` | whether reference is exact |
| `oracle_feasible` | reference feasibility |

Warehouse reports must distinguish CPLEX exact references from feasible/time
limit references.

## Mechanism Analysis

v0.4 should add a separate mechanism report:

```text
campaign trajectory != final quality
```

Questions to answer:

- Did more promotions produce better final quality?
- Did a branch with fewer promotions beat a branch with more promotions?
- Which operator families transferred across cases?
- Did weight optimization add value after structure promotions?
- Did performance feedback reduce timeout/perf-guard failures?
- Did branch broadcast improve exploration or cause over-exploitation?
- Did one seed produce a compounding trajectory while others plateaued?

The four-promotion synthetic v0.3 run should be treated as the first case study,
not as a general conclusion.

## Acceptance Criteria

v0.4 evidence is acceptable only if:

- every final campaign has a final quality/runtime comparison against baseline;
- every promotion has complete validation/frozen evidence;
- timeout/crash/incomplete pairs are counted, not skipped;
- production and CVRP reports use the same top-level evidence schema;
- the docs state which artifacts are current and which are superseded;
- promotion counts are reported as search dynamics, not final solver quality.
