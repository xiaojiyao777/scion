# Scion v0.1 调优报告（2026-04-08 下午）

*Date: 2026-04-08*
*Phase: v0.1 MVP 完成后调优*

---

## 1. 背景

v0.1 MVP 框架在上午完成（全链路跑通、343 tests pass），但实验效果不佳：
- Screening WR 聚集在 ~0.50（等同随机）
- 从未成功 Promote
- code_content 工具调用频繁失败

本报告记录下午的诊断和修复过程。

---

## 2. 诊断发现

### 2.1 算子质量分析（Operator Quality Analysis）

通过 subagent 深度分析 campaign 日志 + 生成代码 + 现有算子 + prompt 上下文：

**🔴 P0 — 字段名 Bug**：5/7 生成的算子用 `order.subcategory`（不存在），正确字段是 `order.vehicle_subcategory`。`getattr(order, 'subcategory', None)` 静默返回 None，所有 subcategory-targeting 逻辑变成死代码。
- 根因：interface spec 写 `Order: order_id, locked_vehicle_id, hazard_flag, spu_list, etc.` —— 那个 `etc.` 藏了全部关键字段

**🟠 P0 — 缺目标函数公式**：LLM 不知道 `subcategory_splits = Σ(|vehicles_per_subcategory| - 1)`

**🟡 P1 — 缺 greedy init 原理**：LLM 不知道 splits 产生的结构性原因

**🟡 P1 — 缺 VNS 动态**：LLM 不知道 pool-based acceptance，不知道需要高方差算子

### 2.2 compute_delta 字典序对齐

`compute_delta()` 只算 cost delta → splits 大幅改善但 cost 增加的算子被 frozen CI 拒绝。
- 实际表现：frozen wr=1.0 但 ci_low < 0 → FROZEN_FAIL
- 修复：splits 差异 × 100K + cost 差异

### 2.3 max_tokens 截断

`max_tokens=4096` 导致 Opus 生成 10K+ 字节代码时被截断，`code_content` 字段丢失。

### 2.4 V5_state_leak

LLM 用 `list(set(...))` 引入遍历不确定性。

### 2.5 CC 源码启发

通过 subagent 研究 CC 源码的 prompt 工程：
- Tool description 300-4000 字（Scion 之前 1 行）
- 负面约束优先（"Don't" > "Do"）
- System prompt 分层（static/champion 分块缓存）
- 先读后写强制链

## 5. 实验结果

### 修复前（上午）
- 10 轮 campaign，最佳 WR=0.625
- 0 次 Promote
- 4/8 轮 code_content 失败

### 修复后（下午）
- **5 次连续成功 Promote**
- Screening WR 典型值 0.75-0.81
- Validation WR 典型值 0.94-1.0
- Frozen WR = 1.0（全部通过）
- Splits 改善 30-60%（fro_x01: 89→38, fro_x02: 131→55）
- 0 次 code_content 失败
- Cache hit rate 33%（首轮 create 拉低平均值）

---

## 6. 结论

v0.1 框架设计正确，问题出在 LLM 上下文不足导致算子质量差。通过系统性诊断 + CC 源码经验迁移，从"从未 Promote"提升到"稳定 Promote + splits 减半"。

框架已进入可用状态，下一步是 v0.2（参数层搜索、多问题泛化、论文实验）。
