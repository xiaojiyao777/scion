# Scion v0.4 P0-D Campaign Controller Decomposition Design

*Date: 2026-05-04*
*Branch: v0.4-dev*

## Position

P0-D is the next cleanup after P0-C promotion integrity. P0-C made promotion
commit safer and extracted a focused `PromotionLifecycleService`, but
`campaign.py` is still too heavy.

The v3 architecture treats the Campaign Controller as the coordinator of
branch state, scheduling, context exposure, lineage, gates, protocol, and
decision. It should not be the implementation owner for every step in the
research loop.

## Current Gap

`CampaignManager` still owns these large control surfaces directly:

- campaign run loop and finalization;
- branch scheduling dispatch;
- EXPLORE step orchestration;
- eval-only validation/frozen orchestration;
- stale reconcile orchestration;
- decision finalization;
- scattered failure/lineage/status glue around those paths.

This means the current state is still an integration hub, not the clean
Campaign Controller implied by v3.

## v3 Target Shape

The controlling loop should read as:

```text
select branch
-> stale reconcile OR proposal branch step OR eval-only step
-> contract/verification/protocol
-> safe features
-> decision
-> state transition/finalization
-> evidence
```

`CampaignManager` should provide dependencies and lifecycle ownership, but
the executable branch-step logic should live behind focused services.

## Design Decisions

### D1: Extract `BranchStepRunner` Before `CampaignLoop`

The heaviest responsibility is not the outer `for round in max_rounds` loop.
It is branch-step execution. Extract that first.

`BranchStepRunner` should own:

- branch dispatch after scheduling;
- create-branch path;
- stale reconcile path;
- explore path;
- eval-only path;
- eval-only state-loss handling.

`CampaignManager.run_one_step()` should become a thin call into this runner.

### D2: Extract `DecisionFinalizer` From Step Execution

The decision finalization block is shared by explore, eval-only, and reconcile.
It should become a dedicated service before or during `BranchStepRunner`
extraction.

`DecisionFinalizer` should own:

- lineage timing for non-promotion and promotion decisions;
- CONTINUE_EXPLORE workspace preservation/discard;
- PROMOTE commit handoff to `PromotionLifecycleService`;
- ABANDON cleanup/archive/status mutation;
- branch-state persistence.

This prevents `BranchStepRunner` from becoming the next large object.

### D3: Keep `CampaignLoop` Last

After branch-step execution is extracted, `CampaignLoop` can cleanly own:

- initial status write;
- per-round weight-opt drain;
- termination/circuit breaker checks;
- one-step execution;
- stagnation checks;
- final weight-opt join/drain;
- final summary/status writes.

Extracting `CampaignLoop` before `BranchStepRunner` would only move a thin loop
while leaving the complex part in `CampaignManager`.

### D4: Services Must Not Reverse-Depend On `CampaignManager`

New services may receive callbacks during staged extraction, but their public
methods should accept explicit request/state inputs and return explicit
outcomes.

Avoid creating a new service that simply stores `owner` and calls arbitrary
`owner._private_method()` everywhere. That reduces line count but not
architecture weight.

## Proposed Slices

### P0-D1: `DecisionFinalizer`

Move `_apply_decision_and_finalize()` into `core/decision_finalizer.py`.

Acceptance:

- Promotion store failure still leaves no champion/stale/lineage side effects.
- CONTINUE_EXPLORE, PROMOTE, ABANDON, and soft-abandon paths keep current tests
  green.
- `CampaignManager` does not directly implement decision-specific side effects.

Status:

- Implemented in `scion/scion/core/decision_finalizer.py`.
- `CampaignManager._apply_decision_and_finalize()` is now a compatibility
  wrapper that delegates to `DecisionFinalizer`.

### P0-D2: `BranchStepRunner`

Move `run_one_step()`, `_run_eval_step()`, and `_run_reconcile_step()` into
`core/branch_step_runner.py`, using `DecisionFinalizer` and existing services.

Acceptance:

- Scheduler dispatch and state transition semantics remain unchanged.
- Eval-only missing `HypothesisRecord` still fails before protocol budget.
- Stale reconcile still runs patch replay, contract, verification, and
  re-screening.
- `CampaignManager.run_one_step()` is a thin wrapper.

Status:

- Implemented in `scion/scion/core/branch_step_runner.py`.
- `CampaignManager.run_one_step()`, `_run_eval_step()`, and
  `_run_reconcile_step()` now delegate to `BranchStepRunner`.
- Compatibility wrappers remain for legacy tests and staged migration.

### P0-D3: Explore Step Extraction

Move `_run_explore_step()` into `BranchStepRunner` or a subordinate
`ExploreStepPipeline`.

Acceptance:

- Hypothesis retry, code retry, contract failure, verification-light fix,
  verification-heavy blacklist, workspace setup, and screening decision tests
  remain green.
- Proposal generation remains tainted and cannot feed Decision directly.

Status:

- Implemented in `scion/scion/core/explore_step_pipeline.py`.
- `CampaignManager._run_explore_step()` is now a compatibility wrapper that
  delegates to `ExploreStepPipeline`.
- `BranchStepRunner` calls the pipeline directly for normal manager instances.

### P0-D4: `CampaignLoop`

Move `run()` into `core/campaign_loop.py`.

Acceptance:

- `CampaignManager` owns construction/configuration and exposes the public API.
- `CampaignLoop` owns run/finalization.
- Full Scion suite remains green.

Status:

- Implemented in `scion/scion/core/campaign_loop.py`.
- `CampaignManager.run()` is now a compatibility wrapper that delegates to
  `CampaignLoop`.
- The loop receives explicit callbacks for status writes, termination, one-step
  execution, stagnation checks, and weight-opt finalization.

### P0-D5: Evaluation And Governance Cleanup

Move remaining protocol/decision glue and campaign stagnation policy out of
`CampaignManager`.

Acceptance:

- `CampaignManager._evaluate()` remains a compatibility wrapper, but no longer
  owns protocol execution, budget accounting, decision reason code assignment,
  or T4 soft-abandon cleanup.
- `CampaignManager.should_stop()`, `_run_stagnation_check()`, and
  `_check_soft_stagnation()` remain compatibility wrappers, but no longer own
  early-stop, hard-stagnation escape, forced-locus diversification, or diagnosis
  construction.
- Legacy private tests that replace transient campaign state remain green.

Status:

- Implemented in `scion/scion/core/evaluation_orchestrator.py`.
- Implemented in `scion/scion/core/campaign_governance.py`.
- Compatibility factories/fallback adapters were moved to
  `scion/scion/core/campaign_adapters.py`, keeping `campaign.py` focused on
  manager wiring and public compatibility.

## Non-Goals

- Do not change promotion thresholds.
- Do not change CVRP objective policy.
- Do not move final evidence generation into the campaign loop in this pass.
- Do not rewrite all campaign tests at once.

## Exit Criteria

P0-D is complete when:

- `campaign.py` no longer directly owns branch-step execution or decision
  finalization.
- `CampaignManager` reads mostly as dependency construction plus public API
  compatibility.
- New services have focused unit tests or are covered through existing campaign
  integration tests during staged extraction.
- Full Scion suite remains green after each slice.
