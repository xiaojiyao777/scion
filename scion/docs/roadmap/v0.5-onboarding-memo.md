# v0.5 Memorandum: Problem And Algorithm Onboarding

*Date: 2026-05-08*
*Status: v0.5 scope memo*

## Decision

v0.5 should make **Problem And Algorithm Onboarding** a first-class Scion
capability.

v0.4 should not try to finish that product layer. v0.4 should close around a
more limited and testable claim:

```text
Scion autoresearch is feasible on manually onboarded problem packages.
```

In v0.4, CVRP and warehouse delivery may remain manually adapted research
objects. CVRP is useful because it exposed the onboarding gap, but v0.4 should
not keep expanding until it becomes a general solver-ingestion product.

## CVRP Version Call

Introducing CVRP in v0.4 was useful as a diagnostic, even if it made the
current weakness more visible. It proved that the framework boundary can hold
for a second real combinatorial optimization problem, and it exposed that the
missing product layer is not another prompt or post-operator fix. The missing
layer is the entry path from a human-written solver to a reviewed
Scion-native problem package.

That entry path should be v0.5 work.

For v0.4, CVRP should stay manually onboarded. The v0.4 goal is to verify that
Scion can perform governed autoresearch once a problem package has already
declared its adapter, solver wrapper, surfaces, audit fields, and tests. v0.4
should not try to solve general problem-package admission at the same time.

## Background

The CVRP work showed that Scion core can remain problem-agnostic while still
governing a second real problem class through:

- `ProblemSpecV1`;
- `ProblemAdapter`;
- declared research surfaces;
- solver wrappers;
- runtime audit;
- Contract / Verification / Protocol / Decision boundaries.

It also showed that this is not enough to make an arbitrary solver immediately
researchable.

The hard part is turning a human-written solver into a Scion-native research
object. That requires naming algorithm surfaces, invocation points, safe
bounds, runtime audit fields, and tests. Today this is manual.

## Version Boundary

### v0.4 Scope

v0.4 should focus on:

- validating the autoresearch governance loop;
- keeping Scion core problem-agnostic;
- proving adapter-backed verification and runtime audit on manually onboarded
  problem packages;
- using CVRP and warehouse as concrete research objects;
- documenting where manual onboarding was required;
- avoiding long solver-quality claims until short diagnostics show nontrivial
  screening quality.

v0.4 should not attempt:

- general solver introspection;
- automatic surface manifest generation;
- automatic adapter generation;
- automatic runtime audit synthesis;
- one-click onboarding for arbitrary combinatorial optimization algorithms.

### v0.5 Scope

v0.5 should focus on:

```text
human-written solver
      |
      v
onboarding assistant / generator
      |
      v
reviewed Scion-native problem package
      |
      v
autoresearch campaign
```

The key v0.5 deliverable is not better prompting. It is an onboarding workflow
that can draft, validate, and review the research-object boundary.

## v0.5 Module: Onboarding Assistant

The onboarding assistant should draft:

- solver entrypoint map;
- problem model summary;
- candidate `ProblemSpecV1`;
- adapter checklist;
- candidate research surfaces;
- surface target files and required functions;
- component-library inventory;
- runtime audit field plan;
- protocol/split/seed checklist;
- focused test skeletons.

It should not auto-approve the result.

Human review remains mandatory for:

- objective semantics;
- feasibility semantics;
- allowed data exposure;
- surface safety;
- benchmark split validity;
- runtime audit adequacy;
- any surface that could leak case identity or holdout detail.

## Candidate Algorithm-Family Templates

v0.5 should begin with templates rather than universal solver understanding.

### Route-Based Metaheuristics

Examples:

- CVRP;
- TSP;
- VRP variants.

Draft surfaces:

- construction policy;
- neighborhood portfolio;
- destroy/repair policy;
- acceptance/restart policy;
- perturbation policy;
- budget schedule;
- parameter space.

### Assignment And Local Search

Examples:

- warehouse delivery;
- facility assignment;
- matching-like repair heuristics.

Draft surfaces:

- assignment construction;
- move/swap repair;
- overflow or saturation repair;
- penalty policy;
- neighborhood portfolio;
- parameter space.

### Scheduling Heuristics

Examples:

- job shop;
- machine scheduling;
- dispatching-rule heuristics.

Draft surfaces:

- dispatching rule policy;
- insertion/swap neighborhood;
- critical-path repair;
- idle-time repair;
- restart policy.

### MILP/CP Hybrid Heuristics

Examples:

- relax-and-fix;
- fix-and-optimize;
- rounding/repair loops.

Draft surfaces:

- relaxation policy;
- fixing policy;
- rounding policy;
- repair neighborhood;
- subproblem budget schedule.

## Admission Checklist

A solver should not enter Scion autoresearch until onboarding verifies:

- stable input and solution schema;
- independently recomputable objective;
- independently checkable feasibility;
- fixed screening/validation/frozen splits;
- fixed seed ledger;
- subprocess-compatible solver wrapper;
- declared editable and frozen files;
- declared research surfaces;
- selected-surface runtime audit fields;
- fail-closed runtime failure categories;
- adapter-backed verification tests;
- surface contract tests;
- at least one controlled smoke.

## Relationship To v0.4 CVRP

CVRP should be treated as the manual prototype of v0.5 onboarding.

The v0.4 CVRP work produced useful artifacts:

- route-native adapter;
- objective recomputation;
- formal and controlled protocol assets;
- multiple policy/config surfaces;
- `solver_design` backed by `policies/main_search_strategy.py`;
- selected-surface runtime audit;
- compact APS surface reads;
- singleton semantic novelty.

It also showed what is missing:

- automated identification of algorithm components;
- component-library inventory;
- stronger route-based metaheuristic template;
- automated draft `research_surfaces`;
- automated runtime audit checklist;
- human review workflow for onboarding approval.

## Success Criteria For v0.5

v0.5 is successful if a new solver can go through this workflow:

1. User provides a baseline solver and problem description.
2. Onboarding assistant produces a draft problem package.
3. Human reviews objective, feasibility, surfaces, and data exposure.
4. Generated adapter and surface tests pass.
5. A short controlled campaign proves selected surfaces load, execute, and fail
   closed when invalid.
6. Only then does Scion run autoresearch.

The success criterion is not immediate solver-quality improvement. It is a
repeatable path from solver to trustworthy research object.

## Open Questions

- How much solver structure can be inferred statically?
- How much should onboarding rely on human-provided annotations?
- What is the minimum viable adapter generator?
- Should v0.5 include only route-based templates first?
- How should component libraries be shared across problem packages without
  leaking domain semantics into Scion core?
- What review UI or checklist is enough for human approval?
