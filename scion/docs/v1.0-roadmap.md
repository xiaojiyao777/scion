## 0. 路径决策更新

| 项 | 当前决策 |
|---|---|
| 路径选择 | 继续走“真正的多问题 OR autoresearch framework”路线 |
| v0.4 第二问题 | **CVRP**，已决定前移到 v0.4 |
| v1.0 重点 | warehouse + CVRP 双问题证据固化、工程化、机制研究、文档化 |
| FCMCNF + Benders | 延后到 v1.x，作为 network-design/decomposition track |
| 前置条件 | v0.4 performance-aware hardening + CVRP baseline/adapter/campaign evidence |

早期路线把 FCMCNF + Benders 作为第二个真实 problem。这个方向仍然有价值，尤其适合验证 lower-bound / optimum-gap / decomposition-aware adapter。但作为 Scion 的下一步泛化验证，CVRP 更直接：

- CVRP 是标准组合优化问题，有成熟 benchmark 和经典算子族。
- 它与 warehouse delivery 的 assignment/bin-packing 语义明显不同。
- 它能直接暴露 route-sequence operator interface、distance objective、capacity feasibility、runtime complexity 等问题。
- 它更适合检验 Scion 是否真正离开了 warehouse-specific 假设。

---

## 1. v1.0 定位

v1.0 不再定义为“首次接入第二 problem”。第二 problem（CVRP）前移到 v0.4。

v1.0 的定位是：

> **把 warehouse + CVRP 两个真实问题上的 Scion 能力固化为一个可复现、可审计、可扩展的 OR autoresearch framework。**

v1.0 应回答四个问题：

1. Scion 的 adapter boundary 是否能稳定支持两个真实 OR 问题？
2. performance-aware promotion 是否能阻止慢算子和不完整证据进入 champion？
3. 不同模型在两个问题上的失败模式是否一致，还是问题相关？
4. 哪些机制真正贡献了效果：hypothesis loop、feedback memory、objective steering、weight optimization、performance feedback？

---

## 2. v1.0 不做什么

- 不追求 CVRP SOTA。
- 不把 CVRP 扩展到所有 VRP 变体。
- 不做线上 shadow deployment。
- 不做完整论文级大规模 ablation matrix。
- 不把 Scion 宣称为“任意组合优化框架”。
- 不在 v1.0 强行接入 FCMCNF/Benders，除非 v0.4 CVRP 已完全稳定且有额外时间。

---

## 3. Phase 1：Evidence 和文档收口（1-2 周）

目标：消除 v0.3 暴露的 evidence scattering，让结论、artifact、报告一一对应。

| # | 任务 | 验收 |
|---|---|---|
| 1.1 | 维护 [evidence-manifest.md](evidence-manifest.md) 作为 single source of truth | 每个 claim 都能追到 experiment dir / file |
| 1.2 | 标记 superseded artifacts | 旧 production quality artifact 不再被误用 |
| 1.3 | 为 CVRP baseline 增加 evidence entry | `vrp/` baseline path、验证命令、质量/runtime summary 入档 |
| 1.4 | 更新 README / docs index / blog | 对外叙述反映 v0.3 final 与 v0.4 CVRP |
| 1.5 | 统一 final report 模板 | 每个版本都有 campaign、quality、runtime、failure、limits 五块 |

退出标准：

- [ ] active docs 中没有把旧 FCMCNF 决策描述为当前第二 problem；
- [ ] v0.3 warehouse 证据和 v0.4 CVRP 证据有清晰边界；
- [ ] superseded artifacts 在文档中明确标注。
- [ ] v0.4 evidence harness 要求每个 campaign final champion 与 baseline 做完整 quality/runtime 对比。

---

## 4. Phase 2：工程化收敛（2-4 周）

目标：让 campaign 执行、artifact 输出、失败诊断更像一个可长期维护的研究系统。

| # | 任务 | 验收 |
|---|---|---|
| 2.1 | 继续拆 `CampaignManager` 的非核心职责 | 主循环、决策应用、lineage 写入边界清楚 |
| 2.2 | 移除 strict production/CVRP 路径中的 silent fallback | 缺 adapter / metric specs / runtime hooks 直接 fail closed |
| 2.3 | 统一 quality/runtime harness | warehouse 和 CVRP 使用同一输出 schema |
| 2.4 | launcher/queue 管理固化 | 不依赖手写 bash 并发；失败状态可恢复/可审计 |
| 2.5 | 清理剩余 legacy aliases | deprecated path 只保留有测试覆盖的兼容面 |

退出标准：

- [ ] warehouse 和 CVRP campaign 都能通过同一 runner 入口启动；
- [ ] quality/runtime 结果进入同一 manifest schema；
- [ ] 完整测试通过，或所有 xfail 都有明确原因和 owner。

---

## 5. Phase 3：跨问题验证（3-5 周）

目标：在 warehouse + CVRP 上跑可复现实验矩阵，形成 v1.0 主证据。

推荐矩阵：

```text
problems = warehouse_delivery, cvrp
models   = claude-sonnet-4-6, gpt-5.4-mini
seeds    = 11, 29, 47
rounds   = 100
```

可以根据 v0.4 runtime 成本缩小为：

```text
warehouse_delivery: reuse v0.3 final evidence + targeted rerun
cvrp: 2 models x 3 seeds
```

必须产出的报告：

- per-campaign summary；
- promotion table；
- complete-evidence audit；
- final champion vs baseline quality/runtime table；
- failure breakdown；
- model comparison；
- problem comparison。

退出标准：

- [ ] 两个真实 problem 都有完整 evidence manifest entry；
- [ ] 所有 promoted candidates 都有 complete validation/frozen evidence；
- [ ] timeout/crash 在 final quality 中被计入，而不是被跳过；
- [ ] 对“Scion 能泛化到第二个问题”的 claim 有实验证据。

---

## 6. Phase 4：机制研究（2-4 周）

目标：从“系统能工作”推进到“知道哪些机制有用”。

最小 ablation：

| 机制 | 对照 |
|---|---|
| Weight optimization | structure-only vs structure+weight |
| Objective steering | off vs on |
| Performance feedback | off vs on |
| Search memory / failure sharing | local-only vs cross-branch |
| Model capability | Sonnet vs GPT-mini |

不要求一次做完整论文矩阵，但至少要让 v1.0 的技术叙述避免“所有机制混在一起，无法区分贡献”。

退出标准：

- [ ] 每个关键机制至少有一个可解释的对照结果；
- [ ] 文档区分 framework correctness、search effectiveness、model capability boundary；
- [ ] 失败案例不只记录为 bug，而是转化为下一轮机制设计输入。

---

## 7. FCMCNF / Benders Deferred Track

FCMCNF + Benders 延后到 v1.x，不从路线图中删除。

它适合验证：

- lower-bound 多来源接口；
- mathematically meaningful optimum gap；
- network-flow / fixed-charge 决策结构；
- decomposition-aware adapter；
- 与 CVRP/warehouse 完全不同的 operator family。

推荐进入条件：

- CVRP adapter 和 campaign 已稳定；
- quality/runtime harness 已通用；
- performance-aware promotion 已在两个问题上验证；
- Scion core 已无 warehouse/CVRP special-case 侵入。

---

## 8. v1.0 成功标准

达到以下条件可宣告 v1.0：

- [ ] ProblemAdapter 在 warehouse_delivery + CVRP 两个真实问题上跑通完整 campaign；
- [ ] 两个问题都有 final quality/runtime/evidence completeness 报告；
- [ ] performance-aware gate 阻止 timeout/incomplete-evidence promotion；
- [ ] Scion core 没有 warehouse-only 或 CVRP-only 的隐式控制逻辑；
- [ ] campaign execution 和 evidence manifest 足够清楚，第三方能复现主要结论；
- [ ] 文档诚实区分已证明内容和未证明内容。

v1.0 的合理 claim：

```text
Scion is an adapter-based, auditable autoresearch framework for improving
modular combinatorial-optimization heuristics. It has been validated on
warehouse-delivery assignment and CVRP routing baselines under complete-evidence
and performance-aware promotion gates.
```

不应宣称：

```text
Scion solves arbitrary OR problems.
Scion beats specialized SOTA solvers.
Scion's improvements are guaranteed outside the benchmark/protocol.
```
