## 1. 为什么 FCMCNF + Benders 是好的第二 problem

### 1.1 问题类结构与 warehouse 真正不同

| 维度 | Warehouse Delivery | FCMCNF |
|---|---|---|
| 决策变量 | 订单 → 车辆分配 | 弧流量 + 设施开关 |
| 解表示 | `{vehicle → [orders]}` | 网络流 + 固定费率决策 |
| 目标结构 | 字典序（splits > cost） | 加权（固定 + 变动成本） |
| 约束类型 | 容量 + 业务聚合 | 守恒 + 容量 + 整数开关 |
| 下界来源 | 不完备 MILP（无效） | **Benders decomposition（有效）** |

不是"把 warehouse 换个名字"级别的改动，而是真正检验 adapter 抽象的泛化能力。

### 1.2 顺手解决的已知问题

- **RQ4 无效下界问题**：warehouse 上 MILP 不是真下界，`LowerBoundEstimate` 接口形同虚设。FCMCNF + Benders 给出数学上严格的下界，`optimum_gap` 首次有意义。
- **ObjectiveComparison 单问题偏见**：warehouse 是字典序（业务聚合 > 成本），FCMCNF 是纯成本最小化。泛化后的 comparator 只有接上"只有一个目标的问题"才能验证是否真泛化。
- **Operator 范式偏见**：warehouse 的算子是"池 + 概率选择 + VNS 主循环"。FCMCNF 可能是 path-based / arc-level / LNS 风格，能暴露 `operator_interface` 的隐藏 warehouse 假设。

### 1.3 预先存在的工程资产

用户已有 Benders 实现（工作项目中）。Phase 2 不是从零写求解器，而是：
- 把现有 Benders 代码迁入 `scion/problems/fcmcnf/solver.py`（或作为外部依赖包装）
- 写 adapter 适配 Scion 接口
- 设计 FCMCNF 的 operator 库（LLM 可演化）

**这是 Phase 2 的关键加速点**。如果从零写 Benders，Phase 2 要 8-10 周而非 4-6 周。

## 2. 四 Phase 执行序列

### Phase 1：技术债清理 + 工程化（3-4 周）

目标：在接第二 problem 之前，把 warehouse 残留和 god object 都处理掉。

#### P0 任务

| # | 任务 | 验收 |
|---|---|---|
| 1.1 | `campaign.py` 拆成 `ProblemRuntime` / `AsyncWeightOptCoordinator` / `PlateauController` | 单文件 < 800 行；原有 973 tests 不回退 |
| 1.2 | 去掉所有 `if adapter is not None` silent fallback；adapter 缺失直接 raise | `_build_problem_summary` / `_build_operator_interface_spec` 无 fallback 分支 |
| 1.3 | fix_code / light retry / V5 全路径加 INFO log | `grep "fix_code"` / `grep "V5.*retry"` 在 campaign.log 可命中 |
| 1.4 | `run_validation_campaign.py` 接 `systemd --user` 或 `nohup+setsid` 封装 | 断开终端后 campaign 继续跑完 |
| 1.5 | 删除所有 DEPRECATED 代码（`ObjectiveBreakdown` / `compare_with_breakdown` / `_MECHANISM_KEYWORDS` 硬编码 / CaseAggregateFeedback aliases） | grep 不到任何 DEPRECATED 标记 |

#### P1 任务（应做）

| # | 任务 | 验收 |
|---|---|---|
| 1.6 | 修 4 个 pre-existing test failures（或明确标为 xfail 并记录原因） | pytest 无未解释 failure |
| 1.7 | Campaign launcher：queue + 并发控制 + 失败自动重启 | 手工不再 bash 并发 |
| 1.8 | Observability：campaign 进度 JSON 实时输出（`status.json`） | 不用 grep log 就能知道进度 |

#### Phase 1 退出标准

- [ ] `campaign.py` < 800 行，god object 拆分完成
- [ ] adapter 接口无 silent fallback
- [ ] fix_code 路径可 grep
- [ ] 至少一次完整 campaign 用 systemd 启动、断开终端跑完
- [ ] 所有 DEPRECATED 代码删除
- [ ] 完整 pytest 通过

**不做的事**：
- 不动 Benders / FCMCNF 任何东西
- 不碰 MILP 下界逻辑（留给 Phase 3）
- 不扩展并发架构（留给 Phase 3）

### Phase 2：FCMCNF 接入 + adapter 真伪检验（4-6 周）

目标：接入 FCMCNF 作为第二个真实 problem，在过程中暴露并修复所有 warehouse 残留。

#### 任务

| # | 任务 | 验收 |
|---|---|---|
| 2.1 | 把现有 Benders 代码迁入 `scion/problems/fcmcnf/solver.py`（或包装外部依赖） | solver 能独立跑一个小实例并产出 feasible flow |
| 2.2 | `scion/problems/fcmcnf/problem.yaml` 按 ProblemSpecV1 strict 格式写 | `load_problem_adapter` 可加载 |
| 2.3 | `FcmcnfAdapter` 实现 `ProblemAdapter` 9 方法 | Protocol 满足 + toy 实例跑通全链路 |
| 2.4 | 设计 FCMCNF operator 库（至少 5 个基础算子） | 算子能在小实例上修改解并保持可行 |
| 2.5 | Benders 下界接入 `estimate_lower_bound` | 对小实例，下界 ≤ 任何可行解的 objective |
| 2.6 | 冒烟 campaign（10-20 rounds）跑通 | hypothesis → code → verification → protocol → decision 全链路 |
| 2.7 | **暴露的每一处 warehouse 残留修掉** | 代码里不再有 `subcategory_splits` / `vehicle_level` / `order_level` 硬编码 |

#### 预期暴露的 warehouse 残留热点（今天已观察到）

- `_MECHANISM_KEYWORDS` 关键词表（warehouse 风味）
- `CaseAggregateFeedback` legacy aliases
- `extract_champion/candidate_metrics` 字段名硬编码
- Stagnation handler 的 `order_level ↔ vehicle_level` 切换逻辑
- 单测里可能有 warehouse 特有的 fixture 假设

每发现一处，产出一个 TD（technical debt）条目，在 Phase 2 内解决。**不允许在 Phase 3 再处理 warehouse 残留**。

#### Phase 2 退出标准

- [ ] FCMCNF 冒烟 campaign 跑通（≥ 1 次 promotion）
- [ ] Benders 下界对所有 frozen 实例严格成立
- [ ] Scion core 代码（非 `problems/` 目录）无 warehouse 术语 grep 命中
- [ ] ProblemAdapter 接口在两个 problem 上都不需要 special case

### Phase 3：下界多来源 + 并发升级（2-3 周）

目标：让下界接口真正泛化（不是"MILP 专用"），并解决 thread + 共享状态的并发脆弱性。

#### 任务

| # | 任务 | 验收 |
|---|---|---|
| 3.1 | `LowerBoundEstimate` 接口支持多来源（MILP / Benders / LP relaxation / custom） | `kind` 字段区分，`value` 保证 ≤ optimal |
| 3.2 | `estimate_lower_bound` 签名允许多下界并行 + 取最紧 | 两 problem 各自可返回多个下界 |
| 3.3 | Async weight-opt 从 thread 改为独立 subprocess + queue | 跑并发 campaign 时不再依赖 `sys.path` trick |
| 3.4 | Campaign 级并发：n 个 campaign 可以真正独立运行 | 2 个 campaign 并发时 workspace / lineage / log 完全隔离 |
| 3.5 | `_champion_lock` 进程全局锁改为 per-campaign | 消除跨 campaign 串行点 |

#### Phase 3 退出标准

- [ ] 两个 problem 都能用 adapter-specific 下界跑完整 campaign
- [ ] 4 campaign 并发无 race condition（压测）
- [ ] warehouse + FCMCNF 可同机并行各跑一个 campaign，互不干扰

## 3. 跨 Phase 不变量

这些约束贯穿四个 Phase，任何时候都不得破坏：

1. **v3 基石边界**：Decision Input Guard / Creative-Decision 分层 / 三级协议 / Champion 池级别语义 —— 全部保持。
2. **测试不回退**：每个 Phase 结束 pytest 全绿。Phase 2 期间允许 FCMCNF 相关新 test 逐步增加，但 warehouse 侧测试数量不得减少。
3. **adapter 抽象不漏**：Phase 2 起，任何新的 warehouse-specific 代码路径都是回归。
4. **文档同步**：每个 Phase 的接口/行为变化必须反映在 `design/` 或 `docs/` 下的相应文档。

## 6. v1.0 不做的事

明确划出范围避免蔓延：

- ❌ 第三个 problem（v1.x 再考虑）
- ❌ 多语言 solver 支持（Python-only）
- ❌ 分布式 / 多机 campaign
- ❌ Shadow deployment / 生产接入
- ❌ 结构级搜索（改 solver 主体框架）
- ❌ 论文级完整 ablation matrix（W17 是 lightweight）
- ❌ Benders 求解器本身的算法改进（用现有实现）
