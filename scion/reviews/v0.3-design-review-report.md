## 1. Executive Summary

### Overall Verdict

**结论：条件通过（Conditional Pass）**  
Scion v0.3 的方向是对的，而且相对于 v0.2 的遗留问题，回应也基本命中要害：  
- W1 对准了 production weight opt 失效  
- W2 对准了 async skeleton 未闭环  
- W3 对准了 plateau 浪费  
- W15 对准了问题耦合  
- W16 对准了“大规模验证”诉求

但是，**当前 Draft v2 还不是“可以直接开 Sprint N 干活”的实现级设计**。最主要的不是“目标错了”，而是：

1. **W15 的抽象边界还不够深，且 acceptance 太弱**  
   现在更像“去术语化 + 配置外提”，还不是“问题适配层真正成立”。

2. **W2 的并发/快照语义不安全**  
   现在的思路仍然容易形成 snapshot 原地改写、stale 语义过粗、stage 进度丢失等问题。

3. **W16 当前实验矩阵无法回答 Draft 声称的全部 RQ**  
   特别是 **RQ1/RQ2/RQ3/RQ7 现在是答不出来的**，最多得到描述性结果，得不到机制性结论。

4. **Sprint 顺序需要更细拆**  
   W15 不应只是 Sprint N 的一个 work item，而应成为 N 的前置 foundation。  
   同时 W16 不应等到最后才第一次做较真回归。

5. **memory-layer 有“事实层 / 视图层”混杂风险**  
   W5/W8/W9/W7 如果都按现在草案做，很容易新增三套相互重叠的“记忆系统”，形成新的技术债。

---

### Most Important 5 Recommendations

1. **把 W15 从“grep 去 warehouse 化”升级为“Problem Adapter 边界建立”**  
   不是去掉几个术语就算解耦。要把：  
   - objective comparison  
   - solver output 反序列化  
   - feasibility / objective recomputation  
   - solution consistency  
   - saturation lower-bound logic  
   - family taxonomy / mechanism labeling  
   全部收口到 problem adapter / problem hooks。

2. **W2 改成“不可变 champion 快照 + 原子切换指针”，禁止原地 chmod / 回写 live snapshot**  
   当前 draft 的 async 设计如果继续原地改 snapshot，会留下竞态和审计问题。  
   正确做法是：**immutable snapshots, mutable pointers**。

3. **重写 W16 的研究问题映射，补一个 lightweight ablation 计划（可叫 W17）**  
   当前 18 campaigns 只能较好回答 RQ4/RQ5 的一部分、RQ6 的一部分。  
   不能回答 structure alone / parameter alone / synergy / early-stop gain。

4. **Sprint N 拆成 N0 → N1 → N2，并在 N1、O/P 后插 mini-validation**  
   否则问题会在 Sprint Q 才暴露，代价太大。

5. **把 W5 从“持久化 SearchMemory”改成“持久化事实、按需重建视图”**  
   也就是：**persist facts, not views**。  
   family_id、hypothesis outcome、failure code、promotion 等事实入库；  
   SearchMemory / ResearchJournal / FailureSummary 由 lineage 计算得出。

#### P0-2：不要把 `Objective`/objective handling 粗暴改成 `dict[str, float]`

**为什么是 P0**  
这是典型“为了泛化牺牲类型稳定性”的改法，很容易把大量已有测试、序列化、渲染、比较逻辑都打散。

**结论**
- **不建议** `@dataclass -> dict` 这种改法
- 更好的做法是：
  - 保留 solver/problem 侧自己的 objective 表达
  - 在 Scion core 引入一个 generic 的比较模型，例如：

```python
@dataclass(frozen=True)
class MetricDelta:
    name: str
    candidate_value: float | int
    champion_value: float | int
    raw_delta: float | int | None
    decisive: bool = False

@dataclass(frozen=True)
class ObjectiveComparison:
    decisive_objective: str
    metrics: tuple[MetricDelta, ...]
```

这样：
- protocol feedback 不再绑定 `subcategory_splits / total_cost`
- 但也不需要把整个代码库都改成随手乱用 dict

**一句话建议**  
> **genericize the comparison layer, not the whole objective type system**

---

#### P0-3：W15 里的 `lower_bound_rule` 字符串设计不可靠，甚至示例本身就可疑

Draft 例子里：

```yaml
lower_bound_rule: "|distinct(order.subcategory)| - 1"
```

这个设计有两个问题：

1. **规则字符串求值本身就是坏信号**
   - 你们要么得写 DSL
   - 要么得 `eval`/解析表达式
   - 两种都不值得

2. **这个例子本身在数学上就很可能不是你们真正想要的下界**
   - warehouse 的 `subcategory_splits` 若定义是 `sum_s (vehicles_used_by_s - 1)`
   - 那“绝对下界”不是这个字符串
   - 甚至 instance-level 理论下界也应该依赖 pallet volume / allowed vehicle type，而不是简单 distinct count

**正确做法**
把 lower bound 分成两类：

- `absolute_floor`：指标理论下界，例如 splits 的 floor 可能是 0
- `instance_lower_bound_fn`：问题特定 hook，按实例集计算更强下界

建议改成：

```yaml
objectives:
  - name: subcategory_splits
    direction: minimize
    priority: 1
    lower_bound:
      kind: problem_hook
      fn: "warehouse_hooks.subcategory_split_lower_bound"
```

#### P0-4：W2 必须改成“不可变快照 + 原子替换”，不能原地修改 champion snapshot

这是本次最明确的工程 P0 之一。

当前 v0.2 已经有 D10：
- 先 freeze snapshot
- 后台线程再 make writable
- 写 registry
- 再 freeze

这个模式在 W2 若继续沿用，会有这些问题：

- live champion snapshot 被原地改写，审计语义差
- 半写状态 / chmod 中间态可能被别的 reader 看到
- cancel signal 很难做正确
- double-promote 时版本切换不干净

**推荐做法**

后台 weight opt 成功后不要改原 snapshot，而是：

1. 复制 `champion_vN` → `champion_vN_opt_tmp`
2. 写入新权重
3. freeze 新目录
4. 计算 hash
5. 用锁原子更新 `current_champion_pointer`
6. 老 snapshot 保留不动

也就是：

> **Champion snapshot is immutable. Champion selection is mutable.**

---

#### P0-5：`STALE_WEIGHT_UPDATE` 不能一刀切“只 re-screening”

Draft 里这句我不认同：

> reconcile 对 `STALE_WEIGHT_UPDATE` 只重做 screening，对 `STALE` 走完整 reconcile

这太粗。

因为 branch 被 weight update 打 stale 时，它原来可能处在：
- EXPLORE
- READY_VALIDATE
- VALIDATING
- READY_FROZEN
- 甚至更晚

不同 stage 的“失效范围”不同，不能全部压回“只 screening”。

**建议做成 stage-aware invalidation matrix**

| 原状态 | weight update 后最低需要重做的阶段 |
|---|---|
| EXPLORE / EXPLORE_EXPAND | screening |
| READY_VALIDATE / VALIDATING / VALIDATING_EXPAND | validation（可选先做 cheap screening sanity） |
| READY_FROZEN | validation 或 frozen 前复核 |
| FROZEN_TESTING | 明确策略：要么禁止 weight update 干扰，要么直接 invalidate 当前 frozen |

不然你会出现两类错误：
- **过度丢进度**
- **保留了不再可比的统计证据**

---

#### P0-6：W3 early-stop 默认开启前，必须先定义“硬饱和”和“经验饱和”的区别

当前设计里 plateau 信号之一依赖 `at_absolute_minimum=True`，但你们当前的实现其实更多是“经验上接近 0”，不是严格理论证明。

**必须区分**
- **hard lower-bound reached**：由 problem hook / exact bound 证明
- **empirical saturation**：历史改善趋缓，但不代表理论到头

我建议：
- `hard saturation` 可以参与 early-stop hard condition
- `family_exhausted` 和 `empirical plateau` 只能先触发一次 diversify/escape，不应直接 stop

另一个硬问题是：
- `no_promote_window=40` 没有充分依据
- F4-A 恰好 41 轮没 promote 不等于 40 就合理

**必须补**
- replay calibration on F1-F6
- manual override：`force_continue=True` / `--disable-early-stop`
- 一次 escape branch 机制（类似 I4）

### 4.2 Q1.1 — W15 问题解耦是否完整？

**结论：不完整。**

#### 已识别的遗漏耦合点
除了 Draft 写的 5 个点，我认为至少还漏了 6 类：

1. **Objective feedback 数据模型**
   - `ObjectiveBreakdown`
   - `CaseAggregateFeedback`
   - `compare_with_breakdown`
   都是 warehouse metrics 命名绑定

2. **Solver output → domain object reconstruction**
   - 这是 `verification/feasibility.py` / `objective.py` 真正最深的耦合点

3. **V5 solution consistency / state mutation**
   - 当前 JSON shape 假设不 generic

4. **search memory / classifier taxonomy**
   - 现在本质上是 warehouse-specific mechanism ontology

5. **problem summary / interface prompt builder**
   - 不是读几个术语就行，而是整段 reasoning scaffold 都是 warehouse 语义

6. **saturation lower-bound logic**
   - 当前设计没把“理论下界”“实例下界”“经验平台期”拆开

#### 关于 `Objective dataclass -> dict`
**不建议做。**  
如果一定要 generic，优先 genericize：
- comparator
- breakdown
- feedback schema

而不是把整个 objective 表达改成裸 dict。

#### 关于 dynamic import oracle
**本身不冲突于“Contract 禁止 shell/subprocess”的精神**。  
因为：
- import 发生在 trusted problem package 内
- 不是 LLM patch 任意触发的系统调用
- 只要 problem files frozen，就属于受控 hook

但要满足 3 个前提：
1. 只能从 `problem_root` 解析，不能任意 `importlib.import_module("foo")`
2. 应该 path-pinned / hashable
3. 不要在 runtime 中反复全局改 `sys.path`

#### toy_tsp 是否只放 README 就够
**不够。**

必须有一个 **minimal working example**，至少覆盖：
- `problem.yaml`
- `solver.py`
- `oracle.py` / hooks
- `operators/base.py`
- 1~2 个 operator
- 1 个 canary / 1 个 screening case
- 1 组 tests
- 一次 mock/end-to-end smoke

**没有真实 MWE，W15 的验收就是伪验收。**

### 4.3 Q1.2 — Decision Input Guard 是否被破坏？

**当前草案下，W9/W10 本身不会破坏。**

原因：
- 它们只是扩 LLM context
- 决策仍然来自 `DecisionFeatures`

但我认为要补一个明确规则：

#### 必须新增的架构约束
- `family_id`
- `search_memory summary`
- `research_log`
- `weight_opt textual feedback`
- `classifier output`

这些都属于 **Creative metadata / tainted enums**，默认：

- 可供 LLM 读
- 可供研究报表读
- **Decision 不读**
- **Termination 默认也不读**

#### 唯一例外
如果未来真的要让某类枚举参与 deterministic control（例如 plateau diagnosis），必须满足：

1. taxonomy 是 closed-set
2. source 有 `family_source`
3. version 有 `taxonomy_version`
4. 进入的是单独的 deterministic mapping，而不是原始标签文本

但我的建议是：  
**v0.3 不要让 classifier family_id 进入 hard stop / scheduling / promotion。**

### 4.4 Q1.3 — W4 MILP integration scope 是否过小？

**结论：不算过小，但目前还少一个“sanity use-case”。**

我认为 v0.3 把 MILP 放在：
- report
- gap anchor
- benchmark calibration

这是合理的。  
我**不建议**把 MILP optimum 直接接入 promotion 决策路径，因为：

- exact only exists on small subset
- 会对不同 problem variant 形成不对称
- 会让 search loop 对 small exact cases 过拟合

但我建议增加一个强 sanity check：

### 应补的用法
对于已知 exact optimum 的实例：
- 如果 candidate objective **声称优于 exact optimum**
- 或 report 中出现 negative gap beyond epsilon

那么这不是“candidate 太强”，而是：
- objective recomputation bug
- oracle bug
- MILP mapping bug
- solver output bug

这个 check 可以是：
- report warning
- verification sanity flag
- canary-like veto for impossible claims

但不是“没达到最优就 veto”。

### 4.5 Q6.1 / Q6.2 — 主循环与权限矩阵要补哪些？

#### classifier 应插在主循环哪里？
我建议放在：

```python
Round 1 hypothesis
-> ContractGate.validate_hypothesis (通过后)
-> classify family_id
-> persist hypothesis record
-> Round 2 code
```

不建议放在 Contract 之前，因为：
- C1/C2/C3/C10 失败的 hypothesis 不值得花 classifier 成本
- W7 本质是“已批准 hypothesis 的语义归类”

#### async weight-opt 是否需要写进主循环伪代码？
**需要。**

不是把 thread 细节写进 while-loop，而是在 §18 旁边补一个 side-effect lifecycle：

- `on_promote()` spawns async parameter search
- async result may atomically update champion pointer
- branches receive `stale_reason = weight_update`
- reconcile follows stage-aware invalidation policy

否则 v0.3 文档会留下“主循环看不到的控制流”。

#### 权限矩阵新增建议
建议新增至少 4 行：

| 数据类型 | LLM 可写 | Contract 可校验 | Verification/Protocol 可写 | Decision 可读 |
|---|:---:|:---:|:---:|:---:|
| family_id | ✅（via classifier） | ✅（taxonomy validate） | ❌ | **❌** |

这是比原顺序更稳的做法。

### 5.2 Q2.1 — W2 Async Weight Opt 的工程风险

#### 结论
当前草案**低估了 W2 的复杂度**。

#### 具体风险

##### 1) stale 不是单一语义
weight update 带来的 stale 和 champion structural promote 带来的 stale，本质不同：
- 前者：candidate code 没变，comparison baseline 变了
- 后者：candidate base code / champion code 都变了

所以 `STALE_WEIGHT_UPDATE` 不该只是另一个 enum，而应带：
- `stale_reason`
- `stale_from_stage`
- `stale_from_champion_hash`
- `resume_policy`

##### 2) “re-screening 后回 EXPLORE”太粗暴
会丢掉分支当前阶段语义，也让 lineage 难解释。

##### 3) double-promote / cancel signal
Python thread 不是说 cancel 就 cancel。  
如果后台线程已写了一半 snapshot，你要如何保证：
- 不污染 live champion
- 不写坏 registry
- 不留下半冻结目录

**答案是：不能靠原地 snapshot 改写来保证。**

#### 推荐 redesign
- 后台线程只在独立 temp snapshot 上工作
- champion 用 version + epoch token
- update 时做 compare-and-swap
- stale 采用 stage-aware policy
- 旧证据不删，只标记 `superseded_by_weight_hash`

---

### 5.3 Q2.2 — W3 Plateau 误停风险

#### 我对当前 design 的判断
**误停风险明显存在。**

尤其是这两点：

##### 1) `at_absolute_minimum=True` 目前并不总是“理论下界命中”
生产数据里“splits≈0”更多像经验现象，不是数学证明。

##### 2) `no_promote_window=40` 没有足够统计依据
它可能是对 F4-A 的 hindsight fitting。

#### 建议
1. 先做 replay calibration（F1-F6）
2. 采用两阶段 stop：
   - Phase 1：plateau warning + 强制 diversify
   - Phase 2：warning 后仍无新正信号才 stop
3. `family_exhausted` 只做 soft signal，不做 hard stop
4. 增加：
   - `force_continue=True`
   - CLI `--disable-early-stop`

#### Acceptance 建议
不要用“R80-R120 之间停住”这种宽松条件当主验收。  
应增加：
- missed-promotion rate
- false-stop replay count
- saved rounds distribution

---

### 5.4 Q2.3 — W7 Classifier 接电的成本与一致性风险

#### 成本
如果按 18 campaigns × 100 rounds = 1800 次分类，大概率**成本不是最大问题，反而是 consistency 和 plumbing 才是问题**。

但我仍建议：
- 只对 **hypothesis contract 通过** 的 proposal 调 classifier
- 缓存 `normalized_hypothesis_text_hash -> family_id`
- 记录 `family_source / taxonomy_version / classifier_model`

#### 更大的问题：当前 classifier skeleton 甚至还没到“可直接接电”的程度
你们当前 `proposal/classifier.py` 里：
- 假设 `llm_client.call_simple()` 存在
- fallback 分支又调用 `llm_client.call(prompt, model=...)`
- 但当前 `LLMClient.call()` 的签名需要 `response_schema`

也就是说：  
**即使把 CampaignManager 接上，当前 classifier 实现也很可能只会一直 fallback 到关键词法。**

这不是 wiring 问题，是 **实现契约都没对齐**。

#### 15% 分类差异不是好验收标准
“差异”只能说明不一样，不能说明更好。

更好的验收应是：
- 重复 family proposal 比例下降
- 同义机制在 SearchMemory 中更少碎裂
- branch diversity 指标改善
- downstream promotion rate / screening redundancy 改善（哪怕只在 replay 中）

---

### 5.5 Q2.4 — W16 大规模验证实验的可靠性

当前设计的几个 confound 必须明说：

#### 1) model comparison 不纯
如果 proposal model 比的是 Opus / Sonnet / GPT-5.4，但 classifier 固定是 Sonnet，那么结果是：
- **proposal model effect in a hybrid system**
- 不是纯单模型 agent comparison

这个没问题，但必须写清楚。

#### 2) synthetic vs prod 不一定可比
因为你们现在不仅数据不同，可能还伴随：
- protocol threshold 不同
- case 数不同
- frozen/validation sample size 不同

如果不 harmonize，RQ5 就会变成：
> “数据分布 + protocol 配置 + case count + objective saturation 的混合差异”

#### 3) case 数差异影响统计功效
18 vs 9 instances，不做处理直接比 promote rate / win rate，会有功效偏差。

#### 4) API/provider drift
对 RQ4 尤其重要。应记录：
- model id
- provider endpoint
- inference params
- rate-limit / timeout incidents
- retry counts

#### 建议
- harmonize protocol where possible
- 或至少 pre-register “variant-specific protocol”并明确不做直接 promote-rate 同口径对比
- blocked execution / randomize launch order
- infra-aborted campaigns 不计入主分析，按同 seed rerun
- 用 uncertainty intervals，不只报点估计

## 6. Research Value Assessment

### 6.1 当前 W16 矩阵到底能回答哪些 RQ？

这是 Draft v2 最大的研究设计问题之一。

| RQ | 当前 18-campaign matrix 能回答吗？ | 结论 |
|---|---|---|
| RQ1 结构搜索 alone 的效果 | **不能** | 因为所有 campaigns 都是 full system，没有 structure-only 对照 |
| RQ2 参数搜索 alone 的效果 | **不能** | 没有 parameter-only condition |
| RQ3 结构 + 参数叠加收益 | **不能严格回答** | 缺 structure-only / parameter-only 对照 |
| RQ4 模型对 Scion 的影响 | **部分能** | 仅限选定 3 个 proposal models，在 fixed hybrid setup 下 |
| RQ5 合成 vs 生产泛化 | **部分能** | 但 confounded by dataset saturation / protocol / N |
| RQ6 距最优还有多远 | **部分能** | 仅 small instances，不能外推到大实例 |
| RQ7 Plateau 早停收益 | **不能** | 没有 early-stop on/off 对照，只有“节省了多少轮”这种非因果量 |

### 6.2 所以当前更像什么？
**更像 validation campaign，不足以叫“搜索研究体系化”。**

如果你们坚持这个定位，我建议加一个轻量级 **W17 SEARCH-MECHANISM-STUDY**，不需要 full factorial，但至少补最关键对照：

#### 最低配可行方案
1. **RQ1 / RQ3**
   - 只在 1 个主模型（例如 Opus）上做：
   - full system vs weight-opt off
   - 2 variants × 3 seeds = 6 extra campaigns

2. **RQ2**
   - 不一定需要 full campaign
   - 可用 offline weight-opt on base/promoted snapshots 做 parameter-only analysis

3. **RQ7**
   - 最便宜的是 replay / trace analysis
   - 或对少量 campaigns 做 early-stop on/off paired runs

#### 如果不愿意加 W17
那就应当**下调 v0.3 的研究表述**：
- 不叫“搜索研究体系化”
- 改叫“validation + instrumentation + benchmark anchoring”

### 6.3 Q3.1 — 对 RQ4/RQ5/RQ6 的具体修正建议

#### RQ4
现在最多能说：
> “在 Opus / Sonnet / GPT-5.4 三个选定 proposal models 下，Scion 在该协议中的表现差异如何”

不要说成：
> “模型对 Scion 的影响”

后者太大了。

#### RQ5
需要加一句 scope 限定：
> “在当前 synthetic/prod 切片下，Scion 的行为差异如何”

不要暗示成“泛化研究”大结论。

#### RQ6
建议改成：
> “在可 exact 求解的小实例上，当前 champion 到 exact optimum 的 gap 分布如何；该 gap 与 larger instances 上的 heuristic trend 是否存在相关迹象”

不要说成“大规模上距离最优还有多远”。

### 6.4 Q3.3 — 问题解耦要做到多深，才不会给 v1.0 留双倍技术债？

我的建议是：

#### v0.3 现在必须做的
建立一个 **adapter boundary**，让 Scion core 不再知道 warehouse 的：
- field names
- objective names
- oracle function specifics
- output serialization shape
- family taxonomy

#### v1.0 再做的
不需要现在就把所有 operator hierarchy 泛型化到极致。  
Scion 仍然可以保持：
- “面向 operator-pool local search solver”
- registry/operator file convention 不变

也就是说：
> **不要为了“通用”而过度抽象 registry / pool / operator architecture。**  
> v0.3 只需要把“warehouse 特有知识”从 core 拔出去，而不是把 Scion 变成任意 CO 问题框架。

### W1 — SCORING-DECOUPLE

#### 但要补两点
1. **adaptive scoring 必须 lexicographic-safe**

### 9.5 代码层面的总建议

#### 建议 1：确立“事实层 vs 视图层”
- 事实层：SQLite lineage / hypotheses / champions / weight_opt
- 视图层：SearchMemory / ResearchJournal / FailureSummary / Report

这样能大幅减少 O/P Sprint 的技术债。

#### 建议 2：确立“Problem Adapter”而不是“Problem YAML 万能神表”
一些东西适合 YAML：
- objectives order
- categories
- prompt fragments
- file paths

一些东西更适合 code hooks：
- deserialize output
- objective recomputation
- lower bound
- solution consistency
- taxonomy

#### 建议 3：并发一定走 immutable snapshot
不要再让 campaign thread 和 bg thread 共享一个会被 chmod / 改写的 snapshot。
