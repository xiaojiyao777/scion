## 0. 本版结论先说清楚

基于你已有的 v2 和本轮审核意见，我建议直接把本版蓝图升级为 **v2.1**，核心不是“再加模块”，而是把几个关键组件**协议化、边界化、可执行化**：

### 本版最重要的 5 个升级

1. **Verification Gate 升为一等公民**
   - 明确与 Contract Layer 分离
   - Contract 管“越界与结构”
   - Verification 管“语义正确性与 solver 安全性”

2. **实验协议从 2-stage 升级为 3-stage**
   - `screening / validation / frozen`
   - 增加 `SplitManager + SeedLedger + ExposurePolicy`
   - 明确谁能看到什么，防止 validation 被隐式训练

3. **新增 Decision Input Guard（决策输入白名单）**
   - Decision 层**只能读数值与枚举**
   - LLM 自由文本一律视为 **tainted（污染态）**
   - 杜绝 LLM 通过 `confidence / summary / improvement_axes` 间接控制调度与晋升

4. **Scheduler 形式化**
   - 不是“validate debt first”这种口号式规则
   - 给出**可实现的优先级函数 / 词典序规则**
   - 加入 stale branch 语义、饥饿避免、预算约束

5. **4 周目标重新定义为 MVP，而不是完整可信版**
   - **4 周做出“能跑、可审计、边界清晰”的核心原型**
   - 完整论文级可信协议留到 MVP 之后继续补强
   - 这更符合工程现实，也更容易做成

# 3. 核心设计原则

## 3.1 方法论是内核，agent 只是承载

- 真正的价值不在“几个 agent 对话”
- 真正的价值在：
  - 分支治理
  - 决策边界
  - 协议可信性
  - 实验可追溯

---

## 3.2 能用确定性代码解决的，不交给 LLM

- LLM 只负责：
  - 提假设
  - 产 patch 提案
  - 解释失败原因（仅存档/供人读）
- 确定性系统负责：
  - Contract 检查
  - Verification
  - 实验协议
  - 调度
  - 晋升 / 淘汰
  - Artifact persistence

---

## 3.3 LLM 只能提案，不能决策

这句话在 v2.1 里不再是口号，而是**接口规则**：

- LLM 输出全部视为 **tainted 数据**
- 只有经过：
  - Contract
  - Verification
  - Protocol
  - Safe Feature Extractor
- 才能形成 `DecisionFeatures`
- 决策层**只允许读取 `DecisionFeatures`**

---

## 3.4 先把边界做硬，再考虑搜索做大

工程上最怕的是：
- 候选变多了
- 日志变多了
- 但信号更脏了
- 最后人不敢信

所以 v2.1 的顺序是：

1. **先保证实验可信与边界明确**
2. 再增加搜索深度 / 搜索空间 / 参数层

# 5. 控制模型：三层控制 + 双硬闸门 + 决策输入白名单

> 为了兼容你原来的“三层控制模型”说法，本版保持“三层”原则不变，但在工程实现上明确为：  
> **Creative Layer → Contract Gate → Verification Gate → Decision Layer**  
> 其中 Contract / Verification 是两个**硬闸门**，Decision 只读白名单输入。

---

## 5.1 控制流分层

### Layer A：Creative Layer（LLM，污染态）
负责输出：
- `HypothesisProposal`
- `PatchProposal`
- `FailureAnalysis`
- `BranchIdea`

特点：
- 全部是 **tainted**
- 只能进入 Contract / 存档
- 不得直接驱动调度与晋升

---

### Gate 1：Contract Gate（结构边界）
负责检查：
- JSON schema
- touched files 是否在白名单
- forbidden files 是否未修改
- patch 行数
- AST / 接口签名
- import 白名单
- 输出字段白名单

---

### Gate 2：Verification Gate（语义正确性）
负责检查：
- import / syntax
- interface compliance
- unit tests
- regression tests
- feasibility oracle
- objective recomputation
- no state leak
- wall-clock / memory guard
-（可选）RNG 接口合规

---

### Layer B：Decision Layer（确定性）
只读取：
- Contract 结果
- Verification 结果
- Protocol 结果
- Branch 元信息
- Budget / stale / failure counters

不读取：
- LLM 自由文本
- confidence 文本
- evidence_summary
- narrative-style observed_effect

---

## 5.2 Decision Input Guard（本版新增关键组件）

### 原则
任何进入 Decision 层的数据必须先被转换为 `DecisionFeatures`。

### 推荐数据结构

```python
from dataclasses import dataclass
from typing import Optional, Literal, Tuple

@dataclass(frozen=True)
class DecisionFeatures:
    branch_id: str
    stage: Literal["screening", "validation", "frozen"]
    contract_passed: bool
    verification_passed: bool
    canary_passed: bool

    n_cases: int
    win_rate: Optional[float]
    median_delta: Optional[float]
    ci_low: Optional[float]
    ci_high: Optional[float]

    stale: bool
    recent_retry_count: int
    recent_failure_codes: Tuple[str, ...]
    budget_remaining_ratio: float
```

### 关键约束
- **没有任何自由文本字段**
- 没有 `confidence_text`
- 没有 `evidence_summary`
- 没有 `hypothesis_text`

---

| 数据类型 | LLM 可写 | Contract 可改写/过滤 | Verification 可写 | Protocol 可写 | Decision 可读 |
|---|---:|---:|---:|---:|---:|
| hypothesis_text | ✅ | 仅校验 | ❌ | ❌ | ❌ |

| pass/fail label | ❌ | ❌ | ❌ | ✅ | ✅ |

# 6. 实验协议层（v2.1 正式版）

> 这是本版最关键的升级之一。  
> 不再是“配对评估 + fresh cases”的松散组合，而是一个正式的协议层。

---

## 6.1 协议目标

实验协议层必须同时解决 4 件事：

1. **噪声控制**
2. **信息泄漏控制**
3. **重复试探导致的错误晋升**
4. **回归与安全退化检测**

---

## 6.2 协议对象与统计单位

### 统计单位的设计
建议将**主统计单位定义为 case**，而不是 `(case, seed)`。

#### 原因
- 多 seed 的作用是降低 case 内噪声
- 真正比较时，应以 **case-level delta** 作为主要统计对象
- 这样更稳，也更接近 reviewer 会接受的设定

### 计算方式
对于每个 case：

```text
delta_case = mean_over_seeds(transformed_score(candidate) - transformed_score(champion))
```

其中：
- `transformed_score = score`（如果 higher_is_better）
- `transformed_score = -score`（如果 lower_is_better）

---

## 6.3 三层 split 设计

### Level 1：Screening Set
- 用途：快速探索 / 粗筛
- 可重复使用：是
- 暴露级别：允许较细结果
- 推荐默认：
  - `N_cases = 6`
  - `N_seeds = 2`

### Level 2：Validation Set
- 用途：正式验证
- 每个分支对当前 champion **一次性验证**
- 暴露级别：只给 aggregate
- 推荐默认：
  - `N_cases = 12`
  - `N_seeds = 3`

### Level 3：Frozen Holdout Set
- 用途：最终确认
- campaign 级有限使用
- 暴露级别：只给 pass/fail + aggregate
- 推荐默认：
  - `N_cases = 12~20`
  - `N_seeds = 3`
  - `max_uses_per_campaign = 3`

---

## 6.4 暴露控制（Exposure Policy）

| 信息类型 | Screening | Validation | Frozen |
|---|---|---|---|
| per-case 原始结果 | 可见（LLM/人类） | 不可见 | 永不暴露 |
| subgroup breakdown | 可见（LLM/人类） | 不可见 | 永不暴露 |
| aggregate（win_rate / median_delta / CI） | 可见 | 可见 | 仅 aggregate |
| pass/fail | 可见 | 可见 | 可见 |
| 结果是否允许反馈进后续搜索 | 是 | 有限（aggregate only） | 默认否（campaign 结束前不进 LLM） |

### 工程建议
- **Frozen 结果默认不回喂 LLM**
- Frozen 只用于：
  - Decision promote / reject
  - 人工审核
  - 最终记录

---

## 6.5 SplitManager + SeedLedger

### SplitManager
负责：
- 固定 split 版本
- case id 列表
- strata 信息
- split hash
- frozen 使用计数

### SeedLedger
负责：
- 固定每个 stage 的 seed bank
- 不允许临时“挑 seed”
- infra retry 必须复用同一 seed
- 任何变更都会写 protocol version

### 推荐文件

```yaml
# split_manifest.yaml
version: "split-v1"
screening:
  cases: ["inst_01", "inst_02", "inst_03", "inst_04", "inst_05", "inst_06"]
validation:
  cases: ["inst_07", "inst_08", "inst_09", "inst_10", "inst_11", "inst_12"]
frozen:
  cases: ["inst_13", "inst_14", "inst_15", "inst_16", "inst_17", "inst_18"]

# seed_ledger.yaml
version: "seed-v1"
screening: [11, 29]
validation: [11, 29, 47]
frozen: [11, 29, 47]
```

---

## 6.6 Retry 规则（必须预注册）

### 允许的 retry
#### 1）Infra Retry
仅限：
- benchmark infra failure
- LLM API failure
- 机器故障
- 文件系统瞬时错误

规则：
- 重试同一个 case / 同一个 seed
- 最多 2 次
- 不改变 split / seed / budget

#### 2）Statistical Expand（不是 retry）
如果结果 `unclear`：
- 不能简单“再跑一次碰碰运气”
- 只能按预注册规则**扩大样本**
- 例如：
  - screening: `6 -> 10`
  - validation: `12 -> 20`

### 不允许的 retry
- 换 seed 重跑
- 换 case 重跑
- 反复刷到过门槛为止

---

## 6.7 Canary Regression Check（安全 veto）

### 作用
不是为了证明 candidate 更好，而是为了防止明显退化/污染。

### Canary set 特点
- 固定
- 小
- 快
- 覆盖安全边界 case

### 默认 veto 规则
- feasibility violation > 0 → 直接 fail
- objective recomputation mismatch → 直接 fail
- timeout rate 超阈值 → 直接 fail
- 相对 champion 明显负退化超阈值 → fail

### 重要说明
Canary 只做 veto，不做 improvement 证据。

---

## 6.8 Promotion Gate（分阶段）

> **本版明确把 `N>=6 + win_rate>=2/3` 降级为 screening gate，不再作为最终 promotion 依据。**

---

### 6.8.1 Screening Gate（粗筛）
默认规则：
- `N_cases >= 6`
- `win_rate >= 2/3`
- `median_delta >= δ_screen`

结果：
- `pass`
- `unclear`
- `fail`

---

### 6.8.2 Validation Gate（正式验证）
默认规则：
- `N_cases >= 12`
- `win_rate >= 2/3`
- `median_delta >= δ_validate`
- `bootstrap_ci_low >= 0`

结果：
- `pass_to_frozen`
- `expand_sample`
- `fail`

---

### 6.8.3 Frozen Gate（最终确认）
默认规则：
- `N_cases >= 12`
- `bootstrap_ci_low >= 0`
- `canary_passed == True`
- 无 critical verification/runtime failure

结果：
- `confirmed`
- `rejected`

---

## 6.9 推荐实现：Bootstrap CI + 实用阈值并用

工程上不必把 p-value 当唯一判据，推荐组合：

- `win_rate`
- `median_delta`
- `bootstrap CI`
- `min_practical_delta`

这样比单纯 t-test 更稳，也更适合 solver 这种噪声分布不一定高斯的场景。

# 7. Contract Layer（结构边界层）

> Contract Layer 负责的是：**“候选有没有越界、结构是否符合约束”**  
> 它不负责证明候选“语义上是对的”。

---

## 7.1 Contract 检查范围

### 输入产物
- `HypothesisProposal`
- `PatchProposal`
- `FailureAnalysis`

### 主要检查
1. JSON schema 严格校验
2. patch 必须是 unified diff / 结构化 patch
3. touched files 必须在白名单
4. forbidden files 绝对不能改
5. patch size limit
6. AST / 接口签名检查
7. import 白名单
8. 禁止 shell / subprocess / socket 等敏感调用（静态扫描）
9. analysis 字段白名单
10. touched symbols / change_locus 必须可枚举化

---

## 7.2 建议的 Contract 输出

```python
@dataclass
class ContractResult:
    passed: bool
    touched_files: list[str]
    touched_symbols: list[str]
    patch_line_count: int
    violations: list[str]
    normalized_change_locus: str | None
```

---

## 7.3 Contract 与 Verification 的边界

| 维度 | Contract | Verification |
|---|---|---|
| 文件白名单 | ✅ | ❌ |
| patch 行数 | ✅ | ❌ |
| AST / 接口签名 | ✅ | ✅（可复检） |
| import / forbidden API 静态检查 | ✅ | 可选 |
| 单元测试 | ❌ | ✅ |
| feasibility | ❌ | ✅ |
| objective recomputation | ❌ | ✅ |
| no state leak | ❌ | ✅ |
| 时间/内存预算 | ❌ | ✅ |

---

# 8. Verification Gate（语义正确性层）

> 这是 v2.1 必须拉升的一等组件。  
> 没有它，实验协议再漂亮，也可能是在比较“已经偷偷改题”的东西。

---

## 8.1 Verification Gate 的定位

位于：

```text
Patch Proposal
  -> Contract Pass
  -> Materialize Candidate
  -> Verification Gate
  -> Protocol
```

### 它解决的问题
- 候选是否仍然遵守原求解问题语义
- 是否破坏可行性
- 是否把 objective 计算搞坏
- 是否有状态泄漏
- 是否引入不可接受的复杂度退化

---

## 8.2 检查清单（建议 fail-fast）

### P0 必做（MVP）
1. `check_import_syntax`
2. `check_interface_compliance`
3. `check_unit_tests`
4. `check_regression_tests`
5. `check_feasibility_oracle`
6. `check_objective_recompute`
7. `check_no_state_leak`
8. `check_wall_clock_guard`

### P1 建议（MVP 后）
9. `check_memory_guard`
10. `check_rng_interface_compliance`
11. `check_thread_safety`

---

## 8.3 推荐伪代码

```python
class VerificationGate:
    def run(self, candidate, spec) -> VerificationResult:
        checks = [
            self.check_import_syntax(candidate),
            self.check_interface_compliance(candidate, spec),
            self.check_unit_tests(candidate, spec.unit_tests),
            self.check_regression_tests(candidate, spec.regression_suite),
            self.check_feasibility_oracle(candidate, spec.feasibility_cases),
            self.check_objective_recompute(candidate, spec.objective_tolerance),
            self.check_no_state_leak(candidate, spec.state_leak_cases),
            self.check_wall_clock_guard(candidate, spec.time_budget_ratio),
        ]
        return VerificationResult.from_checks(checks)
```

---

## 8.4 推荐的失败路由

| 失败类型 | 例子 | 是否允许 LLM 修复重试 | 是否消耗分支预算 |
|---|---|---:|---:|
| Syntax / import | 语法错误、模块导入失败 | ✅ | 否 |
| Interface violation | 签名不匹配、返回类型不对 | ✅ | 否 |
| Unit test failure | 小逻辑错误 | ✅（限次） | 否 |
| Feasibility violation | 破坏约束 | ❌ | 是 |
| Objective mismatch | 增量 cost 与全量重算不一致 | ❌ | 是 |
| State leak | 连续调用互相污染 | ❌ | 是 |
| Wall-clock excessive | 明显超时 | 视情况 | 是 |

# 9. Runtime Isolation（运行隔离）

> MVP 不上 Docker 是合理的，但不能只做“目录隔离”。  
> 最低标准应是 **subprocess-per-run + 资源限制 + 只读快照 + 环境净化**。

---

## 9.1 MVP 最低隔离标准

### 必做
1. **每次 run 在独立 subprocess 中执行**
2. **每个分支独立 workspace**
3. **champion snapshot 只读**
4. **benchmark harness / split manifest / protocol 文件只读**
5. **资源限制**
   - timeout
   - memory cap
   - file descriptor limit
6. **环境变量净化**
7. **临时目录隔离**
8. **run 后清理 workspace cache**

### 能做最好做
9. 禁网
10. 禁止 shell 调用
11. 限制非白名单 import

---

## 9.2 为什么 subprocess-per-run 是必须的

它一次性解决很多 Python 污染问题：

- `sys.modules` 缓存污染
- import side effects
- 全局单例泄漏
- 随机数状态延续
- monkey patch 污染
- 上一轮 run 的临时状态残留

---

## 9.3 Runner 抽象（建议）

```python
class Runner(Protocol):
    def run(self, workdir: str, cmd: list[str], limits: ResourceLimits) -> RunResult:
        ...
```

### MVP 实现
- `LocalSubprocessRunner`

### 后续扩展
- `DockerRunner`
- `RemoteRunner`

这样后续升级执行沙箱时，不需要改上层控制逻辑。

# 10. 分支治理与调度（Scheduler 形式化）

---

## 10.1 分支状态机（推荐）

```text
NEW
  ↓
EXPLORE
  ├── screen_pass      -> READY_VALIDATE
  ├── screen_unclear   -> EXPLORE_EXPAND
  └── screen_fail      -> ABANDONED

READY_VALIDATE
  ↓
VALIDATING
  ├── validate_pass    -> READY_FROZEN
  ├── validate_expand  -> VALIDATING_EXPAND
  └── validate_fail    -> ABANDONED

READY_FROZEN
  ↓
FROZEN
  ├── frozen_pass      -> PROMOTED
  └── frozen_fail      -> ABANDONED

任意状态
  ├── champion_changed -> STALE
  ├── infra_incident   -> BLOCKED_INFRA
  └── budget_exhausted -> ABANDONED
```

---

## 10.2 stale branch 语义（本轮必须补清楚）

### 为什么必须有 stale
如果：
- Branch A 从 `champion_0` 分叉
- Branch B 先晋升成了 `champion_1`

那么 Branch A 当前的统计结论可能已经不成立，因为它不再是对当前 champion 的比较。

### 规则
每个分支必须记录：
- `base_champion_id`
- `base_solver_hash`

当 champion 变化时：

1. 所有活跃分支标记为 `STALE`
2. stale 分支不能直接继续 validate / promote
3. 必须执行 `stale_reconcile`

### MVP 的 stale_reconcile 简化规则
- 重新将 patch 应用到新 champion
- 重新跑：
  - Contract
  - Verification
  - Screening（against current champion）
- 若仍有正信号，恢复到 `READY_VALIDATE`
- 否则放弃

> 这是工程上最稳的做法，虽然保守，但简单且可信。

---

## 10.3 调度器优先级（建议用词典序 + 分数混合）

### 一级优先级（硬规则）
1. `READY_FROZEN`
2. `READY_VALIDATE`
3. `STALE`（待重新 screening）
4. `EXPLORE` 中已有正信号的分支
5. 创建新分支

### 二级排序（同级内）
按以下分值排序：

```text
priority_score =
    100 * validate_debt
  + 40  * frozen_ready
  + 20  * screen_signal
  + 10  * starvation_boost
  - 15  * stale_penalty
  - 10  * retry_penalty
  - 20  * budget_penalty
```

其中：
- `validate_debt`: 是否已满足 screening 且还没验证
- `screen_signal`: 来自 screening 的安全数值特征
- `starvation_boost`: 等待过久的分支加分
- `stale_penalty`: stale 分支减分
- `retry_penalty`: 最近修复重试过多减分
- `budget_penalty`: 已消耗过多预算减分

---

## 10.4 分支预算规则（MVP 推荐）

- `max_active_branches = 3`
- `max_commits_per_branch = 3`
- `max_llm_fix_retries_per_candidate = 2`
- `max_screen_expands = 1`
- `max_validate_expands = 1`
- `max_frozen_uses_per_campaign = 3`

# 13. Context Manager（防记忆退化）

---

## 13.1 上下文分层策略

### 给 LLM 的上下文只应包含
1. 问题定义摘要
2. 当前 champion 简述
3. 当前分支最近 3~5 次 screen 结果
4. 已验证失败的结构化教训
5. 白名单文件的当前代码片段
6. 不允许暴露的 validation/frozen 细节**不进入上下文**

## 13.2 记忆压缩规则

### 保留完整
- active hypotheses
- promoted hypotheses
- 最近 3 轮 experiment summaries

### 压缩
- 长期 rejected hypotheses
- 重复失败模式
- 过期 blacklist 规则

## 13.3 blacklist 机制改造建议

不要再用简单的 `local/global` 二值。

### 建议改成
- `scope_tags`
- `evidence_count`
- `expiry_round`

例如：

```yaml
blacklist_rule:
  id: "bl-003"
  change_locus: "destroy_operator"
  scope_tags: ["dense_only", "large_scale"]
  reason: "causes systematic regression"
  evidence_count: 3
  expiry_round: 20
```

### MVP 可简化
- 保留 `scope_tags`
- 不做复杂条件逻辑
- 但不要再只有 local/global 两档

# 14. 主循环（推荐伪代码）

```python
while termination.not_reached(campaign_state):
    branch = scheduler.select(campaign_state)

    # 1) stale 分支优先做 reconcile
    if branch.state == "STALE":
        branch = stale_handler.reconcile(branch, current_champion)

    # 2) 构建受暴露控制的上下文
    llm_context = context_manager.build(branch, exposure_policy)

    # 3) LLM 产出 proposal（污染态）
    proposal = proposal_engine.generate(llm_context)

    # 4) Contract Gate
    contract_result = contract_gate.validate(proposal, problem_spec)
    lineage.record(contract_result)
    if not contract_result.passed:
        failure_router.handle_contract_failure(branch, contract_result)
        continue

    # 5) 物化候选 + 隔离执行
    candidate = workspace.materialize(branch, proposal)
    verify_result = verification_gate.run(candidate, problem_spec)
    lineage.record(verify_result)
    if not verify_result.passed:
        failure_router.handle_verification_failure(branch, verify_result)
        continue

    # 6) Canary 回归检查
    canary_result = protocol.run_canary(candidate, champion)
    lineage.record(canary_result)
    if not canary_result.passed:
        branch_controller.reject(branch, reason="CANARY_FAIL")
        continue

    # 7) Screening / Validation / Frozen
    stage = branch_controller.next_stage(branch)
    proto_result = protocol.run(stage, candidate, champion)
    lineage.record(proto_result)

    # 8) 提取安全决策特征
    features = safe_feature_extractor.from_results(
        branch=branch,
        contract=contract_result,
        verify=verify_result,
        canary=canary_result,
        protocol=proto_result
    )

    # 9) Decision
    decision = decision_engine.decide(features)
    lineage.record(decision)

    # 10) 状态转移
    branch_controller.apply(branch, decision)
```

- 而是“实验治理协议”
- 本地单机把骨架跑通更重要

# 21. 风险与缓解（工程视角）

---

## 风险 1：协议太硬，搜索空间被压缩成“小修小补”
### 缓解
- patch limit 是默认值，不是永恒真理
- 先固定 `200 lines`，MVP 稳了后可开放 `structural branch` 特殊模式
- 不要把 blacklist 做成永久封杀，必须可过期

---

## 风险 2：Verification 成为开发瓶颈
### 缓解
- 先做 8 项硬检查
- memory/thread-safety 后移
- 所有 verification checks 独立可单测
- 为每项 check 建最小 synthetic fail case

---

## 风险 3：Context 退化成日志垃圾堆
### 缓解
- 只保留结构化 hypothesis summary
- 自由文本不进入 decision
- 每 5 轮做记忆压缩
- 压缩前保留原始 event，在 lineage 中可追溯

---

## 风险 4：统计门槛过松导致错误晋升
### 缓解
- screening 不再等同 promotion
- validation 增加 CI
- frozen 作为最后确认
- `unclear` 只能扩样，不能随便 retry

---

## 风险 5：隔离不够造成污染
### 缓解
- subprocess-per-run 必须做
- 所有 benchmark/protocol 文件只读
- run 后清理 workspace
- 同一 candidate 重跑一致性测试纳入 smoke test

---

# 22. 本版相对原 v2 的关键升级总结

---

## 已经明确补硬的地方

### 1）Verification Gate 不再附属化
- 从“流程里一句 verify”变成独立顶层模块

### 2）实验协议从“规则集合”变成“正式协议”
- 有 split
- 有 seed ledger
- 有 exposure policy
- 有 retry 规则
- 有 frozen 使用上限

### 3）LLM 的隐蔽决策路径被技术上截断
- 引入 tainted / safe 数据分层
- Decision 只读 `DecisionFeatures`

### 4）Scheduler 从口号走向可实现
- 有状态机
- 有 stale 规则
- 有优先级顺序
- 有预算限制

### 5）Failure Model 更适合长期维护
- 不再把“语法错 / infra 故障 / 结果不好”混在一个 Enum

## 推荐落地策略

### 技术上
- 物理实现先做：**单强 LLM + 确定性工具链**

- 逻辑上保留角色边界即可

- 先把一个 VRP destroy operator 跑透

- 等 MVP 稳定后，再加：
  - param tuning
  - policy search
  - 多任务评估
  - paper-grade ablation
