## 0. 本文档的定位

这是 OR-AutoResearch Agent 框架的**蓝图基石**。目标是设计一个以 autoresearch 方法论为内核的 agent 框架，使其能够：

1. 接受一个**已定义好的组合优化问题**（含 baseline、benchmark、评价标准）
2. 按照 autoresearch 的科学方法论**自主执行算法探索与优化**
3. 通过统计验证输出**可靠的、优于 baseline 的新算法**

v0.1 会用一个具体问题验证，但框架设计不为该问题定制。

---

## 1. 设计哲学

### 1.1 从两个源头汲取

**Autoresearch 方法论**（已有完整设计）：
- 分支控制器（explore → validate → promoted/abandoned）
- 统计晋升门槛（win_rate ≥ 2/3 + median_delta ≥ min_practical_delta, N≥6）
- 搜索空间三分（Structural / Parametric / Policy）
- 实验记录与可复现性

**Agent 框架工程**（从 CC/MiroFlow 学到的）：
- 控制流是 agent 的灵魂——决策树比 LLM 自由输出更可靠
- 确定性逻辑不交给 LLM——统计门槛、分支状态机必须是代码
- 工具层要简洁——不需要通用 MCP 生态，Python 函数就够
- 容错机制必须显式——rollback/retry 不能靠 LLM "自愈"

### 1.2 核心原则

1. **autoresearch 方法论是内核，agent 是执行器**——框架的价值在方法论，不在 agent 的通用性
2. **确定性逻辑用代码，创造性推理用 LLM**——分支控制器、统计门槛、验证管线全部是确定性 Python，LLM 只负责假设生成和代码编写
3. **不做通用 agent 框架**——这不是下一个 LangChain/MiroFlow，这是一个为 OR 算法自动设计而生的专用系统
4. **人在回路但不在循环里**——人定义问题、审核结果、做最终决策；agent 独立执行探索循环

```markdown
# program.md — Agent 指令（领域知识注入）

## 问题背景
VRP（Vehicle Routing Problem）的 destroy 算子决定了 LNS 框架的搜索效率。
当前 random destroy 在密集场景下效率低，因为随机删除打散了有利的局部结构。

## 搜索层次
Level 1: 算子设计（选择哪些节点删除）← 主要探索方向
Level 2: 参数调优（删除数量 k、权重）
Level 3: 策略选择（什么条件用什么算子）

## 已知有效方向
- 基于距离的 destroy（删除地理相近的节点）
- 基于代价贡献的 destroy（删除对目标函数贡献最差的节点）
- 基于时间窗相关性的 destroy（删除时间窗重叠的节点）
- 组合策略（先聚类再删除边界节点）

## 思维循环
每次实验前：
1. 读取实验历史和当前 breakdown 报告
2. 识别薄弱维度
3. 提出假设："我认为 X 会改善 Y，因为 Z"
4. 设计最小实验验证

每次实验后：
5. 解释结果（不只看涨跌，理解为什么）
6. 更新理解，生成下一假设
```

### 3.2 Control Flow（控制流 — 框架核心）

这是整个框架的灵魂。设计决策全部来自 autoresearch 方法论，借鉴 CC/MiroFlow 的容错思路。

```python
class AutoResearchLoop:
    """主控制循环 — 确定性调度 + LLM 推理"""

    def __init__(self, problem_spec, llm_client, tools):
        self.spec = problem_spec
        self.llm = llm_client
        self.tools = tools
        self.branch_controller = BranchController(spec.branch_control)
        self.promotion_gate = PromotionGate(spec.benchmark)
        self.context_manager = ContextManager()
        self.experiment_log = ExperimentLog()

    async def run(self):
        """
        主循环 — 纯确定性调度，LLM 只在明确标注的点介入。

        设计决策（vs CC/MiroFlow）：
        - CC 用 "无 tool_use 就终止"——我们不用，因为终止条件是明确的
          （预算耗尽 / 目标达成 / 所有分支 abandoned）
        - MiroFlow 用 max_turns——我们用 max_experiments + wall_clock
        - 两者都没有分支概念——这是我们的核心差异化
        """
        while self._should_continue():

            # ── Step 1: 调度（确定性） ──
            branch = self.branch_controller.schedule()
            if branch is None:
                # 无活跃分支 → 尝试创建新分支
                branch = self._try_create_branch()
                if branch is None:
                    break  # 搜索空间穷尽

            # ── Step 2: 构建上下文（确定性） ──
            context = self.context_manager.build(
                branch=branch,
                experiment_history=self.experiment_log.get_branch_history(branch),
                champion_info=self.branch_controller.get_champion_info(),
                problem_spec=self.spec,
            )

            # ── Step 3: LLM 推理 + 代码生成（创造性部分） ──
            if branch.state == BranchState.EXPLORE:
                result = await self._explore_step(branch, context)
            elif branch.state == BranchState.VALIDATE:
                result = await self._validate_step(branch, context)

            # ── Step 4: 统计门槛（确定性） ──
            label, win_rate, median_delta = self._evaluate_with_retry(
                result.candidate_code, branch
            )

            # ── Step 5: 分支决策（确定性） ──
            self.branch_controller.decide(branch, label, win_rate, median_delta)

            # ── Step 6: 记录（确定性） ──
            self.experiment_log.record(branch, result, label, win_rate, median_delta)

    async def _explore_step(self, branch, context):
        """
        探索步骤：LLM 分析 → 假设 → 代码生成 → 验证

        容错设计（借鉴 CC/MiroFlow）：
        - 代码语法错误 → 反馈 LLM 修复，最多 2 次（参考 CC 的 tool validation）
        - 代码不满足接口 → 反馈具体违规项（参考 MiroFlow 的 Rollback 精神：
          不让坏状态继续，而是显式回退重试）
        - 3 次生成失败 → 记录失败原因，跳过本轮，不消耗分支预算
        """
        # LLM 分析当前状态 + 提出假设
        hypothesis = await self.llm.reason(
            context=context,
            instruction="分析实验历史，识别薄弱维度，提出改进假设"
        )

        # LLM 生成代码
        for attempt in range(3):
            code = await self.llm.generate_code(
                context=context,
                hypothesis=hypothesis,
                interface_spec=self.spec.search_space.interface
            )

            # 确定性验证（不用 LLM）
            verification = self.tools.code_verify(code)
            if verification.passed:
                return ExploreResult(
                    hypothesis=hypothesis,
                    candidate_code=code,
                    attempt=attempt
                )

            # 验证失败 → 反馈给 LLM
            context.append_verification_error(verification.errors)

        # 3 次失败 → 记录但不消耗预算
        return ExploreResult(hypothesis=hypothesis, candidate_code=None, failed=True)

    async def _validate_step(self, branch, context):
        """
        验证步骤：不修改代码，只在新 case 集上重新评估。
        这是纯确定性操作，不需要 LLM。
        """
        return ValidateResult(candidate_code=branch.frozen_sha_code)

    def _evaluate_with_retry(self, candidate_code, branch):
        """
        统计门槛 + retry 逻辑（完全确定性，无 LLM）

        直接复用 autoresearch framework 的 promotion_gate 实现。
        """
        if candidate_code is None:
            return "not_better", 0.0, 0.0

        pooled_exp, pooled_champ = [], []
        retry_index = 0

        while True:
            case_set = self.tools.benchmark_run.sample_cases(self.spec.benchmark.n_cases)
            exp_scores = self.tools.benchmark_run.evaluate(candidate_code, case_set)
            champ_scores = self.tools.benchmark_run.evaluate(
                self.branch_controller.champion_code, case_set
            )
            pooled_exp.extend(exp_scores)
            pooled_champ.extend(champ_scores)

            label, wr, md = promotion_gate(
                pooled_exp, pooled_champ,
                self.spec.benchmark.min_practical_delta
            )

            if label != "unclear" or retry_index >= self.spec.branch_control.max_retries_per_sha:
                return label, wr, md
            retry_index += 1

    def _should_continue(self):
        """终止条件（确定性）"""
        return (
            self.experiment_log.total_experiments < self.spec.budget.max_experiments
            and self.experiment_log.total_tokens < self.spec.budget.max_llm_tokens
            and self.experiment_log.wall_clock_hours < self.spec.budget.max_wall_clock_hours
            and not self.branch_controller.all_directions_exhausted()
        )

    def _try_create_branch(self):
        """
        创建新分支需要 LLM 提出新假设方向。
        但如果已有假设都失败且 LLM 无法提出新方向 → 返回 None → 终止。
        """
        if self.branch_controller.active_count >= self.spec.branch_control.max_active_branches:
            return None
        # LLM 基于失败历史提出新方向
        # ... (实现略)
```

### 3.3 Branch Controller（分支控制器 — 确定性状态机）

**直接从 autoresearch framework 搬过来，不需要 LLM 参与。**

```python
class BranchState(Enum):
    EXPLORE = "explore"
    VALIDATE = "validate"
    PROMOTED = "promoted"
    ABANDONED = "abandoned"

class Branch:
    id: str
    state: BranchState
    hypothesis_family: str  # 这个分支探索的方向描述
    step: int = 0
    budget_used: int = 0
    budget_max: int = 3
    consecutive_failures: int = 0
    frozen_sha_code: Optional[str] = None  # validate 阶段冻结的代码
    fork_champion_version: int = 0  # 分叉时的 champion 版本号

class BranchController:
    """
    确定性分支状态机。
    所有状态转换规则来自 autoresearch framework，无 LLM 参与。
    """

    def schedule(self) -> Optional[Branch]:
        """
        调度优先级：
        1. validate 分支（最老的先）
        2. explore 分支（最好信号优先）
        3. 无活跃分支 → 返回 None
        """
        # validate 优先
        validate_branches = [b for b in self.branches if b.state == BranchState.VALIDATE]
        if validate_branches:
            return min(validate_branches, key=lambda b: b.created_at)

        # explore 按信号排序
        explore_branches = [b for b in self.branches if b.state == BranchState.EXPLORE]
        if explore_branches:
            return self._rank_by_signal(explore_branches)

        return None

    def decide(self, branch, label, win_rate, median_delta):
        """分支决策 — 纯确定性规则"""

        if branch.state == BranchState.EXPLORE:
            if label == "confidently_better":
                branch.state = BranchState.VALIDATE
                branch.frozen_sha_code = branch.current_code  # 冻结
            elif label == "not_better":
                branch.consecutive_failures += 1
            # 检查是否该放弃
            if (branch.budget_used >= branch.budget_max
                or branch.consecutive_failures >= 3):
                self._abandon(branch)

        elif branch.state == BranchState.VALIDATE:
            if label == "confidently_better":
                self._promote(branch)
            else:
                self._abandon(branch)

    def _promote(self, branch):
        """晋升：更新 champion，清理 stale 分支"""
        branch.state = BranchState.PROMOTED
        self.champion_code = branch.frozen_sha_code
        self.champion_version += 1
        # 清理所有从旧 champion 分叉的分支
        for b in self.branches:
            if (b.state in (BranchState.EXPLORE, BranchState.VALIDATE)
                and b.fork_champion_version < self.champion_version):
                self._abandon(b, reason="stale_after_promotion")

    def _abandon(self, branch, reason="exhausted"):
        branch.state = BranchState.ABANDONED
        branch.abandon_reason = reason
```

### 3.4 Promotion Gate（统计晋升门槛 — 确定性）

**直接复用 autoresearch framework 的实现，零修改。**

```python
def promotion_gate(experiment_scores, champion_scores, min_practical_delta):
    """完全确定性，无 LLM，无外部库。"""
    assert len(experiment_scores) == len(champion_scores)
    deltas = [e - c for e, c in zip(experiment_scores, champion_scores)]
    N = len(deltas)
    wins = sum(1 for d in deltas if d > 0)
    win_rate = wins / N
    median_delta = sorted(deltas)[N // 2] if N % 2 == 1 else \
        (sorted(deltas)[N//2 - 1] + sorted(deltas)[N//2]) / 2.0

    if N < 6:
        return "unclear", win_rate, median_delta
    if win_rate >= 2/3 and median_delta >= min_practical_delta:
        return "confidently_better", win_rate, median_delta
    if win_rate <= 0.5 or median_delta <= 0:
        return "not_better", win_rate, median_delta
    return "unclear", win_rate, median_delta
```

### 3.5 Context Manager（上下文管理）

**借鉴 CC 的三级压缩思路，但简化为 OR 场景所需。**

```python
class ContextManager:
    """
    管理 LLM 可见的上下文窗口。

    设计决策（借鉴 CC）：
    - CC 有 autocompact/snip/microcompact 三级压缩
    - 我们的场景更简单：实验历史是结构化数据，不需要 NLP 压缩
    - 用显式的窗口策略替代

    设计决策（借鉴 MiroFlow）：
    - MiroFlow 无上下文管理，靠 max_turns 硬截断
    - 我们不能这样——长时间运行的实验需要历史信息
    """

    def build(self, branch, experiment_history, champion_info, problem_spec):
        context = []

        # Layer 1: 问题定义（始终完整）
        context.append(self._format_problem_spec(problem_spec))

        # Layer 2: Champion 状态（始终完整）
        context.append(self._format_champion(champion_info))

        # Layer 3: 当前分支的完整历史（最近 5 轮详细，更早的压缩为摘要）
        recent = experiment_history[-5:]
        older = experiment_history[:-5]
        if older:
            context.append(self._summarize_experiments(older))
        context.append(self._format_experiments_detailed(recent))

        # Layer 4: 失败假设摘要（避免重复探索）
        context.append(self._format_failed_hypotheses())

        # Layer 5: 其他分支的简要状态（跨分支信息共享）
        context.append(self._format_sibling_branches())

        return AgentContext(context)
```

| 步骤 | LLM 参与？ | 说明 |

| 问题定义 | ❌ | 人类定义 |
| 分支调度 | ❌ | 确定性调度器 |
| 上下文构建 | ❌ | 确定性窗口策略 |
| **假设生成** | ✅ | LLM 分析历史，提出改进方向 |
| **代码生成** | ✅ | LLM 基于假设编写算子代码 |
| **结果分析** | ✅ | LLM 解释为什么某个实验成功/失败 |
| 代码验证 | ❌ | 确定性 pipeline |
| Benchmark 执行 | ❌ | subprocess 调用 |
| 统计门槛 | ❌ | promotion_gate() |
| 分支决策 | ❌ | 状态机规则 |
| 实验记录 | ❌ | 结构化日志 |
| 新分支创建（方向判断） | ✅ | LLM 提出不同于已有分支的新方向 |

**LLM 只在 4 个点介入，其余全部确定性。**

## 7. 从 CC/MiroFlow 借鉴清单

| 来源 | 借鉴什么 | 用在哪里 |
|------|---------|---------|
| CC | 工具验证 + 错误反馈重试（最多 N 次） | code_verify 失败 → 反馈 LLM → 重试 |
| CC | 上下文三级压缩的思路 | context_manager 的分层窗口 |
| CC | BashTool 的超时/输出截断 | benchmark_run 的 subprocess 管理 |
| MiroFlow | Rollback 精神（显式检测坏状态，不靠 LLM 自愈） | explore_step 的 3 次生成失败 → 放弃本轮 |
| MiroFlow | YAML 配置驱动 | problem.yaml 问题规约 |
| 两者 | 都没有分支概念 | 分支控制器是我们的核心差异化 |

## 8. 成功标准

v0.1 成功的定义：

1. **框架能跑**：给定 problem.yaml + baseline + instances，agent 能自主执行探索循环
2. **方法论正确**：分支状态转换、统计门槛、调度优先级全部与 autoresearch framework 设计文档一致
3. **不为验证问题定制**：换一个 OR 问题（如 TSP 邻域设计），只需改 problem.yaml + baseline 代码，框架代码不动
4. **实验可复现**：任何一轮实验都能从日志重放
5. **LLM 边界清晰**：把 LLM 调用全部 mock 掉后，框架的确定性部分仍然能正确运转（分支状态机、统计门槛、调度）
